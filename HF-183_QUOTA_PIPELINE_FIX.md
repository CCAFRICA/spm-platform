# HF-183: Quota Pipeline — Target Agent Signature + Per-Row Entity Resolution

## CONTEXT

CRP Plan 2 (Consumables Commission, piecewise_linear) produces $16,156.17 vs GT $28,159.48. Root cause is a TWO-PART failure in the quota data pipeline:

**Failure 1 — Classification:** The SCI Entity Agent wins at ~90% for quota files. The Target Agent's existing `per_entity_benchmarks_static` signature requires `numericFieldRatio > 30%` AND `noTemporalDimension` — quota files have ~17% numeric ratio and DO have a temporal column. No existing signature matches. The file gets classified as entity, enriched into entity attributes, and never reaches committed_data.

**Failure 2 — Entity resolution:** The engine's OB-183 entity resolution takes the FIRST `entity_id_field` found across ALL committed_data rows. Transaction rows use `sales_rep_id`. Quota rows use `entity_id`. When transaction rows are encountered first (they always are — 389 vs 24), ALL rows including quota rows are looked up using `sales_rep_id`. Quota rows don't have `sales_rep_id` in their `row_data`, so they remain unresolved → store-level orphans → monthly_quota never reaches entity metrics → ratioInput denominator = 0 → all entities get 3% base rate.

**Evidence:**
- Quota committed_data row: `row_data.entity_id = "CRP-6007"`, `row_data.monthly_quota = 25000`, `row_data.effective_date = 46023` (Excel serial = 2026-01-01)
- Quota metadata: `entity_id_field: null`, `semantic_roles.entity_id.role = "transaction_identifier"` (not `entity_identifier` — bindings were generated by Entity Agent)
- Transaction metadata: `entity_id_field = "sales_rep_id"` (389 rows)
- Plan 2 calculationIntent: `ratioInput.sourceSpec.denominator = "monthly_quota"` — references the metric by exact name
- `parseAnyDateValue(46023)` already returns `"2026-01-01"` — no code change needed in date parsing

## PHASE 1: Fix 1 — New Target Agent Signature

**File:** `web/src/lib/sci/signatures.ts`

**What to add:** A new signature block for "entity-referencing target/quota data." This fires when:

1. `identifierRepeatRatio` is approximately 1.0 (one row per entity — quota/target pattern)
2. File has an identifier column (from HC or structural detection)
3. At least one numeric column that is NOT an identifier and NOT a date
4. Row count is small (< 200) — this is reference/target data, not transaction volume

**Scoring approach:** This signature alone may not beat the Entity Agent's `entity_single_row_per_id` signature. Two additional mechanisms:

**Mechanism A — Tenant context entity overlap:** If the identifier values in this file match >50% of existing entity `external_id` values, this file REFERENCES existing entities rather than DEFINING them. This is the strongest signal. Check Read 4 output — if this mechanism already exists in tenant context scoring (Phase D of SCI spec), use it. If not, it needs to be built.

**Mechanism B — Entity Agent counter-signal:** When all identifier values match existing entities (0 new entities would be created), reduce Entity Agent confidence. A file that creates 0 new entities is not a roster — it's data ABOUT an existing roster.

**Korean Test:** All signals are structural — identifier overlap percentage, numeric column count, row count thresholds. ZERO field name matching. A file with Korean column headers `직원번호, 이름, 역할, 할당량, 시작일` must score identically to `entity_id, name, role, monthly_quota, effective_date`.

**Implementation notes from the code:**
- Entity overlap requires querying `entities` table for the tenant's existing external_ids — this may need to happen in the scoring function's context, not just in signatures.ts
- The agents.ts `ENTITY_WEIGHTS` and `TARGET_WEIGHTS` arrays contain additive weight rules — a counter-signal for Entity can be added there
- Tenant context data structure (`TenantContext`) is defined in the SCI spec Phase D — check if it's implemented

## PHASE 2: Fix 2 — Per-Row entity_id_field Resolution

**File:** `web/src/app/api/calculation/run/route.ts`

**Current code (from DIAG):**
```typescript
let entityIdFieldFromMeta: string | null = null;
for (const row of committedData) {
    const meta = row.metadata as Record<string, unknown> | null;
    if (meta?.entity_id_field && typeof meta.entity_id_field === 'string') {
        entityIdFieldFromMeta = meta.entity_id_field;
        break;  // ← TAKES FIRST FOUND — problem
    }
}
```

Then later:
```typescript
if (!resolvedEntityId && entityIdFieldFromMeta) {
    const rd = row.row_data as Record<string, unknown> | null;
    const extId = rd?.[entityIdFieldFromMeta];  // ← USES GLOBAL FIELD NAME
```

**Required change:** Each row must use its OWN metadata's `entity_id_field` for resolution. Replace the global lookup with per-row resolution:

```typescript
// REMOVE the global entityIdFieldFromMeta loop entirely

// In the per-row resolution loop, change to:
if (!resolvedEntityId) {
    // Per-row entity_id_field from this row's own metadata
    const rowMeta = row.metadata as Record<string, unknown> | null;
    const rowEntityIdField = (rowMeta?.entity_id_field as string) || null;
    
    if (rowEntityIdField) {
        const rd = row.row_data as Record<string, unknown> | null;
        const extId = rd?.[rowEntityIdField];
        if (extId != null) {
            resolvedEntityId = extIdToUuid.get(String(extId).trim()) || null;
            if (resolvedEntityId) calcTimeResolved++;
        }
    }
}
```

**CRITICAL: Preserve backward compatibility.** Some older committed_data rows may have `entity_id_field: null` in metadata (pre-OB-195 imports). For these rows, the fallback must still work. After per-row resolution attempt, if `resolvedEntityId` is still null AND the row has no `entity_id_field` in metadata, try the FIRST non-null `entity_id_field` from any other row (the current global behavior as ultimate fallback).

```typescript
// Build a fallback field name from the first row that has one
let fallbackEntityIdField: string | null = null;
for (const row of committedData) {
    const meta = row.metadata as Record<string, unknown> | null;
    if (meta?.entity_id_field && typeof meta.entity_id_field === 'string') {
        fallbackEntityIdField = meta.entity_id_field;
        break;
    }
}

// In per-row loop:
if (!resolvedEntityId) {
    const rowMeta = row.metadata as Record<string, unknown> | null;
    const rowEntityIdField = (rowMeta?.entity_id_field as string) || fallbackEntityIdField;
    
    if (rowEntityIdField) {
        const rd = row.row_data as Record<string, unknown> | null;
        const extId = rd?.[rowEntityIdField];
        if (extId != null) {
            resolvedEntityId = extIdToUuid.get(String(extId).trim()) || null;
            if (resolvedEntityId) calcTimeResolved++;
        }
    }
}
```
