## §0 — WHY THIS, WHY NOW
Every user in production was created by a seed script outside any validated path. The unresolved Sabor login defect is the direct product: profiles with design-violating field shapes (capabilities written as JSONB objects carrying mixed role+module flags, e.g. `{"icm":true,"admin":true,"financial":true}`; the design contract is an array of structural capability strings). The structural fix is not another patch — it is making the platform the only thing that can create users, through one validated writer, with contracts enforced at write time. DS-014 §6 designed the surfaces; DS-019 Phase F named the build; the ICM User Production Readiness Audit lists user provisioning, deactivation, and entity-to-user linking among the P0 gaps; this document makes them buildable.

## §1 — DS-014 AMENDMENTS PROPOSED FOR DECISION LOCK
1. **Role canon executed-to-date:** `vl_admin` retired at the RLS layer via `is_platform()` (HF-283); `PLATFORM_ROLE_VALUES` is the app-side declaration. The DS-014 §3 resolution table stands; remaining renames (`sales_rep`→`member`, `tenant_admin`→`admin`) execute in the remediation phase (§6).
2. **Capabilities contract (closes the defect class):** `profiles.capabilities` is a **JSONB array of capability strings** drawn ONLY from the DS-014 §4 matrix vocabulary as encoded in `permissions.ts` (the single PDP). It is **derived from role at write time** by the single writer — never authored by hand, never an object, never free text. A DB CHECK (`jsonb_typeof(capabilities) = 'array'`) backstops the writer. Module access is NOT a profile capability: it is tenant-level state (live schema: `tenants.features` / `tenants.settings`; CC verifies the exact key at Phase 0) — therefore normalizing legacy object-shaped capabilities to role-derived arrays is lossless.
3. **Per-tenant capability overrides and module-specific capability namespaces** (DS-014 docx ABAC layer, `tenant_settings.permissions` pattern) are OUT of this build; the derivation function is the seam where both later attach. "Role templates" from the onboarding spec are satisfied BY derivation — the role IS the template.
4. **Entity-linkage direction (supersedes DS-019 §3):** DS-019 designed `profiles.entity_id → entities.id`. The live schema implements the inverse — **`entities.profile_id` (nullable) → `profiles.id`** — and all built RLS (DS-014 §5.2) traverses that direction. This design canonicalizes the live direction. "Entities without platform access" = `entities WHERE profile_id IS NULL` (tenant-scoped). CC must not re-derive the DS-019 column.
5. **Uniqueness model (HF-282 HALT-2 trace preserved):** flat `UNIQUE(auth_user_id)` is the Option-B target, valid because the tdelcarlo duplicate rows were dispositioned as tenant-create-flow artifacts and deleted, and §6.3 removes the flow that minted them. The composite alternative (`UNIQUE(auth_user_id, tenant_id)` + partial unique WHERE tenant_id IS NULL) is the recorded fallback if a genuine multi-tenant administrator requirement ever materializes; adopting it then is a named follow-on, not a silent widening.
6. **Platform-tier lifecycle is in-product (architect-directed 2026-06-11):** DS-014 §3 marks platform-role assignment "Internal only" — this design defines "internal" as the same single door: platform admins create platform admins (F8) through the §3 writer, never by seed. The seed-created platform personas (platform@, tdadmin, eoadmin) are the same defect class this build retires. DS-014 §6.1's function list is amended to include platform-user invitation.

## §2 — THE DATA CONTRACT (one user, fully specified; schema-verified against SCHEMA_REFERENCE_LIVE profiles 11-col shape)
**auth.users:** created via admin API by the writer; email confirmed per flow (invite link confirms; temp-password path sets `email_confirm: true`); provider `email`.
**profiles (exactly one row per auth user — Option B, SOC2 CC6 governing):**
- `auth_user_id` — the linkage; UNIQUE constraint added in §6 (deferred home from HF-282).
- `tenant_id` — NULL iff role `platform`; required otherwise.
- `role` — one of `platform | admin | manager | member | viewer` (CHECK constraint added §6; column default `viewer` stands).
- `capabilities` — derived array per §1.2 (e.g., admin → `["manage_users","import_data","configure_plans","run_calculation","view_all_results","upload_storage",…]` exactly per permissions.ts/DS-014 §4; note the matrix gives `submit_dispute` to member+manager but not admin — derivation follows the matrix, not intuition).
- `display_name`, `email` (both NOT NULL per live schema), `locale` (tenant default — `tenants.locale` NOT NULL default `en` — overridable per user), `avatar_url` null.
- `status` — active | disabled (new column, §6; see Q-D).
**Entity linkage:** optional at create; effected by setting `entities.profile_id` (§1.4), never by a profile column.
**Invariant:** any row violating this contract is impossible to create through the platform and is flagged by the §6 audit.

## §2A — PERSONAL-DATA MAP & STRUCTURAL PRIVACY INVARIANTS (Decision 123 applied; GDPR Art 25 privacy-by-design as the strictest-regime ceiling, subsuming LFPDPPP/ARCO for MX, LOPDP for EC, LGPD)
**Where personal data lives in this build's scope — the closed set:**
| Store | Personal data | Lifecycle |
|---|---|---|
| `auth.users` | email, hashed credential, sign-in metadata | erased whole by F10 |
| `profiles` | email, display_name, locale, avatar_url | tombstoned by F10 (PII nulled, uuid row persists) |
| `audit_logs.ip_address` | IP (personal data under GDPR) | bounded security-retention window, then nulled (Q-I sets the number) |
| `platform_events` / `audit_logs` payloads | **NONE — by invariant I-1** | retained pseudonymously, indefinitely auditable |
| Resend (subprocessor) | recipient email + delivery metadata | governed by I-3; Resend joins the subprocessor register alongside Supabase/Vercel/Cloudflare/Anthropic |

**Structural invariants (write-time discipline, CI-greppable like the seeder prohibition):**
- **I-1 — uuid spine, never PII copies:** events, audit rows, and every JSONB payload reference users by `profile_id`/`auth_user_id` uuid and structural facts (role, tenant) ONLY — never email, name, or IP. This is what makes erasure a one-row operation instead of a forensic sweep, and lets legally-retained records (SOC/GAAP/MX fiscal) coexist with erasure rights by construction.
- **I-2 — minimization as contract:** the §2 field set is the collection boundary; the writer accepts no fields outside it; capabilities are structural, never behavioral or personal.
- **I-3 — emails carry a link and the recipient address, nothing else:** no payout data, no role detail beyond what the template function requires, no personal data of third parties transits the subprocessor.
- **I-4 — notice at collection:** first login presents the privacy notice (aviso de privacidad for MX tenants, tenant-locale) and emits a version-stamped `privacy_notice.presented` event; an explicit acknowledgment gate is Q-J.
- **I-5 — residency named:** personal data resides in the single production Supabase region; per-tenant residency is an infrastructure seam (out of scope, recorded), not a silent assumption.

## §3 — THE SINGLE WRITER (the only door)
`provision-user.ts` extends to a lifecycle service consumed ONLY via server API routes (service-role; no client-direct profile writes; no new RLS write policies needed):
- `POST /api/users` — create+invite (or create+temp-password). Validates contract → creates auth user → inserts profile with derived capabilities → sets `entities.profile_id` when linkage supplied → emits events → returns invite state. **Atomic:** auth-user creation rolls back (admin delete) if the profile insert fails.
- `POST /api/users/:id/reset` — admin-initiated reset email.
- `POST /api/users/:id/role` — role change → capabilities re-derived → events.
- `POST /api/users/:id/disable|enable` — auth ban/unban + profile `status` flag (Q-D).
- `POST /api/users/:id/send-credentials` — body `type: invite_resend | magiclink | recovery`. One delivery route for all credential emails: service-role `generateLink` mints a time-limited, single-use link of the requested type → branded Resend delivery in the tenant locale → event emitted. `invite_resend` voids the prior invite link; `magiclink` gives demo/offline users a one-time sign-in without any password handling; `recovery` is the F4 admin-reset transport.
- `POST /api/users/:id/erase` — the data-lifecycle terminus (F10): requires prior disabled state + explicit irreversibility confirmation; admin may erase own-tenant users, platform any; the §3 lockout guard applies; deletes the auth user, tombstones the profile (PII fields nulled, `email` → `erased+{profile_id}@anon.invalid` to satisfy NOT NULL/uniqueness, `display_name` → constant), clears `entities.profile_id` (the entity is tenant business data and remains), and emits a PII-free erasure event. The uuid spine (§2A I-1) keeps all retained audit/event/calculation records intact and pseudonymous.
- Public `/(auth)/forgot-password` — self-service reset (Site URL prerequisite — confirmed set 2026-06-11).
**Authorization on every route (single helper):** caller capability `manage_users` + tenant scoping (admin manages own tenant only; platform any; nobody assigns `platform` except platform) **+ AAL2 assertion** per DS-019 Phase D, which names user management as a step-up-gated operation (Q-C; cost ≈ nil since tenant admins are MFA-enforced per exit-criteria B1 ✅). **Lockout guard (writer invariant, generalizing the DS-014 §3 VL-Admin survival rule):** the writer refuses any disable, demotion, or deletion that would leave zero active `platform` users; self-demotion or self-disable requires at least one other active platform admin to exist.
**Password discipline (M2):** no human passwords persisted anywhere, ever — not in directives, reports, seeds, or logs. Temp passwords (F2) are generated to the production policy (12-char, all types, leaked-check — Supabase-enforced, audit §1 ✅), displayed once, and force-changed at first login. **Credential delivery is link-based:** the email IS the credential — time-limited, single-use invite/magic/recovery links — never a password in an email body, per the no-persisted-passwords rule (an inbox is persistence) and NIST 800-63B secure-delivery requirements (Q-H records the disposition seam if literal temp-password email is ever directed). Verification harnesses use the mint pattern (generateLink→verifyOtp), never credentials.
**Email dispatch (one facility, architect note 2026-06-11):** Resend is the canonical platform email service. All credential emails are minted as links via service-role `generateLink` and delivered through a **single platform email module backed by the Resend API** — one verified sender identity (@vialuce.ai), one template set, tenant-locale aware, every send evented (§7). Supabase SMTP (already wired to Resend per onboarding 2.3) remains transport for any residual Supabase-originated system mail, but no credential flow in this build depends on Supabase sending. Consequence CC must implement: API-dispatched mail bypasses Supabase's email rate limits, so the dispatch module carries its own per-identity/per-IP rate limiting, and public-facing sends return uniform success regardless of account existence (anti-enumeration). `RESEND_API_KEY` presence is a Phase-0 environment check.
**Seeders retired:** all seed/demo scripts MUST call these routes (or the service directly under test harness) — direct `profiles` INSERTs outside the writer (e.g., `p3-profiles.ts`) are prohibited and CI-greppable. Bulk/roster provisioning (onboarding spec 2.6, Topic 5B) is out of scope (§9) but the writer's signature is the seam: batch = iterated calls to the same door.

## §4 — FLOWS (functional detail)
**F1 Create/Invite (primary):** admin opens Users surface → "Invite user" → email, display name, role (no `platform` option for tenant admins), optional entity linkage (§1.4 promotion) → submit → writer provisions → invite link minted (`generateLink type:'invite'`) and delivered through the canonical Resend module (§3; templates Phase D) → user sets password → first login (HF-284-safe) → **admin-role invitees pass the existing MFA enrollment surface (/auth/mfa/enroll) per the live role-based enforcement; no new MFA UX is built (§9)** → lands per the existing role-derived routing (member/manager/viewer → `/perform`; admin → the admin workspace per live derivation — Decision 128 candidate makes `/stream` canonical for data-bearing tenants; CLT166-F02/F03 empty-state behavior is a known open finding this build does not own).
**F2 Create with temporary password (demo/offline):** same form, "set temporary password" toggle → policy-compliant generated password shown once → forced password change at first login.
**F3 Self-service reset:** login page "Forgot password" → server route mints a recovery link (`generateLink type:'recovery'`) → canonical Resend dispatch (uniform success response regardless of account existence; route rate-limited per §3) → new password → login.
**F4 Admin reset:** Users surface row action → reset email sent → event logged.
**F5 Role change:** row action → role picker → capabilities re-derived → user's next request reflects new access (capability checks read live profile).
**F6 Disable/Enable:** row action → auth ban + `status` flag → login blocked with honest message; enable reverses.
**F7 Entity-to-user promotion:** "Entities without platform access" (`profile_id IS NULL`) list → Invite → F1 prefilled with entity linkage (DS-014 §6.3 — the key UX bridging data entities and platform users; most entities never become users).
**F8 Platform-user provisioning:** platform admin on `/admin/users` → "Invite platform user" → email, display name → role fixed `platform`, `tenant_id` NULL per §2 → writer provisions → invite email → first login is HF-284-safe and includes the existing MFA enrollment (platform role is MFA-enforced per HF-147/CLT-178) → lands in the platform workspace per live derivation. Tenant admins never see this flow; events follow the §7 platform-target emitter precedent; the §3 lockout guard governs all subsequent disable/demote actions on platform users.
**F9 Credential delivery & resend (architect-directed 2026-06-11):** Users surfaces show credential state per user — **Invited (pending, with sent-at) / Active / Disabled** — with row actions: **Resend invite** (pending users; voids the prior link), **Send sign-in link** (one-time magic link — the credential-by-email path for demo and offline users, no password ever transmitted), **Send password reset** (= F4). All three ride `POST /api/users/:id/send-credentials`, authorize through the same §3 helper, deliver through branded tenant-locale Resend templates, and emit §7 events. Supabase rate limits apply to all link generation.
**F10 Offboard & erase (the data-lifecycle terminus; GDPR Art 17 / ARCO-Cancelación / SOC 2 Privacy disposal):** row action on a **disabled** user → irreversibility confirmation naming what is destroyed (identity) and what is retained pseudonymously (audit, events, calculation provenance — legal-retention records) → `/api/users/:id/erase` executes per §3 → surface shows the tombstone state. First login after invite presents the privacy notice per §2A I-4.
**F11 Roster-to-users bulk promotion & group role application (architect-directed 2026-06-11):** the "Entities without platform access" view becomes multi-select, filterable by imported attributes (the columns CEEC carried) → admin assigns **one role to the selection** → batch invite executes as **iterated calls through the §3 single door** (the seam activates; no batch bypass): per-row atomicity, idempotent re-run, and a partial-failure report naming each entity's outcome. Invite email per entity is sourced from the imported roster where the import's field classification identified an email-semantic field (structural identification per the Korean Test — never a hardcoded header literal), with inline manual fill for gaps before submit. Group application also works on existing users (bulk F5 role change). **Group permissions = group ROLE:** capabilities still derive per §1.2; per-group capability overrides remain the §1.3 seam. CSV-of-users upload stays out of scope (§9) — the roster already arrives through SCI.

## §4A — HIERARCHY FROM DATA: MANAGER/EMPLOYEE CONSTRUCTED, NOT CONFIGURED (architect-directed 2026-06-11)
**Design lineage (locked-in-history, now operationalized):** Entity Model D2 — the typed, directed, temporal relationship graph (`entity_relationships`: relationship_type, direction, confidence, evidence, source ∈ ai_inferred|human_confirmed|human_created|imported_explicit, context, effective dates — live schema matches, 13 columns); D6 — operational users derive scope from the graph, pre-materialized into `profile_scope` (live, `scope_type` default `graph_derived`), regenerated on relationship/role change, consumed as a fast array-containment test; **Contextual Proximity Inference** (TMR candidate, 2026-02-16) — relationships discovered from data through six structural proximity dimensions: shared-attribute, containment, transactional, temporal, hierarchical-by-exclusion, cardinality. Gap analysis 2026-02-20 §1.7 records the inference pass as designed-not-built. D6's bidirectional FK pair resolves to the single live direction per §1.4.
**What this build constructs:**
1. **Derivation (CPI pass):** post-import or on-demand over roster data, the recognition/construction split per Decision 158 — the LLM recognizes proximity signals and emits compact relationship intents; deterministic code constructs graph edges with `source='ai_inferred'`, per-edge confidence and evidence (which fields, which dimension). First-class dimensions for rosters: **containment** (manager-identifier-classified fields), **shared-attribute** (location/zone co-membership), **hierarchical-by-exclusion**, **cardinality**. All signals structural — zero domain or language literals (Korean Test).
2. **Confirmation (hints, never gates):** admin reviews inferred edges confidence-ranked on the Users/roster surface → confirm or reject → confirmed edges flip to `source='human_confirmed'`. Inferred edges NEVER feed scope silently; only confirmed (or imported_explicit/human_created) `manages`-type edges materialize.
3. **Materialization:** confirmed manages-subgraph → `profile_scope` rows (`graph_derived`, `visible_entity_ids`) for manager-role users with entity linkage. The §3 writer triggers regeneration on role change to/from manager, on F11 promotion, and on edge confirmation — D6's regeneration contract.
4. **Consumption boundary (honest):** this build proves scope CONTENTS (A7, harness set-equality), not scope ENFORCEMENT — role-scoped RLS read policies and manager team views remain the named successor (DS-014 Phases 2/4), keeping the §9 RLS-hygiene exclusion intact. Nothing here forecloses them; `profile_scope` is exactly the surface they read.
