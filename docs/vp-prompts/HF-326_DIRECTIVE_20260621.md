## §1 — Problem Statement

Three defects block the calculate flow. All share a root cause: configure and calculate surfaces have zero awareness of tenant state (what data exists, what plans exist, what periods exist, what can be done next). Each is independently demo-blocking.

### Defect A — No plan selector on Lifecycle Cockpit

The Lifecycle Cockpit (`/operate`) shows one plan (the first returned by the API) and provides no mechanism to select a different plan. Tenants with multiple plans (MIR has 5) cannot calculate any plan other than whichever the API returns first. There is no dropdown, no list, no route parameter. The calculate button fires for the hardcoded plan only.

**Evidence:** Screenshot `CLT227-HF325_-Calculate_-_no_plan_selection.pdf` — MIR tenant, 5 plans, cockpit shows "Plan de Incentivos 2025 — Coordinadores de Logistica" with no selector. Three periods visible (Mar/Feb/Jan), all "No calc." No path to calculate any of the other 4 plans.

**Lineage:** CLT102-F17 (no multi-plan selection), F-46 (Plan x Data x Period not user-controllable), CLT111-F39 (Calculate All, no individual selection). Identified across 4+ CLT cycles since February 2026.

### Defect B — Auto-detect periods broken

The `/configure/periods` page has an "Auto-detect from data" button (OB-227 Cluster C, Critical Blocker B1). The button calls `/api/periods/detect` which returns "No data uploaded yet. Upload data to auto-detect periods" despite the tenant having committed_data loaded (BCL: 510 rows, MIR: 75K rows).

**Root cause:** The detect endpoint cannot resolve the tenant_id from the session context to query committed_data. The same defect exists on MIR (noted in prior session as "clean-slate period endpoint bug — tenant_id resolution failure in /api/periods/detect").

**Evidence:** Screenshot `CLT227-HF325_-Configure_Periods.pdf` — BCL tenant, 6 manually-created periods visible, "Detected periods" section says "No data uploaded yet."

### Defect C — Configure/Periods dead-end UX

After periods are created (manually or otherwise), `/configure/periods` is a dead end. The page shows periods in a table with "Status" and delete as the only actions. It has zero awareness of:
- Whether data is loaded (it is)
- Whether a plan is imported (it is)
- Whether all preconditions for calculation are met (they are)
- What the user should do next (calculate)

This fails IAP on all three dimensions: no Intelligence (doesn't know what's present), no Acceleration (no forward guidance), no Performance (doesn't enable the next step).

### Combined treatment rationale

All three defects live on the same two surfaces (`/configure/periods` and `/operate`), share the same root cause (no tenant state awareness), and block the same user journey (import data, create periods, select plan, calculate). Fixing them together ensures the journey works end-to-end. Fixing any one alone leaves the user stuck at the next break.

## §2 — Constraints

**C1 — STATE QUERY, NOT STATE INVENTION.** The surfaces query existing state (plans from rule_sets, data from committed_data, periods from periods table, results from calculation_results). No new tables, no new state machines, no new lifecycle phases.

**C2 — EXISTING API ROUTES.** The plan list comes from whatever API the import page already uses to populate its plan selector (the `allActivePlans` pattern visible in import code). The period detect endpoint already exists at `/api/periods/detect`. CC reads the import page's plan-fetching code and reuses the same query pattern for the cockpit.

**C3 — TENANT_ID RESOLUTION.** The auto-detect fix resolves the tenant_id correctly from session context before querying committed_data. CC must diagnose the specific resolution failure (is the session missing tenant_id? is the query using a different column? is the RLS policy blocking?) by reading the endpoint code and the session/auth context code.

**C4 — IAP ON EVERY SURFACE.** Every changed surface must pass the IAP gate: Intelligence (knows what state the tenant is in), Acceleration (guides the user to the next step), Performance (enables the action). The "dead end" is the anti-pattern being removed.

**C5 — VIALUCE THEME ONLY.** Changes render under `useIsVialuce()`. Non-Vialuce paths unchanged.

**C6 — RECONCILIATION-CHANNEL SEPARATION.** CC reports UI behavior verbatim. No GT comparison.

**C7 — NO SCOPE EXPANSION.** No i18n (OB-scope). No lifecycle state machine changes. No import flow changes. No engine changes. This HF makes existing state visible and actionable on two surfaces.

## §3 — Proof Gates

**PG-1 — PLAN SELECTOR EXISTS.** On the Lifecycle Cockpit (`/operate`), when a tenant has multiple active plans, a plan selector is visible. Switching plans updates the cockpit content (period ribbon, calculation summary, data readiness). Paste the rendered selector component code.

**PG-2 — PLAN SELECTOR SINGLE-PLAN.** For tenants with one plan (BCL), the selector is either hidden or auto-selected with no user action required. The page does not degrade.

**PG-3 — AUTO-DETECT FIRES.** On `/configure/periods`, clicking "Auto-detect from data" for a tenant with committed_data returns detected periods (not "No data uploaded yet"). Paste the fixed endpoint code showing the tenant_id resolution path.

**PG-4 — PERIODS PAGE STATE AWARENESS.** After periods are created, `/configure/periods` displays a contextual guidance section showing: what prerequisites are met (data loaded, plan imported, periods created) and the next action ("Ready to calculate" with a link/button to `/operate`). Paste the component code.

**PG-5 — BCL JOURNEY.** Starting from `/configure/periods` on BCL (6 periods, data loaded, plan imported): the page shows all preconditions met, links to calculate. `/operate` shows the plan (auto-selected, single plan), periods, and "Run Calculation" is actionable.

**PG-6 — MIR JOURNEY.** Starting from `/operate` on MIR (5 plans, 3 periods): the plan selector shows all 5 plans. Selecting a different plan updates the cockpit. Each plan's calculation status is independent.
