## §1 — Problem Statement

### 1.1 The defect

BCL plan calculation produces a correct C1 (component index 0, "Colocación de Crédito") payout for every **Ejecutivo Senior** entity and an incorrect, collapsed C1 for every **Ejecutivo** entity. C2/C3/C4 reconcile exactly for both variants, both periods. The defect is isolated to C1 on the Ejecutivo variant.

The collapse signature, established by entity-level reconciliation against ground truth: for every Ejecutivo entity, C1's cumplimiento (loan-placement attainment) axis lands in the **lowest band** (the `< 0.7` floor), while the calidad (portfolio-quality) axis is **correct**. Ejecutivo Senior entities are correct on both axes.

### 1.2 Root cause (proven from runtime trace + persisted bindings + consumer code)

The BCL plan is variant-structured. The persisted `rule_sets.input_bindings.convergence_bindings` is keyed `component_0` through `component_7` against the **flattened, all-variants** component list (per HF-243: `extractComponents` flattens components across all variants, was `variants[0]`-only). In that flattened keying:

- `component_0` = **Senior** Colocación, with measure binding `colocacion_credito → Monto_Colocacion`.
- `component_4` = **Ejecutivo** Colocación, with measure binding `colocacion_actual → Monto_Colocacion`.

(Both verified in the live `convergence_bindings` JSONB. Both at `match_pass: 1`, `confidence: 0.9`, no `scale_factor`. The two variants use **different numerator field tokens** — `colocacion_credito` vs `colocacion_actual` — for the same underlying column.)

The **consumer** in `web/src/app/api/calculation/run/route.ts` reads the binding by the variant-selected component loop's **positional index**:

```ts
const compBindingKey = `component_${compIdx}`;
const compBindings = convergenceBindings?.[compBindingKey];
```

`compIdx` is the loop index over `selectedComponents` — the entity's *variant-selected* component array (indices 0–3 for any entity, regardless of which variant matched). For a Senior entity, `selectedComponents[0]` is Senior Colocación and `convergence_bindings['component_0']` is Senior Colocación: **aligned, correct.** For an **Ejecutivo** entity, `selectedComponents[0]` is *Ejecutivo* Colocación, but the loop still reads `convergence_bindings['component_0']` — the **Senior** binding. The correct Ejecutivo binding at `component_4` is **never consulted** for an Ejecutivo entity, whose loop only reaches index 3.

Downstream, `resolveMetricsFromConvergenceBindings` (prime_dag branch) walks the **Ejecutivo intent's** DAG references (`colocacion_actual`, `meta_colocacion`, `calidad_cartera`) but reads each against `component_0`'s **Senior** bindings:

```ts
const refs = extractReferencesFromDAG(intent);   // Ejecutivo intent → includes colocacion_actual
for (const field of refs) {
  const fieldBinding = compBindings[field];        // compBindings = Senior's component_0
  if (!fieldBinding?.column) continue;             // ← colocacion_actual undefined in Senior binding → SKIP
  ...
}
```

`component_0` (Senior) has **no `colocacion_actual` key** — it has `colocacion_credito`. So `compBindings['colocacion_actual']` is `undefined`, the prime_dag branch executes `continue`, and the numerator field is **silently skipped** — no binding read, no metric populated, no trace line emitted. `meta_colocacion` and `calidad_cartera` resolve (both present in `component_0`). In the evaluator (`web/src/lib/calculation/intent-executor.ts`), the `reference` to the absent metric returns `ZERO`; the `divide` yields 0; `0 ≥ 0.7` is false at every cumplimiento band → **the conditional falls through to the lowest band.** Calidad resolves, so the calidad axis is correct. This is the exact observed coordinate signature.

### 1.3 Runtime evidence (forensic trace, `CALC_TRACE_VERBOSE=true`, BCL October)

For Senior entities (BCL-5001, 5003, 5004), `component_0` traces **both** measure fields:
```
prime_dag_field entity=BCL-5001 componentIdx=0 | field=colocacion_credito | column=Monto_Colocacion | raw=112918.53
prime_dag_field entity=BCL-5001 componentIdx=0 | field=meta_colocacion    | column=Meta_Colocacion  | raw=150000
```
For Ejecutivo entities (BCL-5006, 5009), `component_0` traces **only** the denominator — the numerator line is **absent** (the silent skip):
```
prime_dag_field entity=BCL-5006 componentIdx=0 | field=meta_colocacion | column=Meta_Colocacion | raw=120000
prime_dag_field entity=BCL-5009 componentIdx=0 | field=meta_colocacion | column=Meta_Colocacion | raw=120000
```
The missing numerator trace line on Ejecutivo entities is the direct fingerprint of `compBindings['colocacion_actual']` being `undefined` in the Senior `component_0` binding and the field being `continue`-skipped.

### 1.4 Defect class and lineage

Two coupled defects, one structural-class each:

- **Defect A (root):** binding consumption is keyed by the variant-selected loop ordinal (`component_${compIdx}`) instead of by stable variant-component identity. Bindings are produced against the flattened all-variants list; they are consumed against the per-entity variant-selected list; the two index spaces coincide only for variant 0. Every non-zero-indexed variant whose components occupy different ordinals in the flattened list than in its own selected array reads the wrong binding. BCL Ejecutivo is the instance of record; the class is any variant-structured plan.

- **Defect B (silent-skip):** the prime_dag branch's `if (!fieldBinding?.column) continue;` converts an unresolvable DAG reference into a silent metric omission, which the evaluator then converts into a `ZERO` that collapses a band to a plausible-but-wrong payout. This is the same loud-vs-silent class HF-272 closed at the interpretation-time field-resolution gate; it recurs here at the **binding-resolution** path, which HF-272 did not touch.

Per **SR-34**, A and B ship together. Fixing A alone (correct binding selected) leaves the silent-skip landmine for the next misalignment or genuine binding gap. Fixing B alone converts the wrong C1 into a loud `failed` but does not restore the correct payout. The combined treatment closes the class: the correct binding is selected (A), and any *genuinely* unresolvable reference thereafter fails loud (B).

### 1.5 Lineage to prior scale-arc sessions

HF-243/HF-244 and the 2026-05-28 dimensional-coherence session repeatedly diagnosed C1-Ejecutivo as a **scale** defect (`cumplimiento_colocacion` scale=1 vs scale=100) and fixed it at the binding/annotation layer; it recurred. This HF establishes that scale is **not** the cause — the evaluator applies no scale to these bands (the cumplimiento compare constants carry no `meta`), the binding records `scale=1` correctly, and the column read returns the right magnitude. The recurrence across sessions was the prior fixes correcting the annotation layer while the actual fault — wrong binding consulted, numerator silently skipped — was never read at the consumer. This HF fixes the consumer.

## §4 — Phase 2: Fix (ships as one PR with Phase 3 per SR-34)

Phase 2 (Defect A) and Phase 3 (Defect B) are authored and committed together. Neither ships alone.

### 4.1 — Defect A: align binding consumption to variant-component identity

Contingent on §3.2 confirming a shared stable `PlanComponent.id`. The fix replaces ordinal-based binding lookup with identity-based lookup.

Today (consumer):
```ts
const compBindingKey = `component_${compIdx}`;
const compBindings = convergenceBindings?.[compBindingKey];
```

The fix resolves, for the variant-selected component at loop index `compIdx`, the **flattened-list ordinal** of that same component (by its stable `id`), and keys the binding by that flattened ordinal:

```ts
// HF-273 Defect A: convergence_bindings is keyed component_N against the
// FLATTENED all-variants component list (extractComponents order — see
// convergence-service.ts). selectedComponents is the per-entity variant-
// selected array; its ordinal does NOT match the flattened ordinal except
// for variant 0. Resolve the binding by the component's stable identity:
// find this selected component's position in the flattened list and key by it.
const flattenedComponents = /* the same flattened list extractComponents produces,
                                reconstructed at calc time from `variants` /
                                `defaultComponents` — Phase 1.1 specifies the exact source */;
const selectedComp = selectedComponents[compIdx];
const flattenedIdx = flattenedComponents.findIndex(c => c.id === selectedComp.id);
const compBindingKey = flattenedIdx >= 0 ? `component_${flattenedIdx}` : null;
const compBindings = compBindingKey ? convergenceBindings?.[compBindingKey] : undefined;
```

Exact reconstruction of `flattenedComponents` at the consumer is specified by Phase 1.1's finding (it is the union of `variants[*].components` in `extractComponents` order, or `defaultComponents` for single-variant plans). For single-variant plans and for variant 0, `flattenedIdx === compIdx`, so `compBindingKey` is unchanged → DD-7 preserved by construction.

**Korean Test compliance:** selection is by `c.id` equality — a structural identity — not by variant name or role string. No language-specific literal is introduced.

**If `flattenedIdx < 0`** (the selected component has no counterpart in the flattened list — a structural impossibility if both derive from the same plan, but a real failure if the plan shape is inconsistent): this is a loud failure, not a silent fallback. Emit a per-component resolution failure (the Phase 3 marker), not `component_0` and not a silent skip.

### 4.2 — Defect B: make an unresolved DAG reference loud (HF-272 marker at the binding-resolution path)

In `resolveMetricsFromConvergenceBindings`, prime_dag branch. Today:
```ts
for (const field of refs) {
  const fieldBinding = compBindings[field];
  if (!fieldBinding?.column) continue;   // ← silent skip
  ...
}
```

The fix records a resolution failure for any DAG reference field that has no resolvable binding, and surfaces it as the HF-272 per-component `resolutionFailure` marker so the component yields a loud `failed` at calc time rather than a silent partial-metrics map that the evaluator turns into a band-collapsing ZERO:

```ts
// HF-273 Defect B: an unresolvable DAG reference field is a loud failure,
// not a silent skip. A skipped numerator becomes ZERO in the evaluator and
// collapses a band to a plausible-but-wrong payout (the exact HF-272 silent-
// $0 class, recurring at the binding-resolution path). Record the failure;
// the per-component resolutionFailure marker (HF-272) converts it to a loud
// `failed` status at calc time.
const unresolvedRefs: string[] = [];
for (const field of refs) {
  const fieldBinding = compBindings[field] as ConvergenceBindingEntry | undefined;
  if (!fieldBinding?.column) {
    unresolvedRefs.push(field);
    continue;
  }
  const rawValue = resolveColumnFromBatch(fieldBinding.column, lookupKey, fieldBinding.filters);
  if (rawValue === null) { unresolvedRefs.push(field); continue; }
  const scaled = fieldBinding.scale_factor ? rawValue * fieldBinding.scale_factor : rawValue;
  dagMetrics[field] = scaled;
  ...
}
// Surface unresolved references through the SAME resolutionFailure channel
// HF-272 established at convergence (per-component `failed`, loud at calc time).
// The exact marker shape and the call/emit site are specified by reading the
// HF-272 resolutionFailure implementation in convergence-service.ts (Phase 2.3
// below) and matching it — do NOT invent a parallel failure channel.
```
