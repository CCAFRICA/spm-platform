## §1 — Problem Statement

Meridian (tenant `5035b1e8-0754-4527-b7ec-9f93f85e4c79`, rule_set `2fb555d4-53fe-42e8-9662-cae3d07da4f4`) has two coupled defects in the fleet/hub path. The four non-fleet components calculate correctly per-employee (line-item exact); only the fleet path and the hub-payee admission are wrong.

### Defect 1 — Fleet component (C5) returns zero for every entity, every period

Code-justified by **DIAG-58** (against refreshed reference `e85a7678`, post-HF-238 prime-DAG engine):

- The post-HF-238 engine is a single `evaluate()` walker over a 10-prime `PrimeNode` union. The `scope` prime (`intent-executor.ts:223-238`) matches **sibling ENTITIES** sharing `entityMetadata[boundary]`, self-excluded; downstream `aggregate` sums `row[field]`. This is **peer-entity aggregation** and it is COMPLETE (harness: a manager's district aggregate resolves nonzero = 300).
- The fleet component needs a **different shape**: take the per-hub utilization ratio (loads ÷ capacity, hub-level) and **project it onto each employee at that hub**. DIAG-58 established this **reference-row→member projection shape was never implemented** — not in the deleted pre-HF-238 `scopeAggregates` (which was itself peer-entity), not in the current scope prime. This is **Condition A: an additive capability gap, not a regression.**
- Live root cause: the fleet intent is a plain `reference` tree relying on convergence to resolve hub columns onto the employee key — which it cannot, the columns are hub-keyed (the binding cross-key). Even if routed through the scope prime, Meridian hub entities carry `metadata = {}`, so `entityMetadata[boundary]` is `undefined` and no projection occurs. Either way: **C5 = 0 for everyone.**
- The fleet **formula is sound** (harness c2: with hub metrics present, `clamp(1200/1000=1.2 ≤ 1.5) × 800 = 960`). C5=0 is **pure input-resolution starvation**, not an arithmetic fault. The fix is resolution, not formula.

### Defect 2 — Hub entities are admitted as payees

HF-261 ADR established: 79 entities, all `entity_type='individual'`; by **provenance**, 67 employees (person-roster/Plantilla row) + 12 hubs (no roster row; `external_id` = hub name; origin `Datos_Flota_Hub` reference sheet; `metadata={}`). All 79 get a `rule_set_assignment` (self-heal `run/route.ts:398-433`, no provenance filter) and the roster population filter (`:1035-1052`) fails to exclude them because roster-sheet detection is a keyword heuristic that Meridian's `Plantilla` sheet does not match. Hubs are paid the safety-component base (phantom ≈ 5,500). Hubs are **scope** for the fleet aggregate, not commission recipients.

### Combined-treatment rationale (DD-7 adjacent; SR-34)

Both defects are the **hub/scope-provenance mis-resolution**: the fix that makes hubs resolvable as the fleet **scope source** (Defect 1) is the same provenance distinction that excludes hubs as **payees** (Defect 2). The reference-provenance entity metadata that carries the hub boundary (the projection lever) is the same provenance signal that identifies a hub as scope-not-recipient. Splitting them re-commits the instance-vs-class pattern. One integrated HF.

### Convergence note (SR-42 surface — architect-dispositioned above this directive)

This HF touches **convergence-binding production** for the fleet component (the binding/intent may move from a plain `reference` tree to a `scope_aggregate` that composes to the scope/aggregate primes). Meridian's fleet convergence currently **fails** its anchor — this repairs a broken path, it does not modify a proven one. The proven, passing components (the four non-fleet, exact to the dollar against the reference) are protected by DD-7 and the Phase-1 ADR isolation gate. The SR-42 disposition ("IRA Class A before touching convergence") is resolved in the architect channel prior to dispatch.

## §2 — Substrate-Bound Discipline Applications

- **AP-17 single path (central).** The projection capability is added to the **existing scope mechanism**, not as a new resolver. The scope prime currently projects entity-metadata-keyed peers; the fix makes the **same prime** able to resolve a boundary-keyed reference row onto members. DIAG-58 names the two viable levers — (L1) populate the boundary key into reference-provenance entity metadata so the existing scope prime matches it, or (L2) a matching path within the scope prime that reads the row's boundary field — both keep one path. The ADR selects between them.
- **Korean Test (IGF-T1-E910 / AP-25).** Hub-grouping and fleet inputs resolve structurally — by field-identity and the entity-to-hub provenance relationship. No tenant/hub/column literal. The existing roster Tier-3 keyword list (`['datos colaborador','roster','employee','empleados']`) is itself a **latent AP-25 literal**; if Defect 2 is fixed at the population filter (§3.2 lever c-ii), it de-literalizes that list structurally as a side benefit. HALT-5 if any path requires a literal.
- **`undefined===undefined` footgun (DIAG-58 latent finding — MUST guard).** The scope prime matches metadata-less siblings to each other (`boundaryValue===undefined` matches all `undefined`). Introducing a hub boundary while hub entities (and any metadata-less entity) carry `metadata={}` is a live hazard: a metadata-less entity could match all metadata-less siblings and aggregate spuriously. The fix MUST guard the boundary match so `undefined`/absent boundary never matches (explicit: skip aggregation when `boundaryValue` is null/undefined). This is in-scope hardening, named here because the fix touches the exact code.
- **Carry Everything (IGF-T1-E902).** Hub reference data is already imported and persisted correctly (`reference … 36 rows`). This HF changes **calc-time resolution + entity metadata/admission only** — not ingestion, not the persisted reference data. Do not narrow or re-commit imported data.
- **Reconciliation-channel separation (operative).** Meridian verification target values (per-period totals; per-entity per-component expected values) are **architect-channel only** and do **not** appear in this directive. CC reports **calculated** values verbatim (per-period grandTotals; per-entity component breakdown incl. fleet for a named sample; the per-hub aggregate it computes; variant distribution; reconciliation footer) and produces **no** reconciliation verdict ("correct"/"expected"/"matches"/"too high"). The architect reconciles against the reference file.

### §3.2 — Phase 2: Fix (contingent on ADR disposition)

Three structural corrections, exactly as confirmed in the ADR:

**(i) Reference-row→member projection (fleet C5).** Extend the **one** scope mechanism (selected lever L1 or L2) so the per-hub utilization ratio (aggregate loads ÷ aggregate capacity from the reference rows) resolves for each employee by their hub boundary, and the existing fleet intent applies the variant rate. Hub-grouping derived structurally (Korean Test). If convergence emission must change (per ADR §3.1(d)), that change is scoped to the fleet component only. AP-17: no second resolver.

**(ii) Hub payee exclusion (selected lever c-i or c-ii).** Hub entities (provenance: reference-sheet origin, no person-roster row) are not admitted as rule_set payees / `calculationEntityIds`. Structural distinction by provenance, never name literal. If (c-ii), the roster-provenance recognition replaces the Tier-3 keyword list (de-literalization). Data-state remediation of existing phantom hub entities/assignments/results per ADR disposition (re-import supersession vs explicit cleanup); if cleanup SQL is required, follow the §0 SQL Verification Gate + migration path. **Hubs remain readable as the fleet aggregate SOURCE** — exclusion is from the payee set, not from the data.

**(iii) `undefined===undefined` guard.** The scope-prime boundary match rejects null/undefined `boundaryValue` (no aggregation, no spurious metadata-less sibling match). DIAG-58 latent footgun closed at the same site the fix touches.

Commit + push per Section D after the phase.
