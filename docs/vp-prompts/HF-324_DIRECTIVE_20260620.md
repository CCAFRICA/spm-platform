## ¬ß1 ‚Äî PROBLEM STATEMENT

Post-HF-321 architect browser verification reveals six defects across the Financial Agent and Reconciliation surfaces. HF-321 corrected the seed data field-name mismatch (43,875 rows, 9/9 pages confirmed via route handler output). Browser-rendered verification exposed defects the data-layer tests missed.

**D1 ‚Äî `/financial/performance` CRASH.** Location Benchmarks renders error boundary ("Something went wrong"). Route handler returns valid data for 20 locations. The React component throws at render time. Code bug, not data bug.

**D2 ‚Äî `/financial/staff` CRASH.** Staff Performance renders error boundary. Route handler returns 40 servers with metrics. Component crashes at render. Related to R-4: `metadata.location_id` holds external_id string, component resolves by UUID ‚Üí undefined ‚Üí null reference.

**D3 ‚Äî SINGLE MONTH OF DATA.** All 43,875 rows dated January 2024. Revenue Timeline shows one period. Trend charts flat. Period-over-period comparisons produce artifacts (‚àí99.6%). Only 4 periods exist in the periods table (3 weekly + 1 monthly, all January). A financial analytics platform with one month of data is not demonstrable.

**D4 ‚Äî NO DRILL-THROUGH ON FINANCIAL SURFACES.** Leakage Monitor shows MX$228K in Descuentos ‚Äî user wants to see which cheques, which locations, which servers. No click handler exists on any financial page. OB-224 built the reusable drill-through library (`web/src/lib/drill-through/`) and wired it into 8 ICM surfaces. Financial surfaces were not in scope. The platform has a Transactions page at `/data/transactions` (Platform Core ‚Üí Data Visibility ‚Üí Transactions) ‚Äî financial pages should offer click-through into transaction-level detail.

**D5 ‚Äî BRAND ENTITY MISCLASSIFICATION (R-1).** Brand entities are `entity_type='team'` with `metadata.entity_role='brand'`. Financial route expects `entity_type='organization'` with `metadata.role='brand'` + location `metadata.brand_id`. Network Pulse collapses all 20 locations into "Other (20 locations)."

**D6 ‚Äî RECONCILIATION PAGES GREY BACKGROUND.** The Reconciliation Studio (`/operate/reconciliation`) and its results view render with the old grey/dark background instead of the Vialuce theme. The theme arc (OB-221‚ÜíHF-316) adopted 46 routes but missed these pages. The 5-layer comparison depth assessment, period matching grid, benchmark upload, and confirm mappings sections all render on grey backgrounds while the rest of the platform uses the Vialuce deep indigo + gold palette. Screenshots from BCL tenant (Banco Cumbre del Litoral) confirm both the initial reconciliation view and the populated results view have this defect. This is the same pattern as HF-313/314/315 ‚Äî components not consuming Vialuce design tokens.

**CROSS-TENANT NOTE:** D1‚ÄìD5 target Sabor Grupo (Financial Agent). D6 targets the reconciliation pages which are cross-tenant (BCL, CRP, MIR, Sabor all use them). The fix is in the component CSS/tokens, not tenant-specific.

**OB-227 CONTEXT (for CC awareness):** OB-227 discovered that `calculation_results.components` is an Array for BCL but an Object keyed by name for Sabor. CC guarded this with a `breakdownToRecord` normalizer (Array.isArray check). If the `/financial/performance` or `/financial/staff` crashes (D1/D2) involve rendering `calculation_results.components`, CC should check whether the normalizer is being applied on the financial code paths.

## ¬ß2 ‚Äî OBJECTIVES

CC achieves all six. Sequencing, phasing, and implementation approach are CC's to determine. The only hard ordering constraint is: **read the codebase before changing it** (Phase-0 read-before-build per Standing Architecture Rules).

**O1 ‚Äî CRASH ELIMINATION.** `/financial/performance` and `/financial/staff` render without error boundary activation. Fix root cause (SR-34), not symptom. If D2 is caused by the ext_id‚ÜíUUID mismatch on `metadata.location_id`, fix the data or the lookup ‚Äî do not wrap in try-catch. If the crash involves `calculation_results.components` shape inconsistency, apply or extend the `breakdownToRecord` normalizer from OB-227.

**O2 ‚Äî SIX MONTHS OF DATA.** Extend from January 2024 to January‚ÄìJune 2024. ~43K cheques per month, ~262K total rows. Revenue Timeline shows 6 periods with meaningful variation. Period-over-period comparisons produce real trends. **This includes creating period records for Feb‚ÄìJune** (weekly + monthly per month, matching January's pattern of 3 weekly + 1 monthly) since the Operating Summary period selector and Revenue Timeline derive their period lists from the `periods` table.

**O3 ‚Äî FINANCIAL DRILL-THROUGH.** Users can click into detail from financial surfaces. At minimum:

| Surface | Click Target | Drill-Through Shows |
|---|---|---|
| Leakage Monitor | Category row (Cancelaciones/Descuentos/Cortes√≠as) | Individual cheques matching that leakage category |
| Leakage Monitor | Location row | That location's leakage cheques |
| Staff Performance | Server row | That server's cheques |
| Location Benchmarks | Location row | That location's cheques |

OB-224's component library (`DrillThroughPanel`, `TransactionRows`, `EntityResultsList`, `ComponentCards`) is the integration target. CC reads the library API and wires it. If the library's data contract assumes `calculation_results` structure and financial needs `committed_data` structure, CC adapts (within ‚â§ 30 lines per component ‚Äî beyond that, HALT-2).

**O4 ‚Äî BRAND RESOLUTION.** Network Pulse groups locations by brand (Cocina Dorada, Taco Veloz, Mar y Brasa). Fix the entity data: brand entities ‚Üí `entity_type='organization'` + `metadata.role='brand'`; location entities ‚Üí add `metadata.brand_id` (UUID of parent brand). Read `entity_relationships` for the mapping.

**O5 ‚Äî STAFF LOCATION LABEL (R-4).** Staff Performance shows the location name next to each server, not blank. Fix server `metadata.location_id` from external_id ‚Üí location UUID. The ext_id‚ÜíUUID mapping comes from the entities table.

**O6 ‚Äî RECONCILIATION THEME ADOPTION.** The Reconciliation Studio (`/operate/reconciliation`) and its results view render with the Vialuce theme ‚Äî no grey backgrounds. Same pattern as the theme arc HF-313/314/315: identify components that aren't consuming Vialuce design tokens (CSS custom properties, `useIsVialuce()` hook, three-layer token architecture), apply the tokens. The fix covers:
- Reconciliation page background and card surfaces
- 5-layer comparison depth assessment component
- Period matching grid
- Benchmark file upload area
- Confirm mappings section
- Results table and column headers

The Vialuce theme system is documented at `docs/design/vialuce-theme/`. The three-layer token architecture: Layer 0 (primitives), Layer 1 (semantic), Layer 2 (component slots). Components consume tokens via CSS custom properties (`var(--vl-surface-primary)`, etc.). If the reconciliation page uses components that bypass the token system with hardcoded colors or the old dark-mode classes, replace with token references.
