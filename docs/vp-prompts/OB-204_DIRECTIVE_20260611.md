## §1 — PROBLEM STATEMENT
Every user in production was created by seed scripts outside any validated path. The defect class is live: the three Sabor Grupo Gastronómico users (FRMX tenant) fail login in production; the prime untested suspect is `profiles.capabilities` written as a JSONB **object** (mixed role+module flags, e.g. `{"icm":true,"admin":true,"financial":true}`) where every working user carries an **array** of capability strings. RCA is NOT established — six hypotheses were eliminated with receipts (data/linkage, RLS policy, query-under-JWT, session survival, provider, cache); the profiles row demonstrably reaches the failing browser (200, 1.1 kB). Beyond Sabor: there is no validated way to create, invite, reset, disable, or erase a user; no tenant-admin or platform-admin user-management UI is complete (OB-67 partials untested post-OB-178); the Users-page read defect (CLT166-F10) stands; five orphan auth accounts exist; `UNIQUE(auth_user_id)` was deferred out of HF-282 to this build; hierarchy/manager-scope inference (Entity Model D2/D6, CPI) is designed-not-built (gap analysis 2026-02-20 §1.7). The structural fix is one validated writer with contracts enforced at write time, surfaces over it, remediation of the existing estate through the same contracts, and data-protection invariants established architecturally (Decision 123). Combined treatment is justified: every element shares the writer, the contract, and the Users surface; splitting them re-creates the multi-door condition that caused the defect.
