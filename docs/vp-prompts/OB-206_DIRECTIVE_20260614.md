## §1 — PROBLEM STATEMENT

### 1.1 The Recurring Defect Class

The March 16 Platform Experience Assessment named the root-cause pattern precisely: **the engine operates in Paradigm 3 (Intent-Based Outcome Specification); the experience layer is still Paradigm 2 (pages, sidebars, click-navigate-click).** Every prior attempt to close this gap (OB-105 Bloodwork, OB-166 manager fixes, OB-205 carrier expression) patched a symptom and left the structural gap intact. The assessment's FP-67/73/74 pattern — "backend capability built and proven, frontend surface built and renders, the connection between them never wired" — has recurred across the entire build history.

OB-205 + HF-291 are the latest instance. They added carrier-metadata cards to /stream. They did not address: the tenant-landing routing, the Manager surface, the Rep surface, component-level data in the team grid, or the Paradigm 2 → Paradigm 3 gap. The production screenshots prove three live failures.

### 1.2 The Three Live Failures (production, 2026-06-14)

**F-1: The tenant landing is the wrong page.** Selecting BCL from Observatory lands the admin on `/operate` ("Operations Overview") — a Pipeline Readiness stepper + an ICM module summary card + a "View Intelligence →" button. This page has been flagged across many CLTs (CLT-166-F01/F02/F03, and the March 3 and Feb 17 sessions). DS-015 §4.1 is explicit: **/stream is the canonical landing route; there is no /operate landing, /perform landing, or /briefing landing.** The platform routes tenant selection to /operate anyway. If Intelligence is the destination (the page literally has a button to go there), it must be the landing.

**F-2: The Manager surface is a flat, unactionable grid.** The Manager /stream shows Team Health ($58,406, 22/21/42 triage badges — this part is correct) above a Team Performance Grid: 85 entities, one row each, with "–" in every component column (Colocación de Crédito, Captación de Depósitos, Productos Cruzados, Cumplimiento Regulatorio) and a value only in TOTAL. This is the exact CLT-165-F03 defect ("Manager grid shows all dashes — no component breakdown") that OB-166 attempted and failed to fix. An 85-row flat list with one number per row is not coaching intelligence. The manager cannot answer "who needs coaching on which component?" DS-008-A2, the March 10 design session, and the Feb 19 IAP scorecard all specify the answer: an **entity × component heatmap sorted by coaching priority**, plus **Acceleration Cards (Certify/Coach/Intervene)** with proximate actions. None of it is built.

**F-3: No persona surface delivers Intelligence.** Neither the Admin nor Manager /stream passes the DS-013 test battery. There is no Bloodwork health panel, no trajectory with reference frame, no Five Elements intelligence, no Action Proximity, no Cognitive Fit diversity. The surfaces are thermometers — they display numbers without telling the user what to do.

### 1.3 Why One OB, Not Three

The March 16 assessment is unambiguous: *"This is not a collection of hotfixes. The findings trace to a structural gap between engine capability and user experience. The fix is a single, comprehensive Experience Architecture OB that addresses the user journey end-to-end, governed by DS-013."* The routing, the three persona surfaces, and the component-data path are mutually dependent: routing to /stream is pointless if /stream is empty for the Manager; the Manager surface is pointless without component-level data; the component data path serves all three personas. One vertical slice. One PR.

## §2 — SUBSTRATE-BOUND DISCIPLINE

**DS-013 (governing).** /stream is the single adaptive surface. Persona determines which intelligence ranks highest and which agency tier governs — not which page is shown (§5). Five Elements on every element (§2). The 10-test battery is the ship gate (§7). Ambient environment per persona: Admin indigo, Manager amber, Rep emerald (§5.1–5.3).

**DS-015 (implementing).** /stream is canonical landing (§4.1, Decision 128 candidate). The State Reader computes TenantContext deterministically (§4.2). The V1 priority order per persona is specified (§4.3). Action Proximity per element is specified (§7). Container Stability (Decision 129): section positions stay consistent across visits; content adapts. This OB implements DS-015 Phases A–C (Five Elements + Action Proximity + Persona Density) as one slice, since BCL now has the calculated periods that Phase B trajectory requires.

**DS-003 (component vocabulary).** Every visualization form matches its decision task. The per-persona compositions are specified in DS-003 §4 (Tenant Dashboard — Admin / Manager / Rep). Diversity Minimum (Rule 1): 3+ component types for 4+ elements. Reference Frame Mandate (Rule 3): every number answers "good or bad?" Persona Density Gradient (Rule 4): Admin high, Manager medium, Rep low.

**DS-008-A2 + A3 (intelligence + agency).** Allocation Intelligence drives the Manager coaching priority (entity × component with highest marginal return) and the Rep allocation card. Action Cards follow the Bloodwork Principle (A3 §4.4): they surface only when materially actionable; silence = health. Agency tiers per persona (A3 §4.3): Admin Tier 2–3, Manager Tier 2, Rep Tier 1–2.

**TMR-2 (Wayfinder).** Ambient character per persona environment. State communication via completeness/opacity, not alarm color. Core interaction patterns (search, sort, drill) consistent; persona extensions layered.

**TMR-7 (Persona Psychology).** Admin = governance cognitive task. Manager = people development. Rep = self-mastery. Color psychology: indigo analytical, amber mentorship, emerald growth. Density and reference frames per persona.

**TMR-8 (Cognitive Fit).** Every form passes the six-question Cognitive Fit Test before implementation. The flat grid fails it (a table forces serial reading for a pattern-detection task); the heatmap passes it (pre-attentive two-dimensional pattern detection).

**Korean Test (AP-25).** Every label derives from data: `data_type` values, `entity_type` values, `rule_sets.components` keys, persona structural identifiers. Zero hardcoded domain strings ("commission", "banking", "franchise"). Component column headers come from the rule_set's component names, whatever language they are in.

## §6 — PHASE 3: MANAGER COACHING SURFACE (F-2)

The Manager surface is governed by DS-013 §5.2 (team acceleration cognitive task, amber, Tier 2), DS-015 §6.2 (Manager trajectory), DS-003 §4 (Tenant Dashboard — Manager), DS-008-A2 (Allocation Intelligence → coaching priority), the March 10 design session (entity × component heatmap), and the Feb 19 IAP scorecard (Acceleration Cards).

### 6.1 The entity × component heatmap (replaces the flat grid)

The flat 85-row Team Performance Grid is replaced by an **entity × component heatmap** (March 10 design, DS-003 §1.4 monitoring + pattern detection):

- **Rows:** team entities, sorted by **coaching priority** (highest marginal return first), NOT alphabetically and NOT by total.
- **Columns:** the rule_set's components (names from `rule_sets.components`, Korean Test — whatever the tenant's components are).
- **Cells:** per-component attainment, color-encoded by attainment-relative-to-target (the data the audit in §3.3 located). Cell intensity is the pre-attentive signal. The cell value (the number) remains visible for anyone who needs it.
- **Rightmost column:** trajectory arrow per entity (improving/declining).
- **Interaction:** tapping any cell opens that entity × component detail inline (progressive disclosure, Type 1 → Type 2).

This is the fix for F-2. The manager's eye sees the pattern: which entities are dim across all components (system-level), which are dim in one column (component-specific). The flat grid's "–" cells are replaced by real per-component data.

**HALT reference:** If §3.3 HALT-2 fired (no per-component data anywhere), the heatmap renders attainment from whatever IS available and the completion report documents the gap. Do not fabricate component values.

### 6.2 Acceleration Cards (Certify / Coach / Intervene)

Above the heatmap, three Acceleration Cards (Feb 19 IAP scorecard, DS-008-A3 §4.3 Manager) — the Bloodwork triage of the team:

- **Certify (emerald):** entities exceeding — "Recognize [entity]". Recognition action.
- **Coach (amber):** entities with opportunity — "Ana → Captación de Depósitos: +$N projected. View Ana's detail." The coaching priority names the **entity × component** intersection (DS-008-A2), not just the entity.
- **Intervene (rose):** entities declining 3+ consecutive periods — "Schedule coaching for [entity]". Intervention action.

The existing Team Health card's "22 Exceeding / 21 On Track / 42 Needs Attention" breakdown feeds these three cards — the badges become actionable triage, not a static count. Each card has a proximate action (DS-013 Action Proximity).

### 6.3 Manager trajectory + scope

### 6.4 Manager ambient + Cognitive Fit
