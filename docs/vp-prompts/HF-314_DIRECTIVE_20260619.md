## §1 — Problem Statement

Under `data-theme="vialuce"`, content pages look washed out compared to the same pages under Bliss. The sidebar and topbar (built from scratch with design spec classes) render correctly. The content area (existing React components) does not.

Root cause: Bliss's `[data-theme="bliss"]` block in `globals.css` contains comprehensive utility remaps built across HF-305 through HF-311 — every `bg-zinc-800`, `bg-slate-900`, `text-white`, `border-zinc-700`, `bg-zinc-950`, modal scrim, navbar, and dozens of other dark utility classes are mapped to their light equivalents. Vialuce's `[data-theme="vialuce"]` block has an INCOMPLETE subset of these remaps. The gap between Bliss's remap coverage and Vialuce's remap coverage is the exact cause of every washed-out surface.

Additionally: the per-user theme toggle (`.sb-theme`) in VialuceSidebar is persistently visible above the persona switcher. It should be hidden behind a user profile expansion, consistent with how the other themes handle theme selection.

## §3 — Phase Prose

### §3.0 — CSS Parity with Bliss (the actual fix for "washed out")

**Objective:** Under `data-theme="vialuce"`, every content page renders with the same visual quality as under `data-theme="bliss"`. CSS-only. No React changes. Fixes ALL pages simultaneously.

**Procedure — this is the critical step and must be done exhaustively:**

1. **Open `globals.css`.** Find the `[data-theme="bliss"]` block. Read EVERY utility remap rule it contains. This is the reference — Bliss works on all pages because of these rules.

2. **Find the `[data-theme="vialuce"]` block.** Compare rule by rule against the Bliss block.

3. **For EVERY remap in Bliss that has no equivalent in Vialuce, add one.** The Vialuce equivalent uses the Vialuce token values (Layer 0/1/2 from OB-221), not Bliss's values. Example:

```css
/* If Bliss has: */
html[data-theme="bliss"] .bg-zinc-800 { background: var(--bliss-surface) }

/* Vialuce needs: */
html[data-theme="vialuce"] .bg-zinc-800 { background: var(--vl-surface, #FFFFFF) }
```

4. **Categories to audit (non-exhaustive — use the actual Bliss block as the authoritative list):**
   - Dark background utilities: `bg-zinc-*`, `bg-slate-*`, `bg-gray-*`, `bg-neutral-*` (800, 900, 950 variants)
   - Light text utilities: `text-white`, `text-zinc-*`, `text-slate-*`, `text-gray-*` (100, 200, 300 variants on dark backgrounds)
   - Dark border utilities: `border-zinc-*`, `border-slate-*` (700, 800 variants)
   - Dark divide utilities: `divide-zinc-*`, `divide-slate-*`
   - Dark ring utilities: `ring-zinc-*`, `ring-slate-*`
   - Modal scrim / overlay backgrounds
   - Navbar / app bar overrides
   - Input / form field dark backgrounds
   - Hover states that assume dark context (`hover:bg-zinc-700`, etc.)
   - Gradient utilities with dark stops
   - Placeholder text colors on dark inputs
   - Focus ring colors on dark backgrounds
   - Scrollbar styling
   - Code block / pre styling

5. **ALSO check:** Are there Bliss remaps that AREN'T scoped to `[data-theme="bliss"]` but instead use `.bliss-*` custom classes, `data-*` attributes, or Tailwind `dark:` modifiers? If so, replicate the pattern for Vialuce.

6. **Token mapping for Vialuce equivalents:**
   - Dark surfaces (`bg-zinc-800/900/950`) → `#FFFFFF` (white card) or `#F4F5FB` (page background)
   - Dark text (`text-white`, `text-zinc-100/200`) → `#1A1A2E` (near-black) or `#606880` (slate body)
   - Dark borders (`border-zinc-700/800`) → `#E8EAF3` (line) or `#F0F1F8` (line-soft)
   - Dark hovers → light hovers (`#FBFBFE` or `#EEF0FB`)
   - Modal scrim → lighter scrim appropriate for light theme

**The test:** After this phase, toggle between Bliss and Vialuce on `/stream`. If there is ANY visible difference in card backgrounds, text colors, borders, or surface treatments between the two themes, there are still missing remaps. Find and fix them.

### §3.1 — Theme Toggle Placement Fix

**Objective:** Move the per-user theme toggle (Dark/Bliss/Vialuce) from its persistent position above the persona in VialuceSidebar to behind the user profile expansion — consistent with how themes are selected in Dark and Bliss.

The current VialuceSidebar footer shows: `.sb-theme` (Dark/Bliss/Vialuce toggle) → `.persona` (Admin/Manager/Rep) → `.sb-user` (avatar + name). The theme toggle should NOT be permanently visible — it's a settings action, not a navigation element.

**Fix:** The theme toggle should appear when the user clicks the chevron-up (`ti-chevron-up`) on the `.sb-user` block, revealing a dropdown or expansion panel that includes the theme options. When collapsed, only the persona and user info are visible.

If the other themes (Dark/Bliss) use a specific expansion pattern in their sidebars, follow that pattern. If they use UserIdentity.tsx's built-in expansion, wire VialuceSidebar to use the same mechanism under the `.sb-user` chevron.
