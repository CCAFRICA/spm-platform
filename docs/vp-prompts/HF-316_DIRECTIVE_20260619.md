## §1 — Problem and Pattern Analysis

Five prior HFs defined scope by tracing from known surfaces inward. Every time, dark components outside the trace surfaced. The methodology is broken — the platform has 315+ unconditional dark utilities across 58+ files, and trace-based approaches always miss branches.

Browser verification of HF-315 shows dark card backgrounds on `/operate`, `/operate/import`, and `/operate/lifecycle` — components that were either missed by HF-315's import-tree trace or partially converted (useIsVialuce exists in the file but not all dark utilities are inside the conditional).

This HF takes a different approach: two prongs that together guarantee elimination.

---

## §2 — Objective (Two Prongs)

### Prong 1 — CSS Safety Net (the floor: every page is readable)

Add comprehensive utility remaps to `globals.css` under `[data-theme="vialuce"]` that override EVERY dark Tailwind utility to its light equivalent. Not just zinc/slate/gray — ALL color families:

- `bg-{zinc,slate,gray,neutral,stone}-{700,800,900,950}` → white or surface token
- `bg-{indigo,blue,purple,violet}-{800,900,950}` → white or surface token
- `bg-black`, `bg-[#hex]` where hex is dark → white or surface token
- `text-white`, `text-{zinc,slate,gray}-{100,200,300}` → near-black or text token
- `border-{zinc,slate,gray}-{600,700,800}` → line token
- `divide-{zinc,slate,gray}-{600,700,800}` → line token
- Gradient stops with dark values
- Any `from-{dark}`, `to-{dark}`, `via-{dark}` gradient utilities

To find what actually needs coverage: grep the ENTIRE `web/` tree for every dark utility pattern, count unique patterns, and write a remap for each. The remaps must use specificity or `!important` that beats the component's own styles.

After this prong: toggle to Vialuce, visit ANY page — no dark backgrounds should survive. If any do, the remap set is incomplete. This prong is the SAFETY NET — it catches everything the component-level conversion misses.

Commit: one commit for the CSS safety net.

### Prong 2 — Component Conversion (the ceiling: pages look polished)

Convert remaining unconverted components to use the design spec vocabulary under `useIsVialuce()`.

**Critical: do NOT filter out files that already have useIsVialuce.** HF-315 added useIsVialuce to 64 components but may have only converted SOME dark utilities in each file. A file with useIsVialuce can still have unconverted dark utilities OUTSIDE the isVialuce branch.

The correct search: find every dark utility usage in `web/components/` and `web/app/` and verify it is INSIDE a `!isVialuce` branch. Lines with dark utilities OUTSIDE any isVialuce conditional need conversion — even in files that already import the hook.

Where design spec classes apply (`.card`, `.kpi`, `.tbl`, `.pill`, `.insight`, `.empty`), use them. Where they don't apply (layout utilities, spacing, minor visual elements), the Prong 1 CSS safety net handles it.

Commit: one or more commits for component conversion.

---

## §3 — Why Two Prongs

Either prong alone has failure modes:

- **CSS-only (Prong 1 alone):** Makes everything readable but pages look like "dark theme painted white" — no design-spec card shells, no DM Mono KPI treatment, no 3px accents.
- **Component-only (Prong 2 alone):** Misses components (proven five times). One missed component = one dark card on one page = another HF.

Together: Prong 1 guarantees the floor (no dark surface survives). Prong 2 raises the ceiling (pages look polished). If Prong 2 misses a component, Prong 1 catches it. The user never sees a dark card.
