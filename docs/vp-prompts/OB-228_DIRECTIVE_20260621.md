## §1 — Problem Statement

The platform produces a rich interpreted rule-set (`rule_sets.components` JSONB: per-component, per-rate, per-condition structure + confidence) and has **no surface that lets a permitted user see, understand, or adjust it.** Plan editing UI is NOT BUILT (gap analysis Category 2, P1). Plan simulation ("what if I change this rate") is NOT BUILT.

The market gap this exploits (DS-029 §1.1): every competitor renders the plan as a static artifact (spreadsheet / form / formula engine) that is a black box to the rep, and every AI move — including CaptivateIQ's May-2026 Agents — bolts a *production accelerator* onto that static representation. The representation itself is never intelligent: it does not explain itself, simulate on real data, or surface its own improvement opportunities.

OB-228 builds the foundation of the answer (DS-029 Slices 1+2): a plan surface that **is** a living, self-explaining, self-simulating organism — the canvas (①), its consequence preview (②), its confidence topology (③), and its provenance thread (④), rendered through a persona-resolution seam (⑧). Defect class addressed: the absence of any plan-expression surface. This is net-new construction, not a fix; behavior-preservation default (DD-7) applies to all *existing* surfaces touched (none should regress).

## §2 — Substrate-Bound Discipline Applications

Each invariant below binds a specific construction decision. Violating it fails the relevant proof gate.

- **Korean Test (D154 LOCKED, IGF-T1-E910).** The canvas renders whatever `rule_sets.components` contains. Component-type rendering dispatches on the `componentType` value present via a **dispatch map with a generic fallback renderer** for any type lacking a bespoke visual — never an enumerated whitelist, never a `switch` that throws/drops on unknown (AUD-009: the catalog is the next bug). No hardcoded field names, no language-specific label dictionary; labels render from the `components` payload (MIR es-PE labels render natively). The `normalizeComponents` function (Phase 2) carries every component and every field through; it tags unknowns, never discards.
- **Carry Everything (T1-E902 v2).** Read the full `components` JSONB and full `row_data` for distribution. Confidence is a hint, not a gate — a low-confidence component is flagged (③), never hidden. Narrowing happens only by *expression context* (persona scope, viewport), never by discarding data at the read layer.
- **Decision 158 (recognize / construct).** Explanations, confidence heat, and provenance are *recognition* (advisory, LLM/anomaly/Allocation-sourced). Rendered values and any committed edit are *deterministic* — read from / written to `rule_sets.components` exactly. Consequence (②) is a **deterministic recompute over committed_data**, not an LLM estimate.
- **Prove Don't Describe (IGF-T1-E905).** Every proof gate verifies LIVE RENDERED state in the browser against MIR's 5 real plans. A component card rendering from a fixture is not proof; it renders from `rule_sets` rows or the gate fails.
- **IAP Gate (§A.9).** Every surface element scores Intelligence / Acceleration / Performance (§3.6). Elements failing all three are cut.
- **Vertical Slice Rule.** Engine and experience evolve together. Concept ② ships the recompute (engine) *and* the consequence tray (experience) in the same phase; the canvas never renders a plan the engine cannot recompute under it (subject to the HALT-4 gate).
- **Three-Scope Flywheel.** Every admin correction (acknowledged confidence flag, committed edit) emits a `classification_signals` row (Tenant scope immediate). One correction, four improvements.

### §3.1 — Phase 1: Diagnostic & Architecture Decision Gate

**No implementation code in this phase.** Output is `docs/diagnostics/OB-228_PHASE1_DIAGNOSTIC.md` + the Architecture Decision record + a commit. Per §B, the Architecture Decision Gate completes before Phase 2.

   ```
   npx tsx -e "<service-role client>.from('tenants').select('id,name,metadata').ilike('name','%MIR%' /* and any other MIR identifier */)"
   ```
2. **FP-49 schema verification.** Query `information_schema.columns` for `profiles`, `profile_scope`, `rule_sets`, `committed_data`, `entity_period_outcomes`, `calculation_results`, `calculation_traces`, `periods`. 4. **Binding resolution.** For each component, read `rule_sets.input_bindings`, resolve the bound column, and confirm that column is present in `committed_data.row_data` for MIR (sample real rows). 5. **Recompute-path discovery (HALT-4 unknown).** Read the engine/calculation code. Prefer reusing the engine's per-entity component-evaluation logic, invoked over the period's committed rows, scoped to the edited rule_set. Record the finding precisely: which function(s) can be reused, what inputs they require, whether a scoped adapter is feasible from CC's authority. **Architecture Decision:** (a) lightweight path exists/assemblable → Phase 4 wires ② fully; (b) only the full batch path exists and a scoped adapter requires an architectural decision beyond this OB → HALT-4 (Phase 4 builds the tray + edit + commit, leaves the recompute adapter as the single surfaced seam, does not block Phases 5/6).
6. **Routing convention.** Inspect the existing app route structure (`/operate/*`, `/configure/*`). Confirm the canvas lands at `/configure/plans` (rail) and `/configure/plans/[ruleSetId]` (canvas). Adjust to the live convention if it differs; record the decision.

### §3.2 — Phase 2: Persona-Resolution Seam + Canvas Data Layer

Build `lib/plan-surface/` (mirror the established `lib/insights/` data-layer pattern from OB-227). Define canonical types first, then the read functions. All functions tenant-scoped and period-aware where noted.

**Canonical types** (`lib/plan-surface/types.ts`):
- `PersonaScope { persona: string; visibleRuleSetIds: string[]; visibleEntityIds: string[]; visiblePeriodIds: string[] }`
- `PlanStructure { id; name; version; status; effectiveFrom; effectiveTo; variants: PlanVariant[] }`
- `PlanVariant { variantId; variantName; components: CanonicalComponent[] }`
- `CanonicalComponent { id; name; componentType: string; config: unknown /* the raw typed config, preserved whole */; binding: { column: string | null; matchReason?: string; tokenOverlap?: number }; confidence?: number; calculationIntent?: unknown; isKnownType: boolean }`
- `ComponentDistribution { componentId; periodId; buckets: { label: string; rangeMin?: number; rangeMax?: number; entityCount: number }[]; totalEntities: number }`
- `BaselineOutcome { entityId; totalPayout: number; componentBreakdown: Record<string, number> }`

**Functions:**
- `resolvePersona(profileId): PersonaScope` — reads `profiles.<personaColumn>` (Phase-1 confirmed) + `profile_scope` (visible_rule_set_ids / visible_entity_ids / visible_period_ids). **This is the refraction seam.**
- `getVisiblePlans(tenantId, scope): PlanStructure[]` — MIR plans filtered by `scope.visibleRuleSetIds` (admin scope = all). Calls `getPlanStructure` per plan.
- `getPlanStructure(ruleSetId): PlanStructure` — reads `rule_sets`; runs `normalizeComponents` on `components`.
- `normalizeComponents(componentsJson): PlanVariant[]` — **the Korean Test core.** Handles the MIR dialect confirmed in Phase 1 *and* the OB-227 alternate dialect via a shape guard (the `breakdownToRecord`-style normalizer). Carries every component and every config field through. Sets `isKnownType` by membership in the bespoke-renderer set, but **never drops** an unknown type — unknowns flow to the generic renderer. No enumerated whitelist gates inclusion.
- `getComponentDistribution(ruleSetId, componentId, periodId): ComponentDistribution` — reads `committed_data.row_data` joined to the component's bound column (via `input_bindings`), buckets entities against the component's tier/matrix structure. **Aggregated server-side** (§A.2 — no per-row payload to the client; produce bucket counts). Returns empty-with-flag if the binding did not resolve (HALT-2 case), never fabricated buckets.
- `getBaselineOutcomes(ruleSetId, periodId): BaselineOutcome[]` — reads `entity_period_outcomes.component_breakdown` (preferred) or `calculation_results.components` as fallback. Baseline for the consequence engine (Phase 4).

Commit. **Phase-2 proof gate:** paste each function signature + the `normalizeComponents` body verbatim (architect inspects for whitelist/drop violations per DD-5-style inspection); paste a real `getPlanStructure(MIR plan)` result and a real `getComponentDistribution` result showing non-empty buckets from MIR committed data.

### §3.4 — Phase 4: The Consequence Engine (Concept ②)

Gated by Phase-1 step 5. Build the edit affordance, the consequence diff, and the commit path. The recompute function is built against the Phase-1 finding; **if HALT-4 fired, build the tray + edit + commit and leave the recompute adapter as the single surfaced seam (clearly labeled "recompute pending architect disposition" — not faked values). Do not block Phase 5/6.**

- Edit affordance in `TierRenderer`/`RateRenderer`/`MatrixRenderer`: drag a tier boundary, edit a rate, edit a matrix cell → produces an in-memory edited component set (does not mutate `rule_sets` until commit).
- `lib/plan-surface/consequence.ts`:
  - `recomputeConsequence(ruleSetId, editedComponents, periodId, scope): BaselineOutcome[]` — runs the edited component set over `committed_data` for the period's entities using the Phase-1 recompute path (deterministic, D158). Reuses engine per-entity evaluation; does not re-implement calculation logic (no parallel math — §A.3-adjacent: one source of calculation truth).
  - `diffConsequence(baseline, recomputed): { aggregateDelta; thresholdCrossers; newZeros; perEntity: { entityId; before; after; delta }[] }`.
- `ConsequenceTray` (Zone C, `components/plan-surface/ConsequenceTray.tsx`): on edit, shows baseline → proposed, aggregate Δ, threshold-crossers count, new-zeros count, per-affected-entity drill list. Commit / Discard.
- Commit path: an API route (server-side Supabase client) writes the edited `components` back to `rule_sets.components` (deterministic write, D158) **and** inserts a `classification_signals` row (scope='tenant') recording the correction (Three-Scope Flywheel). On commit, re-read so the canvas reflects the saved state.

Commit. **Phase-4 proof gate:** if recompute available — browser demo of a tier-boundary edit on a MIR plan showing live aggregate Δ + crosser/zero counts, then a commit that persists to `rule_sets` and emits a signal (paste the new `rule_sets.components` diff + the `classification_signals` row). If HALT-4 — paste the diagnostic confirming the constraint, show the tray + edit + commit built, and surface the recompute seam explicitly in the completion report.

### §3.5 — Phase 5: Confidence Topology + Provenance (Concepts ③ + ④)

Additive overlays on the Phase-3 canvas; depend only on Phases 2–3.

**③ Confidence Topology:**

## §4 — HALT Conditions

- **HALT-1 — `components` dialect unmappable.** MIR's `rule_sets.components` does not match the canonical shape and cannot be normalized. Surface the actual shape verbatim; halt for architect disposition before building renderers. Do not coerce silently.
- **HALT-2 — binding unresolved.** A component's `input_bindings` column is absent from MIR `committed_data.row_data`. The distribution sparkline has no source for that component. Surface the unresolved binding; render the card without a distribution (flagged), do not fabricate. (Scoped halt — does not block other components/phases.)
- **HALT-3 — persona column absent.** `profiles` has no persona/role column to drive the seam. Surface the actual `profiles` + `profile_scope` columns; halt for architect disposition. The seam cannot be stubbed with a hardcoded "admin" constant (violates §3 persona architecture).
- **HALT-4 — recompute path requires architectural decision.** Only the full `calculation_batches` lifecycle exists and a scoped single-rule_set recompute adapter requires a decision beyond this OB's authority. Surface the constraint precisely (which functions, what's missing). Phase 4 builds tray + edit + commit; the recompute adapter is the single surfaced seam. Phases 5/6 proceed. The working surface ships.
