## §1 — Problem Statement

OB-221 delivered:
- Three-layer token architecture (168 scoped CSS rules, zero leakage)
- VialuceSidebar with real navigation, docked persona, gold Calculate CTA
- Full component CSS vocabulary (`.page`/`.kpi`/`.tbl`/`.pill`/`.insight`/`.empty`)

**Architect browser-verification found three blocking defects in OB-221's delivery:**

0a. **Vialuce toggle non-functional.** The UserIdentity desktop toggle displays "Vialuce" as an option but clicking it does nothing — the theme does not switch. The registration (VALID set, AppTheme type, getResolvedTheme) is correct but the click handler or state update is broken. This blocks ALL visual verification.

0b. **Observatory Settings missing Vialuce.** The Observatory settings page (admin-level theme selector) only lists Current and Bliss. Vialuce must appear as a third option. This is a separate settings surface from the UserIdentity toggle — likely a different component or configuration.

0c. **"Current" → "Dark" rename.** The theme name "Current" is an internal development label, not a user-facing name. Rename to "Dark" in all user-facing surfaces (toggle UI, Observatory settings, any labels). The `data-theme` attribute value can remain `"current"` internally — only the display label changes.

**Five additional gaps remain between OB-221's delivery and the design spec's 7-screen prototype:**

1. **Topbar bar absent.** The ChromeSidebar shell has no topbar slot. The design spec's `.top` bar (breadcrumb, search ⌘K, gold Calculate CTA, utility icons) is the primary horizontal navigation element — without it, the Vialuce theme lacks the consistent page header that unifies all screens.

2. **Per-page component adoption incomplete.** The CSS classes (`.page`, `.phead`, `.kpi`, `.tbl`, `.pill`) are shipped and scoped in `globals.css`, but content pages still render their existing React structure without using them. Under Vialuce, pages get brand colors from tokens but not the unified page template, KPI card relief, or table typography.

3. **Login gold CTA.** The login button renders indigo (`--primary`) instead of gold. One scoped CSS override.

4. **Rail width.** VialuceSidebar renders at 264px (matching ChromeSidebar) vs the design spec's 252px. Needs architect-informed correction.

5. **Icon library.** OB-221 substituted lucide-react for the design spec's Tabler icons. Audit needed to confirm the substitutions match intent.

### §3.0 — Blocking Fixes: Toggle + Settings + Rename + Rail Width + Login CTA

**Objective:** Fix the three blocking defects first (theme cannot be selected), then two surgical corrections. These five items are prerequisite to all visual work in Phases 1–3.

**0a — Vialuce toggle non-functional (BLOCKER):**
The UserIdentity desktop toggle displays "Vialuce" but clicking it does not switch the theme. Diagnose the click handler. The registration is correct (VALID set includes `'vialuce'`, AppTheme type includes `'vialuce'`, getResolvedTheme accepts it) — so the issue is in the toggle's onClick/onChange handler or the state update path. Likely candidates:
- The toggle cycles through an array that doesn't include `'vialuce'`
- The API call to `api/user/theme` silently rejects the value
- The state update doesn't trigger a re-render or cookie write
- The toggle renders the label but the clickable area doesn't cover it

Diagnostic-first: trace the click from the toggle button through to the `data-theme` attribute update. Fix structurally — do not patch with a workaround.

**0b — Observatory Settings missing Vialuce:**
The Observatory settings page has a theme selector that lists only Current and Bliss. Find this component (likely under admin/settings, observatory settings, or tenant settings — search for the theme selector that renders radio buttons or a dropdown with "Current" and "Bliss"). Add "Vialuce" as the third option. This component may have its own hardcoded list of theme values separate from the UserIdentity toggle — both must be updated.

**0c — "Current" → "Dark" rename (display label only):**
In every user-facing surface where the theme name appears:
- UserIdentity toggle: "Current" → "Dark"
- Observatory Settings theme selector: "Current" → "Dark"
- Any other UI that displays theme names to the user

The internal `data-theme="current"` attribute value, the `'current'` string in the VALID set, and the `profiles.preferences` stored value all remain `"current"` — do NOT rename the internal value. Only the display labels change. This is a mapping: internal `'current'` → display `'Dark'`, internal `'bliss'` → display `'Bliss'`, internal `'vialuce'` → display `'Vialuce'`.

If the display labels are currently derived from the internal values (e.g., capitalizing the string), add a label map:
```tsx

  current: 'Dark',
  bliss: 'Bliss',
  vialuce: 'Vialuce',
};
```

### §3.1 — Topbar Bar Insertion

**Objective:** Under `data-theme="vialuce"`, insert the `.top` bar between the sidebar and content area. This is the single largest gap between the design spec and OB-221's delivery.

**Architecture:** The ChromeSidebar shell currently renders `sidebar | content` with no topbar. Under Vialuce, the layout becomes `sidebar | (topbar + content)` — the topbar sits atop the content column, not spanning the full viewport width.

**Implementation approach:** In the layout component that wraps authenticated pages (likely where ChromeSidebar is mounted), add a Vialuce-conditional topbar component:

```tsx
{theme === 'vialuce' && <VialuceTopbar />}
```

**VialuceTopbar anatomy (from design spec §4.2):**
- 60px height, sticky at top of content column, `backdrop-filter: blur(8px)`, white/translucent background, bottom border `--vl-line`
- **Breadcrumb** (`.crumb`): Tenant name › Section › Page (active leaf bold). Derive from the current route — use existing breadcrumb logic if available, or derive from `pathname` segments.
- **Calculate button** (`.btn-calc` / `.btn-gold`): Gold background, `#3a2606` text, lightning/activity icon. This is the primary platform action. If a gold Calculate CTA already lives in VialuceSidebar (OB-221 relocated it there), move it to the topbar — it belongs here per the design spec. Do not duplicate it in both locations.
- **Search bar** (`.top-search`): Styled input with search icon + `⌘K` kbd hint. This can be non-functional (styled placeholder) if the platform has no global search — the visual presence matters for the design language. If global search exists, wire it.
- **Utility icons** (`.top-right`): Help (`ti-help` / `HelpCircle`), Alerts with ping indicator (`ti-bell` / `Bell` + `.ping` dot), Language selector (`ti-world` / `Globe` + "EN"/"ES"), Tenant pill (`ti-building-bank` / `Building` + tenant name + chevron). These can link to existing routes or be presentational where the route exists.

**Positioning:** The topbar must be `position: sticky; top: 0; z-index: 5` within the content column. It must NOT overlay the sidebar. It scrolls with the content column, sticking at the top.

**When Dark or Bliss is active:** No topbar renders. Zero regression.

Commit: `"HF-312 Phase 1: VialuceTopbar inserted"`.

**HALT-1:** If the layout component's structure prevents inserting a topbar above the content area without restructuring the entire authenticated shell, halt and report the layout architecture. Describe what would need to change and what the regression risk is.

### §3.2 — Per-Page Component Adoption

**Objective:** Under `data-theme="vialuce"`, wrap each content page's output in the design spec's page template classes so the shipped CSS vocabulary activates.

**The pattern (applied to every page reachable via VialuceSidebar navigation):**

```tsx
// Before (current state — page renders its own structure)
return <div className="existing-classes">...</div>

// After (Vialuce theme conditional wrapping)
const isVialuce = theme === 'vialuce';
return (
  <div className={isVialuce ? 'page' : 'existing-wrapper-class'}>
    {isVialuce && (
      <div className="phead">
        <div>
          <h1>{pageTitle}</h1>
          <div className="sub">{pageSubtitle}</div>
        </div>
        <div className="pactions">
          {/* existing action buttons */}
        </div>
      </div>
    )}
    {/* existing page content */}
  </div>
);
```

**What to apply per component type:**

- **Page wrapper:** Wrap in `.page` (padding, max-width, centered)
- **Page header:** Replace existing header with `.phead` (title left, actions right — same pattern every screen)
- **KPI cards:** Where the page renders metric cards, wrap in `.kpis` grid + `.kpi` card structure (3px accent, DM Mono number, delta chip, hover relief). The KPI card's number must use the `.kpi-val` class (DM Mono 500, 27px).
- **Data tables:** Where the page renders tables, apply `.card.flush` wrapper + `.tbl` class (DM Mono uppercase headers, hover rows, right-aligned numbers)
- **Status indicators:** Replace button-styled status badges with `.pill` classes (`.pill.success`, `.pill.danger`, `.pill.open`, `.pill.neutral`) — informational, no cursor:pointer
- **Empty states:** Where a page renders empty or stub content, apply `.empty` (icon chip + title + action guidance — never a dead end)

**Scope constraint:** Apply to pages reachable via VialuceSidebar navigation. Do NOT attempt to apply to Observatory, admin-only pages, or routes not in the Vialuce nav tree. Walk the VialuceSidebar's `WORKSPACES` config to enumerate the reachable routes — those are the scope.

**Priority order:** Start with the pages that have the most visual components (likely the analytics/dashboard page, then calculate, then configure pages). If time or complexity limits the number of pages, apply to as many as practical and document the remainder as residual.

Commit per logical group: `"HF-312 Phase 2: [page-group] vialuce template adoption"`. Multiple commits are expected.

**HALT-2:** If a page component is structured so that conditional wrapping would require refactoring the component's core layout (not just adding wrapper classes), skip that page, document it in the completion report, and move to the next.
