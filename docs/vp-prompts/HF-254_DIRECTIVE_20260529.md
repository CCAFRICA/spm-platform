## §1 — Problem Statement

### 1.1 — Symptom

On a clean-slate tenant, the **first** import of a transaction structure classifies correctly (cold start: LLM runs). Every **subsequent** same-structure import misclassifies `transaction` → `entity`, commits with null `source_dates`, and yields no transaction data for those periods. Live evidence: BCL Datos Oct = correct; Nov/Dic/Ene/Feb/Mar = all committed `data_type=entity`, all 13 columns `unknown@0.85`, `llmCalled=false`.

### 1.2 — Root cause: two uncoordinated warm-reuse caches; one is unfinished and fabricates

The platform has **two** independent import-time "skip the LLM" caches:

**Cache A — fingerprint flywheel** (`structural_fingerprints`, `fingerprint-flywheel.ts`). Keyed by structural hash. Stores `column_roles` + `classification_result.fieldBindings`, `match_count`, Bayesian `confidence`. Has a real learning loop (increment/EMA, HF-218 decrement, tier escalation, HF-247 refuses to persist `'unknown'`). **This cache is sound.**

**Cache B — vocabulary bindings** (`classification_signals.vocabulary_bindings`). Keyed by tenant. Stores `Record<columnName, semanticMeaning_string>` — **meaning strings only; no `columnRole`, no `confidence`, no structural context.** Its full-fidelity builder `prepareVocabularyBindings` (which would carry `columnRole` + confidence) **has no call site** — the persistence was never completed (comments still read "Phase E wires storage"). **This cache is unfinished scaffolding.**

The two defects that compound:

- **D1 — fabrication gate (the corruption source).** `comprehendHeaders` (`header-comprehension.ts`) Step 1–2 calls `lookupVocabularyBindings` → `recallVocabularyBindings` (returns meaning strings) and wraps each into a `VocabularyBinding` by **fabricating** the fields the cache cannot carry: `columnRole: 'unknown'`, `confidence: 0.85`, `confirmationCount: 2`. Step 2's skip gate then tests `confirmationCount >= 2 && confidence >= 0.85` — values hardcoded to pass. Result: once every column has any recalled meaning string, the LLM is skipped and **all roles resolve to `unknown`** → no measure detected → `entity` (HC pattern `[NO measure, NO currency]`). This violates Principle 1 (fabricates the AI's role decision) and AP-7 (hardcoded placeholder confidence). It is the direct cause of §1.1.

- **D2 — HF-236 compensation routes into the broken path.** Cache A's warm replay is gated in `analyze/route.ts` by `insufficientFlywheelCache`: if any `fieldBindings[].columnRole` is absent/non-native, the sheet is diverted out of the fingerprint-skip path into "fresh-LLM HC." But the fingerprint **write** does not reliably carry native `columnRole` on `fieldBindings` (it depends on the analyze→client→execute-bulk round-trip preserving `classificationTrace.headerComprehension`). When it doesn't, HF-236 fires on every warm import and diverts into `comprehendHeaders` — straight into D1. HF-236 is a compensation pattern (SR-34) papering over an unreliable write, and it routes the catch into the fabrication.

### 1.3 — Why this is a regression, and the timing

Cold start has no prior fingerprint and no prior vocabulary signal → LLM always runs → correct. The warm path broke when **HF-239** turned on flywheel-signal emission for the execute-bulk path (`emitFlywheelSignals`, "the bulk path used to write zero flywheel signals; this restores emission for every import"). That emission populates the caches that arm D1/D2. October never exercises the warm path, so it was not caught while the convergence work (HF-253) was the focus.

### 1.4 — Against the platform vision

Progressive Performance is constitutional: first import costs, subsequent same-structure imports are free **and correct**, with signals enduring across flywheels as the moat. A free-but-wrong warm path is worse than none. Cache A realizes the vision; Cache B, as built, learns nothing usable (strings without roles) and actively corrupts once emission is on. The fix restores a single correct skip authority and completes Cache B as a genuine enduring signal rather than retreating from it.

### 1.5 — The three changes (architect-specified)

1. **Remove the vocabulary-binding LLM-skip gate** (D1). The fabricated-`unknown` path is deleted; LLM-skip authority returns solely to the fingerprint flywheel.
2. **Delete HF-236's divert gate and fix the underlying write** (D2). The fingerprint write always carries native `columnRole`, sourced server-side from HC at write time — not from the client round-trip. With the write reliable, the HF-236 compensation is removed (SR-34: fix structurally, not by workaround).
3. **Complete the lexical flywheel as a role-bearing, additive prior.** `vocabulary_bindings` persists the full interpretation (`columnRole` + `semanticMeaning` + `confidence`) via `prepareVocabularyBindings`; it is consumed as a **classification prior boost** (mirroring `lookupPriorSignals`), **never** as an LLM-skip gate and never narrowing persistence.
