## 1 — INTENT

Two surgical changes to ONE file: `web/src/app/api/calculation/run/route.ts`.

**Change A:** Move HF-208's existing [CalcRecon] summary block from handler-exit to immediately after the entity loop, BEFORE any [CalcTrace] forensic emissions. Architect's first-screen view becomes the totals.

**Change B:** Cap route.ts [CalcTrace] emissions at the first 5 entities the calc loop processes. After entity 5, those lines do not emit.

That is the entire scope. No cross-file changes. No new helpers beyond one boolean check. No buffer. No parameter threading.

---

## 2 — WHY THIS IS THE FIX

- HF-208's [CalcRecon] code is already in route.ts. It just needs to fire earlier in the log stream, not at handler-exit.
- HF-204's always-on inline trace at ~32 lines/entity × 85 entities = ~2,720 trace lines drowning the log. Capping route.ts emissions to first 5 entities cuts that to ~160 trace lines from route.ts.
- intent-executor.ts trace lines (~10/entity × 85 = ~850 lines) are NOT addressed in HF-210. They remain. Total log per period: ~1,000 lines. [CalcRecon] block lands near the top because it emits BEFORE all of them, and route.ts emissions immediately after [CalcRecon] are capped.
- Pre-HF-204 log shape is approximated, not exactly restored. That's acceptable — the architect's stated requirement is reconciliation visibility plus enough forensics for diagnosis. 5 entities of full route.ts trace + all 85 of intent-executor trace = enough.

## 3 — PATCH

### 3.1 — Add cap state at top of handler scope

Right after `const tenantId = ...` is established (or wherever handler-scope variables are declared near the top), add:

```typescript
// HF-210: Cap route.ts [CalcTrace] emissions at first N entities
const TRACE_CAP_N = 5;
const tracedEntityIds = new Set<string>();
function shouldEmitTrace(entityId: string): boolean {
  if (tracedEntityIds.has(entityId)) return true;
  if (tracedEntityIds.size < TRACE_CAP_N) {
    tracedEntityIds.add(entityId);
    return true;
  }
  return false;
}
```

### 3.2 — Wrap each [CalcTrace] addLog in route.ts with the conditional

For every `addLog(\`[CalcTrace] ...\`)` call inside route.ts where an entity ID is in scope, wrap with:

```typescript
if (shouldEmitTrace(entityExternalId)) {
  addLog(`[CalcTrace] ...existing line...`);
}
```

Use whichever local variable already holds the entity external ID at that site (`entityExternalId`, `entity`, `r.entity_id`, `eid`, etc. — match the existing variable name; do not introduce new ones).

CC: run `grep -nE "\[CalcTrace\]" web/src/app/api/calculation/run/route.ts` to find every site. Wrap each one. If a site has no entity ID in scope (shouldn't happen but verify), surface to architect — do not skip.

Find the existing [CalcRecon] block (~route.ts:2231 area per HF-208 completion report). It currently emits at handler-exit, after entityResults loop, before HF-207's period_complete/batch_complete.

Move it: emit it IMMEDIATELY after the entityResults loop completes, BEFORE the existing handler-exit emissions (HF-207 period_complete, batch_complete, COMPLETE).

The block stays exactly as-is — same lines, same fields, same format. Only its position in the file changes.

After move, handler-exit emission order:
1. (entity loop completes, entityResults populated)
2. **[CalcRecon] block** (moved from #4 to here)
3. HF-207 period_complete

The "top" framing means top-of-the-noisy-section, not literal-top-of-log. Setup lines and per-entity totals (which interleave with the entity loop) emit before [CalcRecon]. The [CalcTrace] forensic blizzard ALSO emits during the entity loop — but with the §3.2 cap, only 5 entities' worth. So the architect reads:

Note on intent-executor traces: those interleave with the route.ts trace during the entity loop. They are NOT capped by HF-210. So entities 1-5 have ~32 route.ts + ~10 intent-executor = ~42 trace lines each = ~210 lines. Entities 6-85 have ~10 intent-executor lines each = ~800 lines. Total trace volume: ~1,010 lines. [CalcRecon] block at ~line 1,150 of paste.

**That's still not first-screen.** I see the problem. Let me adjust §3.3.

### 3.3 REVISED — Place [CalcRecon] block immediately after the entity loop, BEFORE flushing or after — irrelevant since trace is already in the log stream.

Actually the issue: trace lines from route.ts AND intent-executor are streamed real-time during the entity loop. They land in the log BEFORE handler-exit code runs. There is no way to put [CalcRecon] "above" them without buffering — which the architect rejected as overcomplicated.

**Therefore the only path that achieves visible-at-top is the cap.** [CalcRecon] stays at handler-exit (where it already is). Trace gets capped enough that handler-exit becomes visible.

Revised approach: **drop the §3.3 move entirely.** [CalcRecon] block stays where HF-208 placed it. The cap in §3.2 is what makes it visible.

With route.ts capped at 5 entities AND keeping intent-executor uncapped:
- ~25 setup + ~12 variant-diag + (5 × ~32 route.ts trace) + (85 × ~10 intent-executor trace) + (85 × 2 variant + total) + ~95 recon = ~1,452 lines.

Vercel typical visible: ~4,000-4,500 lines. **[CalcRecon] block IS visible at line ~1,357 in this scheme** — within first-screen scroll, definitely within the paste buffer.

**That works.** [CalcRecon] doesn't need to be at literal top. It needs to be visible without truncation. ~1,400 lines is comfortably under Vercel's truncation threshold.

### 3.3 FINAL — Do nothing. [CalcRecon] stays at handler-exit. Cap in §3.2 is sufficient.
