## §1 — PROBLEM STATEMENT

The Vialuce data intelligence layer (internal name: Thalamus, formerly PRISM) is a partial implementation of the substrate architecture DS-021 v1.0 specifies. DS-031 names the gap and the architecture that closes it. Four specific deficits exist:

**Deficit 1 — The Normalizer is beside the surface, not on it (OB-249).** The Normalizer runs as a discrete remediation stage with one LLM call, operating as its own pipeline step. It does not post to the Synaptic Surface for joint recognition. It does not consume other agents' signals. It does not participate in the negotiation protocol. It produces its output in isolation, which means it cannot benefit from (or contribute to) the co-present analysis that the SCI negotiation protocol already proves works for classification.

**Deficit 2 — The read-path is half-open (G11 violation).** Signals are written to `classification_signals` and `synaptic_density` but the next ingestion run does not read consolidated signal to set predictions. The predictive loop is open: prediction-downward is not closed because the accumulated model is not consumed to set expectations for incoming data. This means every run treats every input as if the platform has never seen anything like it before — the opposite of the cost-decreases-with-usage law (P3/P4).

**Deficit 3 — Remediation is positioned inside ingestion, not in front of it.** The current PRISM/Thalamus layer operates as a step within the ingestion pipeline rather than as a perceptual front that comprehends input before ingestion begins. Comprehension should happen first, writing atoms and signals to the shared surface that ingestion and calculation then consume. The current topology has comprehension mixed into the pipeline, preventing the calculation engine and convergence layer from reading comprehension output independently of import status.

**Deficit 4 — The density execution modes are not precision-weighted for trust.** The three execution modes (silent ≥ 0.95 / light_trace 0.70–0.95 / full_trace < 0.70) exist in schema (`synaptic_density.execution_mode`) but there is no mechanism to override density toward surfacing when consequence is high and exposure is thin. A confident model can silently predict-away a genuine anomaly in high-stakes data — the worst failure mode for a trust platform. The efficiency path and the trust path must be reconciled by a precision-weighting override.

**Combined-treatment rationale:** These four deficits are not independent bugs — they are four expressions of one architectural gap: the data intelligence layer is not yet operating as the perceptual front of one generative model. Fixing any one in isolation leaves the others structurally unchanged and creates new interfaces that the unified architecture would not have. They are addressed together in four phases, each building on the verified output of the prior.

## §2 — SUBSTRATE-BOUND DISCIPLINE APPLICATIONS

This section names which substrate commitments are exercised by which phase and how. CC does not need to re-derive these; they are the invariants the implementation must satisfy.

**G1 (Carry Everything) — all phases.** No column, field, or structural attribute is narrowed away at any point in the Thalamus layer. Atom composition (Phase 2) operates over complete input. Joint recognition (Phase 3) sees complete atoms. Precision-weighting (Phase 4) evaluates complete signal. If a phase implementation would drop a column to simplify processing, that is a G1 violation and a HALT condition.

**G3 (Calculation-Time Binding) — Phase 2.** Thalamus comprehends input into atoms with confidence/precision. It does NOT decide what period data belongs to, what entity it maps to, or what component it feeds. Those are calculation-time bindings. Thalamus writes comprehension; the engine reads it and binds at calculation time. If the Thalamus layer would pre-bind data to periods or entities, that is a G3 violation.

**G7 (Single Canonical Signal Surface) — Phases 2, 3, 4.** All comprehension output writes to the ONE shared signal surface (`classification_signals` + `synaptic_density` + `structural_fingerprints`). The calculation engine, convergence layer, and Thalamus itself all read from this same surface. If a phase implementation would create a second signal surface (a private channel, a shadow table, a local cache that persists), that is a G7 violation.

**G8 (Korean Test) — all phases.** Every structural identification in the Thalamus layer is based on value distribution, type, cardinality, and pattern — never on header text, language, or domain vocabulary. If a Korean company uploaded Hangul data with completely different column names, every Thalamus operation must produce the same structural output. If an implementation hardcodes a field name, column pattern, or language-specific string, that is a G8 violation and a HALT condition.

**G9 (Deterministic Calculation Boundary) — Phase 2.** The calculation engine's internal commitments are NOT modified by this OB. Thalamus is a producer that the engine consumes. The interface between them is: Thalamus writes comprehension to the shared surface; the engine reads it via existing convergence bindings. If an implementation would modify the calculation engine's evaluation path, that is out of scope and a HALT condition.

**G11 (Read-Path Closure) — Phase 2.** The read-path MUST close. This is not optional, not deferred, not "nice to have." DS-021 says the write-only signal surface is a substrate violation that must close. Phase 2 closes it: the next run's classification MUST read consolidated signal from prior runs to set predictions. If the Phase 2 implementation writes signals but does not read them for prediction-setting in the next run, Phase 2 has not delivered.

**Decision 158 (LLM boundary) — Phases 3, 4.** The LLM expresses at the apex on irreducible surprise — structure no deterministic level can predict. It does NOT drive the recognition surface, does NOT orchestrate agents, does NOT control the pipeline. In Phase 3 (joint recognition), the LLM is invoked ONLY when the negotiation protocol's deterministic rounds cannot resolve a value's classification — genuine novelty at the apex. In Phase 4 (precision-weighting), the override toward surfacing is DETERMINISTIC (a function of consequence and exposure), not LLM-decided. The LLM does not decide what to surface; the precision-weighting function decides, and the LLM reasons about what was surfaced only if deterministic resolution fails.

**Vertical Slice Rule — all phases.** Engine and experience evolve together. Each phase produces a verifiable result that cuts through comprehension → surface → consumption, not a disconnected fragment. If a phase would produce only a library function with no wired consumer, that violates the Vertical Slice Rule.

## §3A — PHASE 2: THALAMUS AS THE PERCEPTUAL FRONT + CLOSE THE READ-PATH

**Objective:** Position the atom/fingerprint layer as comprehension IN FRONT OF ingestion, writing to the shared signal surface; close the G11 read-path so prediction-downward actually consumes consolidated signal to set the next run's predictions.

**This phase is largely ELEVATION of what exists** (the atom layer, the fingerprint computation, the density table) **plus the mandated read-path closure** the substrate already requires. It does not introduce new concepts; it wires existing concepts into the architecture DS-031 names.

**Deliverables (each is a guarantee, not an investigation):**

**3A.1 — Comprehension before commitment.** When a file arrives (via any path: Data Operations upload, standard Import, API), structural comprehension (fingerprint computation, atom extraction, structural signal writing) completes BEFORE data is committed to `committed_data`. The comprehension output (atoms, fingerprint, classification signals) is written to the shared surface (`structural_fingerprints`, `classification_signals`) and is readable by any consumer (the calculation engine, the convergence layer, the Thalamus layer itself on the next run) independently of whether the data has been promoted/committed.

**Implementation invariant:** this is a TOPOLOGY change (comprehension moves in front of commitment), not a new feature. The fingerprint computation and classification logic already exist. The work is re-ordering the pipeline so comprehension is the first act, writing to the shared surface, and commitment follows as a downstream consumer of comprehension output. Use existing functions; re-position them. If existing functions are tightly coupled to the commitment path (e.g., fingerprint is computed inside the commit transaction), decouple them — the comprehension must be independently queryable.

**3A.2 — Read-path closure (G11).** At the START of a classification/comprehension run, the platform reads consolidated signal from the shared surface to set predictions for the incoming data:

- Read `structural_fingerprints` for the tenant (and cross-tenant Tier 2 entries where `tenant_id IS NULL`): if the incoming file's structural fingerprint matches a known fingerprint, the match sets the prediction (the model expects this structure and has a prior classification for it). This is the existing flywheel read path from DS-016/DS-017 — if it exists, verify it; if it doesn't, build it.
- Read `synaptic_density` for the tenant: the density record's `confidence` and `execution_mode` set the prediction strength. High confidence (≥ 0.95, silent mode) means the model predicts this structure and does not invoke full classification — it uses the prior classification directly, at near-zero cost. Lower confidence means the model is less certain and invokes classification with progressively more effort (light_trace, full_trace).
- The RESULT of reading the prior signal is a PREDICTION that the incoming data will have a known structure with known classification. If the prediction matches (the fingerprint matches, the structural signals align), the classification is confirmed at near-zero cost (silent mode). If the prediction FAILS (the fingerprint doesn't match, or structural signals diverge from expectation), the PREDICTION ERROR propagates upward — and that prediction error is what triggers full classification, the negotiation protocol, and (at the apex) LLM reasoning over the unrecognized residue.

**Implementation invariant:** this MUST produce a measurable difference between a first-ever file and a repeat-structure file. A first-ever file has no prediction (no prior fingerprint, no density record); it runs full classification. A repeat-structure file has a prediction (matching fingerprint, density record with accumulated confidence); it should resolve in a fraction of the time and token cost. This is the cost-decreases-with-usage law (P3/P4) made real. If the implementation does not produce this measurable difference, the read-path is not closed and the gate is not met.

**3A.3 — Consolidation (the write-back that closes the loop).** After classification/comprehension completes, the run's results write back to the shared surface:

- Update or create `synaptic_density` for the fingerprint signature: update `confidence`, `total_executions`, `last_anomaly_rate`, `last_correction_count`, `execution_mode` (recalculated from updated confidence), `learned_behaviors`.
- Update or create `structural_fingerprints` with the refined classification result, updated `match_count`, updated `confidence`.
- Write classification signals to `classification_signals` with full provenance.

This write-back is what makes the next run's read (3A.2) informed. Today's surprise becomes tomorrow's silent prediction. If the write-back does not happen, the loop is open and the flywheel does not turn.

**Gate (PG-2):** Demonstrated with a real tenant's data file (use an existing proof tenant — Robles, Casa Diaz, or Sabor):
1. Upload a file structure the platform has NEVER seen for that tenant. Full classification runs. Time and token cost recorded. Signals written to shared surface (query `classification_signals` and `synaptic_density` after the run — paste results).
2. Upload the SAME file structure again (or a file with the same structural fingerprint). The platform READS the prior signal, recognizes the structure, and resolves classification significantly faster (measurably lower time and/or token cost). Paste the timing comparison and the evidence that the prior signal was read (log output, query of density record showing incremented `total_executions`).
3. If the second upload does NOT resolve faster, the read-path is not closed and PG-2 FAILS.

## §3B — PHASE 3: RE-FOUND REMEDIATION AS JOINT RECOGNITION (THE PARADIGM SHIFT)

**Objective:** The Normalizer stops being a discrete pipeline stage and becomes prediction-error resolution ON the Synaptic Surface — one facet of joint recognition where normalize/reconcile/dedup/anomaly resolve together via the negotiation protocol applied to repair.

**This is the hardest phase.** It is the architectural transformation that DS-031 specifies — not an incremental improvement but a re-founding of how remediation works. It replaces a pipeline of sequential cleanup steps with one recognizing surface where all analysis is co-present.

**The paradigm (CC must hold this to implement correctly):** The platform is one generative model that minimizes surprise. After Phase 2, the model has a prediction for incoming data (from the accumulated surface). When data arrives that matches the prediction, it's recognized (silent, no effort). When data arrives that SURPRISES the model — a value the model didn't predict — that surprise is prediction error, and it propagates upward. The question "what kind of surprise is this?" — is it a typo (correction), a different entity (identity), a unit mismatch (reconciliation), or a genuine variant (normalization) — is NOT answerable by one operation in isolation. The answer depends on ALL the structural evidence being visible at once: the value's distribution, the entity's history, the field's known patterns, the tenant's prior corrections, the cross-tenant baseline. Sequencing the operations (normalize first, then reconcile, then dedup) decides each one blind to the others and produces the wrong answer when the correct classification depends on their interaction.

**The existing proof that joint recognition works:** The SCI negotiation protocol (Content Ingestion Specification, Layer 4) already resolves classification this way — each agent posts to the Synaptic Surface, sees every other agent's claim, and re-examines its own in subsequent rounds. The answer emerges from all the analysis being on the surface together. DS-031's commitment is that REPAIR is resolved the same way CLASSIFICATION already is.

**Deliverables (each is a guarantee):**

**3B.1 — Remediation facets on the surface.** The four remediation facets — normalization (canonical form), reconciliation (unit/format alignment), deduplication (identity resolution), anomaly detection (genuine outlier) — post their assessments to the Synaptic Surface for every value that surprises the model (prediction error from Phase 2). Each facet writes its assessment as a signal (using the existing `classification_signals` infrastructure with appropriate `signal_type` values). All four assessments are visible to all four facets simultaneously — same surface, same round.

**Implementation invariant (Decision 158):** Each remediation facet's FIRST pass is DETERMINISTIC — structural heuristics, pattern matching against accumulated signal, value-distribution analysis. The LLM is invoked ONLY at the apex: when the deterministic facets collectively cannot resolve the value (genuine novelty that no accumulated pattern predicts). The LLM call is ONE BOUNDED EXPRESSION over the unrecognized residue — it sees all four facets' assessments (co-present on the surface) and produces a single resolution. That resolution consolidates back to the surface (Phase 2's write-back path) so the same pattern is predicted (free) next time.

**Implementation invariant (iterative-joint):** The apex resolution follows the negotiation protocol's iterative shape (architect-settled Q1): assess → post to surface → read all assessments → re-assess → converge. This is NOT one monolithic LLM prompt that tries to solve everything at once (that empirically fails — the platform has hit that wall repeatedly). It is iterative rounds where each round's output narrows the unrecognized residue until convergence or escalation to human judgment.

**3B.2 — The Normalizer re-founded, not deleted.** The existing Normalizer code (OB-249) is not thrown away — it is re-positioned. Its classification logic becomes one facet's deterministic heuristic layer. Its LLM call becomes the apex expression, invoked only on the residue the deterministic layer cannot resolve. Its output writes to the shared surface instead of a private result. The re-founding preserves the Normalizer's existing capability while embedding it in the joint recognition architecture.

**3B.3 — No new production surfaces.** This phase does NOT create a new UI. The remediation output is visible through EXISTING surfaces: the Data Operations workspace (where remediation signals appear as data-quality indicators on the import record), the processing cockpit (OB-251, where remediation status appears in the job progress), and the signal surface itself (queryable by the calculation engine and convergence layer). The reconciliation/forensics surfaces that consume remediation output already exist and read from `classification_signals` — they see the new facet signals without new wiring.

**Gate (PG-3):** Demonstrated with a real tenant's data containing at least one of each:
1. A value that is a CORRECTION (typo, formatting error) — the joint recognition classifies it as correction and applies the canonical form. Paste the value, the four facets' assessments from the surface, and the resolution.
2. A value that is an IDENTITY question (two values that might be the same entity) — the joint recognition classifies it as identity/dedup and resolves it. Paste evidence.
3. A value that is a genuine ANOMALY (a real outlier that should surface, not be corrected) — the joint recognition classifies it as anomaly and does NOT silently correct it. Paste evidence.
4. A value whose correct classification DEPENDS on co-present information (e.g., it looks like a typo to the correction facet alone, but the identity facet's evidence shows it's actually a different entity) — the joint recognition gets it right because all facets are co-present. Paste the value, show what one facet alone would conclude vs. what the joint recognition concludes. This is the load-bearing proof that joint recognition produces a different (correct) answer than sequential processing.

## §3C — PHASE 4: PRECISION-WEIGHTING (THE TRUST SAFEGUARD)

**Objective:** Build the precision-weighting override that prevents a confident model from silently predicting-away genuine anomalies in high-consequence data. This is the safeguard that makes the architecture a TRUST paradigm, not merely an efficient one.

**The hazard this phase prevents:** A model that predicts well is a model that hallucinates — it sees what it expects and absorbs real surprise as "normal." Maximum prediction (silent mode, ≥ 0.95 confidence) is maximum hallucination risk. For a trust platform where customers stake payroll on the output, the worst failure is a real data error that the model confidently predicts-away and the operator never sees.

**The safeguard:** Precision-weighting overrides density (confidence-based execution mode) toward SURFACING when the CONSEQUENCE of a wrong prediction is high and the model's EXPOSURE (how much of this exact structure it has actually seen) is thin. High confidence on thin exposure is exactly where a model hallucinates; precision-weighting catches it.

**Deliverables (each is a guarantee):**

**3C.1 — Consequence signal.** A deterministic function that assesses the consequence of a wrong prediction for a given value/signal. Consequence is derived from STRUCTURAL properties the platform already tracks — not from a new registry or hardcoded list. Candidate structural properties (CC evaluates and selects, per ULTRACODE):
- Does this value feed a calculation component? (Higher consequence — wrong input → wrong payout.)
- Is this value in a field that has historically produced corrections? (Higher consequence — known error-prone field.)
- Does this value's classification affect downstream entity resolution? (Higher consequence — identity error cascades.)
- Is this value part of a high-density tenant's data where silent errors accumulate? (Higher consequence at scale.)

**Implementation invariant (no registry):** The consequence function validates STRUCTURAL PROPERTIES (range, shape, downstream dependency), never set membership. It does not contain a list of "high-consequence fields" or "important columns" — that would be a registry violation. It assesses consequence from the value's structural position in the data graph.

**3C.2 — Exposure signal.** A deterministic function that assesses how much the model has actually seen of this exact structural pattern. Exposure is NOT the same as confidence — a model can be confident (high `synaptic_density.confidence`) because it quickly converged on a pattern it's seen a few times, but that few-times exposure means its confidence is brittle. Exposure is derived from:
- `synaptic_density.total_executions` — how many times this signature has been processed.
- `structural_fingerprints.match_count` — how many files have matched this fingerprint.
- Recency of the most recent match (a pattern last seen six months ago may have drifted).

**3C.3 — The precision-weighting override.** When consequence is HIGH and exposure is THIN, the execution mode overrides toward surfacing regardless of density confidence. Concretely:
- A value that would otherwise be `silent` (confidence ≥ 0.95) is overridden to `light_trace` or `full_trace` when precision-weighting triggers.
- The overridden value's assessment surfaces in the Data Operations workspace as a trust-flagged item — visible to the operator, requiring acknowledgment before silent treatment resumes for that pattern.
- The operator's acknowledgment (confirm or correct) feeds back into the surface: if confirmed, exposure increases and future precision-weighting is less likely to trigger for this pattern; if corrected, the correction enters the flywheel as a high-weight learning signal.

**Implementation invariant (this is a learning surface, not a formula).** The precision-weighting calibration (the exact function that maps consequence × exposure → override decision) is itself a surface that learns from operator feedback. Initial calibration is structural and conservative (override when `total_executions < N` AND consequence indicators are present). The surface refines over time as operator confirmations/corrections accumulate. Do NOT hardcode a fixed threshold — build the feedback loop so the threshold converges.

**Implementation invariant (Decision 158).** The precision-weighting override is DETERMINISTIC — a function of consequence and exposure signals, not an LLM decision. The LLM does not decide what to surface; the function decides. The LLM may reason about what was surfaced (at the apex, if the deterministic resolution fails), but the surfacing decision itself is below the Decision 158 boundary.

**Gate (PG-4) — the falsifiable proof:** Demonstrate with real tenant data:
1. A value that the model "expects" (high density confidence, would be `silent`) but SHOULD be surfaced because consequence is high and exposure is thin. Precision-weighting overrides to surfacing. The operator sees the flag. Paste the density record (showing high confidence), the precision-weighting calculation (showing override), and the surfaced flag in the Data Operations workspace.
2. A value that the model expects AND has high exposure on — precision-weighting does NOT override (the model has genuinely learned this pattern). Paste evidence that the override did not fire and why (high exposure).
3. The operator confirms the surfaced value from (1). The confirmation feeds back: exposure increases, density updates. On the NEXT occurrence of the same pattern, precision-weighting is less aggressive. Paste the before/after density records.

## §4 — HALT CONDITIONS

CC halts and reports with evidence when any of these conditions is met. Halts are not failures — they are correct behavior when a gate condition cannot be met.

- **HALT-NARROW:** CC discovers that a phase deliverable cannot be met without narrowing scope (e.g., "Phase 3 can only work for one facet, not four"). CC halts and reports what would need to change to deliver the full scope. CC does NOT silently deliver a narrowed scope.
- **HALT-G1:** An implementation would require narrowing away a column/field at carry time. CC halts and reports the specific narrowing and why it seems necessary.
- **HALT-G7:** An implementation would require a second signal surface (private channel, shadow table, local persistent cache). CC halts and reports why a single surface seems insufficient.
- **HALT-G8:** An implementation would require a language-specific string literal or domain-specific identifier. CC halts and reports the structural alternative.
- **HALT-G9:** An implementation would modify the calculation engine's evaluation path. CC halts — this is out of scope.
- **HALT-158:** The LLM would need to drive or orchestrate rather than express at the apex. CC halts and reports the architectural pressure requiring this.
- **HALT-SCHEMA:** A required table/column does not exist in live schema and the migration would be a schema change requiring architect-administered SQL Editor execution. CC halts, drafts the migration SQL, and reports it for architect execution. CC does NOT fabricate schema (FP-49).
- **HALT-SURFACE:** A phase would require a NEW production UI surface not listed in §3.7's existing-surfaces inventory. CC halts and reports what the surface would need to show and which existing surface could host it.
- **HALT-DEPENDENCY:** A phase's deliverable depends on a prior phase's gate not yet met. CC halts and reports the unmet dependency.
- **HALT-DIVERGE:** CC identifies two or more implementation paths that are materially different in architecture (not just implementation detail). CC halts and reports the options with tradeoffs for architect disposition. CC does NOT pick one silently.
