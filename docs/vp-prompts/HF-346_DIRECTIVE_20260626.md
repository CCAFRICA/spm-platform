## §1 — Problem Statement (CLT-246 root cause chain + design assessment)

The Rep `/perform` surface was designed as an admin dashboard with persona-switched data scope. It shows admin-designed panels (Entities Paid, Average Payout, Top Result, Data Quality, Period Lifecycle) that fail the DS-013 test battery for an individual contributor. The component breakdown shows $0 while the total is $1,505 (data accuracy P0). Intelligence is split across two duplicate sections with contradictory metrics. Links are broken. Locale is mixed. No drill-through from summary to transactions exists.

**Root cause chain (CLT-246):** F12 (component $0) + F13 (0% attainment) + F14 (RepDashboard $0) → F16 (contradictory intelligence) + F17 (inaccurate coaching) + F18 (wrong focus) → F22 (broken what-if). Fix the data, fix the cascade.

**The spatial assessment:** The Rep surface must be rebuilt around the DS-015 §5.3 Earnings Hero pattern — not patched panel-by-panel. Five to six elements maximum. Hero dominant. Everything else on interaction. Thermostat (what to DO), not thermometer (what IS).

## §3 — Phase Prose

### §3.0 — Phase 0: Data Accuracy Root Cause (CLT-246 F12/F13/F14)

**Objective:** Determine why component breakdown shows $0 and attainment shows 0% when total payout is $1,505. Fix the data path.

**Constraints:**
- Query `entity_period_outcomes` for the sample entity (entity `e939a648...` or whichever entity the VL admin Rep preview resolves to) for BCL March 2026. Paste the `rule_set_outcomes` JSONB field — this should contain per-component breakdown.
- If `rule_set_outcomes` contains component data: trace why `getRepDashboardData` returns $0 for each component. The function may be reading from a different source or parsing the JSONB incorrectly.
- If `rule_set_outcomes` does NOT contain component data: this is a materialization gap — the calculation engine produces component results but the materializer doesn't persist them at the entity level. Report as a data-state finding.
- Attainment requires target data. Trace where attainment is computed for the RepDashboard — if it reads from `rule_sets` tier configuration and the entity isn't assigned to a rule set, attainment = 0 is a data-state gap, not a code bug. Report which is the case.
- The percentage bar (37/32/22/10) on the page-level "COMPENSATION BY COMPONENT" StackedBar IS showing component proportions — meaning SOME data path has component data. Identify which data function produces the percentages vs. which produces the dollar amounts. The mismatch is the bug.

### §3.1 — Phase 1: Rep Surface Redesign (panel relevance + DS-013 test battery)

**Objective:** Rebuild the Rep `/perform` view around the DS-015 §5.3 Earnings Hero pattern. Maximum 5–6 elements. Every element passes the DS-013 test battery.

**Constraints:**

**1a. Earnings Hero (dominant)**
- Total payout ($1,505) as the hero metric — large, prominent, unambiguous.
- Context: period name, entity count (always 1 for rep — don't show), component count.
- Comparison: vs prior period ("+9.5% vs February 2026") — from entity trajectory data.
- Reference frame: attainment % with tier indicator (Standard / Premium / Elite). Requires target from rule_sets. If target unavailable, show payout only with "target not assigned" note — do NOT show 0%.
- Action: "View Components ↓" (scrolls to component breakdown), "Simulate What-If →" (scrolls to or opens what-if).
- Impact: distance to next tier in dollars ("$935 to reach Premium") + which component to focus on.
- **Replaces:** PERIOD TOTAL tile + PERIOD FINDING card + current hero metric (which duplicates data). ONE hero, not three renditions of the same number.

**1b. Remove admin-metric panels for member**
- ENTITIES PAID, AVERAGE PAYOUT, TOP RESULT tiles: do not render for member. (Admin/manager surface unchanged.)
- DATA QUALITY panel: do not render for member.
- PERIOD LIFECYCLE pipeline: do not render for member. Replace with a simple payout status badge on the Earnings Hero: "In Review" / "Approved" / "Paid" — one word, not a pipeline diagram. Sourced from `entity_period_outcomes.lowest_lifecycle_state` or equivalent.

**1c. Calculate button in persistent header**
- The "✦ Calculate" button in the persistent top bar must not render when `effectiveRole` is member/rep (or any role without the `calculate` capability). Gate by `hasCapability('batch.calculate')`.
- For admin/platform: unchanged.

### §3.2 — Phase 2: Component Breakdown with Click-Through Drill-Down

**Objective:** The component breakdown shows dollar values AND is clickable to drill into the constituent transactions for each component.

**Constraints:**

**2a. Component breakdown: dollars + percentages + attainment**
- Each component row shows: component name, dollar earned, percentage of total, attainment bar (% of target for that component).
- Dollar values are the PRIMARY display. Percentages are secondary.
- The data source must be the MSP path — `entity_period_outcomes.rule_set_outcomes` or equivalent materialized component data. Not a raw `calculation_results` query.

**2b. Click-through to transactions**
- Clicking a component row expands inline (Action Proximity — DS-013 §4.3: do not navigate away) to show the transactions that produced that component's payout.
- Transaction data source: `committed_data` filtered by entity + component/rule_set + period. These are the raw input transactions that the calculation engine consumed.
- Each transaction row: date, description/reference, amount, contribution to payout.
- If the transaction list is long (>20 rows), paginate or show "first 20 of N — View All →" with a link to the full `/data/transactions` page pre-filtered for this entity + component + period.
- The inline expansion collapses when clicked again (accordion pattern).
- MSP note: transaction drill-through reads from `committed_data` (source of truth for inputs), not from `calculation_results`. This is a read of the input data, not the output — MSP does not apply to input data reads.

**2c. Summary → Component → Transaction chain**
- The user flow is: Earnings Hero ($1,505) → Component Breakdown (click "Captación de Depósitos" $557) → Inline Transaction List (12 transactions totaling $557).
- Every number at each level must reconcile: component dollars sum to total, transaction amounts sum to component dollar.

### §3.3 — Phase 3: Intelligence Consolidation

**Objective:** ONE intelligence section on the Rep `/perform` surface. Accurate, actionable, concise. Passes DS-013 test battery.

**Constraints:**

**3a. Consolidate AGENT INTELLIGENCE and the lower "Intelligence · AI" section into ONE panel.**
- Remove the duplicate. ONE intelligence presence on the page.
- Title: "Intelligence" or "Performance Coaching" — not "Agent Intelligence" (anthropomorphized voice — CLT51A assessment, DS-013 research pending).

**3b. Intelligence content: Three elements maximum (DS-015 Five Elements pattern)**
1. **Focus** — the single most impactful action the rep can take right now. Based on component attainment gap analysis: which component has the largest gap between current and target, weighted by dollar impact. Actionable: "Focus on Colocación de Crédito — you're $481 short of target. [View Transactions →]"
2. **Trend** — trajectory summary. "+9.5% vs prior period" or "↘ declining 3 consecutive periods." Reference frame, not just a number.
3. **Action** — ONE specific, concrete next step. "Log 1 Colocación transaction today" or "Review your Captación pipeline for pending deals." With a functional link to the relevant action surface.

**3c. Intelligence accuracy**
- Intelligence must NOT present contradictory metrics (CLT-246 F16: "Ranked #1 of 85" with "0% attainment"). If two data sources disagree, the intelligence layer must flag the inconsistency rather than presenting both as simultaneous facts.
- Intelligence derives from the SAME data that the Earnings Hero and Component Breakdown display. Not a separate data path. If the hero shows $1,505, intelligence references $1,505 — never a different number.
- The "Ranked #1 of 85" language: for a member, ranking is shown as anonymized relative position (OB-246 Phase 2 — quartile/percentile). The intelligence narrative must use the same framing: "Top quartile" or "100th percentile" — not "#1 of 85" which exposes the population count.

### §3.4 — Phase 4: Trajectory + What-If + Links + Locale

**Objective:** Fix functional gaps, broken links, and localization.

**Constraints:**

**4a. Trajectory**
- The TRAYECTORIA panel shows period-over-period payout history. Keep it — it passes IAP Gate (Performance) and Five Elements (value + context + comparison).
- Data source: `entity_period_outcomes` across periods for the scoped entity (MSP).
- "View Trajectory →" link must navigate to `/perform/trends` or expand inline. Currently broken (CLT-246 F21) — fix or remove. If the target page doesn't render meaningful data for a member, remove the link rather than creating a dead end.

**4b. What-If simulation**
- The "QUE PASARIA SI..." section shows $0 projected payout (CLT-246 F22). This is downstream of the attainment/component $0 data issue (Phase 0). Once component data loads correctly, verify the what-if simulation computes correctly.
- If the what-if simulation depends on target data that isn't available for the sample entity, show "Simulation requires target assignment" — not $0.

**4c. Link verification**
- Every clickable element on the Rep `/perform` surface must navigate to a functional destination. No dead links, no 404s, no empty pages.
- Specifically verify: "View in Compensation →", "View Trajectory →", "Go to Compensation →", "Review component changes", "Focus on improving your lowest component", every Intelligence action link.
- If a link target doesn't render meaningful content for a member, either: (a) make it render meaningful content, (b) remove the link, or (c) change the link to an inline expansion. Do NOT leave dead links.

**4d. Locale consistency**
- All UI labels on the Rep `/perform` surface must respect the user's locale setting (`profileLocale` from `useAuth()`). If the locale is `es`, every label is Spanish. If `en`, every label is English. No mixing.
- Specifically the CLT-246 finding: "RITMO ACTUAL" (Spanish) alongside "PERIOD TOTAL" (English) on the same page.
- The data values (component names like "Captación de Depósitos") are tenant data, not UI labels — they display as-is regardless of locale. But UI chrome (section headers, button labels, status text) must match the locale.
- This is a localization pass on the Rep surface only. Platform-wide locale enforcement is a separate item.
