## §1 — Problem Statement

A user viewing `/perform` ("My Compensation") — an individual contributor (member/rep) surface — is seeing **tenant-wide aggregate data**, not own-scope data. The following org-wide elements leak into the rep surface:

1. **Component distribution charts** spanning all entities across the tenant.
2. **"Ranked #1 of 95"** — full org positioning with peer count exposed.
3. **Staff-level, location-level, and patterns panels** — org-wide rollups from OB-237 materialized serving paths (`summary_artifacts`, `summary_artifacts_fine`).
4. **Tenant-wide component breakdown table** — every component's aggregate across 85+ entities.

Simultaneously, menu/function access does not enforce role boundaries: navigation items may be visible that the role cannot use, producing dead-end "Access Restricted" pages.

**Defect class:** This is not a styling or layout defect. It is a **data-layer visibility breach** combined with a **function-access enforcement gap**. The root cause is one of three possibilities — §3 Phase 0 determines which, in code, before any fix is applied.

**PDR-05 lineage:** PDR-05 (Feb 2026) identified that persona filtering was keyed to a cosmetic `effectivePersona` switcher rather than the authenticated `profiles.role`. OB-168 (Mar 2026) migrated pages to `RequireCapability` backed by DB-read capabilities. Whether the persona-switcher anti-pattern persists in the live `/perform` read path, or the leak is purely a query-scope failure, is UNKNOWN and must be determined empirically.

### Best-Practice Rep (Member) Surface Scoping

For a member on `/perform`:

- **RETAIN:** Own total payout, own component breakdown, own attainment vs target, own period history/statements, own dispute action, own Intelligence Stream, own lifecycle position.
- **STRIP all raw org breakdowns:** Tenant-wide component distribution, org-wide staff/location/patterns rollup panels, cross-entity aggregate tables. These must not render for a member.

## §3 — Phase Prose

### §3.0 — Phase 0: Diagnose (ULTRACODE — autonomous investigation)

**Objective:** Determine the root cause of the data-visibility leak and menu-access gap. State the diagnosis as **(A)**, **(B)**, or **(C)** with pasted code evidence.

**Three possible diagnoses:**

- **(A) Cosmetic-switcher-only:** A persona dropdown overrides rendered scope, but a real authenticated rep login would be correctly scoped. The switcher is the sole failure surface.
- **(B) Authenticated-role leak:** Even a true member login sees tenant-wide data because `/perform` read paths filter by `tenant_id` alone and never narrow by the authenticated user's `profile_scope` / linked entity.
- **(C) Both:** The switcher is cosmetic AND the underlying read paths lack scope narrowing.

**Constraints:**

- Every finding requires pasted code, grep output, or terminal evidence. Self-attestation ("I verified this") is rejected.
- Do NOT assume the answer. The code determines the diagnosis, not prior knowledge.
- Grep the live repo for `effectivePersona|usePersona|PersonaContext` — report every live usage or confirm absence.
- Trace the actual `/perform` page data loader and paste every read query. Identify every filter applied (tenant_id? entity scope? period? role?).
- Identify where role/persona resolves for rendering decisions — is it `authProfile.role` (DB-backed) or a client-side override?
- Confirm whether `profile_scope` is READ anywhere in the `/perform` path. It has a writer; confirm presence or absence of a reader.
- Trace how a logged-in profile resolves to its own entity (`entities.profile_id` join path). Paste the resolution.
- Trace sidebar/menu construction. Is it derived from `hasCapability` per DS-014 §5.3, or hardcoded per role/persona? Paste it.

**Output:** The diagnosis (A, B, or C) with the specific evidence that selects it, and a complete map of what must change in each subsequent phase. This map IS the Architecture Decision Record — commit it before proceeding to Phase 1.

### §3.1 — Phase 1: Authenticated Scope Resolution (single path)

**Objective:** Establish ONE authenticated-scope resolver consumed by every `/perform` read path.

**Constraints:**

- Input: authenticated profile (role + id from session/auth context).
- Output: the visible entity set.
  - For `member`: own linked entity only (via `entities.profile_id` join).
  - For `manager`: `profile_scope.visible_entity_ids` (team set).
  - For `admin` / `platform`: all tenant entities (existing behavior).
  - For `viewer`: per-assignment scope.
- Every `/perform` read narrows by this visible entity set. No `/perform` read for a non-admin role filters by `tenant_id` alone without scope narrowing.
- If a persona switcher exists, it may only select among views the authenticated user is actually entitled to (or be removed). It is NEVER the security boundary — the scope resolver keys off `profiles.role`.
- The resolver is structural (Korean Test): it resolves entity visibility from role + profile_scope/entity linkage, with zero domain-specific or language-specific logic.
- OB-237 materialized serving paths (`summary_artifacts`, `summary_artifacts_fine`) must also be filtered through this scope resolver — they are the source of the staff/location/patterns panels that currently leak.

### §3.2 — Phase 2: Member Surface Correctness

**Objective:** A member on `/perform` sees ONLY data attributed to their access, with anonymized relative positioning where applicable.

**Constraints:**

- Org-wide panels (component distribution across all entities, staff rollup, location rollup, patterns rollup, cross-entity aggregate tables) do not render for member.
- Member panels that depend on the scope resolver (§3.1) read through it.
- Relative position: if a "rank" or "percentile" indicator is rendered, it shows the member's position in anonymized terms (quartile, above/below average) — no peer names, no peer payouts, no raw org rollup data. If clean anonymization is not achievable for a given panel within this HF, hide the panel for member rather than leak.
- Manager panels: same scoping logic applied to team entity set. Manager sees team-scoped component distribution, team rankings (within team), team rollups. NOT tenant-wide.
- Intelligence Stream content (`AGENT INTELLIGENCE` section) scopes to the role's visible entity set.

### §3.3 — Phase 3: Menu and Function Gating

**Objective:** Sidebar items, workspace access, and in-page function buttons are derived from capabilities, not hardcoded per persona. Zero dead ends.

**Constraints:**

- Sidebar construction uses `hasCapability` (DS-014 §5.3). If a navigation item requires a capability the user's role does not hold, it does not render.
- Per DS-014 §4 capability matrix:
  - Member: Perform workspace (own compensation, own statements, dispute submission). No Operate, no Configure, no Admin.
  - Manager: Perform (team view), and any configured approval surfaces. No Import, no Calculate, no Configure.
  - Admin / Platform: full workspace access.
- Route guards enforce the same capability check. A URL manually entered for a workspace the role cannot access redirects to the role's default landing (member → `/perform`, manager → `/perform`, admin → `/operate`).
- No navigation item reachable from the sidebar produces an "Access Restricted" or empty state for the role that can see it.
- DS-014 §9: no per-route role arrays added one at a time (the whack-a-mole anti-pattern). One capability check function, one source of truth.

## §6 — Out of Scope

- **RLS policy authorship.** Database-layer row-level security policy creation or modification. If needed, surfaces via HALT-A for separate OB treatment.
- **User provisioning / invite flow.** DS-014 Phase 3 (user management UX). Not in this HF.
- **Entity-to-user promotion.** Linking entities to profiles via the invite surface. Existing `entities.profile_id` values are consumed; new links are not created.
- **Financial Agent surfaces.** Sabor/Financial Module persona scoping per `FM_Views_Data_Persona_Analysis.docx`. This HF governs the ICM `/perform` surface only.
- **Dispute submission UI.** The dispute capability (`submit_dispute` for member per DS-014 §4.5) renders a button/link if the member role holds the capability. The dispute flow itself is separate scope.
- **Profile_scope materialization.** The `profile_scope` table writer already exists. Whether `visible_entity_ids` are correctly populated for test entities is a data-state question — if empty, HALT-C fires. Materializer changes are not in scope.
- **Cosmetic design / DS-003 visual treatment.** Panel layout, colors, typography. This HF enforces what data appears and what navigation is accessible, not how it looks.
- **OB-234 (Performance Agent visualization redesign).** Separate scope. HF-343 may hide panels that OB-234 would redesign — that's correct (hide now, redesign later).

## §6A — Residuals and ARTIFACT SYNC

### Residuals

1. **Profile_scope materialization coverage.** If HALT-C fires (entity→profile linkage unpopulated), a follow-on seeding task is required before browser verification can prove member scoping. This is a data provisioning gap, not a code gap.
2. **RLS enforcement depth.** If HALT-A fires, the app-layer scope filtering applied by this HF is necessary but not sufficient for SOC2-grade enforcement. A follow-on OB with IRA advisory would establish DB-layer RLS policies that enforce scope regardless of application-layer behavior.
3. **Viewer role scope configurability.** DS-014 defines viewer scope as "configurable per assignment." This HF treats viewer identically to member (own results only). Configurable scope is a future capability.
4. **PDR-05 closure evidence.** If diagnosis is (A) or (C), the cosmetic-switcher anti-pattern identified by PDR-05 is structurally resolved by this HF. If diagnosis is (B), PDR-05 may already be closed by OB-168's RequireCapability migration — the completion report must state which and provide evidence.
