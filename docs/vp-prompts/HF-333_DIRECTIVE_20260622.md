## §1 — PROBLEM STATEMENT

OB-231 removed the fixed `ColumnRole` vocabulary from the HC producer. The LLM now emits free-form characterizations (`"computed measure"`, `"categorical (boolean-like status flag)"`, `"measure (possibly a snapshot balance or reference amount)"`). Three consumer sites still validate incoming characterizations against the old vocabulary's type expectations. The LLM proposes correct bindings; the gates reject them because `"computed measure" ≠ "numeric"`. All five MIR plans abort at HF-281 (incomplete bindings).

### §1.1 — Regression 1: Convergence §2-S5 type-consistency gate

The `OB-216 §2-S5` validation step checks whether a proposed column's `data_nature` satisfies the plan field's type requirement (`needs numeric`, `needs categorical`). Pre-OB-231, `"measure"` matched `"numeric"`. Post-OB-231, the LLM writes `"computed measure"` and the gate rejects it as role-inconsistent:

```
Monto_Total (computed measure): role-inconsistent, needs numeric → gap
Saldo_Pendiente (measure (possibly a snapshot balance or reference amount)): role-inconsistent, needs numeric → gap
Verificado (categorical (boolean-like status flag)): role-inconsistent, needs categorical → gap
```

This cascades: every component's binding is incomplete, HF-281 fires, calculation is aborted. All five plans, all five aborted.

### §1.2 — Regression 2: Convergence attribute admission gate

`OB-216 §2-S3` labels candidate columns and admits attributes. Pre-OB-231, columns with `role === 'attribute'` were admitted. Post-OB-231:

```
OB-216 §2-S3 labeled candidates: 71 columns across 7 sheet(s); attributes admitted=[none]
```

Zero attributes admitted. The characterizations say `"categorical"`, `"categorical identifier"`, `"categorical (boolean-like status flag)"` — none match the old `attribute` label.

### §1.3 — Regression 3: entity_id_field selection still picks Folio

`commitContentUnit` selects `entity_id_field="Folio"` on all Ventas sheets despite OB-231's `findHcEntityIdColumn` function. The structural sanity check correctly warns but proceeds:

```
[OB-231][entity-id-sanity] resolved entity_id_field="Folio" is ~1:1 across rows (4680/4689 distinct)
```

The LLM's `identifies` field is not distinguishing Folio (transaction) from DNI_Vendedor (entity) with sufficient clarity for the selection logic to pick DNI_Vendedor. Both columns show `data_nature: "identifier"` in the HC diagnostic output.

### §1.4 — Defect class

All three are the same violation: **consumer-side vocabulary gates checking against a producer-side vocabulary that OB-231 removed.** The gates are the remaining half of the old registry. OB-231 removed the registry entries; this HF removes the gates that checked them.

Decision 158: the LLM recognized that `Monto_Total` is a numeric measure, that `Categoria` is a categorical attribute, that `DNI_Vendedor` identifies entities. The construction layer must read and act on that recognition, not re-validate it against ghost labels.

### §1.5 — Fix class: SUBTRACTION

Remove the type-consistency gate from convergence §2-S5. Remove the attribute admission gate from §2-S3. Fix the entity_id_field selection to read the LLM's `identifies` scope correctly. No new gates, no new validation, no new vocabulary. The LLM proposed it; the construction layer accepts it.

## §4 — PHASE 1: REMOVE THE GATES + FIX ENTITY SELECTION

### §4.1 — Remove the convergence type-consistency gate (Regression 1)

The function that checks `"needs numeric"` against the column's characterization and produces `"role-inconsistent → gap"` is the gate. Remove the type-consistency check entirely. When the LLM proposes a binding (e.g., `Monto_Total` for a field that needs a numeric input), the convergence layer accepts it. The column's characterization already says `"computed measure"` — the LLM knows it's numeric. The gate was the construction layer re-recognizing what the LLM already recognized.

If the binding is genuinely wrong (the LLM bound a categorical column to a numeric field), the engine will produce wrong numbers and reconciliation catches it. That is the platform's verification model. A fixed gate that rejects correct bindings is worse than no gate at all.

### §4.2 — Remove the attribute admission gate (Regression 2)

The function that filters columns for attribute admission should admit columns whose characterization describes categorical/descriptive data, not just columns whose label was `"attribute"`. The simplest subtraction: remove the label-matching filter entirely and admit all non-identifier, non-temporal, non-measure columns as attribute candidates. Or: read the `data_nature` field and admit anything the LLM characterized as categorical, descriptive, attribute, or similar. CC determines the minimal change from the actual code — the goal is that `attributes admitted=` shows the real categorical columns (Zona, Cargo, Ciudad, Categoria, Motivo_Devolucion, Metodo_Pago, Verificado, Tipo_Negocio, Distrito) instead of `[none]`.

### §4.3 — Fix entity_id_field selection (Regression 3)

This is the most architecturally sensitive fix. The root cause is one of two branches:

**Branch A — LLM doesn't distinguish entity vs transaction in `identifies`.** If the LLM writes `identifies: "entity"` for both Folio and DNI_Vendedor (or leaves `identifies` empty/ambiguous for both), the HC prompt's guidance for the `identifies` field needs to be more explicit. The prompt should instruct the LLM to assess: "Does this column identify a PERSON or ACCOUNT that appears across multiple rows (entity-scope)? Or does it identify a single TRANSACTION or EVENT that appears once (transaction-scope)?" The LLM already understands this distinction (it named DNI_Vendedor as "seller's national identity document" in the `characterization` field). The `identifies` field just needs to carry that understanding explicitly.

**Branch B — LLM distinguishes correctly but `findHcEntityIdColumn` doesn't read it.** If DNI_Vendedor has `identifies: "entity"` and Folio has `identifies: "transaction"` but the function still picks Folio, the function's entity-scope detection regex is broken. Fix the regex to match whatever words the LLM actually used.

CC diagnoses which branch by reading the characterization objects in the code flow. The Phase 0 §3.3 sub-diagnostic determines the branch. CC fixes the branch that applies.

**Structural sanity enforcement:** After selecting entity_id_field, if the sanity check fires (1:1 ratio, transaction-shaped) AND another identifier column has a repeating pattern (many rows per distinct value), the function should prefer the repeating column. This is not the construction layer overriding the LLM — it is the construction layer verifying the LLM's assessment and correcting when the assessment is demonstrably inconsistent with the data. The LLM described DNI_Vendedor as a seller's identity document; if it also marked Folio as entity-scope, one of those assessments is wrong, and the structural signal disambiguates. This is Decision 158 applied correctly: the LLM recognizes, the construction layer guarantees.
