## §1 — Problem Statement

HF-309 shipped the per-user theme preference mechanism (fallback chain, cookie, API route). The completion report (G5) claimed a "Theme segmented control (Current / Bliss)" in the user-menu dropdown. **On production, the toggle is not visible.** The user menu popup shows "My Profile," "Settings," and "Sign Out" — no theme control.

Additionally, both Settings surfaces are broken:
- The **gear icon** in the top-right navigation bar renders nothing when clicked
- The **"Settings" link** in the user menu popup renders nothing when clicked

The theme toggle is either placed behind these broken Settings surfaces (unreachable) or is not rendering in the dropdown despite G5's claim. Either way, the user cannot change their theme preference. The HF-309 mechanism works (the API route, the cookie, the fallback chain are all wired) but the UI control is inaccessible.

**This HF fixes three things:**
1. The theme toggle renders directly and visibly in the user menu dropdown — not behind a Settings page
2. The gear icon (top-right nav) functions correctly or is removed if it has no destination
3. The "Settings" menu item functions correctly or is removed if it has no destination

## §3 — What Must Be True After This HF

### §3.1 — Theme toggle in the user menu dropdown

The user clicks their avatar/name in the bottom-left sidebar. The popup menu appears showing:
```
VL Admin
platform@vialuce.com
VL Admin [badge]

Theme:  [ Current ] [ Bliss ]    ← THIS MUST BE VISIBLE AND FUNCTIONAL

👤 My Profile
⚙️ Settings
🚪 Sign Out
```

The toggle calls `POST /api/user/theme` (already exists from HF-309) and triggers `window.location.reload()`. The user sees the theme change immediately.

**If the toggle already exists in the component source but is not rendering:** diagnose why (conditional rendering bug, CSS hiding it, wrong component tree). Fix it.

**If the toggle is placed behind the Settings page instead of in the dropdown:** move it to the dropdown. The Settings page is broken and the toggle must not depend on a broken surface.

### §3.2 — Settings gear icon (top-right nav)

Diagnose what happens when clicked. Three outcomes:
- **It navigates to a route that exists and renders** → fix whatever is preventing render
- **It navigates to a route that does not exist** → either build a minimal settings page or remove the icon with a code comment explaining why
- **It opens a panel/modal that is broken** → fix the panel

### §3.3 — "Settings" in user menu

Same diagnosis as §3.2. The "Settings" link in the user menu popup must either work (navigate to a functioning page) or be removed/disabled with explanation.

### §3.4 — Verification: Observatory follows theme

Navigate to Observatory (VL-Admin only). Confirm the page renders under the active theme (bliss or current). If it does not, diagnose and fix. HF-309 G6 claimed Observatory has no layout bypass — verify this is true on production.

### §3.5 — Verification: Login page follows theme

Log out. Confirm the login page renders under the theme from the `vl-theme` cookie (set during the session). If a user was using bliss, the login page should show bliss after logout. If no cookie exists, it should fall back to the global default.
