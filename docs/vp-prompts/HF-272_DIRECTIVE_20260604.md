**THE GOVERNING CONSTRAINT (AUD-009, architect-set, binds the entire HF ‚Äî read before touching any code):**
This HF removes a registry. The registry being removed resolves the LLM's recognized field tokens against an ENUMERATED SET and rejects any token not in it. When that set is the comprehended columns of data physically present in the import, it is complete and harmless; when that set is the plan's declared-input list (`requiredInputs`), it is STRUCTURALLY INCOMPLETE ‚Äî it lists a plan's attainment/headline inputs, never the raw/meta sub-fields a ratio's numerator and denominator require ‚Äî so it rejects legitimately-recognized ratio inputs. "All the known fields are not listed" is the DATA_TYPES death, verbatim, one structural level up. The cure is NOT to complete the enumeration ‚Äî completing it is the trap, and the instinct to "add the missing fields to the set" is the alarm, not the fix. The cure is to DELETE the interpretation-time rejection and let the registry-free pair already in the architecture do the work: the LLM recognizes and names the field (Decision 158), and convergence resolves that named token against the REAL DATA COLUMNS at calc time ‚Äî a set that is COMPLETE-BY-CONSTRUCTION (it is whatever columns the file contains, not an enumeration anyone maintains). The gate's one legitimate function ‚Äî catching a genuinely hallucinated token ‚Äî RELOCATES to that same convergence step, where "maps to no real column" is actually knowable, as a STRUCTURED FAILURE. The relocation is registry-free for the same reason: it compares against the real columns present, never against a maintained list. **The test, applied to every line this HF writes: if it rejects a recognized token by checking membership in a set that can be incomplete, it is the registry; if it fails a token only because that token maps to NO real data column that exists, it is structural and permitted.**

## ¬ß1 ‚Äî Problem Statement

The platform is broken on a cache-cold interpretation of two distinct tenants, and a single cause produces both failures: the interpretation-time field-resolution gate introduced by HF-270 (PR #458) in `web/src/lib/sci/plan-orchestration.ts`. The gate resolves every field token the LLM recognizes (the `reference` leaves of the constructed DAG) against an anchor set, and on a plan-only import (a plan document with no data sheet in the same upload) that anchor set falls back to the plan's `requiredInputs` declared-field list. That list is structurally incomplete, and the gate rejects any recognized token not in it.

**Failure face 1 ‚Äî BCL, outright rejection.** On a cache-cold run (rule_set anchored to `requiredInputs`, plan-only import), the LLM correctly recognized that the Colocaci√≥n and Captaci√≥n components are ratios requiring meta/target sub-fields (`meta_colocacion`, `meta_depositos`). Those tokens are not in the plan's `requiredInputs` (which lists the attainment/headline fields), so the gate REJECTED both components with `cognition_violation`, exhausted all retries, and never persisted them. The plan ran as a 2-component remainder instead of its full component set. The components' field tokens were legitimately recognized and would have resolved against real data columns at convergence ‚Äî but the interpretation-time gate executed first and destroyed them before convergence could run.

**Failure face 2 ‚Äî Meridian, retry-perturbation.** On the same class of cache-cold run, the gate did not necessarily reject Meridian's components outright, but it perturbed their recognition: any component whose first recognition named a token absent from the incomplete `requiredInputs` set triggered `cognition_violation ‚Üí retry`, and the persisted intent is therefore the product of a reject-retry loop, NOT a clean recognition. **The persisted cold-run intents for Meridian are gate-corrupted artifacts; they are not evidence of what the recognizer produces unperturbed.** (Inspection this session confirmed a Meridian revenue-performance component persisted as a banded lookup keyed on a route-volume reference ‚Äî a recognition that survived the gate's retry loop, not a clean one. Whether that recognition is correct CANNOT be assessed while the gate is in the path corrupting it; that assessment is deferred to the first clean cold import after this subtraction, and is explicitly NOT a target of this HF ‚Äî see ¬ß6.)

**One cause, both failures.** A single incomplete-enumeration gate rejected BCL's meta fields outright and perturbed Meridian's recognition via the retry loop. Removing it addresses both. This is the dominant defect class (AUD-009) reforming for the fourth time as a field whitelist; it was built and merged as architect instrumentation (PR #458 was the architect's production telemetry, never an acceptance), and per SR-43 the merge does not close the work item ‚Äî the cold-cycle verification revealed the result is critically unacceptable, and the work item is open.

**The downstream machinery is registry-free and correct** (established by source read across this session's arc): `constructTree` (the intent-constructor) dispatches on four structural primitives with no closed taxonomy and builds whatever the recognized structure specifies (AUD-009 cited in its own header); convergence (`convergeBindings` / `resolveColumnMappingsViaAI`, HF-114) maps a recognized token to a real data column by semantic meaning; HF-271's structural-coherence proofread (declared-ratio-must-be-two-field) catches internal incoherence requiring no plan knowledge. **None of these is the defect. The defect is solely the interpretation-time rejection against an enumeration that can be incomplete.**

**Defect-class statement (SR-34).** A legitimately-recognized field token is rejected at interpretation time because it is checked for membership in an enumerated set (the plan's `requiredInputs`) that is structurally incomplete for ratio sub-fields. The fix is subtractive at its source: delete the interpretation-time membership rejection and the incomplete-enumeration anchor that feeds it; relocate the one legitimate function it served (catching a token that maps to no real column at all) to convergence, where the comparison set is the real columns present (complete-by-construction) and where today that condition surfaces ‚Äî per this session's tracing of the recurring "no convergence binding" gap ‚Äî as a SILENT under-binding (the component computes wrong or zero) rather than a structured failure.

## ¬ß2 ‚Äî Substrate-Bound Discipline Applications

- **AUD-009 (registry preclusion ‚Äî the governing act):** the subtraction REMOVES an enumerated-set rejection. The relocated catch compares a token against the real data columns that exist, not against any maintained list ‚Äî complete-by-construction, not an enumeration. No registry survives and none is introduced.
- **T1-E902 v2 (Carry Everything / hints-not-gates):** the HC comprehension of data sheets, where present, may remain as a PROMPT HINT (context that informs recognition) ‚Äî a hint annotates, it never gates. What is removed is the post-construction REJECTION (the gate) and the `requiredInputs` FALLBACK that feeds it, because an incomplete enumeration must not gate AND must not bias recognition even as a hint. The HC-of-present-data hint is real runtime comprehension of the actual columns, not an enumeration, and is retained.
- **IGF-T1-E910 v2 (Korean Test) / Decision 154:** structural recognition only; resolution against a runtime set that is COMPLETE-BY-CONSTRUCTION (real columns at calc time). A token that maps to no real column is a STRUCTURED FAILURE naming the token ‚Äî never a silent guess, and never a rejection against an incomplete declared-field list.
- **Decision 158 (LLM recognizes / code constructs):** the gate constrained recognition by rejecting its output against an enumeration; removing it restores recognition to its intended job. `constructTree` is unchanged.
- **SR-34 (No Bypass / no half-measure):** removing only the rejection while leaving the incomplete `requiredInputs` enumeration feeding the prompt would be a half-measure ‚Äî the incomplete set would still bias recognition. The subtraction removes the rejection AND the `requiredInputs` fallback at its source.
- **DD-7 (behavior preservation):** components whose recognized tokens already map to a unique real column (and whose recognition is clean) compute identically ‚Äî the subtraction removes a rejection that those tokens passed anyway, and the relocated convergence failure only fires on a token that maps to NOTHING real. The HF-271 proofread is untouched, so coherent compositions remain coherent. ¬ß4 verifies via a clean cold import covering both affected tenants.

It is the minimal honest catch ‚Äî "this token corresponds to nothing real" ‚Äî relocated from the incomplete-enumeration gate to the complete-by-construction column set. Stronger convergence-fidelity assertions are separate, append-only follow-ons, never an enumeration.
- **HF-270's Phase-1 reachability work is not wasted.** HF-270 proved HC-of-data-sheets is reachable on `classificationTrace.headerComprehension` at execute-bulk time; that reachability is what keeps the HC-of-data-sheets prompt hint (retained here) functioning.
