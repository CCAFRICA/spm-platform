## §1 — The Problem Pattern

Seven prior HFs attacked gray/dark surfaces by grepping for dark utility patterns and adding CSS remaps. Every pass found a new dimension it missed (colored families, opacity modifiers, mid-shades, semantic tokens). Five remap passes haven't converged. The approach is broken.

**What has never been done:** Trace a SPECIFIC gray surface back to its EXACT CSS cause. Not grep for patterns — trace the specific element's computed style to the specific CSS rule that produces it.

This HF reverses the approach: diagnose first, then fix structurally based on the actual cause.

## §2 — Objective

### Phase 1: Diagnose (do NOT fix yet)

For each of these specific gray surfaces, trace the EXACT CSS cause:

**Surface A: `/perform` Admin KPI cards** ($49,911 / 85 / $587.19 / 1 Plans)
- Find the React component that renders these cards
- Find the EXACT className or style that produces the gray background
- Determine: is it a utility class (which one?), a semantic token (which one, and what OKLCH value does it resolve to?), an inline style, a Tailwind `dark:` variant firing in a subtree, or a CSS specificity issue?

**Surface B: `/perform` Admin section cards** (Agent Intelligence, Total Compensation, Distribution, Lifecycle, Locations vs Budget, Component Composition, Active Exceptions)
- Same trace. Are these the SAME cause as Surface A, or different causes?

**Surface C: `/operate/results` hero cards** (Total Payout dark indigo, Performance Distribution dark indigo, Assessment gold/amber)
- These are intentionally colored (not accidentally gray). Trace what produces the dark indigo/gold backgrounds — utility class, inline style, or token.

**Surface D: "Operations Center" sidebar item** — still present in VialuceSidebar despite OB-226 absorbing it. Find where in the WORKSPACES config and remove under Vialuce.

### Phase 2: Fix structurally based on diagnosis

**If the cause is semantic tokens resolving to gray (Cause A):**
Fix the token values in the `[data-theme="vialuce"]` block in `globals.css`. The design spec values are:
- Card surfaces: `#FFFFFF` (pure white, OKLCH `1 0 0`)
- Page background: `#F4F5FB`
- Muted/secondary: must be VISUALLY DISTINCT from white but not gray — use the design spec's `#F0F1F8` (line-soft) or adjust lightness

Then verify: does fixing these tokens resolve the gray on OTHER pages too? If `--card` was the cause on `/perform`, it's the cause everywhere `bg-card` is used.

**If the cause is uncovered utility patterns (Cause B):**
Before adding individual remaps, ask: why didn't the prior five remap passes catch this? Is there a CATEGORY of utility the safety net structurally can't catch? If so, is there a broader CSS approach (attribute selectors, layer ordering) that catches the category? Only add individual remaps if the category can't be addressed broadly.

**If the cause is Tailwind dark: subtrees (Cause C):**
Find every component that applies `className="dark"` or `data-theme="dark"` to its own container (not from `<html>`). Under Vialuce, these subtrees should not carry the dark class. Fix at the container level.

**If the cause is inline styles (Cause D):**
These can only be fixed at the component level with `useIsVialuce()`. Add the conditional. The design spec's `.card` + `.kpi` vocabulary replaces the dark-colored surfaces with white + accent.

**If the cause is CSS specificity (Cause E):**
The remap approach fundamentally can't win. Fix at the component level instead.

### Phase 3: Verify the fix is exhaustive

After fixing based on diagnosis, verify that the SAME root cause doesn't exist on other pages:
- If the fix was a token value, `grep` for that token's usage and confirm all instances now resolve correctly
- If the fix was a utility pattern, `grep` for that pattern on ALL pages
- If the fix was a dark subtree, `grep -rn 'className.*dark' web/components/` to find all dark subtrees

**Definition of done:** The architect toggles to Vialuce and visits `/perform` (Admin/Manager/Rep), `/operate/results`, and every other page in the Performance and Calculation workspaces — zero gray card backgrounds. The diagnosis table explains WHY, not just what was changed.
