## §1 — Problem Statement

### The defect class

The SCI pipeline (Synaptic Content Ingestion — the unified intake path) assumes, at **Phase A (Content Profile)** and **Phase B (Header Comprehension)**, that each sheet is *one header row over uniform records*. That assumption holds for transaction extracts and clean rosters. It fails completely on a **banded report**: a hand-authored sheet that stacks a title banner, a header (often split across multiple rows and re-printed once per section), section-label rows embedded as data, repeated sub-headers, subtotal rows, merged-cell carry-down, and free-text policy/process prose — multiple sub-tables on one sheet separated by blank rows.

This is a structure class the pipeline has never handled. It is distinct from the CLT-160 lineage (Entity-vs-Transaction misclassification on clean tables); that arc tuned scoring, it never addressed sheet *shape*.

### The observed failure (Casa Diaz, `COMISIONES___AUTORIZADOS_-_copia.xlsx`, 8 sheets, 257 rows)

Run through the live pipeline, the import produced the degenerate outcome:

- **Header detection picked the wrong row on 7 of 8 sheets.** The real headers sit in row 3 or 4 (under a merged title banner in row 2); the parser used the top row and emitted `__EMPTY`, `__EMPTY_1`, `__EMPTY_2` … as column names. Only `DISTRIBUIDORES` (headers genuinely in row 1) recovered real names.
- **Comprehension produced nothing usable:** `Recognized Patterns 0`, `New Patterns Learned 0`, `Field Mappings Applied 0`. The convergence layer mapped no column to any metric.
- **Every sheet classified as "Team Roster"** (Entity) except `PULL (EXTERNOS)` ("Reference Data"). None recognized as rate-bearing/commission-rule structure.
- **The fingerprint flywheel was poisoned at the root.** Because the fingerprint (DS-017) is computed from the header row and that row was `__EMPTY` garbage, all 8 sheets stored as `tier=3 novel, confidence=0`. Progressive Immunity cannot engage on a structure whose fingerprint is computed from auto-generated empty names.
- **Multi-section sheets were flattened with contamination.** On `FORANEAS REFAC` (101 data-bearing rows), six branch-label rows ("SUCURSAL : MERIDA / TIJUANA / LEON / IRAPUATO / CELAYA / MONTERREY"), sixteen repeated sub-header rows, and several "TOTALES/SUMAS" subtotal rows are structural artifacts — yet the importer reported 100 rows committed, meaning roughly two dozen non-records were committed as if they were vendor commission rows, and the branch grouping (which vendor belongs to which branch — the branch is encoded in a label row, not a column) was lost.
- **Carry-down owners are at risk.** On `DIST Y SUC`, each rep's name appears once (row N) and the rep's second rate line (row N+1) has a blank name cell by merge convention; treated independently, the second rule is orphaned.
- **A value-normalization step flattened two distinct rules.** On `LOCALES REFAC` column E, "…1% en la 1ra Cat **únicamente ventas del vendedor**" and "…1% en la 1ra Cat **x los servicios técnicos**" were collapsed to one bare canonical, dropping the scope qualifier.

### EECI framing (the tetrad, applied to this OB)

- **Efficiency:** one general invariant — a deterministic structural-construction stage that turns any sheet (banded or flat) into clean tidy units. Not an enumerated set of shape-specific fixes (AUD-009 test passes: no registry of layouts).
- **Efficacy:** when shipped, the WHEN-COMPLETE statement becomes literally true in a browser — Andrew imports `COMISIONES___AUTORIZADOS` and the platform recognizes its real structure instead of the degenerate 0/0/all-roster outcome. The acceptance gate in §3.5 is that import.
- **Comprehensive:** substrate (Decision 158, Korean Test, Carry Everything, G7, DS-017, SR-2), live schema (`committed_data`, `classification_signals`, `structural_fingerprints`, `processing_jobs`), the signal architecture, and the full spatial consumer chain (§2) are all consulted.
- **Innovate:** repairs the fingerprint flywheel for a whole structure class, extends Progressive Immunity to banded reports, and generalizes the single-header assumption into a singular unified path — it advances the Synaptic Surface, it does not merely restore parity.

## §2 — Substrate-Bound Discipline Applications

### Reconciliation-channel separation

This OB carries **no ground-truth payout values.** It is ingestion, not calculation. Acceptance is *structural correctness and lossless commitment*, both verifiable from the source file itself. No verification target values enter this directive. There is nothing for CC to reconcile against an architect-held answer; the file is its own oracle.

### The Thalamus boundary (answers the architect's question: does this belong in Thalamus?)

**No — and the reason is Decision 158.** Thalamus is the recognition/comprehension substrate: it lives *above* the Deterministic Calculation Boundary and it *recognizes meaning* (what a column means, what a value represents). Structural de-banding is *deterministic construction* of clean structure. Placing construction inside Thalamus would do one of two prohibited things: make the predictive substrate perform deterministic construction (a Decision 158 violation), or smuggle deterministic code into the recognition surface under the name of recognition.

The correct relationship is **composition, not containment**: this stage deterministically constructs tidy units → Thalamus / Header Comprehension recognizes their meaning. The de-bander is the *construction* half of Decision 158; Thalamus is the *recognition* half. They compose in sequence; they do not merge. This is recorded here so the architectural decision is auditable: **the Structural Construction Stage is a deterministic Phase A stage that precedes and feeds Thalamus; it is not a Thalamus component.**

### Singular unified path (the core invariant)

The stage runs on **every** sheet. There is no `if (banded) { … } else { … }` branch at any consumer. A clean single-header tabular sheet is the *degenerate output*: one block, header at the detected row, zero sections lifted, zero junk removed, zero sidecar rows. The transformation is universal; simple sheets fall out as the trivial case. This is what makes it Decision-158-and-singular-path compliant: one deterministic construction path feeding one recognition surface. Parallel-path creation (a second pipeline for banded files) is a named failure pattern and a HALT.

### Carry Everything, applied to removal

"Removing" a banner/subtotal/repeated-header/blank row from the *record set* is a re-positioning, never a deletion. Removed rows are retained in a structured sidecar (transformation provenance) carried in `metadata`. The narrative/process block is captured as a distinct documentation unit. Range and multi-value cells are preserved verbatim in `row_data`. Nothing the source contained is lost; structure is *added* (recovered headers, lifted `__section` column) and context is *preserved*.

### Spatial consumer chain (assessed spatially — every surface this stage's output touches)

The stage's output (tidy content units) flows spatially into, and only into, the existing downstream surfaces. Phase 1 (§3.1) confirms each against live code; the conceptual chain the stage must not break is:

```
parse (raw cell grid, parse-once companion)
        │
        ▼
[NEW] Structural Construction Stage  ──►  emits L1 structural observations ──► classification_signals (G7)
        │  (tidy content units: 1 header row, __section lifted, junk in sidecar, carry-down filled, dup headers suffixed)
        ▼
Content Profile (Phase A)            ──►  structural profiling now runs per tidy unit
        ▼
Structural Fingerprint (DS-017)      ──►  now computed from a CORRECT header → stable → Progressive Immunity engages
        ▼
Header Comprehension (Phase B/Thalamus) ──► recognizes meaning on clean columns (no __EMPTY)
        ▼
Agent Scoring + SynapticIngestionState (Phase C, 5 agents) ──► score on real structure
        ▼
Classification Signals (Phase E)     ──►  full trace incl. the stage's observations
        ▼
Execute / Routing (Phase F)          ──►  committed_data.row_data carries full tidy row + __section;
                                          metadata carries removed-row sidecar + provenance;
                                          narrative unit committed as documentation
```

Every contract above stays unchanged — each consumer simply receives clean input instead of a banded mess. The stage is additive at the front of Phase A; it does not rewrite Phase B–F contracts. DD-7 applies: pre-OB behavior on already-clean sheets is preserved exactly (proven in §3.4c).

### §3.2 — Phase 2: The Structural Construction Stage (deterministic, singular path)

Build a deterministic module under `web/src/lib/sci/` (CC chooses the exact filename and decomposition — ULTRACODE). It takes one parsed sheet (the raw 2-D cell grid: rows × columns, values as-read) and returns **one or more tidy content units**, where a tidy unit is a clean rectangle: a single header row, real column names, a propagated `__section` column where sections existed, no banner/repeated-header/subtotal/blank rows in the record set, carry-down filled, duplicate header names disambiguated. The module makes **zero LLM calls** and contains **zero language literals as gates** (AP-25).

The stage is one walk over the sheet that segments it into blocks and emits tidy units. The sub-steps below are the required invariants; the implementation shape is CC's.

**2a — Structural row classification.** Classify every row using only structural properties. Build per-column type profiles (numeric-dominant vs text-dominant vs empty) from the data region, then label each row:

```ts
// Sketch — structural, no word matching. CC finalizes shape.
type RowClass = 'HEADER' | 'SECTION_LABEL' | 'DATA' | 'SUBTOTAL' | 'NARRATIVE' | 'BLANK';
// populated = count of non-empty cells in the row
// identPop  = non-empty cells in text-dominant (identity) columns
// measPop   = non-empty cells in numeric-dominant (measure) columns
// HEADER:   text-heavy, populated ≳ half the column footprint, mostly non-numeric, and the
//           rows below it shift to a data signature (numerics present). Also: a row structurally
//           equal (normalized) to an already-detected header is a repeated HEADER.
// SECTION_LABEL: populated small (≤3), all in identity columns, measPop == 0, no numerics.
//           Distinguished from a carried-down DATA row by populated count: a data row with blank
//           identity cells still populates many measure cells; a section label populates few.
// SUBTOTAL: identPop == 0 but measPop ≥ 1 (identity columns empty, aggregate columns filled).
// NARRATIVE: a single long sentence-like text cell, identPop ≤ 1, measPop == 0.
// BLANK:    populated == 0.
// DATA:     everything else.
```

A word list (e.g. total-like or sum-like tokens) may be consulted **only** as a confidence hint that never changes a label on its own; the structural rule must stand alone. If any token match is the deciding factor, that is **HALT-2** (AP-25).

**2b — Block segmentation and header construction.** Walk the classified rows, grouping into blocks. Within a block, detect the **header band** — one or more consecutive HEADER rows (a multi-row header) — and merge them column-wise into a single header name per column (concatenate the band's non-empty cells per column, top to bottom). Disambiguate duplicate header names by positional suffix (`OBSERVACIONES`, `OBSERVACIONES__2`). When a later block's header band is structurally equal to an earlier one, it is a *repeated header*: skip it as a record and reuse the canonical header (this is how per-section repeated headers like `FORANEAS REFAC` are handled inside the one path — not as a special case).

**2c — Section lifting.** A SECTION_LABEL row's value is lifted into a `__section` column and propagated onto every following DATA row until the next SECTION_LABEL or block boundary. The label row itself is removed from the record set (retained in the sidecar — 2e). This recovers the branch/grouping that was encoded positionally (`SUCURSAL : MERIDA` → `__section = "SUCURSAL : MERIDA"` on the vendors beneath it). Lifting is lossless: the grouping becomes a queryable column.

**2d — Carry-down fill.** Within a DATA run, forward-fill blank cells in identity columns from the value above. The authoritative source is the XLSX merged-cell ranges where they exist (read them explicitly from the workbook); where a value is visually carried without a formal merge, forward-fill blank identity cells within the run. This keeps each rep's second rate line bound to the rep (`DIST Y SUC`). Measure columns are never forward-filled.

**2e — Junk removal with retention (Carry Everything).** Rows classified BLANK, SUBTOTAL, or repeated HEADER are excluded from the tidy record set **and retained in a sidecar** carried into `metadata` (row index, original cell values, classification reason). Nothing is deleted. The sidecar makes every removal auditable.

**2f — Narrative capture.** A run of NARRATIVE rows (e.g. the numbered internal-process block on `DISTRIBUIDORES`, including the email distribution list) is emitted as a **separate tidy unit tagged as documentation**, not discarded and not mixed into the rule records. This preserves the human's payment-process description — the auditable/explainable content — without contaminating the rate tables.

**2g — Range and multi-value preservation.** Cells containing a range or multiple values (`0.5% A 2.0%`, `5%-10%-13%`) are preserved verbatim in the tidy cell. The stage emits a structural observation flagging the cell as multi-valued (a structural signal, §3.3) — it performs **no** interpretation of the range. (Calculating a discretionary band is the engine's job, explicitly §6.)

**2h — Degenerate case (singular-path proof).** A clean single-header tabular sheet must produce exactly one block, one header at the detected row, zero sections, zero sidecar rows, and a record set identical to a naive single-header parse. This is the same code path — proving there is no branch.

## §4 — HALT Conditions

Report the actual topology; do not improvise a different insertion.

- **HALT-3 — Second signal surface.** The stage would write observations anywhere other than `classification_signals` (G7 breach), or a value-normalization canonical would overwrite an original cell in `row_data` (Carry Everything breach).
- **HALT-4 — Parallel path.** The implementation introduces a separate code path for banded vs clean sheets at any consumer (singular-path breach). The degenerate case must be the same path, not a branch.
- **HALT-5 — Behavior regression.** The §3.4c diff on a clean proof sheet is non-empty for the record set (DD-7 breach). Stop; the stage is altering already-correct imports.
- **HALT-6 — Lossy removal.** A removed row (banner / subtotal / repeated header / blank) or a captured narrative row is not present in the sidecar / documentation unit (Carry Everything breach).

Any locked-rule conflict surfaced mid-build is an SR-42 architect-verification halt: surface verbatim, halt for disposition.
