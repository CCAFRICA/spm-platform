## §-1 — CRITICAL PATH TO OBJECTIVE

HF-302 closed the convergence wrong-file-binding regression so MIR can calculate. But HF-302's RC-3 introduced a **developer-set arbitrary constant** — a `≥0.5` value-overlap threshold to decide which column is the entity rollup key. This is a **prohibited class** under Decision 110 (Contextual Reliability Resolution) and OB-IGF-25 binding rule #5: *the flywheel produces authority values; developers never touch them; thresholds emerge from data, not guesswork.* The constant passes G1 on MIR **because the MIR data is clean** (seller id on every row → 100% overlap), and ships a fragility that surfaces on the first messy real tenant — exactly how PDR-08 (hardcoded 0.5 confidence) survived six consecutive CLTs before it was killed. This HF removes the constant before HF-302 merges, replacing it with a relative, self-calibrating derivation, and installs a standing gate so the class cannot re-enter silently.

## §1 — THE DEFECT (from HF-302 completion report, exact)

HF-302 §3.2 shipped, in the `dataByBatch` build (`run/route.ts`):
```ts
// per-batch: best column by value-overlap with entityExtIdSet (≥0.5) → batchRollupCol
```
The rollup key is chosen by testing whether a column's values overlap the assigned-entity `external_id` set **by an absolute majority (≥0.5)**. The `0.5` is the prohibited constant. Concrete failure modes it creates:

- **Sparse-but-correct key:** a seller column populated on 40% of rows (rest blank/system-assigned) scores 0.4 < 0.5 → NOT recognized → rows never reach the paying entity → $0, with only a log line as signal. (The real-world version of the clean-MIR success.)
- **Coincidental-match mis-bind:** a client column that happens to overlap the roster by 55% (some client codes resemble seller ids) scores > 0.5 → recognized as the rollup key → sales aggregate to the WRONG entities → plausible-looking wrong numbers (worse than $0).
- **Two candidates over the line:** the absolute test does not adjudicate between two columns both > 0.5; first-wins by iteration order.

The flat line cannot distinguish "strongest key" from "good enough," which is precisely the distinction that matters.

### §3.1 — Replace absolute threshold with relative strongest-membership (Decision 110 / HF-218 pattern)
Rewrite the rollup-key selection so the key is the candidate column with the **highest membership** in the assigned-entity `external_id` set, evaluated **relative to the other candidate columns** — no absolute cutoff:

- For each batch, for each column that is NOT the primary `entityCol`, compute `membership = (count of non-null values that are members of entityExtIdSet) / (count of non-null values in that column)`. This is a structural observation (a ratio of set-membership counts), not a developer threshold.
- **The rollup key is the column with the maximum membership** among the batch's candidate columns. The decision is the *argmax over candidates*, not a comparison to a constant.
- **Disambiguation / structured failure (no silent fallback):**
  - If the maximum-membership column has membership **strictly greater than zero** and is **strictly greater than every other candidate's membership**, it is the rollup key.
  - If the maximum is **zero** (no column's values are entity external_ids), there is NO rollup key for this batch — emit a structured log (`HF-303: no rollup key for batch <id> — no column's values are entity external_ids`). The batch's rows reach entities only via the primary key. This is the honest "this file does not carry the paying entity" state.
  - If there is a **tie** at the maximum (two columns with equal, non-zero, maximal membership), emit a structured log naming both and select neither as the sole key — OR index under both (CC's call, documented), but NEVER silently pick by iteration order. A tie is a real ambiguity, surfaced, not hidden.
- **No bare numerical literal may appear in this logic** except `0` (the structural floor — "membership greater than zero," which is "is this column a foreign key at all," not a tuned cutoff) and integer counts/lengths. If CC finds itself writing any other number, halt and surface.
- **DD-1/DD-2 sweep:** confirm there is no OTHER place in the rollup path (or a sibling in the same function) that re-applies an absolute overlap/confidence cutoff. Close the class in this function, not just the one line.

### §3.2 — Standing detection gate (so the class cannot re-enter silently)
The architecture is fragile to this class precisely because nothing catches a bare constant at write time. Install a lightweight, permanent grep gate as a committed script + a line in the standing rules:

1. Create `scripts/no-developer-numbers-scan.sh`:
```bash
#!/usr/bin/env bash
# HF-303: detect bare numeric thresholds in foundational calc/convergence code.
# Permitted: integer counts, array indices [0]/[1], tier numbers, 0/1 structural floors,
#   and any number on a line carrying an explicit `// RATIFIED:` justification comment.
# Flags: bare floats like 0.4 / 0.5 / 0.7 / 0.85 used as comparison thresholds.
set -euo pipefail
TARGETS="web/src/lib/intelligence/convergence-service.ts web/src/app/api/calculation/run/route.ts"
# float literals used in comparisons, excluding lines marked RATIFIED:
HITS=$(grep -nE '[<>]=?\s*0?\.[0-9]+|0?\.[0-9]+\s*[<>]=?' $TARGETS | grep -vE 'RATIFIED:' || true)
if [ -n "$HITS" ]; then
  echo "NO-DEV-NUMBERS GATE: bare numeric thresholds found (add // RATIFIED: <reason> or remove):"
  echo "$HITS"
  exit 1
fi
echo "NO-DEV-NUMBERS GATE: clean."
```
2. Run it. It MUST come back clean after §3.1 (the rollup logic no longer carries a bare float threshold). If it flags any pre-existing bare-float threshold elsewhere in those two files, **list them in the completion report** as discovered debt — do NOT fix them in this HF (DD-7 scope), but surface every instance for architect triage. This is the "anything else" sweep: it enumerates the full existing population of this class in the two hottest files.
3. Add to `CC_STANDING_ARCHITECTURE_RULES.md` a line under the build steps: *"Run `scripts/no-developer-numbers-scan.sh` before every completion report touching convergence-service.ts or run/route.ts; a bare numeric threshold without a `// RATIFIED:` justification is a Decision-110 violation — halt."*
