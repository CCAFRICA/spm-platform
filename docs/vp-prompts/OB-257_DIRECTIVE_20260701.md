## §1 — PROBLEM STATEMENT

**Problem A — the unserved persona.** The platform tells the compensation story: what was paid, its composition, its audit trail. It does not tell the sales story. A CRO, a sales leader, or a product-line owner asks: what was sold, by whom, in what mix, with what momentum, at what incentive cost per revenue dollar? Every measure needed to answer already exists in `committed_data` (Carry Everything: ALL columns persist at import) and, for served tenants, in write-time materializations (OB-237). No surface reads them through a revenue-persona lens. The Financial agent's modes are adjacent but framed for financial operations; several are directly reusable at the serving layer.

**Problem B — the unproven activation promise.** The architecture promises: a tenant that has run Compensation for N periods can have another agent TURNED ON and its data immediately renders in that agent's surfaces — no reimport. The chain was designed (OB-231: convergence bindings replace the hardcoded route contract; OB-237: universal materialized serving) but the end-to-end browser proof has NEVER been executed. OB-257 delivers the Revenue agent AND the first empirical activation proof, on BCL (tenant `b1c2d3e4-aaaa-bbbb-cccc-111111111111`, 85 entities, 6 closed periods, ICM commission data whose transactions carry the underlying sales measures).

**Defect-class lineage:** entitlement gates that decorate but do not deny (OB-211 findings — inert menu gate, unguarded route); hardcoded per-tenant data contracts (OB-231 lineage, Decision 158 violations); read-time aggregation (OB-237 lineage, MSP violations); dual code paths from copy-instead-of-share (AP-17 violations).

## §2 — SUBSTRATE-BOUND DISCIPLINE APPLICATIONS

- **Decision 158 (LLM recognizes; deterministic code constructs):** Revenue semantic-role resolution reads the convergence layer's per-tenant bindings directly. Zero keyword scans of any prose. Zero hardcoded column names anywhere in the serving path.
- **Decision 154 / Korean Test / AP-25:** zero tenant-language or domain-language string literals in structural code. The platform's own semantic-role vocabulary (e.g., transaction-amount role, product-dimension role, temporal role, entity role) is the only naming layer. Structured failure on unresolved roles — never silent fallback.
- **Decision 108:** where Header Comprehension column roles exist at confidence ≥ 0.80, they override structural detection in role derivation.
- **MSP invariant (OB-237, constitutional):** every Revenue surface serves from a materialization. No raw aggregate reads at render time, at any layer, under any mode. If a required rollup does not exist for a tenant, the answer is a WRITE-TIME materialization extension plus a one-time backfill derivation from `committed_data` — never a read-time aggregate, never an RPC that relocates the reduce into SQL at query time.
- **Carry Everything (T1-E902 v2):** activation derives from carried data; it never re-ingests. A "reimport requirement" discovered anywhere in the activation path is a defect, not a workaround (HALT-2).
- **Progressive Performance (constitutional):** activation cost is paid once (backfill derivation); every subsequent encounter serves at materialization speed. The proof gate measures a repeat load. A Revenue surface producing cold-start-cost results on every encounter is FAILURE.
- **AP-17 (single code path):** Finance visualizations are incorporated by SHARING the serving layer (parameterized aggregate-serving modules), not by copying route-handler code. Revenue-specific pages/compositions are new; aggregate serving is shared. A duplicated aggregate path is a defect.
- **DD-7 (behavior preservation):** Finance agent behavior is preserved exactly. Any shared-module refactor that would alter Finance output → HALT-6.
- **End-State A (locked) + Decision 123 (compliance is architecture):** Revenue is a licensable agent gated per tenant. Both gates required: menu gate (navigation filter honoring the tenant feature set) AND route gate (direct navigation denied for un-entitled tenants). SR-39 evidence mandatory.
- **C2 fail-loud:** an unresolved revenue role for a tenant renders as an explicit structured absence on the surface (named role, named reason), never silent zeros or empty charts.
- **DS-003 Cognitive Fit (CLT-bound):** every Revenue page uses 3+ distinct component types when showing 4+ data elements; every quantitative element carries a reference frame (prior period, target, distribution, or trend); progressive disclosure (glanceable primary surface, detail on interaction).
- **Theme discipline:** all Revenue UI consumes the three-layer CSS token architecture at Layer 2 (component slots) only. Zero hardcoded palette values. Surfaces must render correctly under every theme the theme selector offers.

### §3.3 — O3: REVENUE SURFACES (the CRO/sales-leader lens)

**Objective:** an innovative, simple, logical, insight-and-action-oriented Revenue workspace. CC composes autonomously within these named surfaces and their decision tasks; CC may propose additions (name them in the report — proposals are welcome, silent additions are not).

The surface set (each line: surface — decision task — primary reference frame):
1. **Revenue Pulse** (landing) — "where does revenue stand right now and is the pace healthy?" — current period vs. prior periods; deterministic run-rate projection to period-end (no ML claims; the projection is arithmetic and labeled as pace).
2. **Growth Bridge** — "exactly what added and what subtracted revenue vs. last period?" — waterfall decomposition by the tenant's resolved dimensions (product/entity/location as available); each bar drillable.
3. **Revenue Mix & Mix-Shift** — "what is the composition and how is it moving?" — stacked composition over time PLUS explicit mix-shift deltas (the shift is the insight, not the shares).
4. **Seller Revenue Distribution** — "is revenue production healthy across the team or concentrated?" — leaderboard with rank movement AND the distribution curve/Pareto view of revenue per seller.
5. **Concentration & Risk** — "how dependent are we on few contributors, and who is declining?" — top-N dependency, declining-contributor flags with trend evidence.
6. **Incentive Yield** — "what revenue did each incentive dollar buy?" — revenue measure joined with verified payout materializations, by plan/component/variant. This surface is the platform differentiator; give it first-class placement.
7. **Temporal Patterns** — adapted from the Finance patterns/timeline serving modes where grains exist.
8. **Geography/Location** — adapted from Finance location modes, present only when the tenant's convergence bindings resolve a location dimension (C2: absent dimension = explicit absence, not empty chart).

Cross-cutting constraints:
- DS-003 Cognitive Fit rules enforced per page (diversity minimum, reference frames, progressive disclosure, persona density: leader-level density, not rep-level).
- Theme: Layer-2 tokens only; verify rendering under the theme selector's themes; zero hardcoded colors (EPG: grep new Revenue components for hex literals — scoped per DD-4).
- Every surface MSP-served (§3.2). A surface whose grain is not yet materialized renders the explicit structured absence, and the gap is named in Residuals — it does not fall back to raw reads.
- All UI copy through the platform's existing i18n mechanism; no hardcoded user-facing strings where the platform localizes.
