## §1 — Problem Statement

The theme system shipped in PR #541 is controlled by a single global `platform_settings` row, toggled only by VL-Admin in Observatory > Settings > Appearance. The architect now requires per-user preference: each user chooses their theme, the choice persists across sessions, and the login page and Observatory also respect the selection.

**Decision D2 revised (architect authority):** Global platform default (OB-201 D2) is superseded. New rule: per-user preference, persisted to `profiles.preferences` jsonb, with cookie-backed fallback for pre-auth surfaces.

**What this HF delivers:**
1. `preferences` jsonb column on `profiles` — stores `{ "theme": "bliss" }` (or absent = no preference)
2. Three-level server-side fallback chain in the root layout
3. `vl-theme` cookie set on preference change and on login, for pre-auth surfaces
4. Theme toggle in the user-facing My Profile / Account Settings area
5. Login page respects the cookie (or falls back to global default)
6. Observatory renders under the user's selected theme

**What this HF does NOT change:**
- The global `platform_settings.active_ui_theme` row remains and serves as the fallback when a user has no preference set
- The Observatory > Settings > Appearance toggle (OB-201) remains as the global default control (VL-Admin only)
- No visual/design changes — the theme tokens, pipeline, and vocabulary are complete from PR #541

## §3 — Architecture

### §3.1 — Schema: `preferences` jsonb column on `profiles`

```sql
ALTER TABLE profiles
  ADD COLUMN IF NOT EXISTS preferences jsonb NOT NULL DEFAULT '{}';

COMMENT ON COLUMN profiles.preferences IS
  'User display preferences (e.g. {"theme":"bliss"}). Presentation-layer only. No auth/access data.';
```

Theme preference accessed as `profiles.preferences->>'theme'`. Absent key or `{}` means no user preference — fall through to global default.

This column is extensible for future non-auth user preferences (locale override, dashboard layout, notification settings) without additional schema changes.

### §3.2 — Three-level fallback chain (server-side, root layout)

The root layout (`web/src/app/layout.tsx`) currently reads `platform_settings.active_ui_theme` (OB-201). Extend:

```
1. If authenticated → read profiles.preferences->>'theme' for the logged-in user
   - If set ('bliss' or 'current') → use it
   - If absent/null/empty → fall through
2. Read platform_settings.active_ui_theme (existing global default)
   - If set → use it
   - If absent/error → fall through
3. Hardcoded final fallback: 'current'
```

The `fetchCurrentProfile()` call already happens on every authenticated render (per the auth flow in `web/src/lib/auth.ts`). Add `preferences` to the select. No additional query.

### §3.3 — Cookie for pre-auth surfaces (login page)

**On preference change** (user toggles theme in My Profile) AND **on every successful login**, set:

```
Set-Cookie: vl-theme=bliss; Path=/; SameSite=Lax; Secure; Max-Age=31536000
```

The login page layout reads this cookie server-side and applies `data-theme` from it. First-ever visitor (no cookie) falls back to the global default from `platform_settings`.

**Hard constraint: the cookie contains ONLY the theme name string.** No user ID, no tenant ID, no session token, no auth data. Non-sensitive, non-identifying. This is enforced by HALT-2.

**Cookie persists across logout by design** — the user's visual preference should survive sign-out and re-login. The cookie is a display hint, not an auth artifact.

### §3.4 — User-facing toggle: My Profile / Account Settings

Add a theme selection control in the user's profile or account settings area. Simple toggle or segmented control:

```
Theme: [ Current (Dark) ] [ Bliss (Light) ]
```

On change:
1. Write to `profiles.preferences` via the existing service-role or user-scoped Supabase update path (`{ ...existing_preferences, theme: 'bliss' }`)
2. Set the `vl-theme` cookie (§3.3)
3. Trigger page reload (server re-renders with new `data-theme` from the updated preference)

**If no My Profile page exists today**, add the toggle to the nearest existing user-facing settings surface (e.g., the user menu dropdown, or a new minimal profile section). Do NOT build a full My Profile page — just the theme control.

### §3.5 — Login page theming

The login page renders before authentication, so `profiles.preferences` is not available. The fallback chain for the login page:

```
1. Read vl-theme cookie (set on last login or last preference change)
2. Read platform_settings.active_ui_theme (global default)
3. Hardcoded fallback: 'current'
```

The login page layout must read the cookie server-side (Next.js `cookies()` API) and emit `data-theme` from it, same as the authenticated layout does for the user preference.

### §3.6 — Observatory theming

Observatory pages render inside the authenticated app shell. They already receive `data-theme` on `<html>` via the root layout. No additional work needed unless Observatory has its own layout that bypasses the root. Verify and confirm.

Observatory's inline-styled panels (HF-305 G6: 1,211 inline hex) are NOT remediated in this HF — they're out of scope (§6). The theme attribute reaches Observatory; whether individual panels visually respond depends on their styling, which is a separate HF.

## §6A — Residuals

- **Multi-theme expansion.** The `preferences.theme` field and the fallback chain support arbitrary theme name strings. Adding a third theme requires only a new token set in `globals.css` and a UI option — no mechanism change needed.
- **Cookie cleanup on logout.** The `vl-theme` cookie persists across logout by design. If this behavior is undesired, clearing it on sign-out is a one-line follow-on.
- **Observatory visual remediation.** The theme attribute reaches Observatory (§3.6). Whether Observatory panels visually respond depends on their inline styling (HF-305 G6). Remediation is a standalone HF.
- **Other user preferences.** The `preferences` jsonb column is extensible. Future non-auth preferences (locale override, dashboard layout, notification settings) can be stored here without schema changes.
- **Login page visual polish under bliss.** HF-308 deferred 1 `text-white` occurrence on the login page. This HF wires the cookie mechanism; any remaining login-page visual straggler surfaces during the architect's SR.
