## §1 — Problem Statement

Browser verification of OB-246 shows the Admin, Manager, and Rep persona views are functionally identical on `/stream`:

- **Admin:** $58,406 / 85 entities / full sidebar / Top Accelerator names individual / Population Trajectory tenant-wide / Accelerators & Attention tenant-wide — **CORRECT**
- **Manager:** Identical to Admin. $58,406 / 85 entities / full sidebar / same panels — **WRONG.** Manager should see team-scoped data and reduced sidebar.
- **Rep:** Identical to Admin EXCEPT the INSIGHT card correctly shows "$1,505 across 1 entity · 4 components" (proving entity-scoped intelligence exists). Everything else is tenant-wide: period ribbon $58,406, System Health 85 entities, Top Accelerator names a peer, full sidebar. — **WRONG.**

The cause: OB-246 correctly keyed scope off authenticated identity (platform admin → `{type:'all'}`), but the architect's directive explicitly prohibited narrowing scope or menu when the persona override is active. The persona switcher changes the dashboard component and theme but NOT the data or navigation. This makes it impossible to verify, demo, or experience what each role actually sees.

## §2 — Substrate-Bound Discipline Applications

### Narrowing within entitlement is safe

A VL admin is authenticated as `platform` and entitled to ALL data and ALL capabilities. Previewing a narrowed view (rep's own-entity data, manager's team data, member's reduced sidebar) is ALWAYS within entitlement. No security boundary is crossed. Decision 39 prohibits a real member from widening scope to tenant-wide — it does not prohibit an admin from narrowing their preview.

### The persona override is already gated

OB-246 correctly gated the override in `persona-context` by `isVLAdmin`. A real member/manager cannot set the override. Their scope and menu are determined entirely by their authenticated role, exactly as OB-246 built. This HF changes ONLY what happens when a VL admin uses the override.

### EXTEND, not replace

This HF EXTENDS the existing OB-246 architecture. It does NOT create a parallel auth path, a separate hook, or a second lifecycle. It adds `effectiveScope` and `effectiveCapabilities` to the EXISTING `AuthContextType`, computed from the EXISTING scope + the EXISTING persona override.

## §3 — Phase Prose

### §3.1 — Effective Scope (data narrowing for VL admin persona preview)

**Objective:** When a VL admin has an active persona override, data consumers read a narrowed scope matching the selected persona. Real users are unaffected.

**Constraints:**

- Add to `AuthContextType`:
  - `effectiveScope: AuthScope` — the scope data consumers read.
  - For non-VL-admin (or VL admin with no override): `effectiveScope = scope` (the authenticated-role scope, unchanged).
  - For VL admin with override:
    - Override = `admin` → `effectiveScope = scope` (ALL — current behavior)
    - Override = `manager` → `effectiveScope = { type: 'team', entityIds: [sample team set] }` — use `profile_scope.visible_entity_ids` if available, else fall back to a sample set of entities for the tenant (e.g., the first N entities). If no entities exist, `{ type: 'deny' }`.
    - Override = `rep` → `effectiveScope = { type: 'own', entityId: [sample entity] }` — use the VL admin's linked entity if available, else the highest-payout entity in the tenant (the existing `getRepDashboardData` null fallback). If no entities exist, `{ type: 'deny' }`.
- `effectiveScope` is computed INSIDE auth-context, reacting to the persona override change. No separate hook, no separate lifecycle. It is a derived value from `scope` + `persona override` + `isVLAdmin`.
- Every data consumer that currently reads `useAuth().scope` must read `useAuth().effectiveScope` instead. This is a find-and-replace across the OB-246 call sites (all in Phase 4 surfaces + Phase 2 threading callers). `scope` remains available for security checks (e.g., API route tenant binding always uses `scope`, never `effectiveScope`).
- For a REAL member login: `effectiveScope = scope = { type: 'own', entityId: '...' }`. Unchanged from OB-246.
- For a REAL manager login: `effectiveScope = scope = { type: 'team', entityIds: [...] }`. Unchanged from OB-246.

### §3.2 — Effective Capabilities (sidebar and menu narrowing for VL admin persona preview)

**Objective:** When a VL admin has an active persona override, the sidebar and capability checks reflect the selected persona's capabilities. Real users are unaffected.

**Constraints:**

- Add to `AuthContextType`:
  - `effectiveCapabilities: string[]` — the capabilities used for menu/UI gating.
  - For non-VL-admin (or VL admin with no override): `effectiveCapabilities = capabilities` (the authenticated capabilities).
  - For VL admin with override: `effectiveCapabilities = ROLE_CAPABILITIES[overrideRole]` — the capability set for the persona's canonical role, from `permissions.ts`.
- `hasCapability` changes: evaluate against `effectiveCapabilities`, not `capabilities`. The admin/platform bypass applies ONLY when no persona override is active:
  ```
  if override is active and isVLAdmin:
    return effectiveCapabilities.includes(capability)
  if user.role is platform or admin:
    return true
  return capabilities.includes(capability)
  ```
- `navigation-context` `effectiveRole`: when persona override is active, derive from the override persona (the pre-OB-246 behavior for VL admin). When no override, derive from authenticated role (the OB-246 behavior). This means:
  - VL admin, no override → admin sidebar (authenticated)
  - VL admin, Rep override → member sidebar (reduced: `/stream`, `/perform`, `/notifications`)
  - VL admin, Manager override → manager sidebar (team: adds `/insights*`)
  - Real member login → member sidebar (from authenticated role — unchanged)

### §3.3 — Sample Entity Resolution for VL Admin Preview

**Objective:** When VL admin previews Rep persona, the scope needs an entity to narrow to. Resolve a sample entity.

**Constraints:**

- The sample entity resolution for VL admin Rep preview should use, in order:
  1. The VL admin's own linked entity (`entities.profile_id`), if it exists
  2. The highest-payout entity for the current period in the tenant (existing `getRepDashboardData` null→top fallback)
  3. If no entities exist: `{ type: 'deny' }`
- This resolution runs ONLY for VL admin with Rep override. Real members resolve their own entity from `entities.profile_id` (OB-246 Phase 1, unchanged).
- For Manager preview: use `profile_scope.visible_entity_ids` for the VL admin's profile if a row exists; else a reasonable sample (e.g., first 10 entities in the tenant). Real managers use their actual `profile_scope` (OB-246 Phase 1, unchanged).
- This sample resolution can be async but must complete within the SAME `initAuth` / scope-resolution sequence — no separate lifecycle.
