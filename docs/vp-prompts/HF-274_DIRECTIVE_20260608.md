## §1 Problem Statement

Three Meridian components regressed from a verified-correct state (May 13 run: 185,063 / 175,585 / 196,337, all five components correct) to the current state (June 8: 173,900 / 166,950 / 176,700, three components wrong). The regression was caused by a convergence re-derivation on 2026-06-08T17:03:40Z that replaced passing bindings with structurally different ones.

Two distinct defect classes, sharing a single root event:

**Defect A — Scale not applied in DAG construction (c1 Entrega a Tiempo).** The `compositional_intent` correctly declares `"scale": {"side": "convergence", "unit": "percent", "value": 100}`. The `calculationIntent` DAG compares a raw 0–1 ratio (e.g., 34/43 = 0.7907) against breakpoints of 85, 90, 95, 98 (percent-space values). `0.7907 ≥ 85` is false for every entity → floor to 0 across 201 entity-periods. The metadata for correct construction exists; `constructTree` did not consume it when building the comparison nodes.

**Defect B — Hub-scoped data unresolvable from individual rows (c4 Utilización de Flota, c0 Rendimiento de Ingreso).** The June 8 re-derivation correctly decomposed ratios per HF-271 teaching. c4 now binds to `Cargas_Totales` / `Capacidad_Total` — real columns that exist in committed_data but exclusively on the 12 hub-level rows that HF-263 excludes from individual-entity calculation. On individual rows these columns are null → c4 returns 0 across 201 entity-periods. c0 binds `cargas_mes_hub → Cargas_Flota_Hub`, which resolves but may be the wrong column for the hub-volume dimension of the 2D band (the May 13 passing run used a different binding for this axis).

**Combined-treatment rationale.** Defect A is construction-layer (DAG builder ignores declared scale). Defect B is resolution-layer (engine cannot reach hub-scoped data from individual-entity context). Both surfaced from the same re-derivation event. Both must be fixed for Meridian to reconcile. Defect A is self-contained (Phase 2). Defect B extends HF-262 scope (DIAG-58 Condition A; Phase 3). Phase 0 diagnostic reads confirm the data reality before any code change.

## §3B Phase 2 — Scale Application in `constructTree`

**Scope:** When `constructTree` builds a comparison node (prime: "compare", op: "gte") and the component's `compositional_intent.scale` declares `side: "convergence"`, the breakpoint constant must be divided by `scale.value` so the comparison operates in ratio space. Example: breakpoint 85 with scale.value=100 becomes 0.85 in the DAG.

### §3B.1 Read `constructTree`

```bash
grep -n -A 5 "constructTree\|buildTree\|convertComponent" web/src/lib/calculation/run-calculation.ts | head -100
```

Identify where comparison nodes are built from the `compositional_intent` structure. Paste the function body.

### §3B.2 Read scale metadata consumption

```bash
grep -n "compositional_intent\|scale.*convergence\|scale.*value" web/src/lib/calculation/run-calculation.ts | head -50
```

Identify whether `constructTree` currently reads `compositional_intent.scale` at all. If it does not (HALT-2 path), the fix is: when constructing a comparison node whose operand is a ratio (arithmetic divide), check the parent component's `compositional_intent.scale`; if `side === "convergence"`, divide each breakpoint constant by `scale.value`.

### §3B.3 Implement

The fix is one conditional in the comparison-node builder: if the component has `compositional_intent.scale.side === "convergence"` and the comparison operand is an arithmetic divide, scale each constant breakpoint by dividing by `scale.value`. No enumeration of component names, field names, or scale values. The rule is: the LLM declared the scale; the constructor applies it.

Commit: `git add -A && git commit -m "HF-274 Phase 2: constructTree applies declared convergence scale to breakpoints" && git push origin dev`

## §3C Phase 3 — Hub-to-Individual Data Resolution

**Scope:** When the engine resolves metrics for an individual entity, and the bound column exists only on hub-level rows (entity_id IS NULL), the engine must locate the correct hub row for that entity and read the value from it. This is the general scope-resolution capability DIAG-58 identified as Condition A.

### §3C.1 Read the current data resolution path

```bash
grep -n -A 10 "resolveColumnFromBatch\|resolveMetrics" web/src/lib/calculation/run-calculation.ts | head -150
```

Identify where column values are read from committed_data rows. The current path filters to individual-entity rows only (HF-263). The fix adds a fallback: if a column resolves null on the individual row, check whether it exists on a hub-scope row associated with that entity's hub assignment.

### §3C.2 Read entity-to-hub association

```bash
grep -n -A 10 "hub\|Hub_Asignado\|grouping" web/src/lib/calculation/run-calculation.ts | head -100
```

Identify how the engine knows which hub an individual entity belongs to. The entity's row_data likely contains a `Hub_Asignado` field. The hub rows have external_id matching hub names.

### §3C.3 Implement

The resolution: when `resolveColumnFromBatch` returns null for a bound field on an individual row, look up the entity's hub assignment (from the entity's row_data or the flat data), find the hub-scope row in the same batch with matching hub identifier, and read the column from that row. General to any hub/group/store scope, not Meridian-specific. No hardcoded hub names or field names — the scope association is read from the data.

**HALT-3** if no hub association mechanism exists in the current code and the design requires an Architecture Decision for how scope levels are modeled.
