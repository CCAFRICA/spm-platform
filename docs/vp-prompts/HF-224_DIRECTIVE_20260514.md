## 1 -- Problem Statement

Two defect classes, one combined treatment.

**Defect Class A: Flat-input assumption in convergence and route consumers (AUD-008 findings).**

Three consumers read calculationIntent input shapes assuming every input is a flat IntentSource (has `source` and `sourceSpec` fields at the top level of `intent.input`). When the input is a nested IntentOperation (has `operation` field, as produced by HF-223 for ratio caps via conditional_gate), these consumers return unknown/undefined/null and downstream processing fails silently.

AUD-008 identified the three must-fix surfaces:

Surface 1: `extractInputRequirements` at `convergence-service.ts:1245-1320`. The `scalar_multiply` branch (line 1277) checks `input?.source === 'ratio'`. For nested operations, `input.source` is undefined. Falls through to else, returns `{ role: 'actual', metricField: 'unknown', expectedRange: null }`. All candidates score 0.1. `distinctEnoughToBind` refuses.

Surface 2: `resolveMetricsFromConvergenceBindings` ratio-name extractor at `route.ts:1312-1321`. Reads `calculationIntent.input.sourceSpec.{numerator, denominator}` directly. For nested intent, `input.sourceSpec` is undefined. The calculation route cannot resolve ratio metric names from convergence bindings.

Surface 3: Convergence Pass 4 AI prompt builder at `convergence-service.ts:574-583`. Reads `intent.input.sourceSpec` to build the AI column-mapping prompt. For nested intent, no metric context for the component.

All three share the same structural defect: they walk a fixed path through the intent tree expecting a known shape. This is a registry-of-known-shapes pattern. When the LLM produces any structure not enumerated in the fixed path, the consumer fails silently.

The executor already solves this correctly. `resolveValue` at `intent-executor.ts:159` discriminates `IntentOperation` (has `operation` field) vs `IntentSource` (has `source` field) and recurses. Two branches. Handles any depth, any combination, any future primitive. The convergence and route consumers need the same structural pattern.

**Defect Class B: Import-time convergence (DIAG-044 findings).**

Two import routes run convergence at import time when convergence should only run at calculation time (HF-165):

Site 1: `execute/route.ts:192` (OB-160G wiring). Runs `convergeBindings(...)` for all rule_sets in the tenant during plan import.

Site 2: `commit/route.ts:998` (OB-120 legacy). Runs convergence during the legacy import commit path.

OB-182 correctly removed convergence from the bulk data-import path (CLT-181 F10: sequence dependency violation). The non-bulk plan-import path and the legacy commit path retain it. The calc-time convergence at `route.ts:231` (HF-165) is the correct location: fires when `!hasMetricDerivations && !hasConvergenceBindings`, runs with both plan and data present, persists results.

Import-time convergence produces partial bindings (C1-C4 bind, C5 gaps). These partial bindings then prevent HF-165 from firing at calc time (`hasConvergenceBindings` is true because C1-C4 exist). C5 never gets a second chance. Removing import-time convergence means HF-165 fires fresh at calc time with no partial state.

**Combined treatment rationale:** Both defect classes concern convergence pipeline correctness. Defect A (flat-input assumption) prevents correct binding even when convergence runs at the right time. Defect B (wrong-time convergence) prevents correct binding even when the extraction logic is correct. Both must close for C5 to bind.

### 3.2 Create extractLeafSources

Add the following function to `convergence-service.ts`. Place it ABOVE `extractInputRequirements` (before line 1245). This function is the generic tree traversal that replaces all fixed-path intent reading.

```typescript
/**
 * Recursively traverse an intent tree and return all leaf-level IntentSource references.
 * Follows the same structural discrimination as executor resolveValue:
 *   - Has 'operation' field -> IntentOperation -> recurse into sub-fields
 *   - Has 'source' field -> IntentSource (leaf) -> collect
 * Handles any depth of nesting, any combination of operations, any future primitive.
 */
function extractLeafSources(node: unknown): Array<{ source: string; sourceSpec?: Record<string, unknown> }> {
  if (!node || typeof node !== 'object') return [];

  const obj = node as Record<string, unknown>;

  // Leaf: has 'source' field -> this is an IntentSource
  if (typeof obj.source === 'string') {
    return [{ source: obj.source, sourceSpec: obj.sourceSpec as Record<string, unknown> | undefined }];
  }

  // Branch: has 'operation' field -> this is an IntentOperation -> recurse
  if (typeof obj.operation === 'string') {
    const leaves: Array<{ source: string; sourceSpec?: Record<string, unknown> }> = [];

    // Recurse into all known sub-fields that can carry IntentSource or IntentOperation
    const recurseField = (field: unknown) => {
      if (field && typeof field === 'object') {
        leaves.push(...extractLeafSources(field));
      }
    };

    recurseField(obj.input);
    recurseField(obj.onTrue);
    recurseField(obj.onFalse);
    recurseField(obj.condition);

    // condition is an object with left/right operands
    if (obj.condition && typeof obj.condition === 'object') {
      const cond = obj.condition as Record<string, unknown>;
      recurseField(cond.left);
      recurseField(cond.right);
    }

    // inputs (plural, for bounded_lookup_2d: { row: ..., column: ... })
    if (obj.inputs && typeof obj.inputs === 'object') {
      const inputs = obj.inputs as Record<string, unknown>;
      for (const val of Object.values(inputs)) {
        recurseField(val);
      }
    }

    // segments (for piecewise_linear)
    if (Array.isArray(obj.segments)) {
      for (const seg of obj.segments) {
        recurseField(seg);
      }
    }

    // ratioInput / baseInput (for piecewise_linear with ratio)
    recurseField(obj.ratioInput);
    recurseField(obj.baseInput);

    return leaves;
  }

  return [];
}
```

Replace the body of the `scalar_multiply` case with logic that uses `extractLeafSources`:

```typescript
case 'scalar_multiply': {
  const input = intent.input as Record<string, unknown> | undefined;

  // First: try flat source (pre-HF-223 shape, backwards compatible)
  if (input?.source === 'ratio') {
    const spec = input.sourceSpec as Record<string, unknown> | undefined;
    return {
      role: 'ratio' as const,
      numeratorField: String(spec?.numerator || 'unknown'),
      denominatorField: String(spec?.denominator || 'unknown'),
      expectedRange: { min: 0, max: 2 },
    };
  }

  // Second: traverse nested operations to find leaf sources
  const leaves = extractLeafSources(input);
  const ratioLeaf = leaves.find(l => l.source === 'ratio');
  if (ratioLeaf?.sourceSpec) {
    return {
      role: 'ratio' as const,
      numeratorField: String(ratioLeaf.sourceSpec.numerator || 'unknown'),
      denominatorField: String(ratioLeaf.sourceSpec.denominator || 'unknown'),
      expectedRange: { min: 0, max: 2 },
    };
  }

  // Third: check for metric sources in leaves
  const metricLeaf = leaves.find(l => l.source === 'metric');
  if (metricLeaf?.sourceSpec) {
    const field = String(metricLeaf.sourceSpec.field || 'unknown');
    return {
      role: 'actual' as const,
      metricField: field,
      expectedRange: metricLeaf.sourceSpec.expectedRange as { min: number; max: number } | null || null,
    };
  }

  // Fallback: unknown
  return {
    role: 'actual' as const,
    metricField: 'unknown',
    expectedRange: null,
  };
}
```

No behavioral change for any existing component. C5's nested conditional_gate-wrapped ratio is found by `extractLeafSources` traversing into onTrue and finding the ratio leaf.
