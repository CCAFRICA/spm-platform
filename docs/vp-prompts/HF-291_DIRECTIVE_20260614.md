## §1 — PROBLEM STATEMENT

### 1.1 CLT Findings (production screenshots, 2026-06-14)

OB-205 shipped carrier expression on /stream. Production browser verification reveals four design failures:

**F-1: Persona scoping absent.** Manager and Rep personas see Import Health, Entity Landscape, and Pipeline Readiness — identical to Admin. DS-029 §6.2 specifies: "Manager NEVER sees import health, entity landscape, or pipeline readiness." These are infrastructure surfaces serving Admin's governance cognitive task. The Manager's cognitive task is people development; the Rep's is self-mastery. Neither needs row counts or pipeline steppers.

**F-2: Real estate priority inverted.** The three carrier cards consume the top third of /stream, pushing the existing intelligence elements (System Health $58,406 +9.8%, Trajectory Intelligence +$5,430/period, Top Accelerators / Attention Needed, Optimization Opportunities) below the fold. For BCL with 6 periods of calculation history in PREVIEW state, the lifecycle and payout intelligence is the most important information. Import row counts are the least important.

**F-3: No contextual collapse.** Pipeline Readiness shows a full-width stepper with five green checkmarks when all steps are complete. The Bloodwork principle (OB-102, TMR circadian architecture, February 14 design session): passing checks build confidence silently — problems get immediate visibility. Five green checks taking prime real estate tells the Admin everything is fine, loudly. That is the opposite of Bloodwork.

**F-4: Cards fail the intelligence test.** "595 rows across 2 content units from 7 imports" is infrastructure telemetry, not intelligence. No reference frames ("is that good or bad?"). No deterministic commentary. No Bloodwork health assessment. No Cognitive Fit diversity (every card is text + buttons). Every card fails the IAP Gate (zero Intelligence, zero Acceleration, zero Performance), the Thermostat Test (thermometer, not thermostat), and the Reference Frame requirement (A.70–A.74 findings from restaurant CLT).

### 1.2 Defect Class

Design-layer regression. The OB-205 data infrastructure (API route, hook, data pipeline) is correct. The design application of that infrastructure violates DS-029 §6 (persona matrix), DS-013 (intelligence stream paradigm), TMR-7 (persona visualization psychology), TMR-8 (cognitive fit), and the Bloodwork principle (OB-102/TMR-3). The carrier data should feed INTO existing intelligence surfaces, not compete with them for real estate.

### 1.3 Combined Treatment Rationale

All four findings share one root cause: the carrier expression components were designed as infrastructure telemetry cards rendered identically across personas, instead of being persona-scoped intelligence that integrates with the existing stream elements. One HF, four findings, one structural fix.

## §2 — SUBSTRATE-BOUND DISCIPLINE

**DS-029 §6 (Persona × Carrier Intelligence Matrix):** Admin sees governance surfaces (system health, pipeline readiness, classification confidence). Manager sees team data enriched by carrier freshness — never import metrics directly. Individual sees personal data confirmation — never entity counts or pipeline state.

**DS-013 (Intelligence Stream):** Carrier data feeds the intelligence stream's existing elements. It does not create a separate carrier section above the intelligence elements. The stream is one unified ranked surface.

**TMR-7 (Persona Visualization Psychology):** Admin = governance cognitive task (indigo ambient). Manager = people development (amber ambient). Individual = self-mastery (emerald ambient). Visualization form, density, and content differ per persona. Identical cards across personas is the violation.

**Bloodwork Principle (OB-102, February 14 session):** The platform presents a health assessment, not a data inventory. Passing checks are silent. Problems get visibility. The vital signs per persona are documented:
- Platform Admin: Cycle Status, Employees Processed, Total Payout (delta vs prior), Data Freshness, Pending Actions
- Manager: Team Attainment (trend + sparkline), Team Payout (delta), Coaching Flags (named people), Top Performer
- Rep: My Payout (prominent, large), Attainment (visual gauge), Rank, Period status

**TMR-8 (Cognitive Fit):** Every visualization form must match its decision task. A pipeline stepper serves a planning task — only appropriate for Admin when the pipeline is blocked. When the pipeline is healthy, the planning task is resolved and the stepper adds no value.

**Reference Frame requirement (CLT-100 A.70–A.74):** Every number answers "is that good or bad?" 595 rows, 85 entities, 77.7% confidence — none of these include a reference frame. The reference frame for row count is "vs prior import." For entity count, "vs expected population." For confidence, "high (>90%) / review (70–90%) / low (<70%)."

## §4 — PHASE 2: REAL ESTATE PRIORITY — CARRIER BELOW INTELLIGENCE

### 4.1 Move carrier cards below existing intelligence elements

In the /stream page, reorder the render so that existing intelligence elements (System Health, Trajectory Intelligence, Optimization Opportunities, Lifecycle stepper, Team Health / Team Performance Grid) render FIRST. The persona-gated carrier cards render AFTER — at the bottom of the Admin view, not the top.

The carrier cards are supplementary context for the Admin. They are not the headline. The headline is payout, lifecycle state, trajectory.

### 4.2 Handle the no-calculation branch

OB-205 made carrier cards the primary surface in the no-calculation branch (when calculation_batches is empty). This is CORRECT for that branch — when no calculation exists, carrier intelligence IS the most relevant content. Keep carrier cards as the primary surface in the no-calculation branch, but ONLY for Admin persona.

For Manager in the no-calculation branch: show a single message card — "Data imported for [entity count] team members. Calculation pending — your team performance will appear here once the admin runs the calculation." No import metrics. No pipeline stepper. One informational card.

For Rep in the no-calculation branch: show a single message card — "Your data has been imported. Your commission statement will appear here once calculation is complete." No import metrics. No entity counts. One personal confirmation.

## §5 — PHASE 3: CARRIER CARD DESIGN CORRECTION

### 5.1 CarrierImportHealth — Bloodwork-style health indicator

Replace the current text-heavy layout with a compact health indicator. The card transforms from a data inventory into a health assessment.

**Current (wrong):**
```
IMPORT HEALTH
595 rows across 2 content units from 7 imports
85 entities (85 with external ID) · Classification confidence: 77.7%
[Review Data] [Import More Data]
```

**Corrected:**
```
DATA HEALTH                                          [status indicator: ● green/amber/red]
595 rows · 85 entities · 2 content units             Last import: [relative time]
Classification: High (78%)                            vs prior: [+N rows / +N entities / —]
[Review Data →]
```

Design changes:
- Title: "DATA HEALTH" not "IMPORT HEALTH" — the user cares about data health, not import mechanics.
- Single status indicator (green/amber/red dot) derived from: green = all data present + confidence > 70% + entities bound. Amber = confidence 50–70% OR missing content units. Red = confidence < 50% OR zero entities.
- Compact single-line layout. Two lines of content maximum.
- Reference frame: "vs prior" shows delta from previous import (from the CarrierIntelligence payload's existing data — `imports.totalBatches > 1` enables comparison). If first import, omit.
- One action button, not two. "Review Data →" expands inline. "Import More Data" is an Admin action available elsewhere — it doesn't need prime real estate on a health card.

### 5.2 CarrierEntityLandscape — Compact integration

Replace the current layout with a single-line entity summary that integrates into the Data Health card as a second line, not as a separate card.

**Current (wrong):**
```
ENTITY LANDSCAPE
👥 85 entities discovered
85 individual
[View Entities]
```

**Corrected:** Entity information folds into the Data Health card as the second line:
```
595 rows · 85 entities (all bound) · 2 content units
```

"View Entities →" becomes a secondary inline link within the Data Health card, not a separate card with its own header. The CarrierEntityLandscape component can render as an inline element within CarrierImportHealth rather than a standalone card.

### 5.3 CarrierPipelineReadiness — Conditional collapse

Pipeline Readiness already gets hidden when all steps are ✓ (Phase 1, §3.3). When it IS visible (at least one step is ○), redesign:

**Current (wrong):** Full-width stepper showing all 5 steps with ✓/○, always visible.

**Corrected:** Compact action-oriented card showing ONLY the blocking step:

```
NEXT STEP                                             [● amber]
Upload your compensation plan to enable calculation.
[Upload Plan →]
```

Or when bindings are incomplete:
```
NEXT STEP                                             [● amber]
3 entities need binding configuration before calculation.
[Configure Bindings →]
```

One blocking message. One action. No five-step stepper. The stepper is infrastructure visualization — the user needs to know what to do next, not which of five abstract steps is incomplete.

## §9A — RESIDUALS

**R1:** The Manager and Rep no-calculation-branch messages (§4.2) are static text. In a future OB, these should become Five Elements intelligence elements that derive content from the carrier (Manager: "Data imported for N team members" where N comes from entity count scoped by visible_entity_ids; Rep: "Your data has been imported — N transactions" where N comes from committed_data scoped by the rep's entity_id). This HF uses static text because the persona-scoped carrier queries don't exist yet — they require DS-027 (RBAC) entity-to-user linking.

**R2:** The Data Health card's reference frame ("vs prior import") requires comparing current import batch row counts against the previous batch. The CarrierIntelligence payload includes `imports.totalBatches` and `imports.latestBatch` but not the prior batch's row count. The reference frame may need a payload extension. If the data isn't available in the current payload, show "vs prior: —" and note the extension needed in the completion report.

**R3:** The status indicator logic (green/amber/red) uses classification confidence thresholds (>70% green, 50–70% amber, <50% red). These thresholds are initial estimates. They should be calibrated against actual tenant data patterns once more tenants are onboarded. Do not hardcode as magic numbers — define as named constants at the top of the component.
