## §1 — PROBLEM STATEMENT

The platform has 128 page routes. Only 17 are navigable. ~56 substantive pages are built but orphaned — unreachable through the agent-nav, bouncing when accessed directly because route guards fail outside the app shell.

The architect completed a capability-map-aligned triage (v3 wireframe) mapping every page to its agent and capability. The decisions: 26 KEEP, 19 ABSORB, 10 DISCARD, 8 PLANNED (future, out of scope).

The nav must restructure from the current flat workspace model to the four-agent commercial model:

| Agent | Verb | Commercial role |
|---|---|---|
| Platform Core | Always on | Included with every agent. Data integration, engine, access, audit, intelligence. |
| Calculation | Calculate | ICM product. Plans, calculate, reconcile, statements, disputes, approvals, pay. |
| Performance | Decide | Intelligence upgrade. Analytics, insights, benchmarking, coaching, simulate, acceleration. |
| Finance | Consolidate | Licensable module. Gated via `tenants.features`. |

**But wiring pages into the nav is necessary and insufficient.** Several KEEP pages have broken data pipelines:

| Page | MIR Status | Data Pipeline Problem |
|---|---|---|
| Adjustments | 🔴 CRITICAL | `disputes` table exists (16 cols, RLS) but page writes to **in-memory Map** — data lost on reload |
| Approvals | 🟠 CRITICAL | `approval-service.ts` uses **in-memory cache + demo seed data** — no DB table |
| Statements | 🟠 CRITICAL | 28 data queries — must verify they resolve against live schema; scoping depends on entity context |
| Audit | 🟠 CRITICAL | 8 code emit sites, **0 rows persisted** — may need table creation + emit-site wiring |
| Insights (×7) | 🟡 SUPPORTING | Queries against `calculation_results` / `entity_period_outcomes` — verify queries resolve |

This OB delivers both: nav restructure AND data-pipeline substantiation per page. Each vertical slice wires the nav AND makes the underlying functionality real.

## §3.2 — PHASE 2: CALCULATION VERTICAL SLICES (MIR-CRITICAL FIRST)

**Dependency:** Phase 1 complete, build green.

Each page: wire into nav + verify/fix data pipeline + confirm real data flows.

### 2A: Dispute Resolution — Credits & Corrections (`/performance/adjustments`) — 🔴

**Nav:** Registered in Phase 1 under Calculation > Dispute Resolution.

**Data Pipeline Gate:**
1. Read the page source. Identify where dispute data is read/written.
2. **Known defect (class: in-memory storage):** The page likely writes to an in-memory Map or local state, not to the `disputes` table in Supabase. The `disputes` table EXISTS with 16 columns (id, tenant_id, entity_id, period_id, batch_id, status, category, description, resolution, amount_disputed, amount_resolved, filed_by, resolved_by, created_at, updated_at, resolved_at) and RLS enabled.
3. **Fix:** Replace in-memory reads/writes with Supabase client queries against the `disputes` table. Use the tenant-scoped client (not service-role) so RLS applies.
4. **Minimum viable flow:** Page loads → shows existing disputes for the tenant (reads from `disputes`) → user can submit a new dispute (INSERT into `disputes` with status='open') → dispute appears in the list → admin can update status to 'resolved'/'rejected' (UPDATE on `disputes`).
5. **EPG:** Submit a test dispute via the page → reload → dispute persists. If it vanishes on reload, the in-memory path is still active.

### 2B: Payout Approvals — Approval Center (`/approvals`) — 🟠

**Nav:** Registered in Phase 1 under Calculation > Payout Approvals.

**Data Pipeline Gate:**
1. Read the page source and `approval-service.ts`. Identify the data source.
2. **Known defect (class: in-memory storage):** approval-service.ts uses in-memory cache + demo seed data. No `approval_requests` table exists in the live schema.
3. **Fix — two options (CC evaluates from the code read and picks the structurally simpler):**
   - **Option A (preferred if no table exists):** Create a lightweight `approval_requests` table: `(id uuid PK, tenant_id uuid FK, request_type text, entity_type text, entity_id uuid, status text DEFAULT 'pending', requested_by uuid, approved_by uuid, metadata jsonb, created_at timestamptz, resolved_at timestamptz)`. Enable RLS with tenant_isolation. Wire the page to read/write this table. The architect will paste the CREATE TABLE SQL into the Supabase SQL Editor — CC authors the migration file and the application-layer queries.
   - **Option B (if approval data can ride on existing tables):** If `calculation_batches.lifecycle_state` already serves as the approval mechanism and the page just needs to query batches with lifecycle_state='pending_approval', wire the page to that query instead.
4. **Minimum viable flow:** Page loads → shows pending approval requests → admin clicks approve → status updates → item disappears from pending.
5. **EPG:** At least one approval request visible after creating test data; approve action persists across reload.

**HALT-SCHEMA-APPROVAL:** If Option A is selected (new table), HALT with the proposed CREATE TABLE SQL. Architect pastes it into Supabase SQL Editor. CC then verifies and continues.
