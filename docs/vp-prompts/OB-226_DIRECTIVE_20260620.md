## Â§1 â€” Three Problems, One OB

### Problem 1: Gray cards survive under Vialuce

`/operate/calculate`, `/perform` (Manager persona), and other pages still show gray card backgrounds. Same root cause as HF-317 â€” Tailwind opacity variants or mid-range utility classes not covered by the CSS safety net. Grep, find, remap under `[data-theme="vialuce"]`. CSS-only.

### Problem 2: Three lifecycle pages doing one page's job

The Calculation workspace has three overlapping pages that fragment the admin's workflow:

| Route | Sidebar label | What it shows |
|---|---|---|
| `/operate` | Lifecycle Cockpit | Pipeline Readiness + ICM summary (STATUS only) |
| `/operate/lifecycle` | Operations Center | Lifecycle stepper + data prep + calc summary + attainment (most ACTIONABLE but buried) |
| `/operate/calculate` | Run Calculations | Plan/period selector + component results + Recalculate (ACTION but disconnected from lifecycle) |

Design intent (CLT-39 D1, OB-102): these should be ONE unified Lifecycle Cockpit where the admin sees lifecycle position AND takes the next action. Primary actions (Import, Calculate, Reconcile) should be GOLD CTAs â€” the most prominent elements.

### Problem 3: Pages serving mock/synthetic data instead of real tenant data

Browser verification reveals pages that LOOK correct but serve fabricated data:

**`/insights/compensation`** â€” Shows "Sarah Chen", "Marcus Johnson", "Emily Rodriguez", "Accelerator Plan", "Tiered Plan", "Basic Plan". None of these exist in any proof tenant. **`/perform` (Manager persona)** â€” Shows KPI cards with real totals ($58,406) but "No team data for this period" below. The manager persona should show team-level performance from calculation_results grouped by the manager's team entities.

**Other pages in the Insights workspace** (Analytics, Performance, Trends, My Team, Sales & Finance) may have the same pattern â€” good-looking pages with synthetic or missing data. CC must audit ALL pages in the Insights and Performance workspaces and identify which ones serve mock data.

## Â§2 â€” Objectives

### Objective A: Gray-Card CSS Elimination

Extend the `[data-theme="vialuce"]` CSS safety net to cover whatever utility patterns produce the remaining gray surfaces. After this: zero gray or dark card backgrounds under Vialuce on any page.

### Objective B: Unified Lifecycle Cockpit

After this OB, the Calculation workspace has two pages, not three:

**Lifecycle Cockpit (`/operate`)** â€” The command center. One page showing lifecycle position with actionable controls:

- **Period Ribbon** â€” selectable periods with status
- **Lifecycle Stepper** â€” full-width, all 9 phases, current phase gold-highlighted, action button per phase
- **GOLD CTA for the current phase's primary action** â€” the most prominent element. Import Data / Run Calculation / Start Reconciliation depending on lifecycle state
- **Data Readiness panel** â€” plan status, entity count, data freshness, import timestamp
- **Calculation Summary** â€” if calculated: total payout (DM Mono), entity count, component breakdown, vs-prior delta. If not: the gold CTA
- **Results Preview** â€” distribution, top/bottom entities, anomaly count. Auto-loaded if results exist

**Results Table (`/operate/results`)** â€” Entity-level drilldown. Already exists. No structural changes.

**Eliminated:** `/operate/lifecycle` â†’ redirects to `/operate`. `/operate/calculate` â†’ redirects to `/operate`.

### Objective C: Real Data Connection

Every page reachable from VialuceSidebar that currently serves mock/synthetic data must be connected to real tenant data from the existing database tables.

**The data sources that exist:**
- `calculation_results` â€” per-entity per-component results with amounts, attainment
- `calculation_traces` â€” component-level calculation details
- `committed_data` â€” raw imported data rows
- `plans` â€” plan definitions with components
- `entities` â€” vendedores/employees with attributes
- `periods` â€” configured calculation periods with lifecycle status
- `calculation_runs` â€” run history with timestamps and status

**For each page serving mock data:**

1. Identify the mock data (hardcoded arrays, static objects, fake names/amounts)
2. Replace with a Supabase query against the real tables above
3. If the page's concept doesn't map to existing data (e.g., a "Sales & Finance" page that requires data the platform doesn't collect), render an `.empty` state with honest context â€” not fake data
4. If the page shows real data for Admin persona but mock for Manager/Rep, connect the persona-specific query (filter by the persona's entity scope)

**The standard for "real data":** The page shows the same numbers that appear in the verified proof tenant calculations. BCL shows $58,406 per period for the current period. MIR shows the MIR calculation results. If a page shows $525,000 or "Sarah Chen" for a BCL tenant, it's mock data.

## Â§3 â€” Constraints

1. **Vialuce theme.** Lifecycle Cockpit layout renders under `useIsVialuce()`. Under Dark/Bliss, existing three-page structure unchanged. Else-branches byte-identical. Real data connections apply to ALL themes (not theme-scoped â€” data should be correct regardless of theme).

2. **Design spec vocabulary.** The cockpit uses `.kpi`, `.card`, `.tbl`, `.pill`, `.insight`, `.empty`, `.btn-gold` (primary actions), `.btn-pri` (secondary). Numbers in DM Mono. Page template with `.page` + `.phead`.

3. **Korean Test (T1-E910).** All new labels from i18n.

4. **No engine changes.** No changes to calculation engine, convergence pipeline, or data model. This OB reads existing data â€” it does not compute new data.

5. **No new API endpoints unless needed.** Prefer client-side Supabase queries (the pattern used across the platform). If a new server action or API route is genuinely needed for a data aggregation the client can't do efficiently, create it and document it.

6. **Route redirects.** `/operate/lifecycle` and `/operate/calculate` redirect to `/operate`. Old bookmarks still work.

7. **Honest empty states over fake data.** If a page concept requires data that doesn't exist (no team assignments for Manager persona, no budget data, no reconciliation history), render `.empty` with context and action guidance. "No team assignments configured. Assign entities to managers in Configure â†’ People." is better than fake leaderboard data.
