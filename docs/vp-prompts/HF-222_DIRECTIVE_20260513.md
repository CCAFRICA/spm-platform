## 1 — Problem Statement

Three structurally co-dependent defects, all surfaced by reading the current production code together rather than in isolation. Combined treatment is required because the three surfaces share a common operative path through convergence binding selection and engine verification. Treating them sequentially or in isolation admits the failure mode of either (a) reverifying the wrong defect class at the wrong layer (HF-218 Component 2 pattern) or (b) fixing a band-aid surface while leaving the structural root open for the next instance (DIAG-003 → HF-145 → HF-186 → HF-218 Component 2 lineage).

### 1.1 — Defect 1: Korean Test violation at `convergence-service.ts:1006-1027`

The `tenantAdaptiveBoundaryThreshold` mechanism contains two developer-stated numerical constants:

- `0.50` — the initial-state anchor value, used as `BOUNDARY_FALLBACK_MIN_SCORE` when fewer than 5 `convergence:dual_path_concordance` signals exist for the tenant
- `5` (named `RECENT_N`) — the window size required before the tenant-adaptive average is operative

Both constants violate AP-25 (Korean Test) and GP-2 (Research-Derived Design) in foundational binding-gate code. Source-comment rationalization ("cold-start anchor (preserves existing operative behavior)") frames the anchor as inherited from prior operative state. No prior operative state exists — `convergence:dual_path_concordance` was an open loop before HF-218 introduced it. The phrase is implementation-side framing, not substrate.

### 1.2 — Defect 2: HF-218 Component 2 verification block at `route.ts:1742-1798`

The calc-time verification block reads `dataByBatch.get(eidBindingRaw.source_batch_id)` as a hard read-filter against the binding's recorded `source_batch_id`. For tenants whose `entity_identifier` binding was learned from a period-agnostic data batch (roster, Plantilla, reference): the recorded `source_batch_id` does not match any period-specific `import_batch_id` value that keys `dataByBatch` when the engine calculates non-binding periods.

Observable consequence (BCL, 2026-05-13 verification): 5 of 6 BCL periods return `metrics = {}` and component output `$0` because `engine:structural_exception` fires on verification failure. The binding is correct (`ID_Empleado` is the right column); the data is present (5 of 6 periods have 85 distinct `ID_Empleado` values populated); the verification's read shape is wrong (looks up by binding's `source_batch_id` instead of by column name across operative-period batches).

### 1.3 — Defect 3: Schema-class root at `convergence_bindings.source_batch_id`

A single field carries two semantically distinct concepts:

- **Learning provenance** — "the import_batch_id of the batch where convergence learned this column's identity" (audit metadata; period-agnostic; written once at binding-write time)
- **Data-location** — "the import_batch_id where data for this binding's column lives now for this calculation" (period-specific; varies per calc run; must be resolved at read time)

The schema collapses both into one field. The DIAG-003 comment at `route.ts:656` names the collapse explicitly. Three downstream read sites compensate via column-name-keyed fallback (`dataByBatch` construction at `route.ts:657-680`; `resolveColumnFromBatch` at `route.ts:836-846`; the DIAG-003-aware lookup in measure resolution at `route.ts:~1140`). One read site does not (HF-218 Component 2 verification at `route.ts:1742-1798`).

This is the structural class of failure. Instance closures at HF-109 (March 16), HF-145 (March 18), HF-186 (April 5) closed the instances but did not close the class. HF-218 Component 2 is the latest instance. Closing the class requires schema separation, not another fallback compensation.

### 1.4 — Why combined treatment

Three reasons:

1. **Defect 1 and Defect 2 share the same locked design document** (HF-218). Component 4b (Defect 1 surface) and Component 2 (Defect 2 surface) were specified, reviewed, and shipped in the same HF. Splitting them into separate HFs reproduces the artificial decomposition that the locked design already collapsed into one — and the original HF-218 produced both defects together for that reason.

2. **Defect 3's class-root closure subsumes Defect 2's instance closure.** Adding column-name-keyed fallback at the Component 2 verification site (the obvious instance fix) would be the fourth instance of the same compensation pattern. Per SR-34 (No Bypass) and the Adjacent-Arm Drift Discipline, the structural-class layer is where this defect must be closed. Defect 2 is best closed by Defect 3's class-root fix, not by a fourth fallback.

3. **The clean-slate proof gate fires against all three together.** Clean-slate BCL + Meridian + CRP re-import + recalc against `_Resultados_Esperados.xlsx` ground truth is the only exit criterion that proves all three are closed simultaneously. Splitting the work means three separate clean-slate cycles, three separate ground-truth reconciliations, and three opportunities for one defect to mask another.

```typescript
// scoredCandidates: array of { column, score, scaleFactor, ... } sorted descending by score
// Distribution-derived distinguishability via candidate-distribution standard deviation:
//   - top candidate's score must exceed the next-best score by a margin equal to
//     the candidate-distribution's standard deviation
//   - the threshold is computed from the candidate distribution itself at the moment
//     of binding; no developer-stated constants
//   - cluster cases (scores nearly equal) refuse to bind and surface convergence gap

function distinctEnoughToBind(scoredCandidates: Array<{ score: number }>): boolean {
  if (scoredCandidates.length === 0) return false;
  if (scoredCandidates.length === 1) {
    // Single candidate: bind only if score is strictly positive
    // (cardinality × intersection > 0 — already a substrate test, not a magnitude threshold)
    return scoredCandidates[0].score > 0;
  }
  // Multi-candidate (N ≥ 2): distribution-derived distinguishability
  const scores = scoredCandidates.map(c => c.score);
  const mean = scores.reduce((a, b) => a + b, 0) / scores.length;
  const variance = scores.reduce((s, x) => s + (x - mean) ** 2, 0) / scores.length;
  const stddev = Math.sqrt(variance);
  // Top score must exceed next-best by at least one standard deviation of the
  // candidate distribution. No fixed multiplier; threshold is the distribution's
  // own dispersion.
  return scoredCandidates[0].score - scoredCandidates[1].score > stddev;
}
```

**Korean Test rationale:** Zero developer-stated numerical constants. Every numerical value affecting the binding decision is computed from the candidate distribution at decision time (`mean`, `variance`, `stddev`). The single comparison `> 0` for the N=1 case is the existing substrate test (cardinality × intersection > 0 is the binding-eligibility floor, not a magnitude threshold) — preserved unchanged.

**Small-N behavior verified algebraically:**

- **N=1:** Bind iff score > 0. (Existing substrate test.)
- **N=2:** stddev = |s₁ − s₀| / 2 (population), top − next = |s₁ − s₀|. Condition |s₁ − s₀| > |s₁ − s₀|/2 holds whenever scores differ at all. Binds correctly at the small-N case that boundary fallback selection most commonly hits.
- **N≥3:** Cluster cases (small dispersion relative to top-next gap) refuse to bind; clear-outlier cases (large gap relative to dispersion) bind. Substrate-coherent across the candidate distribution shape.

**Phase 2.4 EPG includes property tests verifying each of N=1, N=2, N=3 (clustered), N=5 (outlier), and invariance under linear scaling.**
