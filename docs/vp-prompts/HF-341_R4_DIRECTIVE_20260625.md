## §1 — THE DEFECT

### The spatial picture (every arrow is a loss)

```
LLM recognition (rich: "DNI_Vendedor is the seller's national identity document")
    ↓ classification
role term ("identifier@0.97")
    ↓ atom cache
stored term ("identifier@0.97" — the expression is gone)
    ↓ SCI-HC-PATTERN classifier
idRepeatRatio ≤ 1.5 → "target" (a heuristic filling the gap the classification created)
    ↓ entity-id selector
highest confidence identifier wins (0.98 beats 0.97 — the only signal left)
```

The LLM spent 31 seconds analyzing MIR's Ventas_Mayo columns. It recognized that `Folio` is a unique sequential sales-order number and `DNI_Vendedor` is a person's national identity document. That recognition is rich, contextual, and sufficient to distinguish the two. The pipeline then asked the LLM to **classify** each column into a term from a vocabulary — `{identifier, measure, temporal, categorical, name, computed measure}` — and stored that term. The distinction between a transaction number and a person was inside the recognition. It is not inside the classification. The classification destroyed it.

OB-231 designed richer fields — `{characterization, identifies, data_nature, relationships}` — but the atom cache stores `role=identifier@confidence`, and every downstream consumer reads that term. The richer fields either don't survive into the cache or are ignored by code that was built to consume classifications.

### The evidence (from the R3 log)

**10 of 12 Ventas/Cobranza sheets classified as `target` instead of `transaction`** by the deterministic `SCI-HC-PATTERN` classifier:

```
sheet=Ventas_Abril classification=target@85% pattern=entity_targets conditions=[HAS measure,
  NO reference key — entity-level record, HAS temporal but idRepeatRatio=1.00 (<=1.5 —
  per-period target, not events), 4 identifier(s), 3 measure column(s)]
```

`idRepeatRatio=1.00` because `Folio` is 1:1 (as transaction IDs should be). The heuristic reads 1:1 as "per-period target." Target-classified sheets pick the highest-confidence identifier as entity_id → `Folio@0.98` beats `DNI_Vendedor@0.97` by 0.01 → the entity is the transaction, not the seller.

Ventas_Mayo got `transaction` because the LLM was freshly called (`llmCalled=true`) and `HAS reference key` triggered. Ventas_Abril — structurally identical columns, identical roles — got `target` because atoms were CLAIMED from cache (`llmCalled=false`, `known=11/11 novel=0`) and the reference-key detection didn't fire without a fresh LLM call. **Identical data, different classification, different entity-id, based on whether the LLM ran or was cached.**

Meanwhile, every plan PDF says: **"Identificación: DNI (Documento Nacional de Identidad)"** — the plan names the entity. Plan comprehension read it. The sheet classifier never consulted it.

### The registries (every heuristic that exists to compensate for the lost expression)

| # | Registry | What it replaces | Where |
|---|---|---|---|
| C1 | **Column-role vocabulary** `{identifier, measure, temporal, categorical, name, computed measure}` | The LLM's expressed understanding of what each column IS | The LLM prompt → atom cache `role=term@conf` |
| C2 | **`idRepeatRatio ≤ 1.5 → target`** | Whether this is a transaction sheet (which the LLM already knows) | `SCI-HC-PATTERN` |
| C3 | **`HAS/NO reference key → transaction/target`** | Whether this sheet has entity-referencing columns (which the LLM already expressed) | `SCI-HC-PATTERN` |
| C4 | **Highest-confidence identifier wins** | Which identifier is the entity (which the plan + the LLM's expression already know) | entity-id selector in `commit-content-unit.ts` |
| C5 | **Atom cache stores `role=term@conf`** | The LLM's full expression | Atom storage/retrieval |

Every one exists because the first one (C1 — classify into a term) threw away what the LLM recognized. Delete C1 and C2–C5 become vestigial.

### The resolution

**Store the expression, not the classification.** The atom cache stores the LLM's full recognition output — what the column IS, what it identifies, its nature, its relationships — in the LLM's own terms (OB-231's `{characterization, identifies, data_nature, relationships}` design, actually honored). The role vocabulary is not the storage format. It is not the query interface. It is deleted.

**Delete the heuristics that fill the gap.** `idRepeatRatio → target/transaction`, `HAS/NO reference key`, confidence-ranking entity-id — all vestigial once the expression survives. The entity-id is determined by connecting two things that already exist: the plan's stated entity identifier ("Identificación: DNI") and the LLM's expressed column understanding ("DNI_Vendedor identifies the seller"). Convergence connects them — it already binds plan metrics to data columns; binding the plan's entity to the data's entity column is the same operation.

**Structural verification replaces classification.** The pipeline verifies structural properties of the expression (the column exists, it resolves to real values, the plan's stated entity maps to a real column) — never classifies the expression into a term.

## §2 — CONSTRAINTS

- **Decision 158.** Recognize is not classify. The LLM recognizes what each column is. Deterministic code verifies structural properties and passes the recognition through. At no point does a developer-curated vocabulary reduce the recognition to a term.
- **OB-231.** The `{characterization, identifies, data_nature, relationships}` structure was designed to carry the expression. R4 makes the pipeline actually honor it — the atom stores and the downstream consumers read these fields, not a `role` term.
- **Validation Premise Law.** Every check passes the acceptance test. `idRepeatRatio ≤ 1.5` fails it (a developer edits the threshold to admit a new sheet pattern). "The plan names DNI as the entity and a column called DNI_Vendedor exists" passes it (referential resolution against carried reality).
- **Carry Everything (T1-E902).** The LLM's expression carries through the atom cache to every consumer. The cache does not strip, reduce, or classify it.
- **Korean Test.** No column-name list, no role-term vocabulary, no sheet-type enum that a developer extends.
- **C2 (fail-loud).** When the plan names an entity identifier and no data column matches, fail loud. When the LLM's expression is malformed or missing required structural fields, fail loud. Never fall back to confidence ranking or ratio thresholds.

### Eradication proof

**PG-R4-12 (No role-term vocabulary).** Grep: zero `{identifier, measure, temporal, categorical, name}` used as a classification term that gates downstream behavior. The terms may survive as human-readable labels in logs/UI — they must not be the field that entity-id selection, sheet classification, or convergence binding reads to make decisions.

**PG-R4-13 (No idRepeatRatio threshold).** The `SCI-HC-PATTERN` `idRepeatRatio ≤ 1.5 → target` heuristic is deleted. Sheet-type determination either comes from the LLM's expression or is unnecessary.

When atoms are CLAIMED from cache for a structurally identical sheet, the full expression survives — identical sheets produce identical entity-id selections.

1. **Progressive Performance on the expression.** The atom cache is the recognition cache — second-encounter ~$0/~100ms. The fingerprint must key on the full expression (not the deleted role term) so recall still works. Confirm the fingerprint mechanism is compatible with the new storage format.
2. **Sheet-type downstream consumers.** If any surface outside comprehension/entity-id reads `data_type=target` vs `transaction` to make a behavioral decision (e.g., source_date extraction, convergence filtering), that consumer must be identified and either connected to the expression or proven vestigial.

The connection mechanism must be structural (the plan names an entity concept; convergence finds the data column whose expression matches), not a column-name lookup.
