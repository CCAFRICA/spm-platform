## §1 — PROBLEM STATEMENT

**Symptom (production, 2026-06-15 23:20, MIR tenant import):**
```
[AIService] header_comprehension failed: Anthropic API error: 404
{"type":"error","error":{"type":"not_found_error","message":"model: claude-sonnet-4-20250514"}}
```
Anthropic returns **404 not_found** for the model string `claude-sonnet-4-20250514`. The model has been **sunset**. Every live LLM call to that string now fails.

**Mechanism (why it looks like "data isn't importing" rather than an outage).** The AIService wraps adapter calls in a graceful-degradation `catch` that returns a **zero-confidence response** (`confidence: 0, fallback: true`) instead of throwing. So each 404 becomes `HC=null`, `avgConf=0.00`, `cols=0`, and the sheet falls through to `[OB-203] failed_interpretation … class=schema_mismatch`. The failure is **silent and success-shaped** — indistinguishable downstream from a legitimate low-confidence classification.

**Blast radius (sharp, and confirmed by the logs).**
- **FAILING — paths that call the LLM live.** Tier 3 (novel structure) is the path that calls out. The two novel sheets fail: `Cuotas` (`tier=3 … hash=e873a3a1e2e3 … confidence=0`) and `Nómina` (`tier=3 … hash=100fabf22b2a … confidence=0`).
- **SURVIVING — everything the flywheel already recognizes.** Every **Tier 1 fingerprint match skips the LLM entirely** and imports normally: `Cobranza_Enero…Junio`, `Clientes_Nuevos` — all `LLM skipped — Tier 1 match`, classified `transaction@85%` with injected field bindings, untouched by the 404. Progressive Performance is behaving correctly: recognized structures cost nothing and never touch the dead endpoint; only cold-start structures hit it. This is *why* the break is masked where the platform is mature and exposed only on novel input.

**Defect-class lineage — two separable defects, both addressed here:**
1. **Instance:** a hardcoded model string (the env default `process.env.NEXT_PUBLIC_AI_MODEL || 'claude-sonnet-4-20250514'`, per the AUD-001 extraction) is stale because the pinned model was sunset. → Phase 1.
2. **Class (AUD-009):** a provider **hard-failure** (any 4xx/5xx from the model API) silently degrades to a zero-confidence response that masquerades as legitimate low confidence, making infrastructure failures undiagnosable in-product. This class predates this string and will outlive it. → Phase 2.

**PDR-10 connection (surfaced for architect determination, not claimed closed).** PDR-10 ("AI field-mapping confidence = 0%, systemic across all file types and tenants") has the **exact symptom shape** this mechanism produces: any provider-call failure → `confidence:0` → "systemic 0% confidence." It is a live hypothesis that some or all historical PDR-10 reports are provider-call failures degrading silently. This HF does **not** assert it closes PDR-10's root cause (DD-7). It does make the class **distinguishable** going forward (Phase 2), which is the precondition for ever determining PDR-10. The architect should check whether the model sunset (or other provider failures) predate the PDR-10 reports. Recorded as an MC determination item in §5.

## §2 — SUBSTRATE-BOUND DISCIPLINE APPLICATIONS

**AUD-009 (close at the structural-class layer, not the instance).** Fixing only the dead string (Phase 1) leaves the silent-degradation class open — the next sunset, an API-key fault, or a rate-limit all degrade silently again. Phase 2 closes the class with **one general invariant and a loud-failure guard**: *any provider hard-error is surfaced loudly and tagged distinctly from legitimate low confidence.* This is explicitly **not** an enumerated catalog of failure shapes (no special-case for 404) — AUD-009 prohibits the catalog; it requires the single general guard.

**Provider abstraction (Principle 6).** Both fixes live in the AIService / adapter seam. The model string is config; the error-distinction is in the one chokepoint every AI surface flows through. No feature code is touched. Switching providers remains a one-file change (and the *UI-driven* control over that remains the separate DS — §6).

**Progressive Performance (constitutional — note, not change).** Tier 1 fingerprint matches are unaffected by this defect (they skip the LLM). Phase 1 restores the **Tier 3 cold-start path** so novel structures can be recognized again and their fingerprints stored for future $0 recognition. The HF does not alter the recognition curve; it repairs the cold-start entry to it.

**Behavior preservation (DD-7).** Phase 2 must **not** break the graceful-degradation fall-through that keeps imports limping rather than crashing. It **adds** a distinguishing tag and a loud (error-level) log to the hard-failure path; it does not change whether the degraded response is returned or how a *genuine* low-confidence classification flows. Expansion beyond this is out of scope.

### §3.2 — PHASE 2: AUD-009 CLASS CLOSURE (provider hard-failure must be loud + distinguishable)

**One general invariant, loud guard — not a 404 special-case.** In the AIService `execute` `catch` path (Phase 0.1 site), distinguish a **provider hard-failure** (the adapter threw because the provider returned a non-2xx HTTP status — 4xx/5xx, including but not limited to 404 not_found) from a recoverable/parse issue, and surface it:

**3.2.1 — Tag the response distinctly.** When the failure is a provider hard-error, set a structural marker on the degraded response so it is **not** indistinguishable from a legitimate low-confidence classification — e.g., add `providerError: true` (and a short `errorClass` string) alongside the existing `fallback: true, confidence: 0`. The field name is structural (no domain vocabulary). **Behavior-preserving:** still return the degraded response (do not throw, do not change the fall-through); downstream that ignores the flag behaves exactly as before.

**3.2.2 — Make it loud in logs.** Elevate the provider-hard-error log from `warning` to `error` level, with the distinct class and the model string in the message (so a future sunset names itself). Do not change logging for genuine low-confidence or parse paths — those stay as they are, which is what keeps the two distinguishable.

**3.2.3 — Do not break degradation.** Confirm by reading the downstream consumer(s) of the AIService response (the SCI path that currently reads `confidence`) that adding `providerError` does not alter their control flow. If any consumer would need to branch on it, that is a *visibility* enhancement that belongs with the AI Substrate health panel (OB-212 §3.1.H / DS) — **do not** add consumer branching here. This HF makes the failure *honest* (tagged + loud); making it *visible in-product* is the panel's job (§6).

**§3.2 EPG (EPG-2):** (1) **Loud-failure proof** — temporarily set the model to a deliberately invalid string in a local-only probe/test, run one AI call, and paste: the **error-level** log naming the provider error + model, and the response object carrying `providerError: true` (distinct from `confidence:0` alone). Restore the valid string. (2) **Distinguishability proof** — show that a *genuine* low-confidence path (e.g., a real but ambiguous classification, or the documented behavior) still returns `confidence` low **without** `providerError`, so the two are separable. (3) **Non-regression** — paste the consumer read confirming no control-flow change (3.2.3). **HALT-DEGRADE** if making the error distinct would require changing the degradation fall-through or a consumer's branching — stop; that is scope beyond this HF.
