## S1 -- Objective

The plan interpretation prompt in `web/src/lib/ai/providers/anthropic-adapter.ts` must contain ZERO pattern-specific entries when this HF closes. The prompt teaches the LLM the compositional grammar -- four structural primitives and how they compose over ReferenceSource types -- and nothing else about what compensation plans look like or what DAG shapes to emit for specific plan patterns.

The constructor (`intent-constructor.ts`, `constructTree`) is already registry-free and unchanged by this HF. The engine evaluator is already proven and unchanged by this HF. The defect is in the prompt, and the fix is subtraction from the prompt.

---

## S2 -- Constraints

**C1 -- SUBTRACTION-ONLY.** The net line count of the plan interpretation prompt text must decrease. Any addition of a new worked example, a new shape illustration, a new "when a plan says X, emit Y" instruction, or a new pattern-specific guidance block violates this constraint.

**C2 -- GRAMMAR-GENERATIVE COVERAGE.** After subtraction, the prompt must describe:
  (a) The four structural primitives (`banded_lookup`, `arithmetic`, `conditional`, `composed`) and what each one means structurally -- not by compensation-pattern example.
  (b) Every ReferenceSource type the constructor supports: `metric`, `ratio`, `aggregate`, `filtered_aggregate`, `reference`, `constant`, `cross_data`, `entity_attribute`, `prior_component`, `scope_aggregate`. Each described by what it IS -- particularly `reference` (reads a numeric field value from entity data; does not count rows, does not aggregate, reads the value).
  (c) The composition rule: any primitive's operands can themselves be primitives or references; the LLM describes what the plan MEANS using this vocabulary; the constructor builds the DAG.
  If any of (a), (b), (c) are absent because the deleted catalog was carrying their function, grammar-generative descriptions replace them. These are the ONLY permitted additions.

**C3 -- KOREAN TEST.** Every line of post-subtraction prompt text passes the Korean Test: would this instruction work identically if every field name, plan name, and compensation concept were in Korean? A description of what `arithmetic` means structurally passes. "For per-category rates, emit a categorized structure with filtered_aggregate" fails -- it encodes a domain-specific compensation pattern.

**C4 -- NO CONSTRUCTOR CHANGES.** `intent-constructor.ts` is not edited. `intent-transformer.ts` is not edited. No new ReferenceSource types. No new structure shapes. The constructor vocabulary is sufficient; the prompt was the bias source.

**C5 -- NO RESOLVER CHANGES.** Resolver, convergence, engine are untouched.

**C6 -- RECONCILIATION-CHANNEL SEPARATION.** CC reports computed values and structural observations verbatim. CC produces no reconciliation interpretation, no comparison to ground truth, no pass/fail claims on numeric accuracy. The architect holds the anchor values.

**C7 -- AUD-009 PRECLUSION.** The defining test: after the subtraction, could a new compensation pattern (one never seen by any tenant) be correctly interpreted by the LLM using only the grammar-generative prompt? If the answer is "only if we add an example for that pattern," AUD-009 is violated -- the prompt is still a catalog. The answer must be "yes, because the LLM composes from the grammar."

**PG-3 -- LINE COUNT DELTA.** Prompt line count BEFORE: ___. AFTER: ___. Delta must be negative.

**PG-4 -- ZERO PATTERN-SPECIFIC BLOCKS REMAINING.** After subtraction, re-run PG-1 classification on the AFTER prompt. The PATTERN-SPECIFIC count must be zero.
