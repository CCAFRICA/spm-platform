The test, applied to every line of prompt text this HF writes: **if it names or shapes a kind of plan component, it is the registry and must be cut; if it describes only a primitive (a reference type, a structural shape) and the rule that they combine, it is the grammar and is permitted.** The amino-acid principle: a finite fixed alphabet of primitives composes unbounded plans; we do not keep a lookup table of finished plans.

## §1 — Problem Statement

A component that previously reconciled now computes catastrophically wrong on a cache-cold interpretation — the same defect class as the field-identity divergence HF-270 closed, one structural level up. Meridian rule_set `ec5fea64-7d43-43d9-b7a0-60253373a66e`, component c4 (Utilización de Flota): the plan declares it as **(Cargas_Totales_Hub ÷ Capacidad_Total_Hub, clamped at 1.5 in ratio-space) × base bonus** (Senior $800 / Coordinador $450). The cache-cold interpretation collapsed this two-field ratio-times-base into a single pre-computed-rate column reference (`hub_utilization_rate → Tasa_Utilizacion_Hub`), dropping both the loads÷capacity ratio and the × base multiply. Per-entity c4 collapsed to single digits (the bare rate ~0.04–0.09); c4 total = 348 (M1) / 348 (M2) against the plan-correct order of magnitude in the tens of thousands. Every other component on the same run is correct — C1 (Rendimiento) reconciled to 44,000 (M1) / 40,950 (M2) exactly, confirming HF-270's field-identity fix holds.

**Root cause (proven this session across the construction + recognition surfaces).** The construction half is registry-free and already supports the correct composition — verified by reading `web/src/lib/plan-intelligence/intent-constructor.ts` (`constructTree`):
- `ReferenceSource.type: 'ratio'` → `buildReferenceNode` emits `arithmetic divide(numerator_field, denominator_field)`. The loads÷capacity ratio is one `ratio` reference.
- `shape: 'arithmetic', operation: 'multiply'` → `constructArithmetic` over two operands. The × base is arithmetic over `[ratio-operand, constant(base)]`.
- `shape: 'conditional'` with `then`/`else` (via `constructBranchOrOperand`, which accepts a nested `kind:'structure'` operand) → the 1.5 ratio-space clamp is a conditional on the ratio before the multiply.
- The constructor's header states verbatim: *"No closed taxonomy of compensation patterns… the four shape kinds are structural primitives, not enumerated compensation patterns"* and cites AUD-009. It dispatches on four structural primitives; it never asks "is this Fleet Utilization." **The ratio-cap-times-base DAG is constructible by `constructTree` as-is. No constructor change is warranted, and none is taken in this HF.**

The defect originates in the **recognition** half — the `plan_component` prompt in `web/src/lib/ai/providers/anthropic-adapter.ts` (`SYSTEM_PROMPTS.plan_component` + the `buildUserPrompt` `plan_component` case). Its illustrations are a **catalog of named component shapes** (a 1D-band example, a 2D-matrix example, a linear rate × constant example). Concrete worked examples of *kinds of plan* train the LLM to **pattern-match the plan against the nearest enumerated shape** rather than to *describe the structure it reads*. Reading "Tasa = Cargas ÷ Capacidad, capped 150%, × base," the LLM has no enumerated example of that shape and collapses to the nearest one (a single metric × something), emitting a single-field reference. This is the DATA_TYPES registry re-manifesting as a prompt catalog: it only works for the shapes already enumerated, and "all the known shapes are not listed" is its signature failure — identical in kind to the named-DATA_TYPE that exposed the original registry, and identical in kind to the field-divergence HF-270 closed.

**Why this is the locked direction (Decision 158, history-confirmed).** Per the IRA Class A invocation of 2026-05-23, **OPT-TEACH-C (compositional grammar) was ranked above OPT-TEACH-A (more worked examples)** — *"the grammar is generative and does not require a closed taxonomy."* The architect's framing of record: *"focus on the strengths — LLM recognition; Code as Construction,"* and the non-registry test stated then: *"the LLM extracts 'a band lookup with two reference fields, breaks at […], the following cell values' — that's not a named pattern."* Recognition = the LLM naming the *structure it sees* in grammar vocabulary; construction = code building it. The catalog corrupts the recognition half. The fix restores recognition to grammar-description — and pairs it with the proofreading the 2026-05-28 record named as the then-missing piece: *"we have not yet built the verification that catches when a free [compositional] choice is dimensionally incoherent."*

**Defect-class statement (SR-34).** A component's composed structure can be wrong-but-grammar-valid because (i) the recognition prompt teaches by pattern-catalog, inducing the LLM to match the plan to an enumerated shape instead of describing the structure it reads, and (ii) no deterministic check verifies that the composed DAG's reference arity is coherent with the inputs the plan describes. HF-271 closes both faces as one: recognition-side grammar teaching (remove the catalog that induces collapse) + a structural-coherence proofread (catch a residual incoherent composition before persistence).

## §2 — Substrate-Bound Discipline Applications

- **AUD-009 (registry/cherry-pick preclusion):** the prompt rewrite removes the enumerated shape-catalog and replaces it with primitive + composition-rule description. No named compensation pattern survives in prompt text. The proofread checks structural coherence (reference arity vs plan-described inputs), never membership in a list of known shapes.
- **IGF-T1-E910 v2 (Korean Test) / Decision 154:** zero language-specific literals, zero field-name literals, zero plan-component-kind names. Recognition describes structure generically (a value that is one quantity over another is a ratio of two fields; a bound is a conditional clamp on the value it bounds; a payout table across thresholds is a banded lookup). Proofread derives "expected inputs" from the plan's own component description at runtime, never from an enumerated catalog.
- **Decision 158 (LLM recognizes / code constructs):** restores the recognition half to its intended job. `constructTree` (construction) is unchanged — it is already the registry-free "ribosome" that builds whatever the recognized structure specifies.
- **DD-7 (behavior preservation):** components whose structure the LLM already describes correctly (every component on the current run except c4 — C1/C2/C3/C5(safety), and every reconciled BCL/CRP component) must remain byte-identical. The prompt rewrite changes *how the LLM is taught to describe*, not the grammar the constructor consumes; correctly-described components round-trip identically. The proofread only rejects an *incoherent* composition; a coherent one passes untouched. §4 proves byte-identity on BCL and on the non-c4 Meridian components.

### Phase 2 — Recognition-prompt rewrite: grammar description, not pattern catalog (A)

**2.1 — Remove the shape-catalog; install grammar-description instruction.** In `web/src/lib/ai/providers/anthropic-adapter.ts`, rewrite the `plan_component` prompt so the LLM's instructed task is: *describe this component's structure using the grammar.* The instruction teaches ONLY primitives and the composition rule, with NO named-component examples:
- **Reference types (how a value is read):** a value that is a single quantity is a metric reference (name the field); a value that is one quantity divided by another is a ratio reference (name BOTH the numerator and denominator fields — never collapse a divided quantity to a single field); a value summed/aggregated over a group is an aggregate or scope-aggregate reference; a value carried from an earlier component is a prior-component reference.
- **Structural shapes (how values combine):** values combined by an operation are arithmetic (state the operation and the operands); a value that changes by a threshold/condition is a conditional (state the condition, the then-value, the else-value); a bound (cap/floor/limit) on a value is a conditional clamp applied to THAT value, before it is combined further; a payout that varies across one or more graduated thresholds is a banded lookup (give the reference field(s), the break points, and the cell values); independently-computed parts combined into one result are composed (sum / max / min / first-match).
- **The composition rule (the generative core):** these primitives combine freely and recursively to match whatever the plan describes — a component is built by composing primitives, not by selecting a named template. Describe what the plan text actually says; do not match the plan to a known shape.
- Examples, if any remain, illustrate **a single primitive or the composition rule** (the textbook "codon → amino acid", not a gallery of finished proteins) — and contain NO field names, NO currency/threshold values tied to a real plan, NO component-kind name. Any example that depicts a *kind of plan component* is the registry and is prohibited (HALT-3 / EPG).

**2.2 — Preserve the CompositionalIntent schema contract.** The rewrite changes the *teaching*, not the emitted schema: the LLM still emits a `compositional_intent` whose `structure`/`scale`/reference-type shapes are exactly what `constructTree` consumes. No schema field is added or renamed. DD-7: a component the LLM already describes correctly emits the same intent and constructs the same DAG.

### Phase 3 — Structural-coherence proofread (A's proofreading half — the ribosome's exonuclease)

**3.1 — Derive the plan-described input expectation per component.** In `web/src/lib/sci/plan-orchestration.ts`, after `constructTree(ci)` succeeds in `callPlanComponentWithRetry`, the LLM's emitted `compositional_intent` IS the structure it recognized; the constructed DAG is what was built. The proofread compares the two for STRUCTURAL coherence — it does not consult any catalog. Concretely, extract from the constructed DAG (via `extractReferencesFromDAG`) the distinct referenced-field SET and the operation shape, and verify internal coherence against the emitted intent: every `ratio` the intent declared must surface as a `divide` over TWO distinct reference fields in the DAG (a ratio that constructed to a single field, or whose numerator and denominator are identical, is incoherent); every arithmetic the intent declared must surface with its declared operands present. The check is purely structural — "does the built structure contain the references and operations the recognized structure declared" — Korean Test clean (no field literals, no shape names).

**3.2 — Structured failure on incoherence (never silent, never a catalog).** If the proofread finds an incoherent composition (e.g., a declared ratio that collapsed to a single referenced field), raise a typed interpretation error (extend `interpretation-errors.ts` with `structural_coherence_failure`, or map to the existing `cognition_violation`) naming the component and the specific incoherence (declared-ratio-but-single-field), route through the existing retry policy, and on exhaustion record a `failed` `ComponentOutcome` — never persist a structurally-incoherent component. This is the AUD-009/Korean-Test "structured failure on a malformed recognition, never a silent wrong answer," applied to composition structure rather than to a pattern lookup. It rejects incoherence; it does not steer toward any named shape.

## §6A — Residuals

- **Coherence-check completeness:** the Phase-3 proofread verifies declared-ratio-is-two-field (the present incoherence class) and declared-operands-present. It does not yet verify every conceivable structural incoherence (e.g., a banded lookup whose break count mismatches the plan's described tier count, or a cap that constructed in payout-space rather than ratio-space). It is the first coherence assertion, append-only by design (new structural-coherence assertions added as new incoherence classes surface — NEVER a catalog of named shapes, only structural invariants). The cap-space question specifically (ratio-space vs payout-space clamp) is the next likely coherence assertion if a future cold import shows the clamp constructed at the wrong position; flagged, not closed here.
- **Recognition robustness vs prompt-only teaching:** HF-271 teaches grammar in the prompt and proofreads the result. If a future component is recognized incoherently in a way the proofread does not yet assert, it would fail-soft (structured failure → retry → failed component) rather than persist wrong — which is correct behavior, but surfaces as a failed component the architect must inspect. Strengthening recognition (e.g., a recognition self-check the LLM performs) is a possible follow-on, not required for this HF's contract.
- **The 2026-05-28 "dimensional coherence" gap, broader form:** HF-271 closes the structural (arity/operation) coherence face. A fuller dimensional-coherence check (units/scale commensurability across composed operands — the Captación-style incommensurate-field concern) is related but distinct and remains a candidate follow-on; HF-271's proofread is structural, not unit-dimensional.
- **constructTree normalization interaction:** the HF-266 `normalizeCompositionalIntent` pass infers missing shape/kind discriminants before construction. The proofread runs AFTER construction, so it sees the normalized/constructed result; a normalization that silently reshaped an intent is therefore visible to the proofread as a structure, but the proofread cannot distinguish "LLM emitted this structure" from "normalization inferred it." If normalization is ever found to mask an incoherence, the proofread point may need to also inspect the pre-normalization raw intent (HF-266 P2 already logs it). Flagged.
