## §1 — Problem Statement

### 1.1 — The violation, in the rule's own terms

Plan interpretation runs in **two functions** in `web/src/app/api/import/sci/execute-bulk/route.ts`:

- **`executeBatchedPlanInterpretation`** — downloads the plan file, extracts its content by format (PDF/PPTX/DOCX/XLSX), calls `interpretPlan`, bridges to engine format, creates the rule_set, emits comprehension signals.
- **`executePlanPipeline`** — reached via the per-unit dispatch `case 'plan':`. It does the **same process**: obtains the plan file's bytes, extracts content by format using the **same extraction logic**, calls `interpretPlan`, creates the rule_set.

This is "two separate code paths for the same feature." AP-17 requires one. The two are not complementary — they are duplicates of one process. The extraction blocks are line-for-line the same logic (PPTX via the same JSZip slide-text regex; XLSX via the same sheet-walk).

### 1.2 — Why this exists and why prior work did not close it

The batched path was added (HF-130) as a multi-sheet plan path on top of the original per-unit path; the per-unit `case 'plan':` was never removed. The dispatch runs the batched path first and marks its units handled (`handledPlanUnitIds`); the per-unit loop then skips handled units. So the per-unit `case 'plan':` is reached **only when the batched path throws an unhandled exception** — and the batched function returns its known failures as values (download/extraction/AI/save all `return` a failure array, they do not throw), so the per-unit plan path is effectively reached only on an unexpected runtime throw. It is duplicate code that is nearly unreachable — but AP-17 forbids the duplicate **existing**, not merely its reachability.

HF-256 restored the document-plan capability by routing documents into the batched path, but its Phase 6 was written as a **conditional** ("retire the body-transport only if no live consumer") and CC found the body field set/forwarded and therefore **preserved the duplication**. The conditional was the defect: a single-pipeline requirement cannot be gated on the duplicate's current reachability. This HF removes the duplication **unconditionally**.

### 1.3 — Evidence at HEAD (re-confirmed in Phase 2)

Lines from the audited source (`AUD-0014` at `c75ad63d`; re-confirm at current HEAD `5ad076b0`):

- **The duplicate dispatch:** `executeContentUnit` switch, `case 'plan': return executePlanPipeline(...)` (route ~line 2151).
- **The batched dispatch + handled-set + per-unit fallthrough:** route ~lines 1840–1856 (batched runs first; `handledPlanUnitIds`; the per-unit loop `if (handledPlanUnitIds.has(...)) continue`).
- **The duplicate function body:** `executePlanPipeline` (route ~line 2978), including its `documentMetadata.fileBase64`-first / Storage-fallback byte source and its format extraction that duplicates the batched function's.
- **The batched function:** `executeBatchedPlanInterpretation` (route ~line 2734) — the path that is to become the sole plan pipeline.

### 1.4 — What is NOT duplicated (do not touch)

The per-unit `executeContentUnit` switch also routes `target`, `transaction`, `entity`, `reference` to their own pipelines (route ~lines 2144–2153). Those are each the **sole** path for their classification — NOT duplicated, NOT in scope. Only `case 'plan':` is the duplicate. This HF touches the plan arm only.

## §2 — Substrate-Bound Discipline Applications

- **AP-17 is the whole objective.** The end-state is ONE function that interprets plans. The batched function is that one path (it already handles single and multi-sheet, and HF-256 routes documents into it). The per-unit `case 'plan':` duplicate is removed.
- **Mandatory, not conditional.** Unlike HF-256 §3.6, there is no "retire only if unreachable" gate. The duplicate is removed because AP-17 forbids it. (If the architect later wants the body-transport `fileBase64` field itself removed from `analyze-document`/`SCIExecution`, that is a separate transport-cleanup item; this HF removes the duplicate *interpretation path* and its execute-side consumption of the body field.)
- **Resilience folded into the one path, not preserved as a duplicate.** If multi-sheet plan interpretation can fail in a way that per-sheet retry would survive, that resilience belongs **inside** `executeBatchedPlanInterpretation` (isolate a failing sheet, continue), not in a second function. Phase 4 addresses whether any such resilience exists to fold; if none does, the duplicate is simply removed.
- **SR-34 (No Bypass).** Removing the duplicate is the structural close. Leaving it in place but unreachable (the HF-256 outcome) is the bypass this HF reverses.
- **DD-7 (preserve working behavior).** The single-file and multi-sheet plan imports that work today run through the batched path; they must remain byte-identical. The four non-plan pipelines are untouched.

### §3.4 — Phase 4: Remove the duplicate plan path (and fold resilience if Phase 3 found any)

File: `web/src/app/api/import/sci/execute-bulk/route.ts`.

- **Remove the per-unit plan duplicate.** The `case 'plan':` arm of `executeContentUnit` (route ~2151) no longer calls `executePlanPipeline`. Plan units are handled exclusively by `executeBatchedPlanInterpretation`. Because the batched dispatch already marks plan units handled and the per-unit loop skips handled units, the `case 'plan':` arm is removed; any plan unit that somehow reaches the per-unit switch (batched threw) returns a clear failure result rather than re-running a duplicate interpreter — surface the batched failure, do not silently re-interpret.
- **Delete `executePlanPipeline`** (the duplicate function body) once no caller remains.
- **Fold resilience only if Phase 3 found unique behavior.** If Phase 3 found per-sheet isolation worth keeping, add it **inside** `executeBatchedPlanInterpretation` (e.g. on a combined-interpretation failure, retry remaining sheets), so resilience lives in the one path. If Phase 3 found no unique behavior, this is a pure removal.
- **Harden the batched throw path (so removal does not lose safety).** Since the per-unit path was the catch for an unhandled batched throw, ensure the batched dispatch's `catch` (route ~1850) records a clear plan failure result for the affected units rather than silently dropping them — the batched function should not leave plan units unaccounted on an unexpected throw. This replaces the duplicate-as-safety-net with explicit failure reporting in the single path.
