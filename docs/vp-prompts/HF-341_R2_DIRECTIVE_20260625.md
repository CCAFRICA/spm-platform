## §1 — THE DEFECT

**MIR Plan 1 fails at construction on the HF-341 branch.** The `composed:"multiply"` grammar (ADR-2, D3a) correctly taught the LLM to merge the three Plan 1 factors into one component. The LLM emitted:

```json
{
  "shape": "composed",
  "composition": "multiply",
  "children": [
    { "shape": "arithmetic", "operation": "multiply", "operands": [{"kind":"reference","source":{"type":"metric","field":"Monto_Total"}}, {"kind":"constant","value":1}] },
    { "shape": "banded_lookup", "dimensions": [{"reference_field":"Categoria", "reference_source":{"type":"attribute","field":"Categoria"}, "breaks":[]}], "outputs":[0.025] },
    { "shape": "conditional", "condition": {"reference":{"type":"scope_aggregate","field":"Monto_Total","boundary":"DNI_Vendedor","op":"sum"}, "operator":"gte", "threshold":150000}, "then":{"kind":"constant","value":1.25}, "else":{"kind":"constant","value":1} }
  ]
}
```

Child 1 (base amount) and child 3 (accelerator) constructed correctly. **Child 2 (category rate) failed:** the dimension validator requires numeric breakpoints (`breaks`), but `Categoria` is a string key (ALI/BEB/LIM/CPE → 2.5%/2.0%/3.0%/3.5%). The LLM emitted `breaks:[]` because there are no numbers to break on. The validator rejected it: `dimension.breaks is empty (need at least 1 break for 2 bands)`.

**The dimension validator enforces conformance to an anticipated form (numeric breaks) and rejects a correctly-recognized structure (string key → rate).** This is the Validation Premise Law: the validator anticipates one key type (numeric thresholds) and fails on another (categorical string) that the LLM correctly identified. "Add `categorical_lookup` as a second shape" would grow the registry — the next key type (date-range, ordinal, compound) requires a developer to add a third.

**A lookup is a lookup.** It has a reference (the key), it has outputs (the values), and the key's nature is carried from recognition. The distinction between "numeric thresholds defining ordered bands" and "string keys mapping to discrete values" is a construction detail derived from the carried key type, not a shape-level vocabulary distinction that a developer curates.

---

## §2 — THE FIX (subtraction + carry, not a new shape)

**Make the lookup dimension key-type-agnostic.** The dimension carries its key reference (already present: `reference_field` + `reference_source`). The key's nature — numeric or categorical — is **carried from recognition**, not checked against an anticipated type. Construction reads the carried nature and builds the appropriate mapping:

- **Numeric key:** ordered thresholds (`breaks`) partition a continuous range into bands. Outputs map to bands. This is today's `banded_lookup` behavior — unchanged.
- **Categorical key:** discrete values (`keys` / the key set) map directly to outputs. No thresholds, no ordering. The key set comes from the LLM's recognition of the plan (it saw ALI/BEB/LIM/CPE and their rates).
- **Unknown/future key type:** construction that cannot structurally determine how to map the key → **fail loud (C2)**. Never silently default to numeric. Never silently default to categorical.

The validator checks **structural properties**: a dimension has a key reference, has outputs, and the outputs count is consistent with the key structure (bands = breaks + 1 for numeric; entries = keys count for categorical). The validator does **not** check which key type it is against an enumerated set — the key type is a property of the data, not a developer expectation.

**Behavior change is scoped (DD-7):** this revision modifies the dimension representation (to carry both `breaks` and `keys` as alternative key structures), the dimension validator (from numeric-only to key-type-agnostic structural checks), and the intent-constructor's dimension builder (to construct from either). It does not alter the engine's prime_dag evaluator (C6) — the constructed `banded_lookup` PrimeNode already resolves by comparing the input against thresholds or matching against values; the change is upstream in how the intent is constructed into that node.
