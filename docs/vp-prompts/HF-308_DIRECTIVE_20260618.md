## §1 — Problem Statement

OB-201 through HF-307A built a complete theme pipeline: toggle mechanism (OB-201), palette remap (HF-305), design language (HF-306), authoritative OKLCH tokens (HF-307), and OKLCH pipeline migration (HF-307A). The solid-color and opacity fidelity are now correct. What remains is a bounded tail of surfaces that bypass the theme system entirely — inline hex, hardcoded dark gradients, `text-white` on now-light backgrounds, and light-context palette classes that invert wrong under bliss.

These were inventoried honestly across two completion reports (HF-305 G6 and HF-307 G5). This HF remediates them.

**Straggler inventory (authoritative — from CC's own gate evidence):**

|---------|-------|-------------|
| `upgrade` page | 4 | Inline hex or dark gradient utilities |

12 specific occurrences across 6 surfaces. Each needs either an `isBliss` conditional or conversion to a semantic token / theme-aware class.

### Category 2: `text-white` body text (HF-305 G6)

109 occurrences. Dual-use: some are body text on dark surfaces (low-contrast on bliss light backgrounds), some are button labels on colored backgrounds (correctly white in both themes). Cannot globally remap — each occurrence needs classification:
- **Body text on a surface that is now light under bliss** → convert to `text-foreground`
- **Button label on a colored background** → keep `text-white` (correct in both themes)
- **Ambiguous** → use `isBliss` conditional or scoped CSS

### Category 3: Light-context palette inversions (HF-305 G6)

~125 occurrences. Classes like `bg-slate-100`, `text-slate-900`, `text-slate-700`, `from-slate-50`/`to-slate-100` gradients used as light fills / dark text on light elements within the dark app. The HF-305 scale inversion flips these wrong under bliss (light fill → dark, dark text → light). Each needs conversion to a semantic token that resolves correctly in both themes:
- `bg-slate-100` (light fill in dark app) → `bg-muted` or `bg-secondary` (light in both themes)
- `text-slate-900` (dark text in dark app on a light element) → `text-foreground` (dark in both themes)
- `text-slate-700` → `text-muted-foreground`
- Light gradients → semantic gradient tokens or `isBliss` conditional

### Out of scope (deferred)

- **Observatory inline hex** (1,211 across 87 files) — VL-Admin-only, not tenant-facing. Genuinely deferrable.
- **`bg-white/5`, `bg-black/50` opacity decoratives** (128) — visually acceptable in both themes.

## §3 — Execution Guidance

This is a surface-by-surface remediation. The approach for each straggler:

1. **Identify the current literal** (inline hex, hardcoded class, `text-white`)
2. **Determine the semantic equivalent** that resolves correctly in BOTH themes
3. **Verify current-theme parity** — the semantic token's `current` value must match the original literal
4. **Apply** — either replace the class/style, or use `isBliss` conditional if no semantic equivalent exists

For `text-white` classification: if the element's parent background is theme-aware (goes light under bliss), the text needs to flip too → `text-foreground`. If the parent is a colored surface that stays the same in both themes (a primary button, a status badge), `text-white` stays.
