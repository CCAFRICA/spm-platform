## §1 Problem Statement

Meridian c0 (Coordinador variant) has `compositional_intent.scale = {side:"evaluator", value:100}` with ratio-space breakpoints [0.8, 0.9, 1.0, 1.3]. The evaluator multiplies the DAG-computed ratio by `meta.scale=100` (1.1031 → 110.31), compares against 1.3 → TRUE → wrong tier → overpay.

HF-276 tried to fix this by pre-multiplying breakpoints × scale.value. This was structurally unsafe: BCL c1 (Depósitos) also matches the `evaluator + ratio` condition with already-commensurate breaks — the pre-multiplication double-scaled them, regressing BCL. Reverted.

**Root insight:** A DAG-computed ratio (`prime:arithmetic, op:divide`) defines its own space. Both operands come from raw data; the divide produces a 0–N ratio. The breakpoints are declared against that ratio. Both sides are in the same space by construction. There is nothing to scale. The recognizer's `scale.value` declaration is noise for this specific configuration.

Scale exists for the case where a single bound column carries a pre-computed value (e.g., `Pct_Meta_Depositos = 0.58`) that might need reconciliation with percent-space breaks. But when the ratio is computed in the DAG, the computation defines the space — the evaluator should not apply a scale.

**The fix:** In `buildConstantWithScale`, when `scale.side === "evaluator"` AND the compare operand is a DAG-computed ratio (`arithmetic divide`): do not attach `meta.scale` to the constant. The evaluator sees no meta, does not scale the ratio, compares raw ratio against raw breaks.

This is the mirror of HF-274: convergence-side breaks ARE in percent space (85, 90, 95) and the DAG ratio needs scaling up to match, so HF-274 attaches meta.scale. Evaluator-side breaks are in ratio space (0.8, 0.9, 1.0) and the DAG ratio is already there — so omit.

```
============================
Problem: Evaluator-side DAG-computed ratios carry meta.scale from the
recognizer's declaration. The recognizer inconsistently pairs scale.value=100
with ratio-space breaks on some variants (Meridian Coordinador c0) and
scale.value=1 on others (Meridian Senior c0, BCL c1). Construction-time
pre-multiplication is unsafe (HF-276 proved it — fixed one tenant, broke
another). Proofread-style assertions enumerate failure shapes (registry).

Insight: A DAG-computed ratio (arithmetic divide of two raw references)
defines its own space. Both operands are data values; the quotient is a
0–N ratio. Breakpoints are declared against this ratio. There is nothing
to scale. Scale is meaningful only when a single pre-computed column
(e.g. Pct_Meta_Depositos) needs reconciliation — but that column would
not appear as an arithmetic divide in the DAG.

Option A: Omit meta.scale when evaluator-side + DAG divide   [CHOSEN]
  - Scale test: Yes — any tenant, any DAG-computed ratio
  - Korean Test: Yes — structural (scale.side + ratio operand type)
  - No magnitude heuristic, no proofread assertion
  - BCL safe: already-commensurate bands get same result (no meta = no scale)
  - Meridian fixed: Coordinador c0 ratio compared raw against raw breaks
  - Single code path: same buildConstantWithScale, one conditional

Option B: Pre-multiply breakpoints (HF-276)
  - Rejected: proven unsafe across tenants — double-scales BCL

Option C: Proofread assertion on break/scale commensurateness
  - Rejected: enumerates a failure shape (AUD-009 registry pattern)
```

## §3B Phase 2 — Implement

In `buildConstantWithScale` (`web/src/lib/plan-intelligence/intent-constructor.ts`):

When `scale.side === 'evaluator'` AND the compare operand is a ratio (detected the same way HF-274 detects it: `reference_source?.type === 'ratio'`): **do not attach `ConstantScaleMeta`** to the constant node. Emit the breakpoint value as-is, with no `meta.scale`.

The evaluator's OB-200 compare-site only applies scaling when `meta.scale` is present on the constant. No meta → no scaling → ratio compared raw against raw breaks.

The convergence-side branch (HF-274: `scale.side === 'convergence'` + ratio → attach meta.scale) is preserved exactly as-is.

Korean Test: `scale.side` string + `reference_source.type` structural check. No field names, no component names, no tenant identifiers, no magnitude constants.

Commit: `git add -A && git commit -m "HF-277: omit meta.scale for evaluator-side DAG-computed ratios" && git push origin dev`
