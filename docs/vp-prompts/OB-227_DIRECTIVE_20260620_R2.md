## §1 Problem Statement

This OB addresses four defect clusters discovered during CLT-226 browser verification, all of which block the customer-facing experience.

**CLUSTER A — Insights workspace empty/broken (6 of 7 sub-pages).**
The Performance workspace contains seven Insights sub-pages. OB-224 wired drill-through into My Team. The remaining six are broken or empty. The Compensation page (/insights/compensation) was inspected live: it renders a frozen view of October 2025 with no period selector, a 100% donut chart for one plan, a flat unsearchable entity table with a useless "Components: 4" column repeated on every row, no drill-through, and no search or filter. At enterprise scale this page is unusable. The other five pages (Overview, Analytics, Performance, Trends, Sales & Finance) are almost certainly stubs or empty.

The data substrate is rich — BCL has 6 calculated periods, 510 entity_period_outcomes rows, 510 calculation_traces, 120K committed_data rows. Cross-period trajectory, distribution analysis, component cost allocation — all derivable from existing tables. The gap is entirely presentation.

**CLUSTER B — Period mismatch between surfaces.**
The Configure → Periods page (/configure/periods) correctly shows 6 periods: October 2025 through March 2026. The Lifecycle Cockpit (/operate) shows period pills: September 2025 through February 2026. March 2026 is missing; September 2025 appears but doesn't exist in the `periods` table. Two surfaces reading the same tenant's periods produce different results. The Lifecycle Cockpit is either querying a different source, applying a date-range window that clips/shifts the result, or caching stale data.

This is not cosmetic — if the period query is wrong in the Cockpit, it may be wrong in other surfaces too. The Insights PeriodSelector must read from the canonical `periods` table correctly, so this must be fixed before the Insights pages can trust their period data.

**CLUSTER C — No auto-detect period on the Periods page.**
The Configure → Periods page has a "+ Create Period" button for manual creation. The `detectPeriods()` function exists in the codebase (runs during import validation, analyzes source_dates, infers period boundaries with frequency and confidence). The API route `/api/periods/create-from-data` exists. But no button on the Periods page calls them. A new tenant with uploaded data cannot auto-detect periods from their data — they must manually create each period. This is a significant UX gap that was flagged as Critical Blocker B1 in the March 2026 Production Readiness Reconciliation and remains open.

**CLUSTER D — Blank tenant landing page.**
A new tenant admin sees a blank page with "No periods configured" and a single "Configure Periods" button. No orientation, no onboarding guidance, no sense of what the product does or what steps to take. The first impression of the product is an empty screen. The blank state should be an onboarding checklist showing the full journey from empty tenant to first payout.

**Combined-treatment rationale:** All four clusters share the same prerequisite (correct period queries), the same data substrate (entity_period_outcomes, calculation_results, periods, entities), and the same goal (a customer-facing platform that works end-to-end). The period fix unblocks the Insights pages. The auto-detect period unblocks new tenant setup. The onboarding checklist guides the user to the Insights pages. Building them together in one PR produces a coherent experience; building them separately produces fragmented surfaces that may drift.

### §3A.3 Function implementations

**`getCalculatedPeriods(tenantId): Promise<PeriodSummary[]>`**
Read `periods` joined with `calculation_results` grouped by period. For each: COUNT entities, SUM/AVG/MIN/MAX total_payout. Join `calculation_batches` for lifecycle_state. Order by start_date DESC.

**`getPayoutDistribution(tenantId, periodId, binCount?): Promise<DistributionResult>`**
Read payouts for the period. Compute histogram bins (default 10). Compute mean, median, std_dev. Count zero-payout entities.

**`getComponentTotals(tenantId, periodId): Promise<ComponentTotal[]>`**
Read `calculation_results.components` JSONB, flatten, group by component name, SUM amounts. Compute percentage_of_total. Order by total_amount DESC.

**`getEntityTrajectory(tenantId, entityId?): Promise<EntityTrajectory[]>`**
Read `calculation_results` across all periods. Compute delta (N vs N-1), velocity (avg delta last 3), direction. Per DS-015 §6: delta needs 2 periods, velocity needs 3.

**`getPopulationTrend(tenantId): Promise<Array<{ label: string; total: number; avg: number }>>`**
SUM and AVG total_payout per period across all entities.

**`getAttainmentSummary(tenantId, periodId): Promise<...>`**
Read `calculation_results.attainment` JSONB. If data exists: compute distribution. If null: return null (page renders fallback).

**`getEntityTableData(tenantId, periodId, options): Promise<EntityTableResult>`**
Read entity_period_outcomes or calculation_results. Support: `{ search?, sortBy?, sortOrder?, variant?, page?, pageSize? }`. Search = ILIKE on display_name. Compute top_component and delta_prior per entity. Server-side pagination.

**`getTenantOnboardingState(tenantId): Promise<TenantOnboardingState>`**
Query: `rule_sets` (has plan?), `import_batches` (has data?), `periods` (has periods?), `calculation_batches` (has calculations?), `entity_period_outcomes` (has results?). One query per table, simple COUNT. Returns the state needed for the onboarding checklist.
