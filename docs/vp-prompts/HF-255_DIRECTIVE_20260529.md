## §1 — Problem Statement

### 1.1 — Symptom

A clean-slate plan import for a document-format plan (PDF / PPTX / DOCX) fails at the UI with:

```
storagePath required: HF-239 unified import requires Storage transport for plan units
```

(Witness: Meridian tenant `72bee86b-8988-497d-b9b1-c4bba9017525`, whose plan is a document format, not an `.xlsx`.)

### 1.2 — This is a capability REGRESSION, not a missing feature

Any-format ingestion — multi-sheet XLSX, PDF, PPTX, DOCX, CSV, TSV — was a working platform capability (built in OB-133; document plans imported successfully in production, e.g. BCL/CRP plan PDFs with plan interpretation AND signal emission). HF-239 unified the import transport (collapsed `execute`/`execute-bulk` into a single Storage-based path) and, in doing so, regressed the document path. The vision is unchanged: any file, any format, is read. This HF restores that capability within the unified architecture.

### 1.3 — Root cause (evidenced at HEAD; re-confirmed in Phase 2)

Two coupled facts in `web/src/app/operate/import/page.tsx`:

- **The Storage-upload set excludes documents.** Line ~105: `const spreadsheetFiles = files.filter(f => f.rawFile && !f.parsedData.documentBase64);`. Only non-document files are uploaded to `ingestion-raw`. A document-parsed plan (`documentBase64` present) is never uploaded → no Storage path is produced for it.
- **`storagePath` therefore resolves to `undefined`.** The single-path derivation (`Object.values(storagePaths)[0] || undefined`, line ~310) is empty; the empty value is passed as the `storagePath` prop to `<SCIExecution>` (lines ~481–482). `SCIExecution`'s plan branch requires `storagePath` (HF-239) and throws.

The throw in `SCIExecution` is a CORRECT guard catching an empty path — it is not the bug. The bug is upstream: the unification widened transport to Storage for spreadsheets but never widened the client upload to include documents.

### 1.4 — The blast-radius subtlety that makes this NOT a one-line change

`spreadsheetFiles` (line ~105) double-serves:
1. the Storage-**upload** loop (line ~106), and
2. the spreadsheet-only **async processing-jobs** path (line ~140: `if (!isDocument && spreadsheetFiles.length > 0)` → creates `processing_jobs` records consumed by the XLSX worker).

Naively widening line ~105 to `f => f.rawFile` (the surface reading) would route document files into the spreadsheet-specific async-jobs path, which parses tabular data — a new defect. The correct fix DECOUPLES the two concerns: **upload set = all files with `rawFile`; async-process set = non-document files only.**

### 1.5 — Why signals come back automatically (and must still be proven)

Plan signal emission is transport-independent: `emitPlanComprehensionSignals` is called from inside `executeBatchedPlanInterpretation` and `executePlanPipeline` (`web/src/lib/sci/plan-interpretation.ts` lines ~275 and ~556) — the exact functions `execute-bulk`'s `case 'plan'` arm invokes (route line ~415). It is NOT part of `emitFlywheelSignals` (which is the structural-fingerprint flywheel for data units and correctly short-circuits on `!structuralFingerprint`; plans carry no structural fingerprint). Therefore, the moment a document plan reaches the `execute-bulk` plan arm via a valid `storagePath`, it interprets AND emits `comprehension:plan_interpretation` signals — restoring the moat with no new signal wiring. Because this is the moat clause, it is verified by EPG (`[PlanComprehensionEmitter] Emitted N …`), not assumed.

### 1.6 — Server readiness

`execute-bulk`'s plan arm already downloads the file from `storagePath` and `plan-interpretation.ts` extracts by extension (XLSX/PPTX/DOCX/PDF). No server change is required. The fix is client-side transport restoration only.

## §2 — Substrate-Bound Discipline Applications

- **Capability restoration, not new build:** the document path worked pre-HF-239; this HF restores it within the unified single-transport architecture. DD-7: behavior that currently works (XLSX plan import, the document ANALYZE path via `analyze-document`) must be preserved exactly.
- **AP-17 (single pipeline):** the correct end-state is ONE transport — every format uploads to Storage and executes through the `execute-bulk` plan arm. The fix moves toward that (documents join the Storage upload), it does not add a second execute transport. Relaxing the front-door guard to accept `fileBase64` (re-legitimizing the body path) was rejected at ADR for exactly this reason.
- **SR-34 (No Bypass):** the structural cause is the coupled upload/async-process set; the fix decouples them. It does not patch the symptom by special-casing documents at the guard.
- **Principle 1 / AP-5/AP-6 (Korean Test):** the upload decoupling adds no format-name or language literal. `documentBase64` is an existing structural property of the parsed file (set for non-tabular formats by `SCIUpload`), not a hardcoded extension check introduced here. EPG-5 confirms zero new format/language string literals.
- **AP-1 (no data in HTTP bodies):** the `documentMetadata.fileBase64` execute-time body path (`execute-bulk` line ~90, "legacy single-unit fallback") is the residue of pre-HF-239 transport. This HF does not extend it; §6A names it for separate retirement. The ANALYZE-phase base64 to `analyze-document` is a distinct, legitimate path (documents cannot be parsed client-side) and is preserved.
- **Vertical Slice:** import → classify → execute → **signals** → calculate → result, one path, document formats restored, in one PR. Signal emission (EPG-2) and calculation (EPG-7) are part of the slice, not follow-ons.

## §6A — Residuals

- **`documentMetadata.fileBase64` execute-time vestige (Design Gate, separate decision):** `execute-bulk` line ~90 advertises an execute-time `fileBase64` body fallback ("legacy single-unit fallback"). After this HF the document plan path uses Storage transport and this fallback is unused. Retiring it (and the AP-1 body transport it represents) is a separate, named decision — NOT deleted inline in this HF.
- **Upload-once end-state (forward platform item):** the fully-unified design uploads every file to Storage once and has BOTH `analyze` and `execute` read from there — so `analyze-document` would read the file from `ingestion-raw` rather than receiving `fileBase64`. Documents currently transfer their bytes twice (base64 to `analyze-document`, plus the Storage upload this HF adds). Collapsing to a single upload is a larger, separate platform item; this HF restores the capability first.
- **Multi-file plan import:** `Object.values(storagePaths)[0]` takes only the first uploaded file's path. Multi-file imports (and multi-file document plans) remain constrained by this; the long-standing multi-file P1 (F07 / CLT111-F8 / CLT102-F44 / CLT109-F15) is the home for that work. Recorded, not renumbered.
- **Verification dependency:** EPG-1/2/4/7 require a clean-slate import of the document-plan tenant (the witness). Cleanup + live import is an architect-run verification step, not a code change in this HF.
