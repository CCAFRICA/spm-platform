## §1 — Problem Statement

### 1.1 — Symptom

BCL October: the **Ejecutivo** variant's *Captación de Depósitos* component pays $0 for all 72 Ejecutivo entities. The **Ejecutivo Senior** variant's *Captación de Depósitos* reconciles. The two component intents are structurally identical except for the ratio numerator field name (Senior: `depositos_netos`; Ejecutivo: `depositos_actuales`).

### 1.2 — Confirmed root cause (binding mis-resolution; live evidence)

Inspection of the persisted convergence binding for the Ejecutivo Captación component (production rule_set `ebfdc935-b86b-4b67-931d-69a873f3c04e`) shows:

```json
"depositos_actuales": { "column": "Pct_Meta_Depositos", "match_pass": 1, "confidence": 0.9,
  "field_identity": { "structuralType": "measure", "contextualIdentity": "deposits_target_achievement_ratio" } }
"meta_depositos":     { "column": "Meta_Depositos",     "match_pass": 1, "confidence": 0.9,
  "field_identity": { "structuralType": "measure", "contextualIdentity": "deposits_target_amount" } }
```

The numerator reference `depositos_actuales` is bound to `Pct_Meta_Depositos` — an already-computed achievement ratio (value range ~0–1.3) — instead of `Depositos_Nuevos_Netos`, the raw measure the Senior variant's `depositos_netos` correctly resolved to. The denominator is correctly bound to `Meta_Depositos` (currency, ~10^5). The engine therefore computes ratio ≈ 0.95 / 100000 ≈ 0.0000095, which falls below the lowest band, yielding $0. No `scale_factor` is present on the binding (and a scale factor would not fix it — this is a wrong-column error, not a scale error).

The mis-binding has two interacting causes, both in `web/src/lib/intelligence/convergence-service.ts`:

**Cause A — cross-variant column contention under one-column-once.** `extractComponents` flattens all variants into one global component list (HF-243 flatten loop), and `generateAllComponentBindings` runs a single `resolveColumnMappingsViaAI` call over all components with a single shared exclusion map (`boundColumnToField`). The AI prompt instructs "Match each metric field to the best data column. **Each column used at most once.**" The two variants use *different field names for the same conceptual input* (`depositos_netos` vs `depositos_actuales`). Senior's `depositos_netos` claims `Depositos_Nuevos_Netos` first; the one-column-once constraint then forces Ejecutivo's `depositos_actuales` off that column. Because the engine's variant router (HF-119) evaluates exactly one variant per entity, columns are NEVER contended across variants at calculation time — the global exclusion is spurious for variant plans and is what forces the wrong column.

**Cause B — no distribution signal in the mapping prompt.** Once forced off `Depositos_Nuevos_Netos`, the AI chooses among remaining columns using only the prose `contextualIdentity` (`resolveColumnMappingsViaAI` builds each column line as `` `${i + 1}. "${c.name}" (${c.fi.contextualIdentity})` ``). With "deposits_target_achievement_ratio" reading close to "actuales", it lands on `Pct_Meta_Depositos`. The per-column value statistics that would distinguish a 0–1.3 ratio column from a currency-magnitude column are **already in scope** on `measureColumns[].stats` (`ColumnValueStats { min, max, mean, sampleCount }`) but are discarded when the prompt is built. `match_pass: 1`/confidence 0.9 is vacuous here: ratio operands carry no `expectedRange`, so `scoreColumnForRequirement`'s validation passes unconditionally (`isValidated = !req.expectedRange || boundaryScore > 0.1`).

The HF-203 plausibility check did not catch the mis-binding because its rejection is asymmetric: `checkCalculationPlausibility` flags `ratioToMedian > 10` (result too large); this error is ~10^5× too *small*, well under the floor of the detector.

### 1.3 — Combined-treatment rationale

Cause A and Cause B are treated in one HF because neither fix is sufficient alone:

- Fixing A without B: per-variant scoping stops the *forced* mis-bind, but with both `Depositos_Nuevos_Netos` and `Pct_Meta_Depositos` available in the Ejecutivo-scoped call, the AI may still mis-pick `depositos_actuales` on prose similarity. Unreliable.
- Fixing B without A: the distribution signal cannot help while the one-column-once exclusion has already removed the correct column from Ejecutivo's candidate set before scoring. The contention dominates.

Both ship together. Scope is strictly these two causes; no other convergence behavior changes (DD-7 / SR-38).

#### 3.3.2 — Scope the binding pass per variant

In `generateAllComponentBindings`, the single pass over `matches` with one shared `boundColumnToField` and one `resolveColumnMappingsViaAI` call must become per-variant. Group `matches` by `match.component.variantId` (undefined → a single implicit group, preserving exact current behavior for non-variant plans), and for each variant group:

- build the requirements/measure-column inputs for that group,
- call `resolveColumnMappingsViaAI` for that group's metric fields,
- apply bindings with a `boundColumnToField` map that is **reset per variant group**.

This is the DD-2 close at both contention sites: the AI-mapping exclusion and the boundary-fallback `candidates` filter both consult `boundColumnToField`; resetting it per variant group fixes both simultaneously. Do not change the within-group one-column-once semantics (within a single variant, distinct fields still must not share a column — that guard is correct).

Behavior-preservation requirement (DD-7): for a non-variant plan (all `variantId` undefined), the grouped path must produce byte-identical bindings to the pre-HF single-pass path. State this explicitly in the completion report and verify via EPG-3 (Meridian).

#### 3.3.3 — Surface distribution statistics in the column-mapping prompt

In `resolveColumnMappingsViaAI`, the column list is built as `` `${i + 1}. "${c.name}" (${c.fi.contextualIdentity})` ``. Extend each line to include the per-column distribution already carried on `c.stats` (`min`, `max`, `mean`). The AI then has the structural discriminator to avoid binding a ratio-scale column (e.g. 0–1.3) where a currency-magnitude operand is required.

Sketch:

```ts
const columnList = measureColumns.map((c, i) => {
  const s = c.stats;
  const range = s ? ` [min=${s.min}, max=${s.max}, mean=${s.mean.toFixed(2)}]` : '';
  return `${i + 1}. "${c.name}" (${c.fi.contextualIdentity})${range}`;
}).join('\n');
```

Add one neutral, domain-agnostic instruction line to the user prompt directing the AI to use the value ranges as a consistency signal when a metric field participates in a ratio (i.e. prefer a column whose magnitude is consistent with the role the field plays). Phrasing must contain zero field-name literals and zero language-specific tokens (AP-5/AP-6). Keep `maxTokens` adequate for the enriched column list.
