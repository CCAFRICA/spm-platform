**THE GOVERNING CONSTRAINT ‚Äî the unified pathway (AUD-009 + Decision 158, binds the entire HF):**
The platform's pathway is: the LLM **recognizes, defines, and categorizes** a plan's structure from the document and names it in grammar vocabulary; deterministic **code constructs** the DAG from that recognition; **convergence resolves** each named field token against the **REAL DATA COLUMNS** present, by semantic meaning; and a token is rejected ONLY when it maps to **no real column that exists**. This pathway accepts any file, any format, any component, with no registry, precisely because the LLM is free to define and categorize whatever the plan describes, and code's only authority is to construct what was recognized and to fail what corresponds to nothing real. The interpretation-time field-resolution gate (HF-270) broke this pathway by inserting code in the position of REJECTING the LLM's recognized tokens against an ENUMERATED SET (`requiredInputs` on a plan-only import) ‚Äî "all the known fields are not listed," the DATA_TYPES death one level up, the registry's fourth reformation. The cure is to delete that rejection and restore the unified pathway ‚Äî AND to carry the gate's one legitimate function (catching a token that maps to nothing real) to convergence, where "no real column" is actually knowable, landing it as a loud per-component failure rather than the silent $0 it degrades to today. **The test, on every line this HF writes: if it rejects a recognized token by membership in a set that can be incomplete, it is the registry ‚Äî cut it. If it fails a token only because that token maps to NO real data column that exists, it is structural ‚Äî permitted. The landing in ¬ß3 compares against the real columns convergence evaluated, never against `requiredInputs` or any declared/enumerated list.**
