## 1 — INTENT

The architect cannot read a calculation total or component breakdown from the current Vercel log because the [CalcRecon] block (HF-208) emits at handler-exit, below where Vercel's paste delivery truncates. HF-207, HF-208, HF-210 did not solve this.

HF-211 makes the reconciliation summary visible at the **top** of the noisy section by:

1. Emitting a structured `[CalcRecon]` block IMMEDIATELY AFTER the entity loop completes, BEFORE any per-entity forensic trace is flushed
2. Buffering all per-entity `[CalcTrace]` forensic emissions during the entity loop (instead of streaming them)
3. Flushing the buffered forensic trace AFTER the [CalcRecon] block, capped at 5 entities per variant
4. Including per-component totals in the summary (not just grand total)

This is the design the architect specified earlier and that I incorrectly scoped out of HF-209/HF-210 as "overcomplicated." It is correct. Buffering one array and reordering emissions is not complexity — it is the architecturally correct pattern to deliver a top-visible summary.

## 3 — PATCH SCOPE

Two files, both in calc handler scope:

- `web/src/app/api/calculation/run/route.ts` (primary)
- `web/src/lib/calculation/intent-executor.ts` (buffer parameter pass-through only)

### 3.1 — route.ts: Add buffer + per-component total accumulator at handler-scope top

Add immediately after the existing HF-210 cap state (`TRACE_CAP_N`, `tracedEntityIds`, `shouldEmitTrace`):

```typescript
// HF-211: Trace buffer — collects all forensic [CalcTrace] emissions during entity loop.
// Flushed AFTER [CalcRecon] block emits at handler-exit so summary appears above forensics.
const traceBuffer: string[] = [];
function bufferTrace(line: string): void {
  traceBuffer.push(line);
}

// HF-211: Per-component totals accumulator — sum of rounded outcomes per component index.
// Populated at component_complete sites; emitted in [CalcRecon] block.
const componentTotals: Map<number, { name: string; total: number }> = new Map();
```

### 3.2 — route.ts: Convert all wrapped `addLog` `[CalcTrace]` calls to `bufferTrace`

For every existing `if (shouldEmitTrace(entityId)) { addLog(\`[CalcTrace] ...\`) }` block from HF-210, change `addLog` to `bufferTrace`. Keep the `shouldEmitTrace` guard. Pattern:

**Before (HF-210):**
```typescript
if (shouldEmitTrace(entityExternalId)) {
  addLog(`[CalcTrace] resolveColumnFromBatch:exit entity=${entityExternalId} ...`);
}
```

**After (HF-211):**
```typescript
if (shouldEmitTrace(entityExternalId)) {
  bufferTrace(`[CalcTrace] resolveColumnFromBatch:exit entity=${entityExternalId} ...`);
}
```

Apply to ALL 13 wrapped sites identified in HF-210. The unwrapped one-off `[CalcTrace] context` line (route.ts:1088) stays unchanged — it emits via `addLog` as setup context, NOT in the forensic blizzard.

### 3.3 — route.ts: Capture per-component totals at component_complete sites

At every site emitting `[CalcTrace] runCalculation:component_complete`, add a one-line accumulator IMMEDIATELY BEFORE the bufferTrace call. Use the variables already in scope at that site (`componentIdx`, `componentName`, `rounded`).

Pattern:
```typescript
// HF-211: Accumulate per-component total
const _existing = componentTotals.get(componentIdx);
componentTotals.set(componentIdx, {
  name: componentName,
  total: (_existing?.total ?? 0) + (Number(rounded) || 0)
});

if (shouldEmitTrace(entityExternalId)) {
  bufferTrace(`[CalcTrace] runCalculation:component_complete entity=${entityExternalId} componentIdx=${componentIdx} ...`);
}
```

The accumulator runs UNCONDITIONALLY (every entity, every component) — it is the source of per-component totals for the summary. The trace emission stays guarded.

### 3.4 — intent-executor.ts: Accept and use traceBuffer through existing parameter channel

### 3.5 — route.ts: Replace handler-exit emission order

Find the existing handler-exit sequence (HF-208 [CalcRecon] block at ~route.ts:2231 area, HF-207 period_complete/batch_complete after, existing addLog COMPLETE).

Replace with this exact emission order (after the entity loop completes, before any `NextResponse.json` return):

```typescript
// ═══════════════════════════════════════════════════════════════
// HF-211: RECONCILIATION SUMMARY (TOP-VISIBLE — emits BEFORE forensic trace flush)
// ═══════════════════════════════════════════════════════════════

const periodLabel = period?.canonical_key ?? 'n/a';
const reconTotalLookups = entityResults.length * (ruleSet?.components?.length ?? 0);

addLog(`[CalcRecon] ╔═══════════════════════════════════════════════════════════════╗`);
addLog(`[CalcRecon] ║              RECONCILIATION SUMMARY                           ║`);
addLog(`[CalcRecon] ╚═══════════════════════════════════════════════════════════════╝`);
addLog(`[CalcRecon] tenant=${tenantName ?? 'n/a'} period=${periodLabel} ruleSet="${ruleSet?.name ?? 'n/a'}"`);
addLog(`[CalcRecon] batchId=${batch.id} run=${runId ?? 'n/a'}`);
addLog(`[CalcRecon] entitiesCalculated=${entityResults.length} grandTotal=${grandTotal}`);
addLog(`[CalcRecon] ─── Per-Component Totals ───`);
const sortedComponents = Array.from(componentTotals.entries()).sort((a, b) => a[0] - b[0]);
for (const [idx, info] of sortedComponents) {
  addLog(`[CalcRecon]   c${idx} | ${info.name} | total=${info.total}`);
}
addLog(`[CalcRecon] ─── Exception Flags (HF-208 counters) ───`);
addLog(`[CalcRecon]   diag003Fallback=${diag003FallbackCount}/${reconTotalLookups}`);
addLog(`[CalcRecon]   boundaryFallback=${boundaryFallbackCount}`);
addLog(`[CalcRecon]   ob118MergeGuardFired=${ob118MergeGuardFiredCount}/${reconTotalLookups}`);
addLog(`[CalcRecon] ─── Trace Coverage (HF-210 cap) ───`);
addLog(`[CalcRecon]   tracedEntities=${tracedEntityIds.size} cap=${TRACE_CAP_N}`);
addLog(`[CalcRecon] ─── Per-Entity Totals ───`);
for (const r of entityResults) {
  const externalId = ((r.metadata as Record<string, unknown>)?.externalId as string) || (r.entity_id as string);
  const entityName = ((r.metadata as Record<string, unknown>)?.entityName as string) || externalId;
  addLog(`[CalcRecon]   ${externalId} | ${entityName} | total=${r.total_payout}`);
}
addLog(`[CalcRecon] ╔═══════════════════════════════════════════════════════════════╗`);
addLog(`[CalcRecon] ║              END SUMMARY                                      ║`);
addLog(`[CalcRecon] ╚═══════════════════════════════════════════════════════════════╝`);

// ═══════════════════════════════════════════════════════════════
// HF-211: FORENSIC TRACE (buffered during loop, flushed AFTER summary)
// ═══════════════════════════════════════════════════════════════

addLog(`[CalcTrace] ─── FORENSIC TRACE (capped: ${TRACE_CAP_N} entities) ───`);
for (const line of traceBuffer) {
  addLog(line);
}
addLog(`[CalcTrace] ─── END FORENSIC TRACE ─── (${traceBuffer.length} lines)`);

// HF-207 emissions and existing addLog COMPLETE remain AFTER the forensic block per existing handler structure.
```

## 7 — KNOWN CONSTRAINTS

- Cross-file change (route.ts + intent-executor.ts) required to deliver top-visible summary. Single-file scope cannot achieve the directive — intent-executor traces alone produce ~850 lines/period that would obscure the summary if not buffered.
- `traceCollector` parameter is optional with `console.log` fallback for backward compatibility. Production calc passes the buffer; tests/other callers see no behavioral change.
- Buffer holds ~1,000 lines × ~200 chars ≈ 200KB in memory for handler duration. Negligible.
- Per-component accumulator unconditional: O(85 × 4) = 340 Map operations per period. Negligible.
- If a future tenant has more than 4 components, accumulator scales linearly with no code change.
