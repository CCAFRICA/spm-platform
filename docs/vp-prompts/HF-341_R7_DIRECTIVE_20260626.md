## §1 — PROBLEM STATEMENT

### §1.1 — The capability being unlocked

**Dynamic Ingestion** — an enduring engine that accepts any file, any format, any purpose, any calculation, any domain component, any domain objective. The LLM expresses what each column is, what each plan requires, and how they connect. Deterministic code carries the expression, binds it to data, and calculates. MIR (Almacenes Mirasol — 5 plans, 30 sellers, 6 periods, 12 transaction sheets) is the ICM-domain witness that proves this capability works. MIR is not the target. The target is the general property.

### §1.2 — What R6 accomplished

R6 eradicated the construction vocabulary (~2000 lines: `intent-constructor.ts`, `compositional-intent.ts`, `count-metric-discriminator.ts`, 9 registries V1–V9) and the classification producer (~1556 lines: `hc-pattern-classifier.ts`, posteriors/scoring, `seed-priors.ts`, `contextual-reliability.ts`, `weight-evolution.ts`, `workbook-graph.ts`). The LLM now emits PrimeNode DAGs directly. `expression-classifier.ts::deriveClassificationFromExpression` reads only the LLM expression. `initializeTrace` was fixed (it was narrowing trace and dropping `identifies`, silently defeating entity-id across all prior runs).

**Result:** entity_id_field=DNI_Vendedor on ALL 12 transaction sheets. source_dates populated on ALL sheets. Plan 2 EXACT (210,000 × 3 periods). Plan 5 = 0 (no overflow). **Plans 1, 3, 4 fail. Entities = 68 instead of 30.**

### §1.3 — The four fidelity gaps (class-level)

These are NOT four MIR bugs. They are four fidelity gaps in the `LLM expression → convergence binding → calc-time resolution` chain. Each establishes a general property:

**DEFECT A — Resolution Fidelity (Plan 1: grandTotal = 68, should be ~600K)**

Every entity gets total = 1. Two components: c0 = 0 (category commission returns nothing), c1 = 1 (accelerator returns 1.00 as additive standalone). Two sub-defects:

- **A1: Calc-time resolution failure.** Convergence binds `Monto_Total → Monto_Total (reduction=sum)` and `Categoria → Categoria (reduction=snapshot)` — the bindings exist. But `resolveColumnFromBatch` returns no data for these references at calc time. The `data_type` bucket partition, the `row_data` key resolution, the entity_id join, or the period filter is the failure point. **The general property:** a DAG field-reference that binds at convergence time MUST resolve to data at calc time. If it doesn't, the failure class is reported (C2), not silently returned as 0.

- **A2: Composition fidelity.** The accelerator (c1) is emitted as a separate additive component. The plan requires `(base_commission × accelerator_rate)` — one composed multiply, not two additive components. The LLM prompt's `<<PRIME_GRAMMAR>>` section either doesn't teach the single-component-multiply pattern or still defaults to two-component-additive emission. **The general property:** the composition model between components (multiply vs add) comes from the DAG expression, not from a fold default. No component-type→mode registry (Korean Test).

**DEFECT B — Condition Fidelity + Reduction Fidelity (Plan 3: +36–48% overpay)**

Nine sellers who should receive S/0 (gate closed) are paid. Amounts diverge ±40% on remaining sellers.

- **B1: Gate condition.** Plan 3's PDF requires `collection_rate = Monto_Cobrado / Saldo_Pendiente > 70%` as an activation gate. The constructed PrimeNode DAG either does not contain the gate condition, or the condition evaluates against wrong operands. **The general property:** an activation gate expressed in the plan survives into the DAG as a `conditional` prime node. If the plan says "only pay if X > threshold," the DAG contains `conditional(compare(gt, X, constant(threshold)), payout, constant(0))`.

- **B2: Snapshot reduction.** Plan 3's PDF specifies `Saldo_Pendiente = last record per client per month` (a snapshot/last-value derivation). Convergence binds it as `reduction=sum` (the default). The denominator is inflated → collection rate too high → gate opens for sellers it should block. **The general property:** when the LLM expression or the plan semantics specify a snapshot/last-value/point-in-time derivation, convergence binds `reduction=last` or `reduction=snapshot`, not `sum`. The `sum` default does not override an expressed reduction semantic.

**DEFECT C — Reduction Fidelity (Plan 4: grandTotal = 0, should be non-zero)**

Convergence binds `Verificado → sum`. Verificado holds text "Sí"/"No". Sum of text = 0.

- **C1: Conditional count.** The plan requires `count where Verificado='Sí'` × S/150. Convergence should emit `reduction=count` with `filter={Verificado='Sí'}`, not `reduction=sum`. **The general property:** when a plan component counts qualifying items (items matching a text condition), convergence emits a count+filter binding, not a sum. The R1 S4 prompt change for this was attempted but the R6 calc still produces 0 — investigate whether the prompt change reaches this plan's convergence.

- **C2: Plausibility (D121).** A numeric reduction (`sum`) over a non-numeric column should fail loud (C2), not silently return 0. The plausibility check either doesn't exist or doesn't fire. **The general property:** convergence validation detects `reduction=sum|avg|min|max` on a non-numeric column and fails loud with a diagnostic message, not silently.

**DEFECT D — Identity Fidelity (68 entities, should be 30)**

Nómina's `entity_id_field=Nombre_Completo` (name column, not DNI). 34 name-entities + 34 DNI-entities.

- **D1: Plan-informed entity-id on entity sheet.** The plan states `Identificación: DNI`. All 12 transaction sheets correctly derive `entity_id_field=DNI_Vendedor` from the LLM expression. The Nómina (entity sheet) picks `Nombre_Completo` — the LLM's per-sheet recognition prefers the name column on a roster sheet (understandable). **The general property:** the entity-sheet entity-id is informed by the plan's declared entity identifier routed through convergence (D117: import identifies, convergence routes). The plan says "DNI" → convergence connects this to the Nómina column `DNI` → entity resolution uses `DNI`. Per-sheet column prominence does not override the plan's declared identity.

### §1.4 — Why combined treatment (SR-34)

All four defects share one structural class: **developer-curated defaults silently overriding LLM expression.** Additive fold overrides expressed multiply (A2). `sum` default overrides expressed snapshot (B2). `sum` default overrides expressed count+filter (C1). Per-sheet prominence overrides plan-declared identity (D1). Calc-time resolution failure silently returns 0 instead of C2 fail-loud (A1, C2). The class is: the pipeline has structural defaults at layers where the expression should govern. The fix for the class is: remove the defaults so expression governs.

### §3.1 — TIER 1: PROPERTY ESTABLISHMENT (CC determines implementation strategy)

CC implements fixes for all four defect clusters. CC owns the implementation path: file selection, code structure, ordering, commit granularity. The directive constrains WHAT properties must hold at completion, not HOW to achieve them. Each property is a structural invariant — it holds for any tenant, any plan, any domain.

**PROPERTY P-A1: Resolution fidelity.**
A DAG `reference` node whose field is bound at convergence time resolves to data at calc time. If it cannot resolve, the engine emits a diagnostic naming the failure class (data_type mismatch, row_data key mismatch, entity_id join failure, period filter exclusion) — not silently returning 0. **Subtraction target:** whatever default/fallback/silent-path currently allows c0 = 0 despite valid bindings and valid data.

**PROPERTY P-A2: Composition fidelity.**
The composition model between components (multiply vs add) comes from the DAG expression or component metadata, not from a fold default. **Hard boundary (Korean Test):** no component-type→mode registry (`{accelerator: 'multiply', bonus: 'add'}`). **Subtraction target:** the hardcoded additive fold that overrides expressed multiplicative relationships. CC determines the representation (DAG-internal wiring, component-level mode field, or another structural mechanism).

**PROPERTY P-B1: Condition fidelity.**
An activation gate expressed in the plan (e.g., "only pay if ratio > threshold") survives into the DAG as a `conditional` prime node. The condition's operands resolve to data at calc time. **Subtraction target:** whatever construction/interpretation path drops or fails to emit conditional-activation gates.

**PROPERTY P-B2: Reduction fidelity (snapshot/last).**
When the LLM expression or plan semantics specify a snapshot/last-value/point-in-time derivation, convergence emits a non-sum reduction. The reduction algebra supports `last` (returns the value from the row with the latest date per group key) alongside the existing `sum`/`count`/`avg`/`min`/`max`. **Subtraction target:** the `sum` default that silently overrides expressed snapshot semantics. If the reduction algebra needs extension, extend it as a general capability.

**PROPERTY P-C1: Reduction fidelity (count+filter).**
When a plan component counts qualifying items (items matching a text/boolean condition), convergence emits `reduction=count` with a filter predicate. **Subtraction target:** whatever path causes the R1 count+filter prompt change to be unreachable or the binding to fall back to `sum`.

**PROPERTY P-C2: Plausibility — numeric reduction over non-numeric column.**
Convergence validation (D121) detects `reduction=sum|avg|min|max` proposed on a non-numeric column and fails loud with a diagnostic. Does not silently proceed to calc with a binding guaranteed to produce 0. **This is a new guard, not a subtraction** — acceptable because it's a C2 fail-loud obligation that prevents silent 0.

**PROPERTY P-D1: Identity fidelity.**
The entity-sheet entity-id is informed by the plan's declared entity identifier, routed through convergence (D117). The plan declaration is the primary signal; the per-sheet LLM recognition is the fallback (for cases where data is imported before any plan exists). **Subtraction target:** whatever per-sheet heuristic currently overrides the plan's declared identity on entity sheets. No column-name hardcoding.
