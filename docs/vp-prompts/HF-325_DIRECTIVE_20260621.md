## S1 -- Objective

The engine's DAG evaluator currently selects between multiple field-reading paths based on the LLM-emitted node type: `reference/metric` reads the convergence-resolved scalar from `context.metrics[field]`; `aggregate/count` counts rows; `aggregate/sum` re-sums column values. All three paths produce a number for the same field, but only the `reference/metric` path uses the convergence-resolved scalar. The `aggregate/count` and `aggregate/sum` paths re-derive what convergence already computed, and the LLM's node-type emission selects which path runs.

These aggregate evaluation paths predate convergence. They existed when the engine had to aggregate data itself. Since Decision 111 (convergence bindings authoritative), the resolver already reads the column, applies the reduction (`sum`, `snapshot`, `average`, `last`, `count`), and places the resolved scalar in `context.metrics[field]`. The aggregate paths are vestigial: they re-compute a value convergence already holds, and they expose a Decision 158 violation where the LLM's recognition-grade node-type choice determines whether the engine reads the correct value or a wrong re-derivation.

**Proof of the defect class:** BCL Productos Cruzados. Convergence binds `cross_sell_products_sold -> Cantidad_Productos_Cruzados, reduction=sum` for both variants. The resolver reads the correct value (7, 9, 12...) and places it in `context.metrics["cross_sell_products_sold"]`. The Senior variant's DAG says `metric` so the engine reads the convergence scalar (correct: 7 x 25 = 175). The Ejecutivo variant's DAG says `aggregate/count` so the engine counts rows (wrong: 1 x 18 = 18, should be 7 x 18 = 126). The LLM emitted different node types for the same component across variants. The engine faithfully executed both, producing correct results for one and wrong results for the other.

**The fix is subtraction.** When convergence bindings are present (`HF-108 Resolution path: convergence_bindings`), every bound field resolves from `context.metrics[field]` regardless of DAG node type. The node type (`metric`, `aggregate/count`, `aggregate/sum`) no longer selects between redundant evaluation paths for convergence-bound fields. One path. Convergence is authoritative.

## S2 -- Constraints

**C1 -- SUBTRACTION.** The change removes or bypasses evaluation paths, not adds them. No new evaluator, no new discriminator, no new module. The single `context.metrics[field]` read that `reference/metric` already uses becomes the only path for convergence-bound fields.

**C2 -- GATED ON CONVERGENCE BINDINGS.** The subtraction applies only when convergence bindings are present for the field. Non-convergence paths (sheet-matching fallback, `HF-108 Resolution path: sheet-matching`) retain the existing aggregate evaluator behavior unchanged. Fields without convergence bindings are not affected.

**C3 -- CONVERGENCE REDUCTION IS AUTHORITATIVE.** If convergence assigned `reduction=count` (e.g., MIR P4 count-of-verified-clients), the convergence-resolved scalar IS the count. Using the convergence scalar produces the correct answer because convergence already applied the correct reduction. The engine does not need to re-count.

**C4 -- NO PROMPT, IMPORT, CONVERGENCE, OR CONSTRUCTOR CHANGES.** The LLM still emits whatever node type it emits. `constructTree` still builds whatever DAG the LLM specifies. Convergence still binds fields and resolves values. The change is only in the engine's evaluation of bound fields.

**C5 -- RECONCILIATION-CHANNEL SEPARATION.** CC reports computed values verbatim. No GT comparison. No pass/fail accuracy claims.

**C6 -- NON-REGRESSION.** c0, c1, c3 must produce identical values to the current baseline. The fix changes only c2 behavior because c2 is the only component whose DAG contains `aggregate/count` for a convergence-bound field with `reduction=sum`.
