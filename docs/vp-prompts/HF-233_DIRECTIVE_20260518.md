# HF-233: CLASSIFICATION-AWARE ENTITY_ID_FIELD RESOLUTION

Predecessor: HF-231 (PR #409 unified import pipeline), HF-232 (PR #410
reference_key transaction/target discrimination).

Defect: `commitContentUnit.resolveEntityIdField` always consults the HC
`identifier` role. Correct for entity/target files (the identifier IS or
IS ABOUT the entity); wrong for transaction files (the identifier is the
row's event ID; the entity association is the `reference_key`). Post-
HF-232 import of CRP sales produced 389 ghost entities from `transaction_id`
values; engine ran sheet-matching fallback at $150/$200 floor against 421
"entities".

Fix surface (one file, ~15 lines):
- `web/src/lib/sci/commit-content-unit.ts` — `resolveEntityIdField`
  becomes classification-aware:
    entity / target  → HC `identifier` role  (existing behavior)
    transaction      → HC `reference_key` role
    reference        → null (no entity association)
  Structural fallback to `confirmedBindings.entity_identifier`
  preserved for every classification branch.

Full directive text preserved verbatim in the conversation that
initiated HF-233 work, 2026-05-18. This file marks the prompt commit
per standing rule 5.
