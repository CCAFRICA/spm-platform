## 1 -- Problem Statement

Two defect classes, one combined treatment.

**Defect Class A: LLM invents operation names instead of composing existing primitives.**

CRP Plan 4 (District Override) calculates: "District Manager gets 1.5% of their district's total equipment revenue." The LLM emits `{ "operation": "scope_aggregate", ... }`. The executor throws `IntentExecutorUnknownOperationError: Unknown operation "scope_aggregate"` because `scope_aggregate` is not in the executor's operation dispatch.

The concept decomposes to existing primitives: aggregate (sum revenue for the scope) then scalar_multiply (times the rate). The LLM has never been taught that the existing primitives compose. It invented `scope_aggregate` because no prompt example shows aggregation composed with multiplication. This is the same defect class as HF-223: the prompt constrains LLM expression to patterns it has seen, rather than teaching composition principles.

The executor does NOT need a new operation. The LLM needs to express scoped aggregation as a composition: `scalar_multiply { input: { source: 'aggregate', sourceSpec: { metric: 'equipment_revenue', function: 'sum' } }, rate: 0.015 }`. The scope (district vs region) is a data resolution concern handled by convergence binding, not an operation-level parameter.

**Defect Class B: Period autodetect/creation failure for CRP.**

The period creation UI produces a duplicate key error: `Key (tenant_id, canonical_key)=(e44bbcb1-..., 2026-01) already exists`. OB-188 (March 2026) introduced the new canonical_key format `{period_type}_{start_date}_{end_date}` to prevent collisions between monthly and biweekly periods for the same month. Either the old-format canonical keys from a prior CRP setup survive, or the period creation path is generating old-format keys despite OB-188.

Additionally, the period autodetect panel may not be functioning -- the expected six periods for CRP (4 biweekly: Jan 1-15, Jan 16-31, Feb 1-15, Feb 16-28; 2 monthly: January, February) are not present.
