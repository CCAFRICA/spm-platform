## §1 — Problem Statement

The convergence resolver resolves per-entity metrics from convergence bindings. When a bound column is in an entity-keyed batch, the resolver reads the entity's row and returns the value. When a bound column is in NO entity-keyed batch, the resolver returns `null` with reason `column_in_no_batch` and stops.

This is a premature termination. The binding is valid — convergence correctly identified the column. The data exists — committed_data contains the reference sheet rows. The join path exists — comprehension classified a `reference_key` on the entity/transaction sheet linking to the dimensional key on the reference sheet. OB-216 Phase 4's value-overlap discovery already matches these keys. The resolver stops before following the path that's already been identified.

**Proof of the defect:** Meridian c4 (Utilización de Flota). Convergence bound `hub_total_loads → Cargas_Totales` and `hub_total_capacity → Capacidad_Total` to the Datos_Flota_Hub reference sheet (12 hub-month rows, no entity identifier). The resolver returns null for both. c4 = $0 all three periods. GT c4: $34,913 / $35,135 / $36,287 (Q1 total $106,335). c0-c3 are EXACT across all periods (verified post-HF-328).

The Datos_Rendimiento transaction sheet contains `Hub: reference_key@0.95`. The Datos_Flota_Hub reference sheet is keyed by `Hub`. Each entity's transaction row carries the entity's assigned hub value. The reference sheet has exactly one row per hub per month. The join path is: entity → transaction.Hub → reference_sheet.Hub → Cargas_Totales.

**The fix removes the premature termination.** When `column_in_no_batch` fires for a convergence-bound column:

1. Identify which batch contains the column (convergence binding metadata includes the sheet/batch)
2. If that batch is reference-type (no entity key), find the entity's dimensional key value via the `reference_key` column on the entity's data (the value-overlap mechanism OB-216 Phase 4 already uses)
3. Query the reference batch for the row where the dimensional key matches and the period matches
4. Return the bound column value from that row

No new join mechanism. The value-overlap discovery exists. The reference_key classification exists. The resolver gains the capability to follow what recognition and comprehension already produced.

## §2 — Constraints

**C1 — SUBTRACTION OF HARD-STOP.** The `column_in_no_batch → return null` termination is removed for cases where a classified join path exists. No new heuristic, no new discovery mechanism. The value-overlap mechanism (OB-216 Phase 4) and comprehension classifications (reference_key, reference sheet) already exist.

**C2 — KOREAN TEST.** The join uses value-overlap (matching column values between entity data and reference data), not column names, not field names, not language. Decision 154 applies.

**C3 — DECISION 158 COMPLIANT.** The LLM recognized the correct column (`Cargas_Totales` IS hub total loads, conf=0.97). The construction layer follows the join path. No LLM restriction, no candidate filtering.

**C4 — NO ENGINE, PLAN INTERPRETATION, OR SCI COMMIT CHANGES.** The resolver is in the convergence resolution path (`resolveColumnFromBatch` or equivalent). HF-325's engine gate is untouched. HF-328's commit path is untouched. Plan interpretation is untouched.

**C5 — ACTIVEROWSSCOPED GUARD PRESERVED.** HF-325's filtered/scoped aggregate guard must remain operational. MIR Plan 4's filtered count (Verificado=Si) must not regress. The reference-sheet resolution is a column-level resolution (finding the value), not a row-level scoping change.

**C6 — GRACEFUL FALLBACK.** If no reference_key join path exists (no reference_key on entity data, no value-overlap match with the reference batch), the resolver returns null — the same behavior as today. The enhancement fires only when the join path is classified and discoverable.

**C7 — RECONCILIATION-CHANNEL SEPARATION.** CC reports computed values verbatim. No GT comparison.

## §6A — Residuals

- The reference_key join path assumes one-to-one or many-to-one relationship (each entity maps to one dimensional row per period). Many-to-many joins (entity belongs to multiple hubs) are not in scope and would require a different resolution model.
- If a reference sheet contains period-scoped data (Datos_Flota_Hub has Mes/Año columns), the join must also match the period. The resolver should use the same source_date scoping that entity-keyed batches use, or match by the temporal columns in the reference data. CC reads the existing period-scoping logic and applies consistently.
