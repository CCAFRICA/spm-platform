## §3 — Category A: Code Remediation

### Item 3 — Observatory inline-styled panels (1,211 inline hex across 87 files)

Observatory is VL-Admin-only, not tenant-facing. But it should respect the theme when a VL-Admin user has chosen bliss. The same approach HF-308 used for tenant-facing pages applies: convert inline hex backgrounds to theme-aware `--strag-*` vars or semantic tokens; use the `bliss:` Tailwind variant where direct replacement isn't possible.

**Scope constraint:** Observatory panels are heavily inline-styled. Convert background colors and primary text colors. Do NOT attempt to redesign Observatory layouts or restructure components. If an inline style resists clean tokenization, use a `style` attribute with `var(--strag-*)` (current value = exact original hex, bliss value = light equivalent).

Commit in batches by directory/feature area. Build must pass after each batch.

### Item 5 — `test-ds` dynamic persona gradient `text-white` (1 occurrence)

The `test-ds` page has one `text-white` on a dynamic persona-gradient background (HF-308 HALT-2 deferral). Classify it: if the gradient is theme-aware, add `bliss:text-foreground`. If the gradient is a fixed brand color (stays the same in both themes), keep `text-white`. If the page is genuinely dev/test-only and not user-reachable, document that and move on.

### Item 6 — Login page `text-white` (1 occurrence)

One `text-white` on the login page deferred from HF-308. The login page now follows the theme via the `vl-theme` cookie (HF-309). Under bliss, white text on a light background is low-contrast. Add `bliss:text-foreground` or convert to `text-foreground` if the background is theme-aware in both themes.

### Item 7 — shadcn Dialog/Popover/DropdownMenu internals

Audit every shadcn/ui primitive used in the app (`web/src/components/ui/`). Under `data-theme="bliss"`, verify each renders with appropriate light styling. Where shadcn primitives use hardcoded dark values in their component source (not in the app's usage of them), add bliss-scoped CSS overrides in `globals.css`:

```css
html[data-theme="bliss"] [data-radix-popper-content-wrapper] { /* example selector */ }
```

Do NOT fork or edit the shadcn component files themselves if they were installed via `npx shadcn-ui add` — override via scoped CSS. If a primitive was copied and customized (not installed), it can be edited directly.

List every primitive audited and its disposition (correct as-is, overridden, or deferred with reason).

### Item 8 — Per-component spacing/radius polish

Walk the main tenant-facing surfaces under bliss (Stream/Intelligence, Calculate, Reconcile, Manager, Rep) and compare to the marketing reference. Where spacing, border-radius, or shadow values differ noticeably from the marketing design, add bliss-scoped refinements:

```css
html[data-theme="bliss"] .card { border-radius: 10px; }
html[data-theme="bliss"] .some-component { padding: var(--sp-5); box-shadow: var(--shadow-card); }
```

This is refinement, not remediation. Apply only where the gap is visually obvious. Do not attempt pixel-perfect marketing-to-app matching — the layouts differ. Focus on the design-language consistency: pill CTAs, 10px card radius, warm shadows, proper spacing scale.
