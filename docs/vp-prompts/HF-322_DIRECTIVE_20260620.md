---

## S1 — Objective

When the LLM emits a compositional_intent using `aggregate/count` on a named field, and that field contains varying numeric values in the tenant's committed_data, deterministic code must override to `metric` (read the field value directly). The LLM recognized correctly ("$25 per cross-sold product" bound to `Cantidad_Productos_Cruzados`). The LLM constructed wrong (`count` instead of `metric`). The constructor must apply the structural discriminator that the LLM cannot.

This is Decision 158's exact boundary: LLM recognizes, code constructs. The discriminator uses data properties (does the field have varying numeric values?), not compensation-pattern matching. It is not a catalog entry. It passes the Korean Test.

---

## S2 — Constraints

**C1 — STRUCTURAL DISCRIMINATOR, NOT CATALOG ENTRY.** The fix uses data properties from committed_data to distinguish count-of-rows from read-the-value. It does not match on field names, compensation patterns, or plan language. Korean Test: the discriminator works identically for a Korean tenant with Hangul field names.

**C2 — CONSTRUCTION TIME, NOT CALC TIME.** The discriminator runs in `constructTree` or immediately before it, at plan import time. It queries the tenant's committed_data for the named field to determine whether the field contains varying numeric values. If no committed_data exists yet (plan imported before data), the LLM's emission stands unchanged — the discriminator is additive-only when data context is available.

**C3 — NO PROMPT CHANGES.** HF-320's grammar-generative prompt is unchanged. The fix is in the construction layer.

**C4 — NO ENGINE/RESOLVER CHANGES.** The engine evaluates whatever DAG the constructor produces. The resolver is unchanged.

**C5 — ADDITIVE TO CONSTRUCTOR ONLY.** The discriminator is a new check in `intent-constructor.ts` / `constructTree` or the compositional_intent processing path. Existing construction paths for banded_lookup, arithmetic, conditional, composed are unchanged.

**C6 — RECONCILIATION-CHANNEL SEPARATION.** CC reports computed values verbatim. No comparison to architect-held anchors. No pass/fail claims on numeric accuracy.

**C7 — AUD-009.** The discriminator does not maintain a list of "fields that should be metric." It checks the data: does this field, for this tenant, in committed_data, have varying numeric values? That check is complete-by-construction against the actual data, not membership in an enumerated set.

## S6A — Residuals

- If the discriminator succeeds for BCL c2, verify it does not false-positive on fields that legitimately use count (e.g., MIR P4's count-of-verified-clients). The discriminator should NOT override count when the field genuinely contains binary/boolean/status values where counting rows is correct. The data-property check (varying numeric values vs. categorical/boolean) is the structural distinction.
- The discriminator is the first instance of the constructor using data context at construction time. This is a new capability surface. Future refinements may expand it (e.g., detecting ratio fields, detecting categorical fields for filtered_aggregate). Each expansion must pass C1/C7 (structural, not catalog).
