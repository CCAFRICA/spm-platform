## §1 — Problem Statement

OB-201 (PR #541) shipped a CSS-variable theme system: two token sets (`current` | `bliss`), a global `platform_settings` toggle, and server-side `data-theme` on `<html>`. The mechanism works. The result does not.

When `data-theme="bliss"`, the app remains a dark shell with swapped accents. The reason: hundreds of components use hardcoded Tailwind dark-palette classes (`bg-slate-*`, `bg-zinc-*`, `border-zinc-*`, `text-zinc-*`, `navy-*`, inline hex) instead of the semantic token utilities (`bg-background`, `bg-card`, `text-foreground`, `text-muted-foreground`, `border-border`). These bypass the token system.

**The objective is simple: when the toggle is set to bliss, the entire authenticated app must look like the marketing site.** Every background, every card, every sidebar rail, every heading, every border, every button, every chart, every data table — all of it. Not "mostly." Not "accents changed." The full transformation.

When the toggle is set to current, the app must look exactly as it does today. Byte-for-byte. No regression.

**CLT items this closes:**
- **CLT51A-F2** (P0): font too small / low contrast — semantic tokens enforce readable sizes + contrast ratios
- **CLT51A-F21** (P0): sidebar rail text too small and dull — sidebar fully tokenized
- **CLT195-F03** (P1): "View Intelligence" / "Verify Results" are nondescript text links — converted to proper CTA components

## §3 — Visual Target (Authoritative)

The bliss design is defined by these tokens, already shipped in `globals.css` under `[data-theme="bliss"]` by OB-201. The marketing reference screenshots (architect-supplied) are the visual ground truth.

**Colors:**

| Token | Hex | What it does |
|-------|-----|-------------|
| `--background` / `--app-bg` | `#F8F8FF` (offwhite) | Page background — NOT dark, NOT `#0a0e1a` |
| `--card` | offwhite/white | Card and panel surfaces — NOT `bg-zinc-900` |
| `--foreground` / `--app-fg` | `#0F0A1E` (ink) | Primary text — dark on light, NOT white on dark |
| `--muted-foreground` | `#666080` | Secondary text — NOT `text-zinc-500` |
| `--primary` | `#2D2F8F` (indigo) | Headings accent, primary CTAs |
| `--accent` | `#E8A838` (gold) | Accent CTAs, underlines, dots, highlights |
| `--border` | `#E4E4EE` | All borders and dividers — NOT `border-zinc-800` |
| `--muted` | muted surface | Subtle background fills — NOT `bg-white/5` |

**Typography:** Urbanist (display/headings, weights 300/400/700), Inter (body, 400/500/600), DM Mono (labels/badges/eyebrows, 400/500). Google Fonts import already in `globals.css`.

**Signature primitives under bliss:**
- Pill-shaped CTAs: `border-radius: 9999px` on primary action buttons
- Grid-pattern backdrop: 32×32 SVG at 6% indigo opacity on main content area
- Diamond logo in nav (SVG already in OB-201 ADR)
- Mono-label eyebrows: uppercase, `0.22em` tracking, DM Mono
- Gold dots: 0.5rem / 0.875rem circles as list markers
- Hairline dividers: 1px `--border`, 2px `--primary`
- Subtle warm shadows: `rgba(15,10,30,0.06)` not dark glows

**What NOT to convert:**
- Semantic status colors (`emerald` = success, `rose` = error, `amber` = warning) — these carry meaning independent of theme
- Chart accent colors that already use `--chart-1` through `--chart-5` tokens
- Third-party library internals that can't be overridden without forking

## §4 — Constraints

1. **Zero current-theme regression.** `data-theme="current"` must produce the exact same visual output as pre-HF. The `current` token set maps to today's values — so converting hardcoded literals to semantic tokens is safe as long as the semantic token's current-set value matches the hardcoded literal it replaces.

2. **No new components.** Convert what exists. Do not build new pages, new widgets, or new layouts.

3. **No layout/structure changes.** Only color, font, border-radius, shadow, and typography values change. Spacing, flex/grid structure, component hierarchy stay as-is.

4. **No schema/migration/DB work.** Pure presentation layer.

5. **No auth/RLS/session surface.** SR-39 does not fire.

6. **Commit + push after every meaningful batch.** Do not accumulate a single 500-file diff.

7. **Build must pass after every batch.** `npm run build` + `tsc --noEmit`. If a batch breaks the build, fix it before proceeding.
