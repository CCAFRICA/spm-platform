## §0 — CC STANDING RULES HEADER

`CC_STANDING_ARCHITECTURE_RULES.md` binds throughout.

**R6 ENFORCEMENT RULES (binding — these exist because R1–R5 each silently narrowed scope):**

1. **DELETE means the code does not exist in the codebase after R6.** Not commented out. Not behind a flag. Not disconnected from consumers. Not kept as provenance. The function body, the computation, the write — `git diff` shows them removed.
2. **Every item in §1 is a specific deletion target with a grep pattern.** CC runs the grep, finds the code, deletes it. The keep-list in §2 is the only exception path — enumerated by the architect before execution. If a match is not on the keep-list, it is deleted. There is no mechanism for CC to decide a match should be kept. If CC cannot delete a match without breaking the build, CC fixes the build — the match is still deleted.
3. **A test that fails because deleted code no longer produces a value is correct.** Delete or rewrite the test to assert a behavioral outcome (entity-id correct, source_dates populated, sealed figure unchanged). The only regression is HALT-CALC.
4. **"Inert" is not a state.** Code that executes, computes a value, and writes it to storage is live code regardless of whether anything currently reads the output. It will be read again. Delete it.

## §1 — SPECIFIC DELETIONS (CC runs each grep, deletes every match, reports the diff)

### D1 — The pattern classifier function

**What it is:** the function that produces the `[SCI-HC-PATTERN]` log lines. It reads structural heuristics (`idRepeatRatio`, identifier count, measure count, `HAS/NO reference key`) and outputs `classification=target@85%` or `classification=transaction@85%` with a `pattern=entity_targets` or `pattern=event_transactions` label.

**How to find it:**
```bash
grep -rn "SCI-HC-PATTERN" web/src/
```

**Action:** delete the function. Every call site that invokes it: delete the invocation. If a call site needs a return value the function provided, replace it with a read of the LLM's expression (the atom's `{identifies, characterization, relationships}` from R4). If the call site cannot function without a classification label, that call site is a consumer R5 missed — disconnect it from the label AND delete the producer.

### D2 — The posteriors/scoring computation

**What it is:** the code that computes posterior probabilities (`transaction=28%, target=27%, entity=26%`) and selects a winner (`winner=target@85%`). Produces the `[SCI-CRR-DIAG]` and `[SCI-SCORES-DIAG]` log lines.

**How to find it:**
```bash
grep -rn "SCI-CRR-DIAG\|SCI-SCORES-DIAG\|posteriors\|winner.*target\|winner.*transaction" web/src/
```

**Action:** delete the computation. Delete the scoring function. Delete the winner-selection function. These ARE the classification — they take structural features and produce a label. The log lines themselves may be replaced with expression-based diagnostics (what the LLM expressed for this sheet) if CC judges that useful for observability, but the computation that produces `posteriors=[transaction=28%...]` is deleted.

### D3 — The `idRepeatRatio` computation for classification purposes

**What it is:** the code that computes the ratio of rows to distinct identifier values and uses it as input to the classifier. Produces the `[SCI-PROFILE-DIAG]` line including `idRepeatRatio=1.00`.

**How to find it:**
```bash
grep -rn "idRepeatRatio" web/src/
```

**Action:** R5 claimed to delete this. Verify with the grep. If any instance survives in any code path (not just comments), delete it. The ratio as a *structural fact about the data* (logged for diagnostics) is permitted ONLY if it is not read by any function that makes a behavioral decision. If it feeds any `if`/`switch`/comparison, delete the read.

### D4 — The `data_type` write on new imports

**What it is:** the code in `commitContentUnit` (or its callers) that writes `data_type=target|transaction|entity|reference` to `committed_data` when importing new data.

**How to find it:**
```bash
grep -rn "data_type.*=.*target\|data_type.*=.*transaction\|data_type.*=.*entity\|data_type.*=.*reference\|data_type.*classification\|\.data_type\s*=" web/src/
```

**Action:** delete the write. New imports do not produce or persist a `data_type` classification label. Existing persisted values in already-imported data are untouched (they are historical and read by no behavioral path). The `data_type` column in the database schema is NOT deleted (that's a migration, SR-44) — but no application code writes new values to it.

### D5 — Behavioral reads of `data_type` (R5 verification + cleanup)

**What it is:** any `if`, `switch`, `filter`, `===`, `!==`, `.includes()`, or conditional that reads `data_type` to change pipeline behavior. R5 claimed to disconnect these. Verify.

**How to find it:**
```bash
grep -rn "data_type\s*===\|data_type\s*!==\|data_type\s*==\|\.data_type\b" web/src/ | grep -v "\.test\.\|\.spec\.\|__test__\|// \|/\*\|\.log\|console\.\|logger\.\|trace\.\|diag"
```

**Action:** the grep excludes test files, comments, and logging. Every surviving match is a behavioral read. Delete it — replace with a read of the LLM's expression if the code path needs to make a decision, or delete the code path if it's vestigial. Report every match and its deletion in the diff.

### D6 — Classification-label-asserting tests

**What they are:** tests that assert a specific `data_type` value, or that assert classification consistency (cached and fresh produce the same `data_type` label).

**How to find them:**
```bash
grep -rn "data_type.*target\|data_type.*transaction\|classif.*identically\|classif.*consistency\|classification.*assert\|expect.*data_type" web/src/ --include="*.test.*" --include="*.spec.*"
```

**Action:** delete tests that assert a classification label. Rewrite to assert behavioral outcomes:
- Entity-id is correct (the expression's `identifies` scope drives it)
- Source_dates are populated (the expression's temporal recognition drives it)
- Cached and fresh produce the same **expression** (not the same classification)
- Sealed figures unchanged (HALT-CALC)

### D7 — The `SCI-PLAN-WORKBOOK` `hasTx` dependency

**What it is:** the plan-detection log line `hasTx=true|false` which appears to derive from the classification (`hasTx=true` when classification=transaction). Produces `[SCI-PLAN-WORKBOOK]` lines.

**How to find it:**
```bash
grep -rn "hasTx\|has_tx\|hasTransaction" web/src/
```

**Action:** if `hasTx` is derived from the `data_type` classification, delete the derivation. If plan detection needs to know whether a sheet carries transactions, it reads the LLM's expression (does the expression indicate temporal events with entity references?). If `hasTx` is independently computed (not from the classification), it is on the keep-list — leave it.

## §2 — WHAT IS NOT DELETED (explicit keep-list)

- **Fingerprinting** (`[SCI-FINGERPRINT]`) — recognition cache. Keep.
- **Atom storage** (`[OB-203][atom-claim]`) — the expression cache from R4. Keep.
- **Column comprehension** (`[SCI-HC-DIAG]`) — the LLM's column-level expression (`roles=[...]`). Keep. This is the expression, not the classification.
- **The `scope-predicates.ts`** created by R5 — if this reads the LLM's expression to make behavioral decisions, keep. If it reads `data_type`, it's a D5 match — delete the read.
- **The `data_type` column in the database schema** — not deleted (migration = SR-44). But no code writes new values.
- **`data_type` in diagnostic logging** — a log line that prints the historical `data_type` for debugging is permitted. A log line that computes a new classification to print it is D1/D2 — delete.
