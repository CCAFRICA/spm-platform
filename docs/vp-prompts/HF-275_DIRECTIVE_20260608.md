## §1 Problem Statement

Convergence's column-matching (Pass 2–4 in `convergence-service.ts`) selects the best-matching column for each component metric by semantic similarity. When two columns are semantically similar but live on different row populations — one on individual-entity rows (the calculation population) and one on hub/grouping rows (excluded from calculation by HF-263) — the matcher has no population-awareness and may select the excluded-population column.

On Meridian: the component metric `cargas_totales_hub` matched `Cargas_Totales` (hub-only, 0/268 individual rows non-null) over `Cargas_Flota_Hub` (201/268 individual rows non-null) because the names are more similar. The engine then reads null for every individual entity and c4 returns 0.

**The general invariant:** a column that is null on every row in the calculation population is a worse match than a column with values on those rows, regardless of name similarity. This is a quality signal the matcher should consume, not a gate or a rule — it adjusts the confidence score, not a hard filter.

## §3A Phase 1 — Architecture Decision

Commit ADR at `docs/completion-reports/HF-275_ADR.md`:

```
ARCHITECTURE DECISION RECORD
============================
Problem: Convergence column matcher is not population-aware. A column
with values only on excluded rows can win over a column with values on
calculation-population rows.

Option A: Hard-filter columns with 100% null on individual rows
  - Scale test: Yes
  - AI-first: No — removes candidates before AI sees them
  - Risk: a column might be legitimately sparse (new hires, partial data)
  - Rejected: hard filter is a gate, not a quality signal

Option B: Population-null-rate as a confidence adjustment
  - Scale test: Yes
  - AI-first: Yes — AI picks, then confidence is adjusted by data quality
  - Korean Test: Yes — structural null-rate check, no column names
  - Risk: minimal — a column that is 100% null on calc rows genuinely
    cannot produce a value; penalizing it is correct
  - Chosen

Option C: Include population stats in the AI prompt so the LLM decides
  - Scale test: Yes
  - AI-first: Maximally — LLM gets the information
  - Risk: prompt bloat, non-determinism, the LLM may still prefer name
    similarity over data presence
  - Deferred: valid enhancement but higher variance than Option B
```

## §3B Phase 2 — Implement population-aware confidence adjustment

After the AI column mapping produces its candidates (Pass 2/3 output), before writing bindings (Pass 4/5):

For each proposed column match, compute the null-rate on individual-entity rows (rows where `entity_id IS NOT NULL`) for that column. If the column is 100% null on individual rows, reduce the match confidence to below any column that has non-null values on those rows.

The implementation shape: query committed_data for the tenant, partition by entity_id null/not-null, compute per-column null rates, attach as metadata to each candidate. In the binding selection, a column with 100% individual-null-rate loses to any column with <100% null-rate for the same metric slot, even if the 100%-null column has higher name-similarity confidence.

This is a quality signal, not a hard filter. A column at 95% null is penalized proportionally, not excluded. The severity is: `adjusted_confidence = original_confidence × (1 - individual_null_rate)`. A column at 100% null gets confidence 0 and cannot win. A column at 0% null is unaffected.

Korean Test: the null-rate computation references no column names, no component names, no tenant identifiers. It operates on the structural property of the data.
