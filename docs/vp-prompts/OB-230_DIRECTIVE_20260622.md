## §1 — PROBLEM STATEMENT

### The gap

When a user reports a login issue, the platform admin's only recourse today is to ask the user what they see, then run ad-hoc SQL queries against `platform_events` and Supabase's `auth.*` tables in the Dashboard SQL Editor. There is no in-platform surface for user-level observability. The eoadmin/VLADMIN diagnostic session that produced HF-331 required four rounds of manual query, paste, and analysis across two channels — a process that took hours and would be impossible for a non-architect operator.

### What exists today

The Observatory (`/select-tenant` for platform admins) has five tabs: Command Center, Tenant Fleet, Onboarding, AI Intelligence, and Operations. None of them are user-centric. The closest signals are:

**Data already being captured (available without schema changes):**
- `platform_events` — auth events (login success/failure, MFA verify success/failure, session expiry, permission denied, tenant entered, hydration timeout, redirect events). 8 columns including actor_id, event_type, payload (JSONB with IP, user_agent, email, pathname, timestamps). tenant_id is nullable (platform-scope events).
- `profiles` — 11 columns: id, tenant_id (nullable), auth_user_id, display_name, email, role, capabilities (JSONB array), locale, avatar_url, created_at, updated_at.
- `user_journey` — 6 columns: id, user_id, tenant_id, milestone, completed_at, metadata.
- `audit_logs` — tenant-scoped action log (action, entity_type, entity_id, details JSONB, ip_address, user_agent).
- Supabase Admin API (service-role) provides: `auth.admin.listUsers()`, `auth.admin.getUserById()`, `auth.admin.updateUserById()` (ban/unban, metadata), `auth.admin.generateLink()` (password reset, magic link), MFA factor listing and deletion via admin endpoints.

**Data NOT currently captured (signal gaps):**
- No client-side navigation/page-view tracking (which pages the user visited, how long they stayed).
- No client-side error capture (JS errors, unhandled promise rejections the user encounters).
- No explicit "user is experiencing a problem" detection (rage clicks, repeated login attempts within a window, session churn rate).
- No request-level timing (how long API calls take from the user's perspective).
- Session data from `auth.sessions` is not surfaced (active sessions, AAL levels, creation times, last refresh).

### What the industry does

SaaS admin consoles (Auth0 Dashboard, Firebase Console, Clerk Dashboard, Stripe Dashboard) provide per-user detail views with: identity summary, recent activity timeline, active sessions with device/IP/location, MFA factor management, and admin actions (ban, force logout, reset password, impersonate). The best ones (Auth0's Session Management API, Datadog RUM) add behavioral signals: login frequency, failure rates, session duration patterns, and anomaly detection. The diagnostic triage pattern from HF-331 — event timeline → session state → MFA factor state → profile resolution → mechanism verdict — is exactly the workflow these dashboards codify into a visual surface.

### Design principle

**The User Operations Console is a diagnostic instrument, not a dashboard.** Its job is to help a platform admin answer: "What is this user experiencing right now, and why?" It is organized around the triage workflow proven in the HF-331 investigation: identity → auth state → event timeline → session health → actions. The visualizations narrow from broad (user list / search) to specific (single-user deep dive with event replay). Every data point shown must be actionable — either it answers a diagnostic question or it enables a remediation action.

## §2 — SUBSTRATE-BOUND DISCIPLINE APPLICATIONS

**Decision 142 (httpOnly / server-side resolution):** All user-data API routes use the Supabase service-role client server-side. No auth data is read from cookies on the client. The User Operations Console receives data as props/API responses.

**Decision 143 (auth event audit logging):** The console reads from `platform_events` but also writes to it — every admin action (force logout, ban, MFA reset, password reset) is logged as a new event type with the platform admin as actor.

**DS-014 (access control):** The User Operations Console requires `platform.system_config` capability (platform role only). This is not a tenant-admin surface. The route is scoped under the Observatory namespace.

**DS-019 §8 (audit logging):** Admin actions on users are sensitive operations. Per SOC 2 CC6, every admin action must be logged with: who (admin actor_id), what (action type), whom (target user_id), when (timestamp), from where (IP, user_agent), and outcome (success/failure).

**Korean Test (T1-E910 v2):** Event types displayed in the timeline must NOT be hardcoded string matches in the UI. The timeline renders whatever event_type values exist in the data. Classification for visual treatment (color, icon) uses structural heuristics (prefix-based: `auth.*` = auth events, `tenant.*` = navigation events, etc.), not an enumerated registry. New event types added in future HFs appear automatically without UI code changes.

**SR-2 (Scale by Design):** The user list, event timeline, and session queries must paginate. No "load all users" or "load all events" patterns. The console must work with 1,000 users and 100,000 events without performance degradation.

**Progressive Performance (constitutional):** Event queries for a specific user should be fast on repeat access. Consider indexed queries on `platform_events(actor_id, created_at DESC)` — verify this index exists before assuming it.

## §3 — OBJECTIVES (ULTRACODE)

CC determines execution strategy autonomously. The following objectives, constraints, and proof gates define the contract. The only ordering constraint is Objective 1 before Objective 2 for data-dependent UI.

### OBJECTIVE 1: Data Layer — User Operations API Routes

Build server-side API routes (Next.js API routes under `/api/admin/users/`) that aggregate user state from multiple sources into a unified response. All routes require `platform` role verification via service-role client.

**1A: User List / Search endpoint**
Returns paginated user list with composite state. For each user:
- Profile data (from `profiles`): display_name, email, role, tenant_id, capabilities, created_at
- Auth state (from Supabase Admin API `auth.admin.listUsers()` or `auth.admin.getUserById()`): email_confirmed_at, last_sign_in_at, banned_until, created_at
- MFA status (from Supabase Admin API): factor count, any verified factors (boolean), factor types
- Recent event summary (from `platform_events`): last login timestamp, last event type, login failure count in last 24h, session expiry count in last 24h
- Search by email, display_name, or role. Filter by tenant, by auth health status (healthy / MFA incomplete / banned / never logged in / high failure rate).

**1B: User Detail endpoint**
Returns deep state for a single user (identified by auth_user_id or profile id):
- Full profile row
- Full auth.users row (via admin API)
- All MFA factors with status and created_at
- Active sessions with AAL level, created_at, updated_at (via service-role query against `auth.sessions` — verify table accessibility)
- Event timeline: last 200 `platform_events` rows for this actor_id, ordered by created_at DESC, with full payload
- `user_journey` milestones
- `audit_logs` entries where this user is the actor (last 50)

**1C: Admin Action endpoints**
Each action logs to `platform_events` with event_type prefixed `admin.user.*`:
- `POST /api/admin/users/[id]/force-logout` — calls `auth.admin.signOut(userId, 'global')` (or equivalent session revocation). Logs `admin.user.force_logout`.
- `POST /api/admin/users/[id]/ban` — calls `auth.admin.updateUserById(id, { ban_duration: ... })`. Logs `admin.user.ban`.
- `POST /api/admin/users/[id]/unban` — calls `auth.admin.updateUserById(id, { ban_duration: 'none' })`. Logs `admin.user.unban`.
- `POST /api/admin/users/[id]/reset-password` — calls `auth.admin.generateLink({ type: 'recovery', email })`. Returns the link (admin delivers to user via their own channel). Logs `admin.user.password_reset`.
- `POST /api/admin/users/[id]/reset-mfa` — calls admin MFA factor deletion endpoint for all factors. Logs `admin.user.mfa_reset`. User will be prompted to re-enroll on next login.
- `POST /api/admin/users/[id]/resend-confirmation` — calls `auth.admin.generateLink({ type: 'signup', email })` for unconfirmed users. Logs `admin.user.resend_confirmation`.

**Constraints:**
- Every admin action route checks the calling user's role is `platform` before proceeding.
- Every admin action writes to `platform_events` before returning the response — the audit trail is not optional.
- Error responses include the Supabase error detail so the admin sees why an action failed.
- No direct SQL against `auth.*` tables from API routes. Use the Supabase Admin JS SDK (`@supabase/supabase-js` with service role key). The one exception: `auth.sessions` may require a service-role query if the Admin API does not expose session listing — verify and document which path is used.

### OBJECTIVE 2: Rendered Experience — User Operations Console

A new tab in the Observatory: "Users" (or "User Operations"). Platform admin only. Two views: **User List** (search + filter + summary cards) and **User Detail** (deep-dive diagnostic).

**2A: User List View**
The entry point. A searchable, filterable, paginated list of all users across all tenants.

Per-user row or card shows at minimum:
- Avatar/initial + display name + email
- Role badge + tenant name (or "Platform" for null tenant_id)
- Auth health indicator: a single-glance signal synthesized from multiple data points. Examples: green (healthy — recent login, MFA verified, no failures), amber (attention — high failure rate, MFA not enrolled, long time since login), red (problem — banned, repeated failures, hydration timeouts). The synthesis logic uses structural rules, not an enumerated list of statuses.
- Last login timestamp + relative time ("3 hours ago")
- Device/browser from last login's user_agent (parsed, not raw string)
- Quick actions: Force Logout, Ban/Unban (inline, with confirmation)

Search: real-time filter by email or name. Filters: by role, by tenant, by auth health status, by "has logged in within N days."

**2B: User Detail View**
Reached by clicking a user row. The diagnostic deep-dive. Organized in panels that mirror the HF-331 triage workflow:

**Panel 1 — Identity Card.** Profile data + auth data side by side. Name, email, role, tenant, capabilities, created_at, email_confirmed_at, last_sign_in_at, banned_until. MFA factors listed with status and enrollment date. This is the "who is this person and what is their account state" panel.

**Panel 2 — Session Health.** Active sessions listed with: session ID (truncated), AAL level, created_at, last refreshed (updated_at), device/browser (parsed user_agent), IP address. Admin can force-terminate individual sessions or all sessions. A session with aal1 when the role requires aal2 is flagged visually — this is the "stuck at MFA" signal.

**Panel 3 — Event Timeline.** The centerpiece diagnostic tool. A reverse-chronological timeline of all `platform_events` for this user. Each event rendered as a timeline node with:
- Timestamp (absolute + relative)
- Event type (with visual treatment derived from prefix: `auth.*` = blue, `tenant.*` = purple, `admin.*` = orange, error/failure events = red, success events = green — heuristic, not enumerated)
- Payload details expanded inline: IP, user_agent (parsed), pathname, error details
- Anomaly highlighting: rapid re-login sequences (multiple `login.success` within 2 minutes = "user was struggling"), `hydration_timeout` events, `mfa.verify.failure` events, `permission.denied` events. These are highlighted with a visual marker, not hidden in the timeline.
- Time gaps between events shown visually (a wide gap = user was away; a compressed cluster = user was rapidly retrying)
- Filterable by event type prefix, by time range, by device/browser

**Panel 4 — Admin Actions.** The remediation surface. All actions from Objective 1C rendered as buttons with confirmation dialogs and immediate feedback. Each action logs to `platform_events` and the timeline updates to show the admin action.

**Panel 5 — Journey & Activity.** `user_journey` milestones rendered as a stepped progress indicator. `audit_logs` entries (tenant-scoped actions the user performed: imports, calculations, approvals) rendered as a secondary timeline or activity feed. This answers "what has this user been doing in the platform" vs. Panel 3 which answers "what has the platform been doing to this user."

**Design constraints:**
- Follow DS-003 Visualization Vocabulary: Observatory is the Platform Admin (Indigo Environment). Use the Observatory color palette and component vocabulary already established.
- Follow the frontend-design skill: no templated defaults. The timeline is the signature element — it must be scannable, information-dense, and diagnostic-first (not decorative).
- The event timeline must handle 500+ events without performance degradation. Virtual scrolling or pagination, not DOM-dump.
- Device/browser parsing: use a lightweight UA parser (or write a simple one — CC decides). Raw user_agent strings are never shown to the admin; always parsed to "Chrome 149 / Windows 10" or "Firefox iOS 152 / iPhone iOS 18.7" format.
- All admin action buttons require a confirmation step (modal or inline confirm). No single-click destructive actions.
- The console must be responsive down to tablet. Full mobile is not required (platform admins work on desktop/tablet).

### OBJECTIVE 3: Signal Enrichment (may parallel Objective 2)

Enhance the existing `platform_events` instrumentation to capture signals that are currently missing but are required for the console to be maximally useful.

**3A: Client-side error capture.**
Add a global error boundary and `window.onerror` / `unhandledrejection` handler that logs to `platform_events` via the existing client-side `/api/auth/log-event` route (or a new general-purpose event logging route). Event type: `client.error.unhandled`. Payload: error message, stack trace (truncated to 2000 chars), pathname, component name if available. This surfaces JS errors the user encounters that they may not report.

**3B: Session churn detection.**
Add server-side logic (in middleware or a utility) that detects and flags rapid session creation patterns. If a user creates 3+ sessions within 5 minutes, emit a `platform.user.session_churn` event to `platform_events`. This is the "user is struggling to log in" signal that currently requires manual query to detect.

**3C: Navigation breadcrumb events.**
Add lightweight client-side route-change tracking. On each Next.js route transition, log to `platform_events`: `navigation.route_change` with pathname, previous pathname, and timestamp. This is NOT analytics — it is diagnostic breadcrumbing so an admin can see "the user visited /operate, then /configure, then hit an error." Keep the volume manageable: deduplicate rapid transitions (same route within 2 seconds = one event), and make this toggleable via a platform setting or feature flag so it can be disabled if event volume becomes a concern.

**Constraints on 3A–3C:**
- All new event types use the `client.*`, `platform.*`, or `navigation.*` prefixes. No new `auth.*` prefixes.
- All client-side event logging goes through the existing deduplication guard in `auth-logger.ts` (or its generalization).
- Event volume: 3C is the highest-volume addition. If CC judges the volume is a concern, implement as opt-in (platform setting `enable_navigation_tracking`, default false). Document the decision in the Architecture Decision Record.
- No new database tables. All new events go into `platform_events` using existing schema.

Do not work around by directly writing to `auth.*` tables.

**HALT-3 (Event volume concern).** If CC determines that Objective 3C (navigation breadcrumbs) would generate more than ~100 events per user per hour under normal usage, implement it as disabled-by-default with a platform setting toggle, and document the volume estimate.

**HALT-4 (Sequence number collision).** The sequence number OB-230 is already in use or is not the next available. Report the correct next number and HALT.

**HALT-5 (Index requirement).** If `platform_events` does not have an index on `(actor_id, created_at DESC)`, the event timeline query will be slow at scale. Propose the index DDL for architect to apply via SQL Editor (SR-44 — migrations are architect-only). Do not apply the migration directly. Proceed with the build but note the performance risk in the completion report.

## §6 — OUT OF SCOPE

- **User impersonation / "god mode."** Viewing a user's data is in scope. Logging in as them is not. User impersonation requires a separate security design (session token delegation, audit trail, SOC 2 implications) and is deferred.
- **Tenant-admin user management.** This OB builds the platform admin (VL Admin) surface only. Tenant admins managing their own users (DS-019 Phase F) is a separate OB.
- **Real User Monitoring (RUM).** Full RUM (page load timing, Web Vitals, JS bundle performance) is a separate infrastructure concern. This OB captures only diagnostic signals (errors, navigation breadcrumbs, session churn), not performance telemetry.
- **Email/notification from the console.** The password reset action returns a link; the admin delivers it manually. In-platform notification delivery is a separate capability.
- **User creation/provisioning from this console.** The User Operations Console is diagnostic/remediation. User creation lives in the Onboarding flow.
- **Modifying user role or capabilities.** Role changes are a sensitive operation requiring its own design (approval workflow, audit trail, cascading RLS implications). Not this OB.
- **Historical analytics / reporting.** Login frequency trends, session duration averages, MFA adoption rates across the fleet — these are analytics, not diagnostics. Available as a future Observatory tab using the same data but with aggregation views. Not this OB.

## §6A — RESIDUALS

- **`platform_events` index on `(actor_id, created_at DESC)`.** Required for timeline query performance at scale. If HALT-5 fires, the index DDL is delivered for architect application. Until applied, the timeline works but may be slow for users with 1000+ events.
- **`auth.sessions` direct access.** Availability depends on Supabase project configuration. If HALT-1 fires, session health (Panel 2) degrades to inference-based (derived from login/logout/expiry events in `platform_events` rather than live session rows). Named gap; not silent.
- **Event volume management.** Navigation breadcrumbs (3C) and client error capture (3A) increase `platform_events` row count. A future INF for event retention policy (auto-archive events older than N days, or partition by month) is warranted but not part of this OB.
- **Mobile browser UA parsing.** The UA parser must handle the FxiOS user-agent string format discovered in the VLADMIN investigation (`Mozilla/5.0 (iPhone; CPU iPhone OS 18_7 like Mac OS X) AppleWebKit/605.1.15 (KHTML, like Gecko) FxiOS/152.0 Mobile/15E148 Safari/604.1` → "Firefox iOS 152 / iPhone iOS 18.7"). This is a named test case, not a general UA parsing requirement.
- **Anomaly detection intelligence.** The current design uses rule-based anomaly highlighting (rapid re-login, session churn, failure clustering). A future DS could explore AI-driven anomaly detection using the event stream — but that is Concept ⑦ territory (Plan Improvement Advisor / behavioral intelligence), not this OB.
