## §1 Problem Statement

From the post-HF-280 Meridian cold re-import (architect-channel reconciliation, 2026-06-09, rule_set `be74de80-6885-4b89-a949-3cce2361a786`):

- HF-280 worked: 5 components persisted in both variants; the previously lost component (Rendimiento de Ingreso) reconciles **exactly** to its GT-implied totals (Jan 44,000 / Feb 40,950 — precisely the prior missing deltas).
- The remaining shortfall is one binding: the senior variant group's Utilización de Flota component binding persisted with **only** `period, entity_identifier` — neither of its two intent-required tokens (`cargas_totales_hub`, `capacidad_total_hub`) — while the coordinador variant group's binding for the same component carries both. At calc, every senior entity hits `RESOLUTION_FAILURE token="cargas_totales_hub" reason=no_real_column_match` (the HF-built loud per-component surface, correctly firing) and the component pays 0. The deltas match exactly: Jan −17,878, Feb −18,972 — the senior c4 zeros and nothing else.

**The defect class — the adjacent arm (SR-34) of HF-280:** recognition-layer atomicity now holds (a component that fails recognition fails the import), but the **convergence/binding layer** has the same partial-success absorption: a component binding that failed to map its required tokens persisted as an incomplete binding, and calc proceeded against it, paying flagged zeros. Loud at calc is necessary but not sufficient — a financial calculation must not run against a binding known at bind time to be incapable of resolving the component. The failure belongs at the binding phase, before any calc batch is created.

**The invariant (the HF-280 principle at the binding layer):**

> A component binding is complete only if it maps every token the component's intent requires. The binding phase succeeds only if every component binding of every variant group is complete. Any incomplete binding fails the binding phase as a whole — a structured, user-visible failure naming the variant group, component, and unmapped token(s). Calc never executes against a binding set known to be incomplete.

**Open diagnostic (Phase 0, not assumed):** WHY the senior group's mapping lacks the two tokens while the coordinador group's has them. Candidates the reads must distinguish: (a) the HF-253 per-variant-group AI mapping call for the senior group genuinely omitted them and HF-112 validation accepted the partial mapping; (b) the mapping proposed them but validation rejected them silently; (c) the binding was assembled from a stale/cached prior convergence ("input_bindings already populated") that predates the 5-component rule_set, leaving the new component's tokens unmapped for one group. Each cause aborts identically under the invariant (no per-cause registry), but (c) additionally implies bindings must be invalidated when the rule_set's component set changes — Phase 0 decides whether that secondary fix is in scope.

## §2 Substrate-Bound Discipline

**Decision 154 (Korean Test):** the completeness predicate is structural: `requiredTokens(componentIntent) ⊆ mappedTokens(componentBinding)` — derived from the persisted intent, never from field/component/tenant literals, never from token-name patterns. Names appear only as display data in the failure message.
**AUD-009:** one invariant (binding completeness), loud failure, no enumeration of *why* a token is unmapped; every cause aborts identically.
**SR-34:** this closes the class at the binding layer; recognition (HF-280) and persistence-coherence (HF-279) arms already closed. The three together: no plan persists incomplete, no plan persists incoherent, no calc runs unbindable.
**DD-7:** complete bindings — every currently reconciling component — behave identically. The calc-time T3 RESOLUTION_FAILURE surface stays (defense in depth for data-shape drift between bind time and calc time); this HF prevents the *known-at-bind-time* class from ever reaching it.
**Vertical Slice:** the binding-phase failure surfaces through the same channel the import/calc failure states use; importer/operator-visible, no new chrome.

## §3B Phase 2 — Implement

**2.1 Completeness predicate** (pure, unit-testable, at the binding-assembly seam from Phase 0.1/0.2): for each variant group × component, compute `requiredTokens` from the component's persisted intent (its reference fields / dimension sources — structural read, no literals) and assert `⊆ mappedTokens`. Any miss → structured binding failure carrying variant group, component name, missing tokens (display data).

**2.2 Phase gate**: the binding/convergence phase returns failure if any component binding is incomplete; nothing persists to `input_bindings` on failure (atomic — no partial binding set), and the failure surfaces through the existing pipeline failure channel. Calc-time entry keeps its current behavior as backstop.

**2.3 If Phase 0.5 shows cause (c)**: invalidate persisted input_bindings when the rule_set is superseded/regenerated (bind against the current component set or not at all). Scope per HALT-2.

**2.4 If Phase 0.5 shows cause (a)/(b)**: the mapping request/validation for a variant group must cover all components' required tokens; a validated-partial mapping is a failed mapping for that group (the gate in 2.2 enforces this regardless — 2.4 is only about giving the AI mapping call the full token manifest and retry surface, mirroring HF-280's feedback pattern if a retry exists at this seam).

Commits after each change; korean-test gate; kill dev server → `rm -rf .next` → `npm run build` → `npm run dev` → localhost:3000.
