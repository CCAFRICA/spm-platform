## §1 Problem Statement

Proven from the post-HF-279 cold re-imports of both tenants (architect-channel reconciliation, 2026-06-09):

- **BCL** persisted with variant_1 (Ejecutivo) carrying **3 components** — Captación de Depósitos absent from the variant. HALT-3 SQL shows no ratio dimension for V1 Captación; the plan describes it as a ratio band. Grand totals short −8,400 / −9,580 / −14,380 vs GT. Everything that DID persist is coherent and calculates exactly (c0 exact all periods; V0 ratio bands quotient-space, scale null).
- **Meridian** persisted with **4 components in both variants** — the plan has 5. Grand totals short −44,000 / −40,950 / −48,900: the clean shape of one entire missing component. The four that persisted now behave correctly (Entrega as DD-7 pre-computed metric paying real tiers; Utilización as constructed quotient×base).

**The HF-279 coherence invariant worked.** Nothing incoherent persisted; nothing miscalculated. The missing components are precisely the configurations whose recognition previously emitted the invariant-violating shape (ratio band + binding scale). The reasonable mechanism — to be confirmed in Phase 0, not assumed — is: recognition repeatedly emitted the violating intent, the invariant raised `StructuralCoherenceError → cognition_violation` through the retry budget, retry exhaustion recorded a `failed` outcome, **and the import then persisted the partial plan and proceeded**. Calc later reported `resolutionFailures=[none]` because the failure happened before calc ever saw the plan.

**The defect class:** plan import is not atomic over components. A component-level recognition failure degrades silently into a structurally valid but incomplete plan that calculates clean-looking, wrong totals. This is the verification-layer lesson of AUD-009 recurring one level up: the loud failure existed at the component seam but was absorbed at the plan seam.

**The invariant (one rule, the HF-279 principle applied at the plan level):**

> A rule set persists only if every component of every variant completed recognition successfully. Any component failure after retry exhaustion fails the import as a whole — a structured, user-visible failure naming the component and the violated invariant. No partial variant, no partial plan, is ever persisted.

**Secondary (recognition reachability):** if Phase 0 confirms retry exhaustion on the invariant, the retry loop must feed the violation back to the model (the `StructuralCoherenceError` message: "emit breaks in the quotient's own space; scale: null for ratio-source bands") so a coherent emission is reachable within the budget. A retry that re-asks the identical prompt and gets the identical violation is not a retry.

## §2 Substrate-Bound Discipline

**Decision 158:** unchanged division of labor. This HF touches orchestration (deterministic) and the retry feedback envelope; it does not change what recognition or construction do.
**Decision 154 (Korean Test):** the atomicity predicate is `any component recognition outcome !== success → abort import`. No component names, no tenant literals, no failure-shape enumeration. The retry feedback passes through the structured error verbatim — it does not interpret it.
**AUD-009:** one invariant (import atomicity), loud failure, no registry of drop reasons. We do NOT add per-cause handling for "why" a component failed; every failure cause aborts identically.
**Atomicity precedent:** the HF-279 ADR already states "violation ⇒ structured failure, component NEVER persisted; on retry exhaustion a `failed` outcome is recorded." This HF makes the plan-level consequence of that `failed` outcome what it always should have been.
**DD-7:** no behavior change for any successfully recognized component.
**Vertical Slice:** the import failure must be visible in the import experience (the surface the importer sees), not only in server logs.

## §3A Phase 1 — Architecture Decision

Commit ADR at `docs/completion-reports/HF-280_ADR.md`:

```
ARCHITECTURE DECISION RECORD — HF-280
=====================================
Problem: Plan import is not atomic over components. Post-HF-279 cold
re-imports persisted BCL with a 3-component variant (Captación absent
from V1) and Meridian with 4 of 5 components in both variants. The
HF-279 coherence invariant correctly rejected incoherent component
intents through the retry budget; on exhaustion the orchestrator
recorded a failed outcome and the import persisted the partial plan.
Calc then reported resolutionFailures=[none] — the failure predates
calc. Totals were clean-looking and wrong by exactly one component.

Invariant: a rule set persists only if every component of every
variant completed recognition successfully. Any component failure
after retry exhaustion aborts the import as a whole with a structured,
importer-visible failure naming the component and the violated
invariant. No partial variant or plan ever persists.

Secondary: the retry loop feeds the structured violation message back
to the model on each retry so a coherent emission is reachable within
the budget. The message passes through verbatim (no interpretation —
Korean Test).

Korean Test: predicate is recognition outcome only. No component/
tenant/field literals, no failure-cause enumeration; every cause
aborts identically (AUD-009 — no registry of drop reasons).

Rejected:
  B) Persist successful components, flag failures for later repair —
     rejected: a partial plan calculates plausible wrong totals; the
     two re-imports are the proof. Silent partial state is the defect.
  C) Per-cause handling (treat coherence violations differently from
     other recognition failures) — rejected: registry pattern.
  D) Raise the retry budget without feedback — rejected: identical
     prompt → identical violation; not a retry.
```

## §3B Phase 2 — Implement

**2.1 Import atomicity** (at the Phase 0.1/0.3 seams in `web/src/lib/sci/plan-orchestration.ts` and the persistence call site): collect component recognition outcomes per variant; if any outcome is not success after the existing retry policy, abort before any rule_set persistence with a structured failure that includes variant index, component name (from the plan, for the importer's benefit — display data, not a predicate), and the underlying structured error. The abort surfaces through the import pipeline's existing failure channel to the importer-facing status (Vertical Slice: confirm the import UI/status reflects it; if the pipeline has no importer-visible failure state for this phase, add the minimal one — status + message — not new chrome).

**2.2 Retry feedback** (Phase 0.4 seam): on each retry after a `StructuralCoherenceError`/`cognition_violation`, append the error's message to the recognition request for that component so the model receives what was violated. Pass-through only; no rewriting, no cause-specific templates.

**2.3 No change** to the HF-279 invariant, construction, DD-7, or any successfully recognized component path.

Commit after each change; korean-test gate must pass; kill dev server → `rm -rf .next` → `npm run build` → `npm run dev` → confirm localhost:3000.
