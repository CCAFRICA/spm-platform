## §2 — CONSTRAINTS

### Substrate constraints (standing)

- **Decision 158 (recognize/guarantee).** Recognition guarantees the column's nature and the plan's component relationships. Downstream binding and construction must honor those guarantees.
- **Korean Test / C0 / AP-26.** No nature→operation registry. No component-type→composition-mode registry. No column-name list. No status-value list. Every mechanism must be structural — derived from recognition, extensible without developer edits. **HALT-REGISTRY** if CC cannot achieve this.
- **C6 — scoped exception.** The engine's variant-composition logic is calc-reader territory. Scoped exception: (a) the change must be **additive** (new mode alongside existing, not replacement), (b) existing additive compositions must be **byte-identical**, (c) the mechanism must be **structural** (DAG-declared, no lookup table). **HALT-CONSTRUCTION** if unsatisfiable.
- **C2 (fail-loud).** Unrecognized nature with no structural basis → fail loud, not silent-default. Unrecognized composition mode → fail loud.
- **SR-34 / Adjacent-Arm Drift.** Close the class, not the instance.
- **Carry Everything (T1-E902).** Data imports as-is, one row per transaction. Fan-out and derivation happen at convergence/calc time, never at import.

### Robles-anticipation constraints (the next tenant shapes the design)

Robles Maquinaria is a **distribution** tenant: one sale → 4–6 payout rows via an entity-relationship graph. Every mechanism HF-341 builds must extend to distribution without teardown. These are **design constraints on how CC solves the MIR defects**, not Robles scope:

**RA-1 — N-way multiplicative composition.** MIR D3a needs two-factor multiply (commission × accelerator). Robles needs `participant_rate = ref(base, recipient) × ref(product_factor, category) × ref(channel_factor, channel)` — three reference reads multiplied. **The composition mechanism must express N-factor multiplicative chains declared in the DAG**, not a two-component special case. If CC builds two-component multiply, Robles tears it out.

**RA-2 — Binding must express reference-table joins alongside column aggregations.** MIR D1/D2b are column→aggregation bindings (`Verificado@sum`, `Saldo@last`). Robles's factor-model is three `reference` reads — look up a value from a reference table keyed by recipient/category, not aggregate a committed_data column. **The binding-operation mechanism must represent "ref(table, key_column)" as a first-class operation** alongside "aggregate(column, op)." The Meridian HF-329 reference-join pattern is the precedent; the mechanism must accommodate it. If binding assumes all operations are column aggregations, Robles breaks it.

**RA-3 — Entity resolution must not assume one-entity-per-row.** MIR D3b/D4 fix entity-id selection for the one-entity model (seller = row owner). Robles: one sale row → N recipients resolved via `entity_relationships` graph traversal (originator → branch head → zone manager → director, plus attribute-conditioned overlays for product-specialist and key-accounts). **The entity-id and binding mechanisms must not hardcode "each row has one entity."** The graph-traversal fan-out itself is engine-side (downstream), but **the contract HF-341 writes to `input_bindings` must be structurally extensible** to carry a distribution fragment — a representation of "fan-out over resolved recipients" coexisting with per-entity derivations (HF-234 rule: both coexist). If the `input_bindings` schema can only express per-entity operations, the distribution contract has no home.

**RA-4 — Gate/condition wiring must handle attribute-based conditions.** MIR D2a is a numeric threshold gate (`collection_rate > 70%`). Robles has categorical attribute-conditioned inclusion: product-specialist participates IFF `product_line ∈ machine_lines`; key-accounts participates IFF `channel ∈ {wholesale, tender}`. Same class (conditional activation), different operand type (categorical match vs numeric comparison). **The gate-preservation mechanism must handle both**, or Robles's attribute conditions are dropped at construction. Note: the categorical conditions are **structural** (attribute = category), not a hardcoded set of product lines — the Korean Test applies here too.

**RA-5 — Modifier shapes must be open-vocabulary in the contract.** MIR exercises `cap` and `floor` implicitly. Robles introduces three shapes that don't fit existing types: cross-recipient/per-transaction cap (tope), cross-period-stateful accumulator (streak), cascade-wide atomic reversal (devolución). **HF-341 does not build these** (they're engine-side), but the modifier spec it writes to `input_bindings` must carry shapes as **open-vocabulary strings** (Validation Premise Law), not a closed `{cap, floor, proration, temporal_adjustment}` enum. If the contract uses a closed enum, the three Robles modifiers can't be expressed even after the engine supports them.
