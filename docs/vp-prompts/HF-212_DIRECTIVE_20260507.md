## 1 — INTENT

The Vercel runtime log is the architect's primary diagnostic surface for proof-tenant reconciliation work. It must, in a single Vercel paste (visible window before truncation), provide enough information to:

1. **Confirm calc completion and grand total** — at the TOP, before any per-entity emissions
2. **Identify defect class** — which exception flag classes fired and how many times
3. **Identify defect location** — which entities deviated, which components produced wrong outputs, which lookups engaged fallback
4. **Surface variant decisions** — distribution of variants chosen, default_last triggers
5. **Provide spot-check forensic detail** — for entities that hit exception flags, full trace inline at the flag site (not buried 2,000 lines down)

The current state across HF-207 through HF-211 emits diagnostic data, but at handler-exit, below Vercel's paste truncation. The architect cannot read totals, cannot read per-component breakdowns, cannot read flag counts. Every new tenant requires SQL spelunking instead of single-paste-read diagnosis. This is unacceptable for proof-tenant velocity (Meridian MX$185,063, CRP $566,728.97, and forward).

HF-212 redesigns the emission architecture so diagnostic content emits at the TOP of the log via a pre-pass, with per-entity detail emitted INLINE during the loop, with the per-metric forensic blizzard suppressed by default.

## 2 — TIER ARCHITECTURE

The log surface is restructured into four tiers, emitted in this order:

### Tier 1 — Header Summary (emits BEFORE entity loop runs)

Approximately 10-15 lines. Computed in a pre-pass over the entity assignments and rule set. Contains:

- Tenant name, period label, rule set name, batch ID, run ID
- Entity count assigned, expected components per entity, expected variant types
- Pre-computed exception flag counters initialized to 0 (placeholders)
- Boundary placeholders that will be retroactively filled by Tier 2 emissions

After entity loop completes, Tier 1 footer emits with FILLED totals:
- `grandTotal=X`
- `componentTotals={c0:X, c1:X, c2:X, c3:X}`
- `flags={diag003Fallback:N, boundaryFallback:N, ob118MergeGuardFired:N, no_band_match:N}`
- `variantDistribution={variant_0(senior):N, variant_0(ejecutivo):N, variant_1(default_last):N}`
- `topVariances=[entity:variance, entity:variance, ...]` if expected values known, else absent

CRITICAL: The header section emits at the TOP of the log stream. The footer section emits at handler-exit. **Both are architect-readable in a single paste because the header alone provides enough to confirm calc shape, and the entity loop emissions (Tier 2, ~85 lines) are short enough that the footer fits within Vercel's deliverable buffer.**

### Tier 2 — Per-Entity Summary (emits inline during loop, one line per entity)

Approximately 85 lines for BCL (one per entity). Format:

```
[CalcRecon-T2] BCL-5003 | Gabriela Vascones Delgado | variant=variant_0(senior) | total=1400 | components=[c0:600,c1:400,c2:250,c3:150] | flags=[]
```

Carries: external_id, entity name, variant decision, entity total, per-component breakdown for THIS entity, any exception flags that fired for this entity. Single line per entity. Architect can scan top-to-bottom and identify which entities deviate without scrolling past forensic noise.

### Tier 3 — Exception Detail (emits inline at flag site, only when flag fires)

Suppressed by default. Emits only when an exception condition triggers:

- `[CalcRecon-T3] EXCEPTION entity=BCL-5013 component=2 type=no_band_match input=999 boundaries=[...] reason=...`
- `[CalcRecon-T3] EXCEPTION entity=BCL-5008 component=1 type=boundaryFallback input=83.46 nearest_band=2 fallback_reason=...`
- `[CalcRecon-T3] EXCEPTION entity=BCL-5005 component=0 type=diag003Fallback batchId=... column=... reason=...`
- `[CalcRecon-T3] EXCEPTION entity=BCL-5007 component=1 type=ob118MergeGuardFired existingValue=... derivedValue=... preserved=convergence`

Volume scales with defect count, not entity count. A clean tenant produces zero Tier 3 lines. A defective tenant produces enough Tier 3 lines to characterize the defect class without scrolling.

### Tier 4 — Per-Metric Forensic Trace (suppressed by default)

This is the existing `[CalcTrace] resolveColumnFromBatch:exit`, `resolveMetricsFromConvergenceBindings:scale_applied`, `resolveSource:metric_lookup`, `executeBoundedLookup1D/2D:execution`, `runCalculation:component_complete` blizzard.

Behind environment variable `CALC_TRACE_VERBOSE=true`. Default deployment: SUPPRESSED (zero Tier 4 lines emit). Forensic mode (set env var, redeploy): full trace emits as before.

When suppressed, the per-metric trace lines do not emit at all — not buffered, not capped, not deferred. Just not there. Architect's normal Vercel paste contains only Tiers 1, 2, 3.

## 3 — VOLUME MATH

For a clean BCL October run with default verbosity (CALC_TRACE_VERBOSE not set):

- Tier 1 header: ~12 lines (top of log)
- [CalcAPI] setup lines (Starting, Period, Batch created, etc.): ~25 lines
- [VARIANT-DIAG] (existing 3-entity cap): ~12 lines
- [VARIANT] selection lines (1 per entity): ~85 lines
- Tier 2 per-entity summary (1 per entity): ~85 lines
- Tier 3 exception detail: 0 lines (clean run)
- Tier 1 footer: ~10 lines
- Existing `[CalcAPI] COMPLETE: total=...` line at route.ts:2207: 1 line

Total: ~230 lines. **Comfortably within Vercel's deliverable paste buffer.** Tier 1 header lands at line ~40 (after setup). Tier 1 footer with grandTotal lands at line ~225. Both visible.

For a defective tenant with 10 exception fires: ~240 lines, still within buffer.

For forensic mode with CALC_TRACE_VERBOSE=true: ~3,000 lines, hits truncation as before — but that's the explicit forensic mode tradeoff, not the default.

### 5.2 — route.ts: Tier 1 header emission (BEFORE entity loop runs)

Use existing variables in scope: `tenantName`, `period.canonical_key`, `ruleSet.name`, `batch.id`, `runId`, `entityIds.length`, `ruleSet.components`.

```typescript

// HF-212 TIER 1 HEADER: emits BEFORE entity loop
// ═══════════════════════════════════════════════════════════════
addLog(`[CalcRecon-T1] ╔═══════════════════════════════════════════════════════════════╗`);
addLog(`[CalcRecon-T1] ║              CALC RECONCILIATION HEADER                       ║`);
```

### 5.3 — route.ts: Tier 2 per-entity summary emission (during loop)

Find the existing per-entity total emission line (`Gabriela Vascones Delgado: 1,400 | intent=1,400 ✓` per Phase 0 grep N). Immediately AFTER that line, emit Tier 2 summary using variables already in scope at that emission point.

You will need access to the per-component breakdown for THIS entity. The HF-211 `componentTotals` Map accumulates GLOBAL per-component totals across all entities. For Tier 2, you need PER-ENTITY per-component breakdowns. Capture these by:

1. Adding a per-entity Map declared at the top of each entity iteration (cleared per entity)
2. At each `runCalculation:component_complete` site (Phase 0 grep J), push to the per-entity Map (this is in addition to HF-211's global componentTotals)
3. Emit Tier 2 line after the entity total line

```typescript
// HF-212 TIER 2: per-entity summary line
const entityComponentBreakdown = Array.from(perEntityComponentTotals.entries())
  .sort((a, b) => a[0] - b[0])
  .map(([idx, val]) => `c${idx}:${val}`)

addLog(`[CalcRecon-T2] ${entityExternalId} | ${entityName} | variant=${variantKey} | total=${entityTotal} | components=[${entityComponentBreakdown}] | flags=[${entityFlagsThisIteration.join(',')}]`);
```

### 5.5 — route.ts: Tier 1 footer emission (AFTER entity loop completes)

After entity loop finishes, BEFORE the existing `[CalcAPI] COMPLETE` line. Use existing handler-scope state (HF-208 counters, HF-211 componentTotals Map, entityResults array, grandTotal).

```typescript
// ═══════════════════════════════════════════════════════════════
// HF-212 TIER 1 FOOTER: emits AFTER entity loop
// ═══════════════════════════════════════════════════════════════
addLog(`[CalcRecon-T1] ─── Loop complete; reconciliation footer ───`);
addLog(`[CalcRecon-T1] entitiesCalculated=${entityResults.length} grandTotal=${grandTotal}`);
const sortedComponents = Array.from(componentTotals.entries()).sort((a, b) => a[0] - b[0]);

addLog(`[CalcRecon-T1] ╔═══════════════════════════════════════════════════════════════╗`);
```

## 9 — KNOWN CONSTRAINTS

- Tier 1 header emits BEFORE entity loop runs. Header CANNOT contain post-loop totals (those go in footer). This is the architectural separation.
- Per-entity component breakdown requires a per-entity Map cleared at each iteration start. Adds O(N×C) Map operations across the run — negligible.
- Tier 3 exception emission count scales with defect count. Clean tenant: zero Tier 3 lines. Worst case (every lookup hits boundary fallback): 340 Tier 3 lines for BCL — still within Vercel buffer.
- `variantCounts` Map cardinality bounded by tenant variant types (3 for BCL, similar for others). Negligible.
- Tier 4 suppression is environment-variable-gated. Architect changes Vercel env var to toggle. No code redeploy required for the toggle itself, but Vercel does require a redeploy for env var to take effect.
