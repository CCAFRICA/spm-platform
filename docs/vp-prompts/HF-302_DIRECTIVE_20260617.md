## §1 — ROOT CAUSE (from DIAG-072, proven from live code)

All three compound to produce $0; all three are in scope:

- **RC-1 (the regression): flat cross-data-type candidate pool.** `convergence-service.ts` boundary-matches a component to a data_type, then the HF-228 add-loop (`:2764-2777`) pushes EVERY other data_type's numeric columns into the AI candidate pool tagged `cross_source_numeric` at 0.4 confidence; the prompt (`:2449`) is one flat DATA COLUMNS list. The hard cross-source guard (HF-263 P3.2, commit `dc9c89e3`) was REMOVED by HF-269 Phase A (commit `bc57f1df`). → the AI binds the clawback's four inputs to Cobranza columns.
- **RC-2: resolution may read the wrong batch's rows.** `resolveColumnFromBatch` (`run/route.ts`) has a DIAG-003 fallback that, when the bound `source_batch_id` has no rows for the entity, searches ALL cached batches and takes the first with rows. After RC-3 the vendedor has rows in multiple batches; first-with-rows can pick the wrong file. (Whether `source_batch_id` is reliably populated on the persisted role bindings is the open shape question §3.0/§3.3 resolves against live code.)
- **RC-3: no secondary-identifier aggregation.** `dataByBatch` is keyed by each batch's PRIMARY entity-identifier column. Client-keyed transaction rows never roll up to the vendedor via the seller-identifier column present in those same rows.

**Why single-file tenants (BCL/Meridian/CRP) passed:** one data source → the cross-source add-loop contributes nothing → wrong-file binding is structurally impossible. MIR is the first multi-file tenant.

## §2 — DATA CONTRACT MAP (LOCKED GATE — authored from live code, re-verify in §3.0)

### A. Shape — `input_bindings.convergence_bindings` (persisted JSONB)
The **live** `ComponentBinding` interface (`convergence-service.ts:103-130`, per CC's read — NOT the stale AUD-001 extract) is:
```ts
export interface ComponentBinding {
  column: string;
  field_identity: FieldIdentity;
  match_pass: number | 'failed';
  confidence: number;
  scale_factor?: number;
  failure_reason?: string;
  resolutionFailure?: ...;
  learning_provenance?: { batch_id: string; learned_at: string };  // AUDIT-ONLY — explicitly NOT a resolution source
  filters?: ...;
  // NOTE: the interface declares NO top-level `source_batch_id`.
}
```
Persisted form (HF-108 / Decision 111): `convergence_bindings = { component_N: { <role>: <roleBinding>, ... } }` where roles include `entity_identifier, actual, target, row, column, numerator, denominator`.

**Open shape question (RESOLVE in §3.0 — do not assume):** the `dataByBatch` build and `resolveColumnFromBatch` callers read `roleBinding.source_batch_id` for those roles (HF-111). So the *persisted role objects* appear to carry `source_batch_id` even though the `ComponentBinding` interface above does not declare it. CC must determine from live code whether `source_batch_id` is (i) a declared-but-elsewhere field, (ii) an untyped property written at binding generation, or (iii) absent and the resolver relies on the DIAG-003 all-batch fallback. The fix for RC-2 depends on which is true. **This HF does NOT add a new persisted field** — see §3.3.

### B. Writer
`convergence_bindings` ← written by `convergeBindings` (`convergence-service.ts`), persisted to `rule_sets.input_bindings` by the calc route's HF-165 block (`run/route.ts`) and the import/finalize convergence path. **RC-1 changes WHICH candidate columns the AI may bind (the *content* of the role bindings) — it does NOT change the persisted *shape*.**

### C. Reader
`convergence_bindings` ← read by the calc route's `HF-108 Using convergence_bindings` branch, which builds `dataByBatch` and calls `resolveColumnFromBatch(batchId, column, entityExternalId)` per role. **RC-2 changes this reader's batch-selection logic only. RC-3 changes how `dataByBatch` is indexed.** Shape of `convergence_bindings` is unchanged by any RC. Absence/failure behavior post-fix must be **structured** (logged resolution failure with reason), never a silent 0 masking a mis-binding.

### D. In-memory — `dataByBatch` (calc route, NOT persisted)
`Map<batchId, Map<entityExternalIdValue, Array<row_data>>>`, built keyed by each batch's primary entity-identifier column value (DS-009 5.1; DIAG-003 indexes all rows by any known entity column). **RC-3 adds a secondary index entry: for any batch whose rows contain a column whose values overlap `entities.external_id`, also index those rows under that secondary identifier value** — additive, primary index unchanged.

### §3.1 — RC-1: partition the candidate pool by matched data_type (deterministic, option a)
At the HF-228 cross-source add-loop (`convergence-service.ts:2764-2777`): **gate the loop so a component's candidate pool contains ONLY columns from the data_type(s) the boundary matcher associated with that component**, plus any data_type the component explicitly cross-references via its declared structure — NOT every numeric column tenant-wide.

- Keep matched-data_type columns as the pool; drop the unconditional `cross_source_numeric` injection.
- A genuine cross-data_type need must come from the component's declared cross-reference (boundary-matcher output), not a blanket add-all loop.
- Restores the file affinity HF-269 Phase A removed, deterministically — keyed on `data_type`/`batchId`, no magnitude proxy, no name matching (Korean-Test-clean). **Do NOT restore HF-263 P3.2** (the magnitude-proxy redirect was deliberately deleted as override-class closure; reintroducing it is forbidden).
- **DD-2/DD-3 class sweep:** grep every site that builds the candidate pool / calls `resolveColumnMappingsViaAI` and confirm the partition holds at ALL binding-generation sites, not only the one DIAG-072 cited. Close the class, not the instance.

### §3.2 — RC-3: structural secondary-identifier rollup
Where `dataByBatch` is built (`run/route.ts`, the `entityColsByBatch` / DIAG-003 indexing block): after the primary-key index for a batch, **derive a secondary rollup key structurally**:

- Build the set of `entities.external_id` for the assigned (paying) entities once.
- For each batch, inspect its row columns: if a column's values overlap the external_id set above a clear majority threshold (most rows carry a known external_id in that column), that column is a **rollup key**. Index that batch's rows ALSO under that column's value, in addition to the primary key.
- Fully structural: no column-name literal. The signal is "this column's values ARE entity external_ids." The seller-identifier column is discovered because its values match the roster; the client-identifier column is not.
- Structured failure: if a component needs a batch's data_type and no rollup key is found for that batch, log it explicitly — the component resolves to a logged failure, not a silent 0.

### §3.3 — RC-2: correct batch selection at resolution (NO new persisted field)
Driven by §3.0's findings. Two cases:

- **If `source_batch_id` IS reliably populated on the role bindings** (§3.0 finds it written at generation): make `resolveColumnFromBatch` prefer the bound `source_batch_id`'s rows, and use the DIAG-003 all-batch fallback ONLY when that batch has no rows for the entity — AND when falling back, **select the batch in which the requested `column` actually exists for the entity (non-null), not merely the first batch with any rows.** This is the precise defect: first-with-rows can pick Clientes_Nuevos when the column lives in Ventas.
- **If `source_batch_id` is NOT reliably populated** (§3.0 finds it absent/untyped): the binding can't name its source, so make `resolveColumnFromBatch` resolve the column from whichever cached batch contains that `column` non-null for the entity (column-presence selection), instead of first-with-rows.

Either way: **no new field added to the persisted `ComponentBinding` contract.** RC-2 changes in-memory batch SELECTION logic only — it does not restructure `convergence_bindings`. (This is why RC-2 stays within the Data Contract gate: it reads the existing shape, writes nothing new to it.)

- **DD-1 sweep:** enumerate EVERY caller of `resolveColumnFromBatch` and the `resolveMetricsFromConvergenceBindings` paths (actual/target/row/column/numerator/denominator) and confirm the selection change is consistent across all roles — not just the single-input path.
