## §1 — Problem Statement

Three problems, one surface.

### Problem 1: Platform-plane leak into tenant-plane

The Operations menu inside the tenant workspace currently shows "New Tenant" and "Tenant Management" — these are platform-owner functions that should never appear to tenant users. The "Manage Tenant" action on Observatory tenant fleet cards currently navigates INTO the tenant workspace. Both are the Platform plane leaking into the Tenant plane. The platform owner's tenant-management functions belong exclusively in the Observatory.

### Problem 2: No complete tenant-management surface

HF-352 (PR #616) built a tenant-management surface at `/admin/tenants` with clean-slate, delete-tenant, and agent/feature toggles (including `prism_enabled`). But the tenant identity fields established at creation (country, industry, currency, locale, logo) and the tenant's admin-user management are not part of that surface. An operator who creates a tenant and then needs to edit its configuration, set its agent entitlements, or manage its admin users has no single place to do all three.

### Problem 3: Agent entitlement does not drive menu derivation end-to-end

The mechanism exists: `workspace-config.ts` already drops sidebar sections when `tenants.features[featureFlag]` is false. The toggle exists: HF-352 built agent/feature toggles that write to `tenants.features`. But these are not yet connected into a closed loop where toggling an agent ON/OFF in the Observatory immediately changes what menus, capabilities, and permissions that tenant's users see. The entitlement gate must be deterministic code (Decision 158) — the `tenants.features` field is the single source of truth, the capability matrix derives from it, and the menu renders from the capability matrix. No LLM in the resolution path.

## §2 — Invariants (violations = HALT)

- **I0 — Observatory confinement.** Every surface created or modified by this OB is reachable ONLY from the Observatory, ONLY by platform-scoped users. Zero tenant-plane routes created. If the existing HF-352 surface is at a tenant-plane route, it moves to an Observatory route.
- **I1 — Plane separation.** "New Tenant" and "Tenant Management" entries in the tenant-side Operations menu are removed. The "Manage Tenant" action on Observatory fleet cards opens the management surface WITHIN the Observatory (not by navigating into the tenant workspace).
- **I2 — Entitlement is deterministic (Decision 158).** The resolution path from `tenants.features` → capability set → menu visibility is pure deterministic code. Zero LLM calls. Zero async intelligence. A feature flag is either true or false; the menu item is either present or absent.
- **I3 — Single source of truth for entitlement.** `tenants.features` (JSONB on the `tenants` table) is the sole storage for what agents a tenant is entitled to. No parallel registry, no separate entitlement table, no duplicated flag. The PRISM gate (`prism_enabled`) already lives here — other agent gates follow the same pattern.
- **I4 — Audit completeness.** Every state change (tenant identity edit, entitlement toggle, admin user create/edit/deactivate) writes an audit record: who, what, before→after, timestamp. The audit record survives the change.
- **I5 — UX completeness.** Every writable surface has explicit Save and Cancel actions. Every surface has a clear, always-visible path to exit back to the Observatory. No dead-end pages.
- **I6 — Admin user creation works.** A platform admin can create a tenant admin user from this surface and that user can log in. If the existing admin-creation flow is broken (CLT-48 class), this OB fixes it within scope — no deferral.
- **I7 — SR-39 compliance.** This OB touches access control and menu derivation. The completion report must verify against SOC 2 CC6 (access enforcement at data + menu + API), OWASP (no client-trusted entitlement), DS-014 (capability-derived navigation).

### Phase 1: Plane Separation

**Objective:** Platform-admin tenant-management functions exist exclusively in the Observatory. Zero leakage into the tenant workspace.

What must be true after this phase:

- The tenant-side Operations menu (or wherever "New Tenant" / "Tenant Management" appear) no longer shows those entries to any user at any role.
- The Observatory fleet card "Manage Tenant" action opens the tenant-management surface WITHIN the Observatory layout — not by navigating into `app.vialuce.ai/admin/tenants` inside a tenant context.
- If HF-352's surface is at a tenant-plane route (`/admin/tenants`), relocate it to an Observatory-scoped route, or ensure it renders within the Observatory layout/context when accessed from the fleet card. The surface must never appear inside a tenant's sidebar or navigation.
- Platform-admin scope check enforced server-side on every route and API endpoint this surface uses. A tenant admin hitting the URL directly gets denied.

### Phase 2: Tenant Management Surface — Three Sections

**Objective:** One surface, three sections, opened from the Observatory fleet card. Extends HF-352's existing surface (do not rebuild from scratch — extend and relocate).

**Section A — Tenant Identity.**
Editable fields for the tenant's configuration: name, country, industry, currency, locale, logo. Read the `tenants` table schema (Phase 0 inspection #6) and surface every meaningful identity field. Display the tenant's `slug` and `id` as read-only reference. Explicit Save and Cancel. Before→after audit on every save.

**Section B — Agent Entitlement.**
Toggle controls for each agent the platform offers. The current known set (from the existing `tenants.features` shape and the platform's workspace-config sections): Intelligence, Compensation (ICM), Finance. Platform Core is always on — it cannot be toggled off. Each toggle writes to `tenants.features` on Save. The toggle states displayed must read from `tenants.features` — single source of truth (I3). The PRISM toggle (`prism_enabled`) already exists in HF-352 — it stays, integrated into this section alongside the agent toggles.

**No hardcoded agent list (Korean Test / no-registry).** The set of toggleable agents derives from a structural declaration — the same declaration the workspace-config sections use for their `featureFlag` property. If a new agent is added to workspace-config with a `featureFlag`, it appears in this section automatically. CC determines the best mechanism for this structural derivation.

**Section C — Tenant Admin Users.**
An existing "User Operations" surface already lives in the Observatory Users tab, showing all users across all tenants with role, tenant, last sign-in, device, and health/action controls. Phase 0 inspection #7 determines whether that surface is the EECI-optimal home for per-tenant user provisioning or whether the per-tenant management surface should carry its own user section.

Regardless of placement, these requirements apply:

- **Create Admin User.** A platform admin can create a new user for this specific tenant: email, display name, role selection. Uses the service-role client to call `auth.admin.createUser()` (or `auth.admin.inviteUserByEmail()`) + inserts a `profiles` row with the correct `tenant_id`, `role`, and default capabilities for that role. If an invite-email mechanism exists and works, use it. If it doesn't work, fix it (I6 — no deferral). The user must be able to log in after creation.
- **No duplication.** If user creation lives in the per-tenant management surface (Section C here), the Observatory Users tab should reflect the new user immediately without a separate sync step — both surfaces read from the same `profiles` table. If user creation lives in the Observatory Users tab instead, the per-tenant management surface should either link to it (filtered by this tenant) or show a read-only summary of the tenant's users with a clear action to "Manage Users" that navigates to the Users tab filtered appropriately.
- **No parallel user lists with divergent data.** One source of truth (`profiles` table), zero client-side user caches that can go stale.

### Phase 3: Entitlement → Capability → Menu (Closed Loop)

**Objective:** When a platform admin toggles an agent ON or OFF for a tenant and saves, that tenant's users immediately see the change reflected in their sidebar and capability set. The loop is:

```
tenants.features[agent_flag] → capability resolution → workspace-config featureFlag check → sidebar renders/hides
```

This loop already partially exists (the `featureFlag` mechanism in workspace-config). This phase closes any gaps:

- Every agent workspace section in `workspace-config.ts` that should be entitlement-gated has a `featureFlag` property pointing to the correct key in `tenants.features`. If any section is missing its gate, add it.
- The capability matrix (DS-014 §9 module-aware permissions) respects entitlement: a user in a tenant without the Compensation agent cannot have `icm.*` capabilities, regardless of their role. If the capability resolver does not currently check tenant entitlement, it must. This is deterministic code — a boolean check against `tenants.features` before granting module-specific capabilities.
- No menu entry, no capability, no route access for an agent the tenant is not entitled to.

**Decision 158 enforcement:** This entire resolution path is deterministic. The `tenants.features` value is read from the database. The capability matrix intersects role capabilities with tenant entitlement. The menu renders from the intersection. Zero LLM. Zero async intelligence. Zero heuristics.

### Phase 4: UX Completion

**Objective:** The surface is operationally complete — no dead ends, no missing actions.

- **Save confirmation.** Every section's Save action shows a confirmation of what changed (brief, not a modal wall).
- **Exit path.** A persistent, visible control to return to the Observatory fleet view. Always available regardless of which section is active.
- **Error handling.** Failed saves show the error, do not silently swallow. Failed user creation shows the specific failure (RLS, email conflict, missing field).
- **The tenant creation path.** "New Tenant" from the Observatory (the Onboarding tab or a fleet-level action) should lead to a creation flow that ends at this same tenant-management surface — so the operator can immediately configure identity, entitlement, and admin users after creation. If the existing creation wizard works (Phase 0 inspection #2), it should land here on completion. If it doesn't work, fix the landing — the wizard's persistence + navigation must function.
