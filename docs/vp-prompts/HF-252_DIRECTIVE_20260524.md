## §1 — Problem Statement

HF-251 shipped the construction pathway (Decision 158): the LLM emits CompositionalIntent, the constructor builds PrimeNode trees. All four BCL components constructed successfully, including C0's 30-cell matrix. Two defects surfaced at the BCL October calculation that trace to a single specification gap plus one architectural residual.

**Defect 1 — Component intent encoded role differentiation internally.** The LLM emitted a Productos Cruzados CompositionalIntent that referenced a categorical field (`employee_type`) to differentiate Senior vs Ejecutivo payouts inside one component. Convergence — which binds NUMERIC METRIC references and is proven across three reconciliations (BCL $312,033, Meridian MX$185,063, CRP $364,457.84) — could not bind the categorical reference (`HF-222: employee_type: candidate distribution insufficient to bind`). The evaluator then resolved the unbound reference to the entity's role attribute string ("Ejecutivo Senior") and crashed on Decimal coercion (`[DecimalError] Invalid argument: Ejecutivo Senior`).

The platform already handles role differentiation structurally via variant decomposition. `interpretationToPlanConfig` filters components by `appliesToEmployeeTypes` into per-variant component lists; the engine variant router (HF-119) selects each entity's variant before component evaluation. The BCL import correctly produced 2 variants × 4 components = 8 persisted components. Role differentiation belongs at the variant layer, ABOVE component evaluation — not inside a component's intent.

The specification gap: DS-024 did not scope what a single CompositionalIntent represents. The LLM treated "component" as the plan's logical component (one Productos Cruzados with internal role logic) when the architecture's contract is that a component intent describes ONE variant's calculation, referencing only metrics, with role differentiation handled by variant assignment upstream.

**Defect 2 — Legacy emission-pathway fallback retained.** HF-251 retained the HF-249/250 assembler path as a backward-compat fallback in `callPlanComponentWithRetry`: when the LLM response lacks `compositional_intent`, the orchestrator falls through to the deprecated tree-emission assembler. This is a T0-E03 (Single Pipeline) / AP-17 violation — two pathways doing the same work, with silent dispatch on response shape (an AUD-009 cherry-pick at the orchestration boundary). Decision 158 is not operative while two pathways coexist; an LLM response that omits `compositional_intent` silently produces a tree via the deprecated pathway, masking the discipline failure (T1-E906 v2 silent-fallback anti-pattern).

**Resolution class:** Deliver convergence the input shape it is proven to consume (metric-only component intents), and restore the single pipeline (construction pathway only; structured failure on unrecognized response shape per Decision 154).

**Dependency rationale:** The emission discipline change is the root fix for Defect 1. Phase 2 (fallback removal) is independent but sequenced after because verifying the construction pathway works correctly is prerequisite to removing its fallback.


### §3.2 — Add per-variant-component emission discipline to the prompt

In the `plan_component` system prompt (`anthropic-adapter.ts`), add explicit discipline governing what a component intent represents. The intent describes ONE component's calculation for the entities it applies to. It references only the numeric metrics needed to compute the payout. It does NOT encode role-based, level-based, or category-based payout differentiation internally.

Add to the prompt, in structural (Korean Test-compliant) language:

- A CompositionalIntent describes the calculation for ONE component. Reference only the numeric measures the calculation consumes (attainment ratios, amounts, counts, percentages).
- Do NOT reference categorical entity properties (role, level, tier, type, classification) inside a component's structure. If a plan pays different rates to different categories of people, that is variant differentiation, handled by the platform's variant assignment — not by a conditional inside the component.
- When a component's rate or output differs by a category of person, emit the component as it applies to ONE category, and declare which categories it applies to in the `applies_to` field. The platform routes each person to the correct variant and evaluates the matching component.
- `ReferenceSource.type` values: use `metric`, `ratio`, `aggregate`, `scope_aggregate`, `prior_component` for numeric inputs. Do NOT use `attribute` to drive payout differentiation; entity attributes are for variant routing, which is upstream of the component.

Add an `applies_to` field to the CompositionalIntent schema (`compositional-intent.ts`):

```typescript
interface CompositionalIntent {
  component_id: string;
  component_name: string;
  /**
   * Which entity categories this component applies to.
   * Maps to appliesToEmployeeTypes in the variant decomposition.
   * Empty or ["all"] = applies to all variants.
   * Role differentiation is expressed HERE (which variant), never
   * inside the structure (no internal categorical conditional).
   */
  applies_to?: string[];
  structure: StructuralDescription;
  scale: { /* unchanged */ };
  output_precision: number;
  metadata?: Record<string, unknown>;
}
```

### §3.3 — Update the skeleton call to enumerate per-variant components

In `plan-orchestration.ts`, the skeleton call (Phase A) currently returns a component index. Update it so that when the plan has variant-differentiated components, the skeleton enumerates each (variant, component) pair as a distinct emission target, with the variant's category identifier carried as the `applies_to` context for the subsequent per-component intent call.

The skeleton prompt should instruct: when a component pays differently by entity category, enumerate it once per category in the index, each tagged with which category it applies to. When a component pays uniformly, enumerate it once with `applies_to: ["all"]`.

This makes "component" at the orchestration boundary mean per-variant-component, matching what `interpretationToPlanConfig` expects to filter.

### §3.4 — Carry applies_to through to variant filtering

In `ai-plan-interpreter.ts`, ensure the constructed component's `appliesToEmployeeTypes` is populated from the CompositionalIntent's `applies_to` field. The existing `interpretationToPlanConfig` variant-filtering logic (filtering `allComponents` by `appliesToEmployeeTypes` per employee type) is UNCHANGED — this step ensures the field it filters on is correctly populated from the new intent format.

Verify the existing filter logic consumes the field:

```bash
grep -n "appliesToEmployeeTypes" web/src/lib/compensation/ai-plan-interpreter.ts
```

The persisted component must carry `appliesToEmployeeTypes` derived from `applies_to` so the proven variant filtering routes it. No change to the filter; only ensure its input is populated.

Commit: `git add -A && git commit -m "HF-252 Phase 1: per-variant component intent emission — metric-only, applies_to for variant routing"`

## §4 — Phase 2: Legacy Fallback Pathway Removal (Single Pipeline)

**Dependency rationale:** Phase 1 establishes the construction pathway as correct. Removing the fallback is safe only after the primary pathway is verified to handle all BCL components — which Phase 1's prompt discipline plus HF-251's constructor already establish. This phase enforces T0-E03.

### §4.1 — Locate the fallback dispatch

```bash
grep -n "compositional_intent\|calculationIntent\|assembler\|Mode A\|Mode B\|shouldUseChunking\|interpretPlanComponentWithChunking" web/src/lib/sci/plan-orchestration.ts
```

Identify the dispatch in `callPlanComponentWithRetry` (and any caller) where a response lacking `compositional_intent` falls through to the HF-249/250 assembler path (`interpretPlanComponentWithChunking`, Mode A/B chunking, `prime-assembler` stitching).

### §4.2 — Replace fallback with structured failure

Remove the fallback dispatch. The orchestrator's contract becomes: an LLM response in the plan_component pathway MUST contain a parseable `compositional_intent`. If it does not, raise a structured error per Decision 154 — never downgrade to the deprecated assembler.

Replace the fallback branch with:

```typescript
// Decision 154 / T0-E03: single pipeline. A plan_component response
// without a parseable compositional_intent is a structured failure,
// not a downgrade to the deprecated emission pathway.
if (!result.compositional_intent) {
  throw new MissingCompositionalIntentError(
    `[plan-component] Response for component "${componentName}" lacked ` +
    `compositional_intent. Construction pathway (Decision 158) is the sole ` +
    `plan-interpretation pathway. No fallback. Classify as cognition_failure ` +
    `and retry per error class taxonomy.`
  );
}
```

The `cognition_failure` error class (HF-248 taxonomy) already governs retry-with-prompt-refinement for malformed LLM cognition output. A missing `compositional_intent` is a cognition failure and retries through the existing taxonomy — bounded retry, then surface to architect. This is NOT a silent fallback; it is a bounded, visible retry on the single pathway.

### §4.3 — Retire the dead emission-pathway code paths

Remove the now-unreachable dispatch machinery from the plan_component call path:
- `shouldUseChunking` dispatch (Mode A/B selection)
- `interpretPlanComponentWithChunking` invocation from the orchestrator
- The assembler-stitch branch for LLM-emitted chunks in this call path

Do NOT delete `prime-assembler.ts` as a file in this HF — the constructor (HF-251) may share assembly primitives, and file deletion is a separate cleanup scope (DD-7: no smuggled expansion). Only remove the orchestration-layer dispatch that routes to the deprecated emission pathway. Formal file-level deprecation is HF-255.
