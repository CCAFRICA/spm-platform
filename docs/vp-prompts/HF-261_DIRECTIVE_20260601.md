## §1 — Problem Statement

Meridian (tenant `5035b1e8-0754-4527-b7ec-9f93f85e4c79`, rule_set `2fb555d4-53fe-42e8-9662-cae3d07da4f4`) exhibits two coupled defects in the fleet/hub path. The four non-fleet components calculate correctly per-employee; only the fleet path and the hub-payee admission are wrong.

### Defect 1 — Fleet component (C5) returns zero for every entity, every period

Code-justified from **AUD-005 (calc-execution live reference, SHA `5314c365`)**:

- The fleet component ("Utilización de Flota"; variant component index `c4`; convergence `component_4` / `component_9`) is bound to two hub-level columns (loads, capacity) via convergence (`HF-112` bindings in the live log).
- `resolveColumnFromBatch(batchId, column, entityExternalId)` (`web/src/app/api/calculation/run/route.ts:1213`) resolves a column by `batchEntityMap.get(entityExternalId)` — keyed on the **employee** external_id. The hub columns live in the **hub-keyed reference batch**. An employee id never matches a hub-keyed row → returns `null` → the fleet metric is absent → **C5 = 0**. The DIAG-003 cross-batch fallback also misses: no batch holds both an employee id *and* the hub columns.
- The only hub→member projection in the calc — the OB-146/148 store-merge (`run/route.ts:1442`) — keys on `rd['storeId'] ?? rd['num_tienda'] ?? rd['No_Tienda']`. Meridian employees carry a hub attribute, not a store id → `entityStoreData` is `undefined` → the merge never engages → live log: `Store data: 0 unique stores`.
- The `scope_aggregate` primitive exists (`web/src/lib/calculation/intent-executor.ts:811`) and would perform exactly this resolution, but it reads `data.scopeAggregates`, which is **never populated** with hub aggregates in this path.

Net: **no mechanism projects hub-level fleet data onto an employee by hub.** The PrimeValidator `terminal_completeness` warning that fired on this component was predictive, not cosmetic.

### Defect 2 — Hub entities are admitted as payees

Hub entities are created at import (entity_id_field resolves to the hub) and assigned to the rule set (live: 79 rule_set_assignments = 67 employees + 12 hubs), then paid (live: 11 hubs each receiving the safety-component base via its no-incidents branch). Hubs are **scope** for the fleet aggregate, not commission recipients. The phantom hub payout is the same hub-mis-resolution that starves the fleet input — one defect class, two surfaces.

### Defect-class lineage

Longest-running Meridian defect. `HF-089`/`OB-158` (the entity agent selects the hub identifier for the person-less fleet sheet — correct for the sheet alone; the defect is that the calc expects person-entities). `HF213` reconciliation (hubs fall to `default_last`; ~12 hub rows where ~67 employee rows are expected; the fleet component is the only one whose source is hub-keyed). The historically-named resolution is "ratio + aggregate": aggregate loads ÷ capacity **per hub**, then project onto the employees at that hub. This is the **aggregate-scope-component family** (shared root with CRP Plan 4, "blocked on DAG engine").

### Combined-treatment rationale (DD-7 adjacent)

The two defects share the hub/scope mis-resolution. Fixing fleet projection requires resolving hubs **as scope** (the aggregate source); excluding hubs as payees is the same scope-vs-recipient distinction. Splitting them re-commits the instance-vs-class pattern (SR-34). One integrated HF.

### Convergence note

This HF touches **convergence-binding production** for the fleet component. Meridian's fleet convergence currently **fails** its anchor — this repairs a broken path, it does not modify a proven one. The proven, passing components (the four non-fleet, exact to the dollar against the reference) are protected by DD-7 and the Phase-1 ADR isolation gate. The SR-42 disposition of "IRA Class A before touching convergence" is resolved in the architect channel above this directive prior to dispatch.
