## §1 — Problem Statement

Architect browser-verified HF-312 delivery at `app.vialuce.ai` under `data-theme="vialuce"`. Four blocking defects found:

**Defect 1 — Content area washed out.** Every content page renders with dark/semi-transparent cards on the light `#F4F5FB` background. The design spec requires white cards (`#FFFFFF`) with near-black text (`#1A1A2E`), crisp borders (`#E8EAF3`), and card shadows (`--vl-sh-1`). The sidebar and topbar render correctly (built with design classes). The content area does not. This affects ALL pages, not just one.

**Architect hypothesis:** The OKLCH values in the `[data-theme="vialuce"]` globals.css block for `--card`, `--card-foreground`, and/or the strag neutral scale are producing dark colors instead of light. CC's completion report says "mirrored bliss's proven light values" — but the visual result contradicts this. Alternatively, components may have hardcoded `dark:` Tailwind variants or explicit dark-theme colors that override the semantic tokens.

**Defect 2 — Persona switcher duplicated.** Two persona switchers visible simultaneously: (a) the VialuceSidebar's docked footer version (correct, per design spec §4.1), and (b) the legacy floating persona bar at viewport bottom-center. The legacy one must be hidden when `data-theme="vialuce"` is active.

**Defect 3 — User theme toggle not accessible.** The per-user theme toggle (in UserIdentity — the chevron/expand that lets an individual user override the Observatory-level theme default) is no longer visible. Users cannot switch themes per-user. The Observatory-level selector works, but the per-user affordance is gone. Likely: the Vialuce sidebar styles are hiding the expand indicator, or HF-312 changes to UserIdentity removed the visual affordance.

**Defect 4 — Observatory "Dark" selection not honoring.** Selecting "Dark" in the Observatory Settings theme selector does not switch to the dark theme. Likely: the "Current" → "Dark" rename in HF-312 introduced a value/label confusion — the settings component is writing the display label `'Dark'` to the database instead of the internal value `'current'`. The `THEME_LABELS` map (`current → 'Dark'`) is display-only; the persisted value must remain `'current'`.

### §3.0 — Defect 4: Observatory "Dark" Selection (highest priority — affects all themes)

**Objective:** Fix the Observatory Settings theme selector so selecting "Dark" correctly persists `'current'` (the internal value), not `'Dark'` (the display label).

**Diagnosis procedure:**
1. Open `FeatureFlagsTab.tsx` (the Observatory settings component updated in HF-312).
2. Find the theme selector (radio buttons, dropdown, or select that lists Dark/Bliss/Vialuce).
3. Check what value is sent to the API when the user selects a theme. It must send the INTERNAL value (`'current'`, `'bliss'`, `'vialuce'`), not the DISPLAY label (`'Dark'`, `'Bliss'`, `'Vialuce'`).
4. The `THEME_LABELS` map from `theme-labels.ts` is for DISPLAY ONLY. The selector's `value` attribute must use the AppTheme key, and the `label`/`children` uses `THEME_LABELS[key]`.

**Expected fix pattern:**
```tsx
// WRONG — sends display label as value
<option value="Dark">Dark</option>

// RIGHT — sends internal value, displays label
{THEME_ORDER.map(key => (
  <option key={key} value={key}>{THEME_LABELS[key]}</option>
))}
// key='current' → value='current', display='Dark'
```

Also check: does the API endpoint (`api/user/theme` or the Observatory settings save) validate the incoming value against the VALID set (`{current, bliss, vialuce}`)? If it receives `'Dark'`, it should reject it — trace whether it does.

### §3.1 — Defect 1: Washed-Out Content Area (token-level diagnosis)

**Objective:** Diagnose WHY the content area renders with dark cards on a light background, and fix it.

**Diagnosis procedure (diagnostic-first — read code before changing anything):**

**Step 1 — Inspect the token values.** Open `globals.css`, find the `[data-theme="vialuce"]` block. Read the OKLCH values for these critical tokens:
```
--card
--card-foreground
--popover
--popover-foreground
--secondary
--secondary-foreground
--muted
--muted-foreground
```
For each one, determine what color the OKLCH value produces. An OKLCH value like `0.97 0.005 280` is nearly white (lightness 0.97). A value like `0.15 0.02 260` is nearly black (lightness 0.15). The card tokens MUST produce white/near-white (lightness > 0.95). If they produce dark values (lightness < 0.3), the OKLCH values were copied from the dark theme instead of the light theme.

**Step 2 — Check for dark: variants.** Search the content components for Tailwind `dark:` prefix classes:
```bash
grep -rn 'dark:' web/components/ | grep -v node_modules | grep -v '.test.' | head -30
```
If components use `dark:bg-slate-900` or similar, those classes activate based on the `colorScheme` — and even though HF-312 set `colorScheme` to `light` for Vialuce, there may be components that set their own `dark` class or use `prefers-color-scheme`.

**Step 3 — Check for hardcoded dark colors.** Search for components using explicit dark hex values:
```bash
grep -rn 'bg-\[#[012]' web/components/ | head -20
grep -rn 'bg-slate-[789]\|bg-gray-[789]\|bg-zinc-[789]\|bg-neutral-[789]' web/components/ | head -20
```
These would produce dark backgrounds regardless of the token system.

**Step 4 — Check strag tokens.** The completion report mentions a `--strag-*` scale (12 tokens). If these produce dark neutrals instead of light neutrals, any component consuming them would render dark.

**Fix — depends on diagnosis:**

- **If OKLCH values are wrong (lightness < 0.3 for card/surface tokens):** Correct them. The design spec is explicit: `--vl-surface: #FFFFFF`, `--vl-bg: #F4F5FB`. Convert these hex values to correct OKLCH:
  - `#FFFFFF` → `1 0 0` (pure white)
  - `#F4F5FB` → approximately `0.97 0.005 280`
  - `#1A1A2E` → approximately `0.15 0.02 280`
  - `#E8EAF3` → approximately `0.93 0.01 280`

- **If components use dark: variants or hardcoded dark colors:** Add scoped CSS overrides under `[data-theme="vialuce"]` that force the correct surface/text colors. Example:
  ```css
  html[data-theme="vialuce"] .card,
  html[data-theme="vialuce"] [class*="bg-card"] {
    background-color: var(--vl-surface) !important;
    color: var(--vl-raw-near-black) !important;
  }
  ```
  This is a band-aid — document it as technical debt. The structural fix is per-page adoption where each component uses the design's classes directly.

- **If it's both:** Fix the tokens first (fixes the foundation), then add scoped overrides for any remaining hardcoded dark styles.

Commit: `"HF-313: fix washed-out content area (token/surface correction)"`.

**HALT-1:** If the diagnosis reveals that the washed-out appearance is caused by 50+ components each with their own hardcoded dark styles (not a token-level issue), halt and report the inventory. Do not attempt to override 50 components individually — that's the per-page adoption track.

### §3.2 — Defect 2: Persona Switcher Duplication

**Objective:** Hide the legacy floating persona switcher when `data-theme="vialuce"` is active.

**Diagnosis:** Find the legacy persona switcher component. It's the floating bar at the bottom-center of the viewport showing "Persona | Admin | Manager | Rep". Search for:
```bash
grep -rn 'Persona\|persona' web/components/ | grep -v VialuceSidebar | grep -v node_modules
```

**Fix — either approach:**

Option A (React conditional — preferred):
```tsx
const theme = useTheme(); // or however the theme is read
if (theme === 'vialuce') return null; // VialuceSidebar has its own persona
return <LegacyPersonaSwitcher />;
```

Option B (CSS hide — simpler but less clean):
```css
html[data-theme="vialuce"] .persona-switcher-legacy,
html[data-theme="vialuce"] [data-persona-switcher] {
  display: none !important;
}
```

CC: use whichever is cleaner given the component's architecture. If the persona is deeply embedded in a layout component and conditional rendering risks breaking the layout, use CSS. If it's a standalone component, use the React conditional.

Commit: `"HF-313: hide legacy persona under vialuce theme"`.

### §3.3 — Defect 3: User Theme Toggle Not Accessible

**Objective:** Restore the per-user theme toggle affordance so individual users can switch between Dark/Bliss/Vialuce.

**Diagnosis:** The per-user toggle lives in `UserIdentity.tsx` (desktop). In the previous implementation, there was a chevron or expand control that, when clicked, revealed the theme options. After HF-312, this control is not visible under the Vialuce theme.

Find the expand/chevron element in `UserIdentity.tsx`. Check:
1. Is it still in the JSX? (Was it accidentally removed?)
2. Is it hidden by CSS? (Vialuce sidebar styles may set `display: none` or `visibility: hidden` on elements that look like they belong to the legacy sidebar)
3. Is it obscured by z-index? (VialuceSidebar may overlay the area where the toggle appears)

Also check: where does the UserIdentity toggle render when Vialuce is active? In the legacy sidebar, it's part of the sidebar footer. In VialuceSidebar, the footer has the persona switcher and user info block — but the THEME toggle may not have a slot.

**If the toggle has no slot in VialuceSidebar:** Add a theme selector to the VialuceSidebar's user info block at the bottom. The design spec's `.sb-user` block at the sidebar footer is the right location. It could be a small icon (palette/paintbrush) that opens a dropdown with Dark/Bliss/Vialuce options, or the same three-option pill used in the UserIdentity desktop toggle.

Commit: `"HF-313: restore per-user theme toggle"`.

**Gate 3 (CC-verifiable):**
