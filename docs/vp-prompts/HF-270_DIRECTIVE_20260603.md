## §1 — Problem Statement

The platform produced a result that was previously correct and is now wrong, for the same input — the contract "same input → same output" was broken. Meridian rule_set `9aad7353-6ba1-4e97-9f9d-cbb6e93099fd`, component c0 (Rendimiento de Ingreso), senior variant: the persisted `calculationIntent` reference nodes name the raw field `cargas_mes_hub` (fleet loads) where the prior known-good interpretation named `volumen_rutas_hub` (route volume). These are semantically adjacent in the plan prose but are **different data keys** resolving to different committed_data columns (`Cargas_Flota_Hub` vs `Volumen_Rutas_Hub`) carrying different values. c0 dropped 44,000 → 32,475 on M1 (−11,525), with identical-class drops on M2/M3; c1–c4 byte-identical across the regression. The fleet field belongs to component c4 (Utilización de Flota); the fresh interpretation mis-placed it into the revenue component c0.

**The machinery downstream of interpretation is faithful** (verified by source read this session): `extractInputRequirements`/`extractReferencesFromDAG` extract whatever the DAG names; `convergence-service.ts` maps the named field to its column; `convertComponent` + `validatePrimeNodeTree` validate DAG *grammar* only (the `reference` case is `return true` — it accepts any field string); `plan-comprehension-emitter.ts` forwards the already-built component verbatim; the `intent-constructor` (`constructTree`) deterministically builds the tree from the LLM's `CompositionalIntent`. None of these chooses the field name.

**Root cause (proven, line-level).** The field identity in a `reference` node originates in the `plan_component` (Phase B) LLM emission. The `plan_component` prompt (`web/src/lib/ai/providers/anthropic-adapter.ts`, `SYSTEM_PROMPTS.plan_component` + the `buildUserPrompt` `plan_component` case) presents the LLM with the plan document and the component's `id`/`name`/`briefSemantic`/`rateTableCellCount` ONLY. The `ReferenceSource` spec (`{ "type": "metric", "field": "<name>" }`) places NO constraint on `<name>`. The orchestrator (`web/src/lib/sci/plan-orchestration.ts`, `OrchestrationInput` → `callPlanComponentWithRetry` → `interpretPlanComponent`) passes no field inventory. So the LLM mints `reference_field` from its reading of the plan prose for that component — `volumen_rutas_hub` on one fresh call, `cargas_mes_hub` on the next. Same prose, divergent field key. The non-determinism is structural to the prompt: field identity is unconstrained free-text LLM output.

**This is a locked, recurring defect class.** Decision 147 (Plan Intelligence Forward, LOCKED 2026-04-04) documents this verbatim: *"On April 2 with tenant flywheel patterns, the AI made this connection. On April 5 with a clean slate, it did not. Same data, different result. Non-deterministic."* The structural_fingerprints flywheel was MASKING the defect by replaying a cached correct interpretation; the clean-slate wipe removed the cache and exposed the latent non-determinism. **The fix must hold cache-cold** — re-masking via the cache is the prohibited shortcut. The defect-class statement for SR-34: **a `reference` node's field identity is non-deterministic because it is sourced from free-text LLM emission instead of resolved against a runtime-derived set of the identifiers actually present in the import.**

**The anchor already exists and is correct.** The `header_comprehension` task (same adapter file) already comprehends every data column's semantic meaning at runtime from the data file — `header-comprehension.ts`'s `suppressFalseCurrencyColumns` even pattern-distinguishes `loads` from `volume`, i.e., HC already knows `Cargas_Flota_Hub` (loads) and `Volumen_Rutas_Hub` (route volume) are distinct. HC runs at the `analyze` step and its output is carried on each content unit as `classificationTrace.headerComprehension` (consumed today by `extractFieldIdentitiesFromTrace` in `header-comprehension.ts`, and persisted to `committed_data.metadata.field_identities` for data units per OB-162). At `execute-bulk` (`web/src/app/api/import/sci/execute-bulk/route.ts`), the plan batch (`executeBatchedPlanInterpretation`) runs in the SAME handler that holds `contentUnits[]` with their `classificationTrace` — so the comprehended column identities for the data sheets are IN SCOPE at plan-interpretation time, on the request body, even though `committed_data` is not yet written (plan is `PROCESSING_ORDER` 0, before the data loop). The fix threads that already-present comprehension into the `plan_component` call so the LLM resolves each `reference_field` to a comprehended identity rather than minting one from prose.

This is exactly the structure the architect accepted: a runtime-derived resolution anchor (HC of the data sheets in this import), NOT an enumerated vocabulary. Same plan PDF + same data sheets (hence same HC) → same resolved reference, deterministically. It also collapses the lossy after-the-fact reconciliation the `convergence_mapping` step performs today (matching invented field names back to columns) — but that surface is HF-269's and is out of scope here (§6).

## §6A — Residuals

- **Plan-only import anchor quality:** when no data sheet accompanies the plan (fallback to `plan_skeleton.requiredInputs`), the declared-field set is only as complete as the plan document's own field declarations. If a plan declares fields ambiguously, the fallback anchor may under-constrain emission. This is strictly better than the current no-anchor state (the fallback still bounds emission to the plan's own declared identifiers), but full determinism for plan-only imports depends on the plan declaring its fields. When data is later imported and HC computed, a re-interpretation would anchor to the richer HC set. Tracked as a follow-on if plan-only determinism proves insufficient in practice.
- **HC confidence threshold:** the comprehended set is taken as-is; this HF does not filter by HC confidence. If a low-confidence HC interpretation mis-names a column's meaning, the resolution could match a reference to a wrong-but-comprehended field. The SR-38 variability gate and cold-cycle determinism catch a *wrong* result; they do not catch a *consistently wrong* HC meaning. Whether to gate the anchor by HC confidence (and at what threshold — Decision 108 uses 0.80 for HC override authority) is a follow-on refinement, not required for the determinism contract this HF closes.
- **`metricSemantics` (Decision 147) full realization:** HF-270 anchors field IDENTITY to HC comprehension. Decision 147's broader PIF goal — persisting `metricSemantics` (operation + filters + source field) at interpretation so convergence consumes seeds as highest-confidence input — remains the larger work item this HF touches but does not complete. HF-270 closes the field-identity reproducibility hole; the filtered-derivation seed path is HF-269/PIF territory.
- **Convergence re-matching redundancy:** with references pre-resolved to comprehended columns, the `convergence_mapping` AI call's job narrows to column binding rather than semantic field recovery. Simplifying that path is deferred to HF-269's convergence work; HF-270 leaves it operative (DD-7).
