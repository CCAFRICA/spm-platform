## §1 — Problem Statement

Three nav defects, post-OB-228:

1. **Plans & Canvas is mis-placed and over-exposed.** OB-228 surfaced it by repointing the dead `/design` nav slot to `/configure/plans` — a top-level placement visible to everyone. But **not every tenant/user has a plan**; the surface is meaningful only to those entitled to the **Calculate agent** (and its route already enforces `icm.configure_plans`). The menu entry should live *under the Calculate agent* and render only when the viewer is entitled to that agent — so the menu matches the entitlement the route already enforces.
2. **"Workspaces" mis-names the model.** The platform is agent-centric; the section heading should read **"AGENTS."**
3. **The agents don't stand out.** The four agents render at the same visual weight as utility/config nav items, so the platform's primary organizing concept is visually indistinguishable from secondary navigation.

This is a display-layer correction. The access-control boundary is unchanged: the Calculate/plan route enforcement built in OB-228 remains the security gate; this HF aligns menu *visibility* to it (SR-39 — no auth/isolation surface is weakened; nav gating is display-only and additive to, never a replacement for, route-level enforcement).

## §2 — Substrate-Bound Discipline Applications

- **Entitlement alignment (SR-39-adjacent).** Nav visibility for an agent (and its children, e.g. Plans & Canvas under Calculate) derives from the *same* entitlement/capability signal the route enforces. Read how OB-228 / the platform determines Calculate-agent entitlement (capability such as `icm.configure_plans`, domain registration, or the live equivalent) and gate the nav item on it. Do not invent a new entitlement concept; reuse the live one. The route-level check is the security boundary and stays; the nav check is convenience/visibility.
- **Korean Test (display discipline).** Agent labels, icons, and descriptors render from the nav config data, not hardcoded per-tenant branches. If the live config enumerates agents dynamically, preserve that — the treatment applies to *whatever* set of agents the config yields, not a fixed four hardcoded in the renderer.
- **Wayfinder Layer 1 (Unified Visual Language).** The AGENTS grouping is wayfinding — the primary "districts" of the platform. The elevated treatment is the Layer-1 ambient cue that these are the primary destinations; the utility nav below is subordinate. Use the Vialuce three-layer token system (Deep Indigo #2D2F8F / Gold #E8A838); no ad-hoc colors.
- **Behavior preservation (DD-7).** Every existing agent/nav destination must still route correctly after the restructure; no dead links, no lost entries. Plans & Canvas must remain reachable (now under Calculate) at its existing route `/configure/plans`.

## §3 — Phases (run continuously, no architect return)

### §3.1 — Phase 1: Read live nav config → decisions (flows into Phase 2)

Read the live `workspace-config.ts` and the rail/nav-rendering component(s). Establish, and record in the completion report's Decisions Log:
1. **The actual current agents/workspaces** under the "Workspaces" section — enumerate them verbatim (do not assume the count; the architect's mental model is "four," but render the treatment against whatever the config yields — see §4 if it differs).
2. **The current placement of Plans & Canvas** (the repointed `/design` → `/configure/plans` slot) and how nav entries declare children/sub-items (does the config already support nested items, or must a child structure be added?).
3. **The entitlement signal** that determines Calculate-agent access (the capability/domain check the OB-228 route uses — reuse it for nav gating).
4. **The label/heading source** for the "Workspaces" section (where the string is defined).

Decide the minimal-change approach for each of §3.2/§3.3 against the live structure. Commit any discovery notes. → Phase 2.

### §3.2 — Phase 2: Restructure + rename + entitlement-gate

1. **Rename** the section heading "Workspaces" → **"AGENTS"** (uppercase as written; if the design system lowercases/small-caps headings via CSS, keep the source string "AGENTS" and let the token styling render it — do not fight the type system).
2. **Relocate Plans & Canvas** from its top-level slot to a **child of the Calculate agent.** It keeps its route (`/configure/plans`) and label ("Plans & Canvas"). Remove the standalone top-level entry so it no longer appears at root.
3. **Entitlement-gate** the Calculate agent's Plans & Canvas child (and, if not already, the agent tiles generally) on the live Calculate-agent entitlement signal from Phase 1 — so a viewer without Calculate-agent entitlement does not see Plans & Canvas in the menu. (The route already 401s/redirects unentitled users; this removes the menu affordance to match.)
4. Verify (DD-7): every other agent and nav destination still routes; Plans & Canvas is reachable under Calculate for an entitled viewer and absent for an unentitled one. Commit. → Phase 3.

### §3.3 — Phase 3: Elevated agent design treatment (+ build verify + PR)

**Design specification** (implement this concretely; read frontend-design SKILL for exact token values). The four agents become the visually dominant region of the rail:

- **Agent tiles, not text rows.** Each agent renders as an elevated tile — taller than a standard nav row, with: an **agent icon** (leading), the **agent name** (primary text, heavier weight / larger size), and a **one-line descriptor** (secondary text, smaller, muted) stating what the agent does (e.g., Calculate → "Commission calculation & plans"; derive each descriptor from the agent's purpose in the config, or use a sensible one-liner and flag it in the Decisions Log for architect wording).
- **Container & elevation.** The AGENTS section is a distinct grouped container with subtle elevation (border and/or shadow per the token system) that separates it from the flat utility nav. Deep Indigo base; the **active/selected agent** carries a Gold accent (left border or background tint) — the one place gold appears in the rail, reinforcing "this is where you are."
- **Children subordinate.** An agent's children (e.g. Plans & Canvas under Calculate) render as indented secondary links, revealed when the agent is active/expanded, visually clearly subordinate to the agent tile (smaller, no tile chrome).
- **Contrast below.** Below the AGENTS container, a divider, then the existing utility/config nav at standard (smaller, flat) weight. The visual contrast between the elevated agent tiles and the flat utility nav is what makes the agents "stand out" — that contrast is the deliverable, so keep the utility nav restrained.
- **Entitlement-aware rendering.** Only entitled agents render as tiles (a tenant without, e.g., the Financial agent does not see that tile). The grouping adapts to the entitled set without layout breakage at 1–N tiles.
- **Wayfinder Layer 1.** The treatment is the ambient cue that these are the platform's primary destinations; preserve responsive behavior and the existing rail collapse/expand if present.

Then: build verification (kill dev → `rm -rf .next` → `npm run build` clean → `npm run dev` → confirm `localhost:3000` renders the new AGENTS rail; capture the agent enumeration rendered, Plans & Canvas present under Calculate for an entitled viewer, heading reads AGENTS). **ARTIFACT SYNC** block (MC: nav IA slice of S1 advanced; BOARD: nav/wayfinding delta; SUBSTRATE: Wayfinder Layer 1 exercised, entitlement-aligned nav). Completion report → `docs/completion-reports/HF-332_COMPLETION_REPORT.md` with the Decisions Log + before/after nav description + pasted build output. `gh pr create --base main`. **CC does not merge.**
