## §1 — Problem Statement

The platform has no effective role-based data access control. DIAG-077 documented the full scope:

- **35 scope-sensitive surfaces** render compensation, performance, and intelligence data. Of these, 7+ Intelligence surfaces and `/perform` hero pass `ALL_INSIGHTS_SCOPE` (empty `visibleEntityIds` = "all tenant rows") regardless of role. A member logging in sees every entity's payout, every peer's identity and compensation, and full tenant aggregates.
- **8 shared data functions** hardcode `ALL_INSIGHTS_SCOPE` and accept no scope parameter. The single conduit `getEntityResults` already accepts `EntityScope` — it is the seam — but every caller passes the empty all-scope.
- **The scope reader (`resolveEntityScope`) fails open** to `ALL_SCOPE` on every path, and `profile_scope` has 0 rows, so even surfaces that attempt scoping (`/insights/my-team`, `/data/transactions`, `/perform/statements` team branch) render tenant-wide.
- **Scope lives in a second context** (`persona-context.tsx`) with its own async lifecycle, separate from `auth-context.tsx`. Two scope shapes (`EntityScope` vs `PersonaScope`), two loading states, two resolution chains. HF-343 tried to add a third (`useAuthScope`) — the lesson is: fold into the existing auth lifecycle, not alongside it.
- **The persona switcher override is gated only at the UI render**, not in the context that applies it. The server trusts client-supplied `scopeEntityIds` under RLS-bypass.
- **Menu routes use a role-array fallback** (`roles: ['sales_rep', ...]`) instead of capability derivation, producing dead-ends (Finance visible to member, middleware blocks access).
- **Own-entity resolution** uses brittle email-to-entity string matching instead of the structural `entities.profile_id` linkage.

## §3 — Phase Prose

### §3.1 — Phase 1: Foundation (auth-context scope + empty-means-all closure + switcher gate)

**Objective:** Establish one scope, one lifecycle, one source of truth — in `auth-context.tsx`. Close the empty-means-all trap. Gate the persona override in the context.

**Closes:** AP1, AP2, AP4, AP6.

**Constraints:**

**1a. Fold scope into `auth-context` `initAuth` (AP6 closure, HF-343 lesson)**
- After `fetchCurrentProfile` succeeds (auth-context.tsx:195-199 per DIAG-077), resolve scope in the SAME sequence:
  - Read the profile's `role` (already available from `fetchCurrentProfile`).
  - For `admin` / `platform`: scope = ALL. Zero additional queries. `isLoading` can start `false` for admin (SSR path).
  - For `manager`: query `profile_scope.visible_entity_ids` for the profile. If no row or empty → fail-CLOSED to own entity only (AP4 closure). ONE query.
  - For `member` / `viewer`: query `entities` where `profile_id = profile.id`. If no match → DENY scope (reads nothing). ONE query.
- Store on `AuthContextType`: `scope: AuthScope`, `viewRole: Role | null`, `ownEntityId: string | null`, `canViewAll: boolean`, `canViewTeam: boolean`, `isDenied: boolean`.
- ONE `isLoading` governs user + capabilities + scope. Nothing evaluates until `isLoading` is `false`. No intermediate wrong-renders.
- Do NOT create a new hook, a new context, or a separate loading lifecycle. Do NOT import from `persona-context` for scope resolution.

**1b. Make DENY structurally distinct from ALL (AP1 closure)**
- Define `AuthScope` with explicit discriminant:
  - `{ type: 'all' }` — admin/platform, reads everything.
  - `{ type: 'team', entityIds: string[] }` — manager, reads team entities.
  - `{ type: 'own', entityId: string }` — member, reads own entity.
  - `{ type: 'deny' }` — unlinked member, reads nothing.
- `getEntityResults` (entity-results.ts:100) must handle each discriminant explicitly. `type: 'deny'` returns `[]` — NEVER collapses to "all." `type: 'own'` filters to one entity. `type: 'team'` filters to the set. `type: 'all'` is the current behavior (no `.in()` filter).
- `ALL_INSIGHTS_SCOPE` is retired or redefined as `{ type: 'all' }`. The empty-array-means-all overload is eliminated.

**1c. `hasCapability` admin/platform bypass (DS-014 §4)**
- `hasCapability`: if authenticated `user.role` is `platform` or `admin` → return `true`. This is the capability matrix rule: platform/admin inherit all capabilities.
- This bypass applies to the REAL authenticated role, NOT to the persona override. A VL admin switching to "Rep" persona still has `hasCapability` return `true` (they are authenticated as platform). The persona override changes what DATA scope is used for rendering, not what capabilities the user holds.

**1d. Gate persona override in context (AP2 closure)**
- In `persona-context.tsx`, the `override` (from `sessionStorage.getItem('vl_persona_override')`) must be ignored/cleared unless `useAuth().isVLAdmin` is `true`. Currently this check exists only on the two UI render gates (`VialuceSidebar.tsx:233`, `PersonaSwitcher.tsx:89`) — a manufactured sessionStorage key could widen a real member's scope.
- The fix: in the `useState` initializer (persona-context.tsx:110-116) and in the `setOverride` callback, check `isVLAdmin`. If not VL admin, override = `null`, sessionStorage key cleared.
- For VL admin: the override changes the persona token (theme/vocabulary) and drives which dashboard component renders. But data scope comes from `useAuth().scope` keyed off authenticated role — the override does NOT change `useAuth().scope`. The VL admin demo sees the correct persona dashboard component but with data from auth-scope (tenant-wide for admin). This is correct per Decision 39: scope derives from authenticated identity. The demo preview is approximate (shows admin-scoped data through the rep/manager dashboard lens); the precise role-view verification requires logging in as an actual member/manager user.

**1e. `resolveEntityScope` fail-CLOSED (AP4 closure)**
- `resolveEntityScope` (entity-scope.ts:22) currently returns `ALL_SCOPE` on no-profile, no-row, or empty `visible_entity_ids`. Change: return `{ type: 'deny' }` on all failure paths. This function becomes a utility called by `initAuth` for manager scope resolution — not a standalone path. Its fail-open-to-all behavior is the single most dangerous pattern on the platform.

### §3.2 — Phase 2: Scope Threading (8 data functions + own-entity resolution)

**Objective:** Thread `AuthScope` from `useAuth()` through the 8 shared data functions so every read narrows by role. Replace email-based own-entity resolution.

**Closes:** AP8. **Depends on:** Phase 1.

**Constraints:**

**2a. Thread scope through the 8 shared data functions (DIAG-077 §2)**
- Each function gains an optional `scope?: AuthScope` parameter. Default = `{ type: 'all' }` (preserving existing behavior for any caller not yet updated — backward compatible).
- All 8 functions flow through `getEntityResults` as their conduit. `getEntityResults` is the single seam: it already accepts `EntityScope`, and its `.in('entity_id', ...)` filter is the enforcement point. Update its contract to accept `AuthScope` (the new discriminated type from Phase 1).
- Functions requiring scope threading (from DIAG-077 §2):
  1. `getCalculatedPeriods` (periods.ts:22)
  2. `getPeriodTotal` (intelligence-data.ts:87)
  3. `getComponentTotals` (distribution.ts:66)
  4. `getPayoutDistribution` (distribution.ts:29)
  5. `getPopulationTrend` (trajectory.ts:26)
  6. `getEntityTrajectory` (trajectory.ts:69)
  7. `getEntityTableData` (entity-table.ts:23)
  8. `getEntityResults` (entity-results.ts:92) — update contract to `AuthScope`
- The optional default preserves backward compatibility: existing callers that don't pass scope continue to see tenant-wide (admin) behavior. Each page is updated to pass `useAuth().scope` in Phase 4. This means Phase 2 can merge independently without breaking any surface.

**2b. Replace email-based own-entity resolution (AP8 closure)**
- `/my-compensation` (and alias `/perform/compensation`) currently resolves own entity via `extractEmployeeId(user.email)` matched against `entity_id` (DIAG-077 §5.B). Replace with `useAuth().ownEntityId` from Phase 1 (resolved structurally via `entities.profile_id`).
- If `ownEntityId` is null (unlinked profile), render the existing "no data" empty state — do not fall back to email matching.

### §3.3 — Phase 3: Menu and Navigation (capability-derive, retire role-array fallback)

**Objective:** Every sidebar item derived from `hasCapability`. Zero dead-ends. Retire `roles: []` arrays.

**Closes:** AP5, AP7. **Depends on:** Phase 1 (hasCapability admin bypass).

**Constraints:**

**3a. Capability-gate all routes in `workspace-config.ts`**
- Every route currently using `roles: [...]` fallback (DIAG-077 AP5 — `/perform`, all 5 `/insights*`, `/acceleration`, all 9 finance, `/data*`, `/notifications`, `/integrations/catalog`, `/operations/*`) gains a `requiredCapability` property.
- Mapping per DS-014 §4 (encoded in `permissions.ts ROLE_CAPABILITIES`):
  - `/perform` → `view.own_results` (all roles)
  - `/insights`, `/insights/*`, `/acceleration` → `view.team_results` (manager+)
  - `/financial`, `/financial/*` → `view.team_results` (manager+; removes AP7 dead-end)
  - `/data/transactions`, `/data/quality` → `view.own_results` or `view.team_results` per surface
  - `/notifications` → no gate (all roles)
  - Operate surfaces → already `RequireCapability`-gated (DIAG-077 §5.C — low gap, leave as-is)
- `getWorkspaceRoutesForRole` (workspace-config.ts:368-371): remove the `route.roles.includes(role)` fallback entirely. Only `hasCapability(route.requiredCapability)` governs. One PDP.

**3b. Navigation context: `effectiveRole` from authenticated role**
- `navigation-context.tsx:112`: `effectiveRole` currently derives from `persona` (which includes the override). Change: derive from `useAuth().user.role` (authenticated). The persona override changes theme/vocabulary and dashboard selection, NOT navigation. A VL admin switching to "Rep" still sees the full admin sidebar.
- This is correct because: the VL admin IS an admin. Their navigation should reflect their actual entitlements. The persona switcher lets them preview the rep/manager dashboard content, not restrict their own navigation.

**3c. Dead-end elimination**
- After 3a+3b: verify that every sidebar item reachable by a role has the capability the middleware requires. If visible → accessible. If not accessible → not visible.
- The `/unauthorized` page should be unreachable from visible navigation for any role.

### §3.4 — Phase 4: Surface Enforcement (every scope-sensitive surface reads from auth-scope)

**Objective:** Every scope-sensitive surface identified by DIAG-077 passes `useAuth().scope` to its data functions. Member sees own. Manager sees team. Admin sees all.

**Depends on:** Phases 1 + 2.

**Constraints:**

**4a. Intelligence surfaces (8 pages — DIAG-077 §5.A)**
- `/stream`, `/insights`, `/insights/compensation`, `/insights/performance`, `/insights/analytics`, `/insights/trends`, `/acceleration`, `/insights/my-team`
- Each page: replace `ALL_INSIGHTS_SCOPE` calls with `useAuth().scope` passed to the data functions (which now accept `AuthScope` from Phase 2).
- For member: data functions narrow to own entity → member sees own performance, own trajectory, own component breakdown. Panels that depend on population data (distribution, leaderboard, top accelerator) render in anonymized form (percentile/quartile) or hidden, per DS-014 §8.2.
- For manager: data functions narrow to team → manager sees team aggregates, team trajectory, team standings.
- For admin: unchanged — scope = ALL, same as current behavior.

**4b. Perform/Compensation surfaces (4 pages — DIAG-077 §5.B)**
- `/perform`: hero `getPeriodTotal` passes scope. Persona dashboard dispatch keys off `useAuth().user.role` (authenticated), NOT `usePersona().persona`. Admin → AdminDashboard, manager → ManagerDashboard, member → RepDashboard.
- `/my-compensation` + `/perform/compensation`: own-entity from `useAuth().ownEntityId` (Phase 2b).
- `/perform/statements`: already has SR-39 scoped read layer — verify it reads `useAuth().scope` for the team branch.
- `/approvals`: `DrillThroughPanel` currently uses `allScope` — scope it to the approver's entity set.

**4c. Data/Investigate surfaces (3 pages — DIAG-077 §5.E)**
- `/data/transactions`: replace `resolveEntityScope(user?.id)` with `useAuth().scope`.
- `/investigate/trace/[entityId]`: gate by capability — member may only trace own entity, manager own team, admin any.
- `/performance/adjustments`: scope by `useAuth().scope` — member sees own disputes, manager sees team.

**4d. Operate surfaces (DIAG-077 §5.C — low gap, minimal changes)**
- Already `RequireCapability`-gated. Verify reads still work correctly with the new `AuthScope` type. No behavioral change expected.

**4e. Financial surfaces (11 pages — DEFERRED)**
- Financial module persona scoping is governed by `FM_Views_Data_Persona_Analysis.docx` and is separate scope. Not in this OB.
- However: the Financial API route (`/api/financial/data`) is addressed in Phase 5 (AP3 — server-side scope derivation) because the client-trusted `scopeEntityIds` pattern is a security issue regardless of Financial-specific persona rules.

### §3.5 — Phase 5: API Route Hardening (server-side scope derivation)

**Objective:** API routes that serve compensation/performance data re-derive tenant and scope from the authenticated session, not from client-supplied parameters.

**Closes:** AP3. **Depends on:** Phase 1 (AuthScope type).

**Constraints:**

**5a. Tenant binding**
- Every API route under `/api/` that currently reads `tenantId` from `body`/`query` must instead derive it from the authenticated session: `getUser()` → profile → `tenant_id`. If the caller-supplied `tenantId` differs from the session-derived one, reject with 403.
- Priority routes (from DIAG-077 §2 — cross-tenant readable):
  - `POST /api/financial/data` (route.ts:1386 — trusts body `tenantId` + `scopeEntityIds`)
  - `POST /api/ai/assessment` (trusts body `tenantId`)
  - `GET /api/platform/agent-inbox` (trusts query `tenantId` + `persona`)
  - `GET /api/insights` (trusts query `tenantId`)
  - `POST /api/calculation/run` (trusts body `tenantId`)

**5b. Scope derivation (minimum viable for this OB)**
- For routes that currently trust client `scopeEntityIds` (`/api/financial/data`): re-derive scope server-side from the authenticated user's role + profile_scope/entities linkage (same logic as Phase 1, server-side). Ignore client-supplied `scopeEntityIds`.
- This is the minimum AP3 closure. Full server-side scope enforcement (equivalent to RLS) is a subsequent hardening item — this OB ensures the client cannot override scope, which is the acute vulnerability.

**5c. Agent inbox hardening**
- `/api/platform/agent-inbox`: derive `tenantId` from session; gate `persona` parameter by role (admin→any, manager→`view.team_results`, member→`view.own_results`). Do not allow member to request `persona=admin` inbox items.
