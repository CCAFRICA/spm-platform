## §1 — PROBLEM STATEMENT

The platform's AI surfaces are, today, **single-call LLM-with-prompting** (one request → one structured JSON response → deterministic parse). That pattern is correct for atomic classification (field mapping, sheet classification, assessment narration) and must stay. But three workflows are **investigative** — they require observing an intermediate result and deciding the next query — and are currently performed **manually by the architect** or not at all:

1. **Reconciliation delta diagnosis.** When reconciliation surfaces deltas, root-cause attribution (plan-interpretation error vs. band-assignment vs. metric-derivation vs. data-import vs. boundary handling) is hand-traced across `calculation_traces`, `calculation_results`, `reconciliation_sessions`, `rule_sets.components`, and `committed_data`. The BCL and Meridian proofs each consumed weeks of this. The Reconciliation Report Specification names "**Automated root cause classification**" as Phase 3 (future) and RECON-01e ("variance reasoning from calculation traces") as a P2 sub-item. This OB delivers it as an agent.

2. **Large-plan interpretation.** `AI_ML_Architecture_Briefing` documents the chunked-pipeline gap as **NOT DONE**: monolithic single-call interpretation hits token limits on large plans, and component misinterpretation propagates silently to calculation, caught only at reconciliation. A per-component extraction loop with cross-component validation resolves this.

3. **Dispute investigation.** When a rep files a dispute ("January Optical seems wrong / a transaction is missing"), someone manually traces the calculation, locates transactions, verifies import survival, and checks period attribution. The Structured Dispute Management spec defines seven dispute categories; each maps to an investigation tool-sequence.

**Defect-class lineage.** These are not bugs; they are a **capability gap**: the platform performs Category-C (prompting) work where Category-A (agent: tools + loop + observable trajectory) work is warranted and would materially reduce time-to-diagnosis from hours to seconds. This OB introduces the platform's first genuinely agentic surfaces **without crossing the Deterministic Calculation Boundary** — the agents **read** calculation output and **reason** over it (Decision 158: LLM as recognizer); they never participate in or mutate calculation (deterministic code as constructor).

**Combined-treatment rationale.** All three agents share one harness, one capture table, one signal pattern, and one Observatory section. Building them as three bespoke efforts would triplicate the harness and the capture surface and risk divergence. This is a single ULTRACODE campaign: **prove the pattern once (keystone), replicate in parallel (file-disjoint), sweep the invariants once (batched lens fan-out).**

## §2 — SUBSTRATE-BOUND DISCIPLINE APPLICATIONS

Per-principle application of this build. Each is a binding constraint, not a description.

**Decision 158 (LLM Recognition + Code Construction).** The agents are **recognizers**. Tool handlers return facts from the database; the LLM reasons over them and emits a diagnosis. No agent constructs or alters a prime-DAG tree, a calculation result, or a rule-set component value. The recognizer/constructor split that governs the platform now governs its agents.

**Decision 154 / Korean Test (LOCKED).** Four identification classes (field, operation, primitive, dispatch-surface) carry zero language-specific literals. Applied here: every tool name (`get_entity_calculation_trace`, `find_entities_with_similar_delta`, `check_boundary_resolution`, `extract_component`, `validate_cross_reference`, `search_committed_data`, `get_period_boundaries`), every `agent_invocations` column, every `signal_type` string, and every agent `name` is **structural**. No `commission`, `optical`, `comision`, `venta`, `attainment`-as-literal, etc. The §3.3 sweep greps for domain literals across all three agents and fails on any hit. Tool **descriptions** (which the LLM reads) describe **structural patterns**, not domain semantics — e.g., "retrieve the execution trace for an entity's calculation," not "retrieve the commission trace."

**Carry Everything, Express Contextually (T1-E902 v2, LOCKED).** The agents are **read-only above the boundary**. No agent narrows persistence; no agent writes to `committed_data`. They consume `row_data` as-is. Express-Contextually governs only how the agent's diagnosis is **presented** (agent_inbox severity, Observatory section), never persistence.

**Progressive Performance (constitutional).** Every agent route computes a `request_fingerprint` (a stable hash of the investigation subject + the relevant rule-set version) and checks `agent_invocations` for a recent completed run with the same fingerprint **before** making any paid LLM call. A same-fingerprint second encounter returns the cached trajectory at **$0 cost, `cache_hit = true`**, in roughly the time of one DB query. A surface that re-runs a cold investigation on every identical encounter is a **FAILURE** and the EPG for each agent empirically proves the cached second encounter. (This mirrors the VG IRA's `ira_request_hash` + 24h cache, proven there.)

**AIService mandate (the only way to call AI).** Every per-turn model call goes through the existing Anthropic adapter (`web/src/lib/ai/providers/anthropic-adapter.ts`) via the AI service layer. The agent runner does **not** open its own `fetch` to `api.anthropic.com`. The §3.3 sweep greps agent code for `ANTHROPIC_API_URL` / direct `fetch(...anthropic...)` and fails on any hit outside the adapter. Provider abstraction (Principle 6) is preserved: switching providers remains a one-file change.

**Deterministic Calculation Boundary (sacred).** No agent tool issues an INSERT/UPDATE to `calculation_results`, `calculation_traces`, `entity_period_outcomes`, or any `rule_sets.components` value. The §3.3 sweep greps every tool handler for writes to calc tables and fails on any hit. Agents write **only** to `agent_invocations`, `agent_inbox`, and `classification_signals`.

**Vertical Slice rule.** Engine and experience evolve together — here the "engine" is the agent runtime and the "experience" is the in-browser Diagnose/Investigate action plus the Observatory Agent Operations section. Every agent ships its trigger UI, its capture, and its measurement in the same campaign. No agent ships as a headless API with no surface.

**Reconciliation-channel separation (architect-channel only for GT).** The benchmark the Reconciliation Diagnosis Agent reads is the **customer's uploaded legacy output** already in `reconciliation_sessions`. It is not architect ground truth. CC receives no expected values.

#### §3.1.A — The agent loop harness (`agent-runner`)

Create `web/src/lib/ai/agent/agent-runner.ts`. It is the **only** new place the tool loop lives; both Phase 2 agents import it unchanged.

Contract (structural; no domain vocabulary):
```typescript
// web/src/lib/ai/agent/agent-runner.ts
import { AnthropicAdapter } from '../providers/anthropic-adapter';
// (or invoke through AIService if the adapter is not directly constructable per Phase 0 Target A findings)

export interface AgentToolDefinition {
  name: string;                 // structural, e.g. "get_entity_calculation_trace"
  description: string;          // structural pattern, not domain semantics
  input_schema: Record<string, unknown>; // JSON schema
}

export interface AgentDefinition {
  name: string;                 // structural agent id, persisted to agent_invocations.agent_name
  systemPrompt: string;
  tools: AgentToolDefinition[];
  handlers: Record<string, (input: Record<string, unknown>) => Promise<unknown>>;
  maxTurns: number;             // HALT-RUNAWAY guard (e.g. 12)
  model?: string;               // optional override; default = adapter's env-configured model (Phase 0 Target A)
}

export interface AgentTurn {
  turn: number;
  tool: string | null;         // null on the final text turn
  input: Record<string, unknown> | null;
  outputSummary: string;       // bounded summary of the tool result (NOT the full payload)
}

export interface AgentRunResult {
  finalText: string;
  turns: AgentTurn[];
  turnCount: number;
  tokenUsage: { input: number; output: number };
  // cost computed by caller from token usage + model pricing
}

export async function runAgent(
  def: AgentDefinition,
  initialUserContent: string,
): Promise<AgentRunResult> {
  // messages = [{ role:'user', content: initialUserContent }]
  // loop up to def.maxTurns:
  //   call the adapter ONE turn WITH tools=def.tools (this is the only behavioral
  //     delta from the current single-call adapter path)
  //   find tool_use block in response.content
  //   if none -> capture final text turn, return AgentRunResult
  //   else -> dispatch def.handlers[block.name](block.input); append the assistant
  //     content and a user tool_result block; record an AgentTurn with a BOUNDED
  //     outputSummary; continue
  // if maxTurns exceeded without a text turn -> throw AgentRunawayError (HALT-RUNAWAY)
}
```

- **UI-driven model/provider control (the AI Substrate *control* surface).** OB-212 ships the AI Substrate panel **read-only** (§3.1.H). It does **not** add the ability to *change* a surface's model or provider from the UI. That capability — a config resolver that reads provider/model from `platform_settings` at the `AIService` chokepoint (replacing construct-time env resolution), singleton invalidation on change, optional per-surface model binding, and the **OpenAI provider adapter** (today a commented-out stub in `createAdapter`; note its tool-use format differs from Anthropic's `tool_use`/`tool_result`, so driving the agent loop on it is a deeper lift than the six single-call surfaces) — is a separate vertical slice. It is forward-referenced in §6A. Folding it here would change every existing AI surface's resolution path and is the DD-7 scope-creep this directive refuses.

## §6A — RESIDUALS

Operative-state known gaps, named:
- **Confirm/override control for diagnoses** is introduced minimally in Phase 1 (capture-at-generation) and surfaced in Phase 2; the **Confirmed rate** Observatory metric is only meaningful once admins begin confirming/overriding. The control's richer UX (bulk confirm, dispute-seeding from a confirmed diagnosis) is a follow-on.
- **Model selection for agent loops** defaults to the adapter's resolved provider/model (today env-configured, cached in the `AIService` singleton). A reasoning-capable model improves multi-turn diagnosis. **Forward reference:** UI-driven, per-surface model/provider selection — and the OpenAI adapter that makes a true provider swap real — is its own capability. It should be locked in a short **DS (Model & Provider Configuration)** that decides the three open forks (global vs per-surface binding; expose-but-gate OpenAI vs build the adapter, including whether it must drive the agent tool-loop on day one; platform-global vs per-tenant scope), then built in a follow-on **OB (number derived from live `docs/vp-prompts/`)**. OB-212's agents read the adapter's resolved model, so when that resolver lands they inherit it in one place — nothing here is baked in that the follow-on must rip out. The read-only AI Substrate panel (§3.1.H) is the surface that capability later attaches controls to.
- **`agent_inbox` dedup/expiry** for repeated diagnoses on the same session relies on the existing inbox semantics; a dedup rule keyed on `request_fingerprint` is a follow-on if inbox noise appears.
- **Trajectory depth tuning** (`maxTurns`) starts conservative; empirical run data in `agent_invocations.turn_count` informs whether the ceiling needs adjustment (raise only with pasted evidence, never to mask a HALT-RUNAWAY).
- **Sequence-number confirmation** (the §0 gate) must be closed by the architect against live `docs/vp-prompts/` before dispatch.
