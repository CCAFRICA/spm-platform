## §1 — Objective

Under `data-theme="vialuce"`, three pages tell the full story:

- **`/insights/analytics` — CORRECT.** White KPI cards with 3px indigo accents, DM Mono numbers at 27px, delta chips with semantic color + icon, indigo-ramp chart bars, proper card shadows, gold accent on signal elements. This is what the design spec looks like when applied.

- **`/stream` — WRONG.** Page frame (`.page`/`.phead`) is present from HF-314, but the content sections (System Health, Trajectory Intelligence, Lifecycle, Population Distribution, Data Health) are rendered by imported sub-components that still use their original dark-theme structure.

- **`/perform` — WRONG.** Same problem at deeper scale — dark indigo KPI backgrounds, dark card surfaces, dark-context text. Every content element is a sub-component built for the always-dark app.

**The objective:** Every page under Vialuce must look like `/insights/analytics` — not like `/stream` or `/perform`. HF-314 framed the pages. This HF fills in the content.

**Why `/insights/analytics` works and the others don't:** Its content is rendered inline or by components that already consume the semantic tokens correctly. `/stream` and `/perform` render content through shared sub-components (SystemHealthCard, KPICard, PerformanceGovernance, DataTable, LifecycleStepper, etc.) that have hardcoded dark utilities, inline dark gradients, and dark-context assumptions baked into their JSX.

**The scope is the sub-components, not the pages.** Converting a shared sub-component fixes every page that imports it. Trace the import tree from every Vialuce-navigable page to its content-rendering components. Convert each component to render with the design spec vocabulary under `useIsVialuce()`. The else-branch preserves existing rendering unchanged. Trace deep — if a converted component imports another component that also renders dark content, that nested component also needs conversion.

## §2 — Constraints

1. **Design package is authoritative.** `docs/design/vialuce-theme/vialuce-design-spec.md` defines every component class. `docs/design/vialuce-theme/vialuce-layout-redesign.html` is the visual target. `docs/design/vialuce-theme/vialuce-ui.css` has the CSS.

2. **`useIsVialuce()` conditional on every change.** Vialuce rendering path is the new branch. Else-branch is existing code, byte-unchanged. Dark and Bliss cannot regress.

3. **Design spec class vocabulary (already shipped in globals.css, scoped to `[data-theme="vialuce"]`):**
   - KPI cards → `.kpi` / `.kpi-top` / `.kpi-ic` / `.kpi-delta` / `.kpi-val` / `.kpi-foot` / `.bar`
   - Cards → `.card` / `.card.flush` / `.card-h`
   - Tables → `.tbl` / `th` / `td` / `.num` / `.num.up` / `.num.down`
   - Status → `.pill.success` / `.pill.danger` / `.pill.open` / `.pill.neutral`
   - Insight → `.insight` / `.spark` / `.lbl`
   - Empty → `.empty` / `.ic`
   - Charts → indigo ramp palette (`--vl-raw-indigo-deep` → `--vl-raw-indigo` → `--vl-raw-indigo-light` → `#9A9CE0`, gold accent)

4. **Numbers always in DM Mono.** Non-negotiable. Use `.kpi-val` or `.num` class.

5. **No skipping.** Every content-rendering sub-component reachable from a Vialuce-navigable page gets converted.

6. **Server components that can't use hooks:** Wrap in a client boundary or apply CSS-only treatment. Do not skip.
