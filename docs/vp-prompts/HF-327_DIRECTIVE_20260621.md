## §1 — PROBLEM STATEMENT

The Vialuce theme arc (OB-221→HF-316) adopted 46 routes with the three-layer CSS token architecture (Layer 0 primitives, Layer 1 semantic, Layer 2 component slots). Subsequent HF-324 fixed the reconciliation page's inline `CARD_STYLE` constant. Despite these sweeps, grey/dark backgrounds persist across the platform — individual inline styles, component-level constants, and Tailwind dark-mode classes that bypass the token system.

The defect pattern is always the same: a component uses a hardcoded color value (`rgba(24,24,27,0.8)`, `bg-zinc-800`, `#1A1A2E`, `bg-gray-900`) instead of consuming a CSS custom property (`var(--vl-surface-primary)`) or the `useIsVialuce()` hook. Each instance is a trivial fix, but the DISCOVERY is the bottleneck — we've been playing whack-a-mole across CLT-226 and CLT-227, finding 2–3 grey surfaces per session, fixing them, finding more next session.

**Known grey surfaces (CLT evidence):**

| Surface | Route | Tenant | Persona |
|---|---|---|---|
| Leakage Monitor location ranking cards | `/financial/leakage` | Sabor | Admin |
| Observatory Model Config task rows | `/select-tenant` → Model Config | Observatory | VL Admin |
| Manager persona views (multiple pages) | Various | All | Manager |
| Rep persona views (multiple pages) | Various | All | Rep |
| Import Quarantine (functional, design unclear) | `/operate/import/quarantine` | MIR | Admin |

**Unknown grey surfaces:** An unknown number of pages across the platform's ~46+ adopted routes and their Manager/Rep persona variants still render with legacy grey/dark styling that was missed by the theme arc. Without a systematic audit, every future CLT session will discover more.

**Module-context unawareness (CLT-227, `/stream` Intelligence page, Sabor tenant):**

The Intelligence page assumes every tenant runs the full ICM pipeline. For Sabor Grupo Gastronomico (Financial Agent tenant, ICM rule sets archived per Decision 62), the page displays contradictory and misleading content:

- **"4 periods ready to calculate" with Calculate Now buttons** — Sabor is a financial tenant, not running the ICM calculation engine. These buttons invoke a pipeline that is irrelevant.
- **"19 periods need data" with Import Data prompt** — refers to ICM data import (transaction/sales data for commission calculation), not POS cheque import. Sabor's data IS imported — it's in `committed_data` as POS cheques via the Financial Agent pipeline.
- **"Attention Required" coaching recommendations** — "Below benchmark at 59%" — these benchmarks derive from ICM `calculation_results` attainment. Sabor's `calculation_results` are financial performance indices, not ICM commission attainment. The 59% benchmark is meaningless.
- **Pipeline Readiness "No data imported"** — false. 263,250 POS cheque rows exist in `committed_data` for Sabor. The Intelligence page queries the ICM import pipeline, not the financial data layer.
- **SYSTEM HEALTH MX$520,523.49 "Total payout this period"** — reads `calculation_results` from Sabor's **archived** ICM rule set (Decision 62). This number should not be surfaced at all.

The Intelligence page needs **module-context awareness**: query the tenant's active configuration (financial_enabled, has active rule_sets, has committed_data by type), and adapt its sections accordingly. For a financial tenant: show financial intelligence (revenue trends, leakage alerts, staff performance flags, period-over-period changes), hide ICM pipeline status.

## §2 — OBJECTIVES

CC achieves all five. The INNOVATION is O1 (systematic codebase-wide audit) and O5 (module-context Intelligence page — the page adapts to what the tenant actually uses).

**O1 — THEME HEALTH INVENTORY (THE AUDIT)**

Before fixing anything, CC performs a codebase-wide audit that produces a complete inventory of every non-token-consuming surface in the application. This is the structural innovation — instead of fixing reported pages, CC finds ALL of them.

**Step A — Automated discovery.** CC runs grep/ripgrep across `web/src/` for:

```
CATEGORY 1: Inline style hardcoded colors
- rgba\( patterns in .tsx/.ts files (inline styles with opacity backgrounds)
- Hex color literals (#1A1A2E, #18181b, #27272a, #3f3f46, #52525b, etc.) in style objects
- backgroundColor: patterns not using var(--vl-*)

CATEGORY 2: Tailwind dark-mode bypasses
- bg-zinc-*, bg-gray-*, bg-slate-*, bg-neutral-* class usage in component JSX
- dark: prefix classes that assume the old dark-mode system
- text-zinc-*, text-gray-* that should be semantic token references

CATEGORY 3: Missing useIsVialuce consumption
- Components rendering card/surface/background elements that don't import or call useIsVialuce()
- Components using the old theme system (dark/bliss) without vialuce branch

CATEGORY 4: Inline style constants
- Named constants like CARD_STYLE, PANEL_STYLE, etc. with hardcoded rgba/hex
- Object spread patterns (...CARD_STYLE) that bypass the token net
```

**Step B — Inventory artifact.** CC produces `docs/diagnostics/HF-327_THEME_HEALTH_INVENTORY.md` containing:

```
For each finding:
- File path + line number
- Current value (the hardcoded color/class)
- Component context (what renders — card, background, text, border)
- Token replacement (which CSS custom property should be used)
- Theme scope (affects which personas/views)
```

This inventory is the reusable artifact. Future theme changes, new theme additions, or brand evolution audits run the same grep patterns against this baseline.

**Step C — Fix all findings.** CC replaces every hardcoded value with the appropriate CSS custom property or `useIsVialuce()`-aware conditional. Every fix uses the token system — zero hardcoded Vialuce palette values (#2D2F8F, #E8A838, etc.).

**O2 — PERSONA VIEW SWEEP**

After O1's automated audit, CC manually navigates every accessible route under each persona (Admin, Manager, Rep) on localhost to catch surfaces the grep might miss — dynamically constructed styles, conditional rendering branches, third-party component wrappers.

For each persona, CC navigates:

**Performance workspace:** Intelligence, Performance Overview, and any Manager/Rep-visible dashboards or insights pages.

**Calculation workspace:** Lifecycle Cockpit, Run Calculations, Results Table, Reconciliation, Statements, Dispute Resolution, Payout Approvals, Payroll Reports — under each persona.

**Finance workspace (Sabor):** All 9 financial pages under Admin. Verify Manager/Rep see appropriate subsets.

**Platform Core workspace:** Configure pages, People & Access, Audit Trail, Data Visibility (Transactions, Data Console, Data Quality, Reports), Notifications, Integrations, Operations.

**Observatory (VL Admin only):** Command Center, Intelligence, Model Config, AI Cost, Revenue, Settings.

Any grey/legacy surface not caught by O1's grep is added to the inventory and fixed.

**O3 — THREE-THEME VERIFICATION**

Every surface touched by O1/O2 must render correctly under ALL THREE themes: Dark, Bliss, and Vialuce. The theme switcher (visible at bottom of sidebar: "Dark | Bliss | Vialuce") is the test mechanism.

The principle: **theme fixes consume tokens, never hardcode palette values.** A surface fixed by replacing `rgba(24,24,27,0.8)` with `#2D2F8F` is NOT fixed — it breaks Dark and Bliss. The correct fix is `var(--vl-surface-primary)` which resolves to the right value under each theme.

**O5 — INTELLIGENCE PAGE MODULE-CONTEXT AWARENESS**

The Intelligence page (`/stream`) adapts its content based on the tenant's active module configuration. CC reads the page component and the data sources it queries, then implements conditional rendering based on tenant context.

**Detection logic.** CC determines the tenant's module profile from existing data:
- `financial_enabled` on the tenant record (or `useFinancialOnly()` hook — Decision 40)
- Active (non-archived) `rule_sets` exist → ICM pipeline is relevant
- `committed_data` with `data_type = 'pos_cheque'` → Financial Agent data exists
- CC reads the actual component code to understand how the page currently determines what to show

**For an ICM tenant (BCL, CRP, MIR):** The current Intelligence page behavior is correct — show calculation pipeline status, period readiness, entity benchmarks from `calculation_results`, coaching recommendations. No changes needed.

**For a Financial tenant (Sabor):** The Intelligence page adapts:

| Current (ICM-assumed) | Financial-adapted |
|---|---|
| "X periods ready to calculate" + Calculate Now | **HIDE** or replace with financial period summary (revenue by period, trend direction) |
| "X periods need data" + Import Data | **HIDE** or replace with data freshness indicator (last POS import date, cheque count) |
| "Attention Required" coaching (attainment benchmarks) | Replace with financial attention items: locations below revenue benchmark, leakage above threshold, staff underperformers |
| Pipeline Readiness (ICM import status) | Replace with Financial data health: cheque coverage by period, location coverage, data completeness |
| SYSTEM HEALTH "Total payout" from calculation_results | Replace with financial SYSTEM HEALTH: network revenue, active locations, leakage rate, tip rate |

**Implementation approach — CC determines autonomously.** Options include:
- Conditional rendering in the existing component based on `useFinancialOnly()` or tenant feature flags
- A parallel financial intelligence component that the page renders when financial context is detected
- Adapting the existing data-fetching hooks to query financial data sources (`/api/financial/data` route) when in financial context

The key constraint: **do not create a separate `/financial/intelligence` page.** The Intelligence page at `/stream` is the canonical surface (Decision 14 — one canonical surface). It adapts, it doesn't duplicate.

**HALT-5:** If implementing module-context awareness requires modifying more than 3 files outside the Intelligence page directory (e.g., shared hooks, layout components, context providers), document the scope expansion and proceed only if the additional files are ≤ 30 lines changed each. Beyond that, document as a residual — the adaptation may need its own OB.

## §3 — CONSTRAINTS

**C1 — TOKEN-ONLY FIXES.** Every color replacement uses CSS custom properties (`var(--vl-surface-primary)`, `var(--vl-text-primary)`, `var(--vl-line)`, etc.) or the `useIsVialuce()` hook's conditional object. NEVER replace a hardcoded grey with a hardcoded Vialuce color. The fix must work under all three themes.

**C2 — STYLING ONLY (O1–O4). FUNCTIONAL ADAPTATION (O5 ONLY).** For theme sweep objectives (O1–O4): zero functionality changes. No route additions, no API changes, no data modifications, no component logic changes. If a page is a stub (redirect to /operate), it stays a stub — don't build functionality, just ensure the stub page itself respects the theme. **Exception: O5 (Intelligence page)** is explicitly a functional adaptation — conditional rendering, data source switching, and content adaptation based on module context. O5 changes are scoped to the Intelligence page and its direct data-fetching dependencies.

**C3 — BROWSER-RENDERED EVIDENCE.** Proof gates require screenshots or browser-rendered page descriptions showing the surface under Vialuce theme with no grey backgrounds. Same methodology correction as HF-324.

**C4 — PERSONA SWITCHING.** Use the Demo Persona Switcher bar (bottom of sidebar: Admin | Manager | Rep) to test each persona view. If the persona switcher is not available for a tenant, use direct URL navigation + role context.

**C5 — NO AUTH CHANGES.** The persona sweep reads existing routes under existing roles. CC does not modify authentication, authorization, or role-based access control. If a route returns "Access Restricted" for a persona, that's correct behavior — document it, don't fix it.

**C6 — COMPLETION REPORT PATH.** `docs/completion-reports/HF-327_COMPLETION_REPORT.md` — NOT project root. This is now a standing rule.
