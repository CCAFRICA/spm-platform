## §1 — PROBLEM STATEMENT

**Mechanism (fully traced, QD-1/QD-2):** `vialuce-last-activity` (and `vialuce-session-start`) are browser-session-scoped cookies that outlive the auth session they describe. On login, `signInWithEmail` (auth-service.ts:32-48) awaits `logAuthEventClient('auth.login.success')` → POST `/api/auth/log-event` → **traverses middleware** before `fetchCurrentProfile` runs. Middleware (middleware.ts:204/:241-246) reads the stale cookie, computes idle against a **dead session's** clock, and `clearAuthCookies` destroys the just-created sb-* session. `getUser()` then nulls at auth-service.ts:182-183 — short-circuiting BEFORE the profiles query — and auth-context.tsx:256 mislabels the failure "Account found but profile is missing." Evidence anchors: admin@saborgrupo.mx sign-in 17:12:39Z → `auth.session.expired.idle` 17:12:40Z; gerente/mesero failed 2026-06-09 (predates HF-283 — exonerated). The absolute-timeout path (:234, `vialuce-session-start`) carries the identical dead-cookie exposure. Lineage: HF-151's awaited log POST made the middleware traversal deterministic at exactly the vulnerable instant. The "profile is missing" string also over-claims: it wraps BOTH the session-absent null (auth-service:182, the live branch here) and the true zero-rows null (resolve-identity:91-94).

**Class:** cross-session state leakage — per-browser bookkeeping adjudicating a session it doesn't belong to. Closing only the idle path would be instance closure; the invariant closes idle AND absolute together.

**Combined arms (named per DD-7):** (1) the structural invariant (middleware); (2) honest error surfaces — session-absent vs profile-absent are distinct user messages; (3) observability — the silent client-side branches emit via the existing client-capable path. Arms 2–3 are explicit scope decisions: the mislabel and silence directly extended this diagnosis across rounds.
