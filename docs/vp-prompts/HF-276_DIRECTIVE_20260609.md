## §1 Problem Statement

c0 (Rendimiento de Ingreso) overpays by MX$15,800 in January (59,800 vs GT 44,000). Four of five Meridian components reconcile exactly after HF-274 + HF-275; c0 is the sole remaining defect.

**Root cause:** c0's `compositional_intent.scale` declares `side: "evaluator", unit: "percent", value: 100`. The OB-200 evaluator multiplies the ratio operand × 100 before comparison (1.1031 → 110.31). But the breakpoint constants in the DAG are in ratio space (0.8, 0.9, 1.0, 1.3) and carry `meta: { scale: 100 }`. The comparison `110.31 ≥ 1.3` is trivially true for nearly every entity, so all entities enter the ≥130% tier regardless of actual attainment.

**Same class as HF-274, opposite side.** HF-274 fixed convergence-side: the ratio was unscaled (0–1) against percent-space breakpoints (85, 90, 95, 98). c0 is the mirror: the ratio IS scaled (×100) but the breakpoints remain in ratio space (0.8, 0.9, 1.0, 1.3). Both produce a magnitude mismatch between the value being compared and the threshold it's compared against.

**The fix:** When `buildConstantWithScale` constructs breakpoint constants for a component with `scale.side === "evaluator"` and the comparison involves a ratio, pre-multiply the breakpoint values by `scale.value`. So 0.8 becomes 80, 0.9 becomes 90, 1.0 becomes 100, 1.3 becomes 130. The evaluator then compares: 110.31 ≥ 130 → FALSE, 110.31 ≥ 100 → TRUE → correct tier.

(×100). Nearly every entity enters the top tier.

Option A: Pre-multiply breakpoint constants by scale.value in constructTree

  - AI-first: Yes — consumes LLM's declared scale metadata
