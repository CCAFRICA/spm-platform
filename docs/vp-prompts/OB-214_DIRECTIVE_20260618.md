## §1 — PROBLEM STATEMENT

The platform interprets compensation plan documents (PDF, PPTX) into executable `rule_sets.components` — the bridge between a natural-language plan and the deterministic calculation engine. HF-248 shipped a two-phase per-component orchestration that replaced the monolithic single-call path. That was a real improvement. But four gaps remain, all documented with evidence:

**Gap 1 — No self-correction.** When the LLM extracts a component incorrectly, the code accepts whatever the LLM returns and passes it downstream. Five consecutive hotfixes (HF-156 through HF-159) all failed because the prompt constrained the AI's vocabulary — the LLM picked "tiered_lookup" as the closest match for a linear formula because the prompt only offered `[matrix_lookup, tiered_lookup, percentage, flat_percentage, conditional_percentage]`. An agent can validate its extraction against the source text and re-extract on mismatch. The current path has no mechanism for this.

**Gap 2 — No cross-component reasoning during extraction.** HF-248's cross-component validation runs *after* all components are extracted. An agent extracting Component 3 can already know what it extracted for Components 1 and 2 and use that context: "Component 1 uses individual attainment as its input; Component 3 references store attainment — these are different scope levels requiring different data sources." The agent validates *during* extraction, not post-hoc.

**Gap 3 — Token-limit fragility on large plans.** The AI Architecture Briefing documents this as Backlog 7.11 and Open Item #26 (Critical): "Any plan exceeding model context limits = complete failure with silent degradation." HF-248's per-component calls mitigate this for the extraction phase, but the initial component-identification pass still sends the full document. An agent with a `read_plan_section` tool reads the document in sections, so no single LLM call needs the full text. Scales linearly with document size.

**Gap 4 — Plan structure intelligence is extracted and discarded.** During extraction and validation, the LLM reasons deeply about the plan's structure: it notices tier-table gaps, boundary ambiguities, cross-component scope conflicts, unreachable incentives, and field references that are derived concepts rather than literal data columns. Under HF-248, all of that reasoning is consumed internally and thrown away. The admin receives components but not the intelligence the AI gathered *about* those components. The Thermostat Principle (Principle 4) says: act on data, don't just display it. The agent should surface what it finds — not as an error report, but as structural observations the admin and downstream agents can act on.

**Defect-class lineage.** The five HF-156→159 failures, the active HF-237 regression (source type enumeration at the prompt layer — pipeline observatory documents this as "registry limits what the LLM can express"), and Backlog 7.11 are all instances of the same structural class: *the plan interpretation path is code-orchestrated with no LLM-side reasoning loop.* An agent with tools, a loop, and self-validation closes the class. Gap 4 extends the closure from "better extraction" to "plan intelligence" — the agent's reasoning becomes a platform asset, not a disposable intermediate.

**Reuse.** OB-212 shipped the agent infrastructure: `agent-runner.ts` harness, `agent_invocations` capture table, Observatory Agent Operations section, AI Substrate panel, `classification_signals` capture path, `agent_inbox` write, and the Progressive Performance `request_fingerprint` cache mechanism. This OB builds **one new agent** on that proven infrastructure — no new tables, no new Observatory sections, no harness changes.

## §2 — SUBSTRATE-BOUND DISCIPLINE APPLICATIONS

**Decision 158 (LLM Recognition + Code Construction).** The agent is a **recognizer**: it reads the plan document and emits two outputs: (1) structured per-component definitions (CompositionalIntent JSON or the current component structure — Phase 0 determines the live output contract), and (2) structural observations about the plan that the agent noticed during interpretation. Deterministic code downstream (`convertComponent`, `intent-transformer.ts`) constructs the executable intent from the component definitions. The agent never writes to calculation tables, never constructs prime-DAG trees, never touches the engine. The boundary holds by construction: the agent's tools read plan content and return structural observations; the agent's component output feeds the same downstream path HF-248 feeds; the agent's observation output is additive (new consumers read it; existing consumers ignore it).

**Downstream coupling (OB-216 discovery).** The field names this agent emits in component definitions are consumed by the convergence binding layer, which binds plan-referenced fields to data columns. A dedicated Convergence Binding Agent OB (sequenced after OB-214, same OB-212 harness) will design against this agent's actual output. The agent's observations about field references — specifically, free-form structural descriptions of whether a field is a literal data column or a derived concept and how it derives — are the mechanism by which the downstream binding layer understands the handoff. This is NOT a fixed classification. The agent describes each field reference in its own structural language, the same way it describes any other observation. The Convergence Binding Agent reads these descriptions; it does not match against an enum.

**Korean Test (Decision 154, LOCKED).** Every tool name is structural. Tool *descriptions* describe structural patterns ("extract a component's calculation structure from the plan document"), not domain semantics ("extract the commission formula"). Observation output uses structural language: "gap in boundary definitions between S/50,000 and S/60,000" not "commission tier gap." The Phase 3 sweep greps for domain literals.

**Progressive Performance (constitutional).** The agent computes `request_fingerprint = hash(document_content_hash + tenant_id + rule_set_version)`. If the same plan document has been successfully interpreted before (same content, same tenant, same rule set version), the cached interpretation — components AND observations — is returned at $0 with no LLM call. A plan document re-uploaded unchanged gets the cached result. A plan document that has been edited gets a fresh interpretation. The EPG proves the cached second encounter empirically.

**Carry Everything, Express Contextually (T1-E902 v2).** The agent reads the full plan document (persistence-complete). Its per-component extraction is an *expression* choice (focused context for better accuracy), not a *persistence* narrowing. The full document remains available to every tool call.

**Vertical Slice.** The agent ships with its trigger (in the SCI execute path for plan-classified content units), its capture (`agent_invocations` trajectory + `classification_signals` per component + per observation), and its measurement (Observatory Agent Operations + AI Substrate rows go live for `plan_interpretation`). The existing reconciliation-match-rate serves as the downstream ground-truth proxy — no new ground-truth surface needed.

**Plan Anomaly Detection (spec subsumption).** The existing Plan Anomaly Detection Specification (`ViaLuce_Plan_Anomaly_Detection_Specification.md`) describes anomaly detection as a separate service sitting "after AI extraction, before plan approval." The agent subsumes this: anomaly detection is a natural byproduct of the agent's validation loop, not a separate pass. The specification becomes a reference (patterns the agent might notice), not a gate (patterns the agent is limited to). The agent is free to notice patterns not in the specification — it describes what it sees in free-form structural language. No fixed taxonomy. No developer update required when a new pattern appears.

### §3.1 — PHASE 1: ADR + TOOLS + AGENT + ROUTE

#### §3.1.0 — Architecture Decision Record (mandatory, committed before code)

`OB-214_ADR.md` — answers three questions grounded in Phase 0:

**Q1: Replace or layer?** Does the agent *replace* HF-248's two-phase orchestration at its call site (the agent now IS the plan interpretation path), or does it *layer on top* (the agent orchestrates calls to HF-248's existing per-component extraction)? The answer depends on what Phase 0 found:
- If HF-248 is a thin orchestrator that calls the LLM per component with no validation — **replace it.** The agent subsumes its function with tools + loop + self-validation. The output contract is matched, so `convertComponent` / downstream is untouched.
- If HF-248 has substantial logic beyond LLM calls (non-trivial validation, structural transformation, variant handling) — **layer on top.** The agent uses HF-248's extraction as a tool and adds the observe-reason-act loop. Preserves the proven logic.
- If the boundary is unclear — **HALT-ADR** for architect disposition. Do not guess.

**Q2: Tool set.** Based on the document content shape (Target B) and the output contract (Target A), specify the tools with exact input/output schemas.

**Candidate tools (refined against Phase 0 findings):**
- `read_plan_section(document_ref, section_hint?)` — returns a section of the plan document text. If Target B shows the document arrives as full text, this is a substring; if it arrives as pages, this is page-ranged. The tool signature depends on Phase 0.
- `extract_component(document_ref, section_or_page_range, prior_components_summary?)` — focused single-component extraction. The `prior_components_summary` carries what previous components use (scope levels, metrics, field references) so the LLM has cross-component context *during* extraction, not after. Output shape matches the downstream contract (Phase 0 Target A).
- `validate_extraction(component_json, source_text_excerpt)` — the self-correction tool. The LLM re-reads the relevant plan text and checks whether its extracted structure (tiers, rates, boundaries, scope) matches what the plan actually says. Returns validation result + discrepancies. If discrepancies found, the agent re-extracts.
- `validate_plan_completeness(components_json, observations_so_far)` — after all components are extracted, checks: are component metrics distinct? Do cross-references resolve? Are there orphan references? Are there boundary gaps, unreachable incentives, or ambiguous scope references? Returns completeness assessment + any final observations.

**Q3: Observation output shape.** Based on Target E (output contract extensibility), how does the agent emit its plan observations? The observations are the agent's free-form structural descriptions of what it noticed during interpretation — boundary gaps, ambiguous references, field derivation characterizations, cross-component scope conflicts, unreachable conditions.

The critical constraint: **no fixed taxonomy, no enum, no developer-maintained classification.** The agent describes what it sees in its own structural language. Each observation carries:
- A free-form structural description of what was noticed (the agent's own words)
- The component reference(s) it relates to (if any)
- The source text excerpt that triggered it
- A severity signal (the agent's assessment of how consequential the finding is)

This is NOT a typed anomaly registry. The Plan Anomaly Type Registry (`ViaLuce_Plan_Anomaly_Detection_Specification.md`) exists as a reference the agent can draw on — patterns it might notice. But the agent is unconstrained: if it observes a pattern not in the registry, it describes it. No developer adds a type. No enum expands. The agent reads the plan and says what it found.

The answer depends on Phase 0 Target E:
- If the output contract has an existing extension surface (metadata field, sidecar) — observations go there. Zero contract change for existing consumers.
- If the output contract has no extension surface — add an optional `plan_observations` array that existing consumers ignore. The downstream `convertComponent` path reads only the component fields it always read; the new field is additive.
- If additive output would break the downstream path — emit observations exclusively through `classification_signals` and `agent_invocations` trajectory. The Observatory surfaces them; the component contract is untouched. The Convergence Binding Agent reads them from signals.
- If the boundary is unclear — **HALT-Q3** for architect disposition.

Commit the ADR before code.

3. Validate each extraction against the source text; re-extract on mismatch.

## §6A — RESIDUALS

- **The replace-vs-layer decision (§3.1.0 Q1)** is Phase 0–derived, not pre-decided. This is deliberate — HF-248's code may have non-trivial logic that's worth preserving as a tool rather than discarding. The ADR locks it before code.
- **The observation output shape (§3.1.0 Q3)** is Phase 0–derived (Target E determines extension surface). The critical constraint is not where observations live but that they are free-form structural descriptions with no fixed taxonomy. The agent describes what it sees; no developer maintains an observation type registry.
- **Self-correction loop exercisability.** Simple plans (3–4 clear components, no ambiguity) may validate on first extraction every time. The loop's value shows on complex plans (ambiguous tiers, cross-referencing components, mixed scope levels). The EPG honestly records whether the loop was exercised.
- **Plan document availability for proof.** Five MIR PDF plans are provided by the architect and available on disk. OB-133 (PR #152, March 2026) built PDF acceptance into the SCI upload surface. If the PDF → SCI path has regressed since OB-133, Phase 0 Target B will find the regression. HALT-PLAN-DOC governs only if no viable import path exists after Phase 0. If the SCI pipeline path is broken but the agent route accepts extracted text directly, the proof can proceed via the route with extracted plan text — the agent is exercised even if the SCI-to-agent wiring is proven separately.
- **Downstream sibling: Convergence Binding Agent OB.** Sequenced after OB-214, built on the same OB-212 harness. Takes plan-recognized requirements and binds them to data columns agentically. Designs against OB-214's actual output — both component definitions and field-characterization observations. OB-214 is the template; the Binding Agent is the second occupant of the pattern.
- **Observation richness depends on plan complexity.** MIR Plans 1 and 2 are well-structured single-page plans. The observation output may be light (field characterizations, boundary descriptions) rather than rich (ambiguities, unreachable incentives, cross-component conflicts). Richer observations will surface naturally on more complex plans (multi-variant, multi-page, ambiguous structures). The EPG records what the agent actually observed, not what it was expected to observe.
- **Sequence-number confirmation** closed — architect-confirmed OB-214 on 2026-06-18.
