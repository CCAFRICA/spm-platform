## §1 — PROBLEM STATEMENT

**Architect disposition (2026-06-10, binding):** one auth user maps to **exactly one** `profiles` row. Multi-row profiles are retired as an identity model; role inheritance is a presentation concern (persona layer), never duplicate identity rows. SOC 2 CC6 (unique identification → single authorization record) governs.

**Defect class:** post-auth identity resolution has no canonical read, and user provisioning has no canonical writer. DIAG-060 evidence:

1. **Row-count fork (the account-keyed divergence).** `middleware.ts:304-308` resolves the profile with `.maybeSingle()`, which **errors on >1 row** (PGRST116). platform@ (two rows: `vl_admin` 2026-03-05, `platform` 2026-03-07) fails this query → `isPlatformAdmin=false` → never enters the server-side platform-admin branch. tdadmin (one row) succeeds → is the **first account ever to execute** that branch (cookie check :315, `/select-tenant` :319). Four read shapes answer "who is this user": `.maybeSingle()` (middleware:304), array+find-platform (auth-service:191-213, server-auth:53-56, middleware:283-285), oldest-row `.maybeSingle()` (`web/scripts/fix-sabor-users.ts:30-33`). The divergence **is** the defect.
2. **Tenant-entry race (the repaint).** Tenant selection persists in per-tab `sessionStorage` + a session-scoped client cookie, hydrated asynchronously (`tenant-context.tsx:154-164`). `auth-shell.tsx:128-131` pushes `/select-tenant` on `isVLAdmin && !currentTenant` with no tenant-loading guard visible in the DIAG-060 paste. Auth resolving before tenant hydration → Observatory repaint; two bounces <3s → loop detector (`auth-shell.tsx:97-123`) clears `sb-*` **and** `vialuce-tenant-id` → bare `/login`. Latency-sensitive → intermittent → cost a full diagnostic night.
3. **Torn provisioning.** `fix-sabor-users.ts` repairs Sabor `auth.users` identities but never writes Sabor `profiles` rows → live "account found but no profile" on the Sabor login screen (the `fetchCurrentProfile` zero-rows branch, auth-service:203). Accounts exist in `auth.users` without profiles, or with drifted role literals — the HF-280 atomicity class at the identity layer (lineage: HF-027 cascade-deleted profile, HF-063 skipped creation, HF-086/088 auto-create + cleanup).
4. **Silent ejection.** DIAG-060 §6: **13 redirect branches emit no `platform_event`** (only `middleware.ts:229/239/358` emit). Every bounce in this incident traversed unlogged branches.

**Combined treatment rationale:** one class — identity artifacts and their resolution have no canon. Reader, writer, data constraint, the gate that consumes them, and the observability of every branch that rejects them ship as one slice (Vertical Slice; SR-34: closing reader without writer or constraint leaves the class open at a sibling surface).

### Phase 1 — Canonical reader: `resolveIdentity()`

1.1 Create `web/src/lib/auth/resolve-identity.ts` exporting one pure async function:

```ts
// Canonical post-auth identity resolution. THE only sanctioned profiles-by-auth_user_id read.
// Contract: array-tolerant during the migration window; deterministic winner; loud on anomaly.
export async function resolveIdentity(client: SupabaseClient, authUserId: string): Promise<ResolvedIdentity | null> {
  const { data: rows, error } = await client
    .from('profiles')
    .select('*')
    .eq('auth_user_id', authUserId)
    .order('created_at', { ascending: true })
    .limit(10);
  if (error) { log anomaly 'identity.resolve.query_error'; return null; }
  if (!rows || rows.length === 0) { log anomaly 'identity.resolve.zero_rows'; return null; }
  if (rows.length > 1) { log anomaly 'identity.resolve.duplicate_rows' (count, authUserId); }
  const winner = rows.find(r => resolveRole(r.role) === 'platform') ?? rows[0];
  return mapToResolvedIdentity(winner); // role passed through resolveRole — alias-normalized
}
```

Properties (verbatim contract, implement exactly): role comparison uses `resolveRole` (alias-normalized; never raw literal equality against `'vl_admin'`); winner rule = first alias-normalized-platform row else oldest; anomaly logging goes through the Phase 2 event channel with `identity.resolve.*` event names; **no `.single()`/`.maybeSingle()` anywhere in this module**.
