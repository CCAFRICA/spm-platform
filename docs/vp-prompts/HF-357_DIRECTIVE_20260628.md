# HF-357 — Data Operations: Tenant Context Resolution

**Date:** 2026-06-28
**Category:** HF (Hotfix)
**Branch:** `hf-357-data-ops-tenant-context`
**Defect source:** CLT-253 / SR-44 browser verification of OB-253
**Governing directive:** `OB-253_DIRECTIVE_20260628.md` (Data Operations is the Vertical Slice surface for Thalamus)

---

## §1 — PROBLEM

The Data Operations page (`/data`, route `app.vialuce.ai/data`) deployed in OB-253 renders the title, subtitle ("The perceptual surface — what the model recognized, learned, and flagged for your judgment"), and a Refresh button — then displays **`⚠ No tenant context`** and stops. No signal data, no metrics, no trust flags.

**Root cause (from SR-44 browser observation):** The Data Operations page sits under the **Platform Core** agent in the left navigation (breadcrumb: "Data Visibility > Data Console"). Platform Core operates at platform scope, not tenant scope. When the page calls `GET /api/data/overview`, it needs a `tenant_id` to know whose signals to load. The Platform Core context does not provide tenant scope the way the Compensation or Finance agents do — those agents resolve tenant context from the session's active tenant selection (the tenant switcher in the top-right header, currently showing `VLTEST2`).

**The consequence:** OB-253's only browser-facing deliverable (the live Data Operations workspace with signal metrics, recognition-by-facet, processing jobs, and precision-weighted trust flags) is unreachable. The backend endpoints work (`/api/data/overview` accepts tenant_id and returns real data — proven in OB-253 completion report). The page works (title, subtitle, Refresh button render correctly). The gap is at the boundary where the page reads tenant context from the session.

**This is the pattern.** HF-352/353/354 hit the same class: code path correct, surface deployed, scope context doesn't resolve at the boundary. SR-34 (Adjacent-Arm Drift) says fix at the class layer, not the instance.

---

## §2 — DELIVERABLE (a guarantee, not an investigation)

After this HF, a platform admin logged in as `Platform@vialuce.com` (or any admin), with a tenant selected in the tenant switcher (top-right), navigating to Data Operations under Platform Core, sees **real signal data for the selected tenant** — not "No tenant context."

Specifically:
1. The page reads the active tenant from the session's tenant selection (the same mechanism Compensation/Finance/Intelligence agents use to know which tenant's data to show).
2. The page passes that `tenant_id` to `GET /api/data/overview`.
3. The endpoint returns real data (classification_signals count, structural_fingerprints flywheel state, processing_jobs history, recognition-by-facet breakdown, precision-weighted trust flags).
4. The page renders all sections with real numbers.
5. If trust flags exist, the Confirm/Correct buttons work (POST to `/api/data/acknowledge` with the tenant_id).
6. If the tenant has no signal data yet (a fresh tenant with no imports), the page shows zeros / empty states — NOT "No tenant context."
7. Switching tenants in the top-right switcher and clicking Refresh (or navigating away and back) loads the new tenant's data.

**Anti-narrowing:** This HF delivers working Data Operations for any tenant the admin selects. It does NOT deliver "an investigation into why tenant context is missing." It does NOT deliver "a fix that works for VLTEST2 only" (SR-2 — Scale by Design). The fix must work for every tenant in the fleet.

---

## §3 — APPROACH (ULTRACODE — CC determines HOW)

**Objective:** Make `/data/page.tsx` resolve tenant_id from the same source every other tenant-scoped page uses, and pass it to the API call.

**Invariants:**
- The tenant context resolution must use the EXISTING session/context mechanism — do not invent a new one. Read the code for how Compensation or Finance agent pages (e.g., Plans & Canvas, Reconciliation) get their tenant_id. Use the same pattern.
- The `GET /api/data/overview` endpoint already accepts tenant_id. Do not modify the endpoint unless the existing parameter handling is broken.
- The `POST /api/data/acknowledge` endpoint must also receive the tenant_id. Verify it does.
- The page must handle the case where no tenant is selected (platform admin at platform scope with no tenant chosen) — show an appropriate message ("Select a tenant to view Data Operations"), NOT "No tenant context" with a warning icon.
- No new routes. No new navigation items. No new surfaces (§6 of OB-253 holds).

**Proof gate (PG-357):** Browser verification (SR-44, architect-only):
1. Log in as Platform@vialuce.com (VL Admin).
2. Select VLTEST2 (or any prism_enabled tenant) in the tenant switcher.
3. Navigate to Platform Core → Data Visibility → Data Console (the `/data` route).
4. The page shows real signal data for VLTEST2 — not "No tenant context."
5. If a second tenant is prism_enabled, switch to it — the page shows that tenant's data.
