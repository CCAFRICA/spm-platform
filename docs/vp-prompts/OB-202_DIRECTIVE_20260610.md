## §1 — PROBLEM STATEMENT

**Defect 1 — No runtime release identity.** The platform has no visible release identity. The Decision 60 versioning framework (Rules 30–33) mints numbers at tag time, but nothing bakes the number into the build or surfaces it at runtime. A support conversation cannot answer "what release are you on" from the UI; the number exists only in git.

**Defect 2 — Stale-client exposure on deploy.** When a release deploys, a user with an open session has no signal that a newer build exists, and (absent Skew Protection, enabled separately in INF-001 R3 §C.4) a stale client can request asset chunks or invoke server actions that no longer exist — breaking a session mid-dispute or mid-import. The desired experience: a non-disruptive prompt that a new release is available, with relaunch as the **user's choice**, their current session never interrupted.

**Defect-class lineage.** DS-019 §4.5 established the precedent surface: a session-expiry warning banner (confirmed in production, CLT-178 — amber banner at 25 minutes, "Stay logged in" / "Log out"). This OB introduces a second banner of the same species (advisory, non-modal, user-choice). The two must compose, not collide (§3.3). The adjacent stale-*data* findings (CLT-195 F02/F08 — stale period state on the Calculate page) are a different defect class at a different layer and are explicitly out of scope (§6).

**Combined-treatment rationale.** Identity stamp, version endpoint, and banner are one vertical slice — the endpoint without the banner is invisible; the banner without the stamp has nothing to compare. One PR.

### §3.2 — PHASE 2: VERSION ENDPOINT (SERVER HALF)

**3.2.1** Create `web/src/app/api/version/route.ts` (path adjusted only if 0.1 shows a different App Router convention):

```ts
import { NextResponse } from "next/server";
import { RELEASE, COMMIT_SHA, BUILT_AT } from "@/lib/release";

const INCLUDE_SHA = true; // architect may set false to omit SHA from the public payload

export async function GET() {
  return NextResponse.json(
    { release: RELEASE, ...(INCLUDE_SHA ? { sha: COMMIT_SHA } : {}), builtAt: BUILT_AT },
    { headers: { "Cache-Control": "no-store" } }
  );
}
```

Unauthenticated by design: a stale client whose session state is unknown must still be able to ask. `no-store` is mandatory — the endpoint must always answer from the newest deployment.

**3.2.2** Verify locally:

```bash
curl -si localhost:3000/api/version
```

Paste full output (headers + body) into the completion report. Expected: `Cache-Control: no-store`; JSON with the Phase-1 release string, `"local"` SHA, ISO `builtAt`.

**Commit:** `OB-202 Phase 2: /api/version endpoint, no-store`

### §3.3 — PHASE 3: CLIENT CHECK + RELAUNCH BANNER (EXPERIENCE HALF)

**3.3.1** Create `web/src/hooks/useVersionCheck.ts`:

- Embedded identity: `process.env.NEXT_PUBLIC_RELEASE`
- Fetches `/api/version` with `{ cache: "no-store" }`: on mount, on a 10-minute interval, and on `visibilitychange` → `visible` (a tab returning from background is exactly when staleness matters)
- Mismatch (`fetched.release !== embedded`) → `{ updateAvailable: true, newRelease: fetched.release }`
- Fetch failure → silent no-op; no retry beyond the next interval tick (a version check must never degrade the session it protects)
- Dismissal state in module-level memory keyed by `newRelease`: dismissing suppresses **that release only**; a subsequent newer release re-prompts. No localStorage.

**3.3.2** Create `web/src/components/RelaunchBanner.tsx`:

- Renders only when `updateAvailable`
- Reuses the session-expiry banner pattern discovered at 0.2 (DS-019 §4.5 precedent): same visual primitive, same placement family, existing design tokens. No new dependency; no new design vocabulary (OB-201 owns theme work).
- Copy: **`Vialuce {newRelease} is available — Relaunch to update`**
- Controls: **Relaunch** → `window.location.reload()` · **Dismiss** → per 3.3.1
- **NEVER auto-reloads.** Relaunch is exclusively user-initiated.
- **Stacking precedence:** if the session-expiry banner is visible, it takes precedence and the relaunch banner does not render concurrently (session expiry is time-critical; release update is not). Implemented as a render guard, not a z-index fight. If the expiry banner's visibility is not observable from this component without modifying an auth file → **HALT-2**.

**3.3.3** Mount `<RelaunchBanner />` once in `web/src/app/layout.tsx`.

**3.3.4** Simulated-mismatch verification — one verbatim path, no second deployment required:

```bash
cd web && NEXT_PUBLIC_RELEASE="alpha.0.0-simulated" npm run dev
```

The embedded identity now mismatches the endpoint's `release.json`-derived value. Verify in browser at `localhost:3000`: banner appears with the endpoint's release string; **Dismiss** suppresses it; stop the server, restart with `npm run dev` (no override) — banner absent. Paste banner DOM (browser devtools outer-HTML of the banner node) for both states into the completion report.

**Commit:** `OB-202 Phase 3: useVersionCheck + RelaunchBanner, expiry-banner precedence, root-layout mount`
