## 1 -- Problem Statement

Two defects, one defect class, one combined treatment.

**Defect A: Transformer reconstruction gate (defect class: LLM-limiting registry).**

`intent-transformer.ts` lines 185-194 reconstruct IntentModifier objects from raw LLM emission using a hardcoded field list. The transformer reads `rawIntent.modifiers[]`, cherry-picks `modifier` and `maxValue`/`minValue`, hardcodes `scope: 'per_period'`, and discards all other fields. Any field the LLM correctly emits that the transformer does not anticipate is silently destroyed. The transformer acts as a vocabulary registry limiting LLM expression to what the transformer author encoded. This violates Decision 153 (plan intelligence flows forward), E903 (three-tier resolution chain -- LLM-Primary tier is gated), and Carry Everything Express Contextually (data destroyed at transformation time instead of carried to calculation time).

Current code (DIAG-043 Phase 4.1, lines 185-194, verbatim, verified at HEAD `70bf9c2a`):

```typescript
  const modifiers: IntentModifier[] = [];

  if (Array.isArray(rawIntent.modifiers)) {
    for (const mod of rawIntent.modifiers) {
      const m = mod as Record<string, unknown>;
      if (m.modifier === 'cap' && m.maxValue != null) {
        modifiers.push({ modifier: 'cap', maxValue: Number(m.maxValue), scope: 'per_period' });
      }
      if (m.modifier === 'floor' && m.minValue != null) {
        modifiers.push({ modifier: 'floor', minValue: Number(m.minValue), scope: 'per_period' });
      }
    }
  }
```

Defects in this code:
- `scope` hardcoded to `'per_period'` regardless of what LLM emitted. LLM's scope determination destroyed.
- Only `'cap'` and `'floor'` discriminants carried. `'proration'` and `'temporal_adjustment'` from LLM silently dropped (even though both are valid IntentModifier discriminants defined in `intent-types.ts:203-207`).
- Any future field on any modifier discriminant (including `applyTo` if added to the type) is silently dropped because the push constructs a new object from known fields only.

**Defect B: Plan-interpretation prompt teaches cap-modifier placement by vocabulary example only.**

The plan_interpretation prompt at `anthropic-adapter.ts:600-611` contains exactly one cap example: `{ "modifier": "cap", "maxValue": 5000 }` on a `linear_function`. No example shows a ratio-space cap. No semantic principle distinguishes input constraints from output constraints. The LLM produces `modifiers: [{ modifier: 'cap', maxValue: 1.5 }]` for Meridian C5 because the only pattern it has been taught places all caps in the modifier array. The LLM correctly understood the plan's intent (cap the utilization ratio at 150% before multiplying by base bonus) but encoded it in the only structural vocabulary the prompt offered.

The conditional_gate primitive already exists in the executor (DIAG-043 Phase 2.3, `executeConditionalGate` at lines 312-355). The executor already handles nested operations in input specifications (`resolveValue` accepts nested `IntentOperation` objects, confirmed DIAG-041 Phase 3.7 CC note at line 1066). The LLM can express a ratio cap as a conditional_gate wrapping the ratio input to scalar_multiply -- it has never been taught to do so.

**Combined treatment rationale:** Both defects concern the same architectural surface: how the LLM's plan comprehension flows through to the executor. Defect A (transformer destroys LLM output) and Defect B (prompt constrains LLM expression) are two faces of the same defect class: the system limits LLM intelligence rather than carrying it. Fixing B without A means the LLM could emit the correct shape but the transformer would strip unfamiliar fields. Fixing A without B means the transformer carries everything but the LLM never emits the correct shape.

**What this fixes:** Meridian C5 Fleet Utilization. After HF-223 plus Meridian plan re-import, the LLM should emit the ratio cap as a conditional_gate in the input specification of the scalar_multiply operation. The executor processes this through the existing single execution path. No new execution branches. No modifier system changes. The cap is an input-level concern resolved during input resolution, not an output-level modifier.

### 3.3 Replace the reconstruction block with validation-passthrough

Replace lines 183-202 (the modifier handling block, from `const modifiers: IntentModifier[] = [];` through the closing `}` of the `meta.floor` check) with:

```typescript
  const modifiers: IntentModifier[] = [];

  if (Array.isArray(rawIntent.modifiers)) {
    for (const mod of rawIntent.modifiers) {
      const m = mod as Record<string, unknown>;
      const modType = m.modifier as string;

      if (modType === 'cap' && m.maxValue != null) {
        modifiers.push({
          modifier: 'cap',
          maxValue: Number(m.maxValue),
          scope: (typeof m.scope === 'string' && ['per_period', 'per_entity', 'total'].includes(m.scope))
            ? m.scope as 'per_period' | 'per_entity' | 'total'
            : 'per_period',
        });
      } else if (modType === 'floor' && m.minValue != null) {
        modifiers.push({
          modifier: 'floor',
          minValue: Number(m.minValue),
          scope: (typeof m.scope === 'string' && ['per_period', 'per_entity', 'total'].includes(m.scope))
            ? m.scope as 'per_period' | 'per_entity' | 'total'
            : 'per_period',
        });
      } else if (modType === 'proration' && m.numerator != null && m.denominator != null) {
        modifiers.push({
          modifier: 'proration',
          numerator: normalizeIntentInput(m.numerator),
          denominator: normalizeIntentInput(m.denominator),
        });
      } else if (modType === 'temporal_adjustment' && m.lookbackPeriods != null) {
        modifiers.push({
          modifier: 'temporal_adjustment',
          lookbackPeriods: Number(m.lookbackPeriods),
          triggerCondition: normalizeIntentInput(m.triggerCondition),
          adjustmentType: (typeof m.adjustmentType === 'string' && ['full_reversal', 'partial', 'prorated'].includes(m.adjustmentType))
            ? m.adjustmentType as 'full_reversal' | 'partial' | 'prorated'
            : 'full_reversal',
        });
      }
      // Unrecognized modifier discriminants: not pushed to typed array.
      // The LLM emission is preserved in rule_sets.components[].calculationIntent (source of record).
      // The executor processes typed modifiers only. No data is lost; the raw emission persists.
    }
  }

  // Legacy shortcut: meta.cap / meta.floor (top-level fields outside modifiers array)
  if (meta.cap != null && Number(meta.cap) > 0) {
    modifiers.push({ modifier: 'cap', maxValue: Number(meta.cap), scope: 'per_period' });
  }
  if (meta.floor != null && Number(meta.floor) > 0) {
    modifiers.push({ modifier: 'floor', minValue: Number(meta.floor), scope: 'per_period' });
  }
```

Key changes from the reconstruction pattern:
- LLM-emitted `scope` carried through when valid; defaults to `'per_period'` only when absent or invalid.
- All four IntentModifier discriminants handled (`proration` and `temporal_adjustment` no longer silently dropped).
- `normalizeIntentInput` used for `proration.numerator`, `proration.denominator`, and `temporal_adjustment.triggerCondition` (these are IntentSource fields that benefit from the existing N1-N7 normalization at lines 86-129).
- Unrecognized discriminants: comment documents the design decision. Raw LLM emission persists in `rule_sets.components[].calculationIntent`. No data lost.
- Legacy `meta.cap`/`meta.floor` shortcut preserved (lines 197-202 equivalent). These are legacy fields from pre-intent architecture; behavior preserved per DD-7.

SEMANTIC PRINCIPLE FOR CONSTRAINTS:
When the plan document specifies a constraint on an INPUT value (a ratio, a metric, an attribute) BEFORE it enters a calculation (e.g., "utilization capped at 150%", "ratio limited to 1.5x", "maximum attainment 200%"), express the constraint INSIDE the input specification using conditional_gate. The constraint is part of the input, not a post-calculation modifier.

When the plan document specifies a constraint on the OUTPUT value (a dollar amount, a payout, a commission) AFTER the calculation (e.g., "maximum commission $5,000", "payout capped at $10,000"), express it as a modifier in the modifiers array.

EXAMPLE calculationIntent for scalar_multiply with input-constrained ratio (ratio capped before multiplication):
{
  "calculationIntent": {
    "operation": "scalar_multiply",
    "input": {
      "operation": "conditional_gate",
      "condition": {
        "left": { "source": "ratio", "sourceSpec": { "numerator": "actual_units", "denominator": "target_units" } },
        "operator": "<=",
        "right": { "source": "constant", "value": 1.5 }
      },
      "onTrue": { "source": "ratio", "sourceSpec": { "numerator": "actual_units", "denominator": "target_units" } },
      "onFalse": { "source": "constant", "value": 1.5 }
    },
    "rate": 800
  }
}
