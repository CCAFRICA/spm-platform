## §1 — Problem Statement

The plan-interpretation pathway currently requires the LLM to emit complete grammar-compliant PrimeNode DAG trees in JSON. For components exceeding ~20 cells (BCL C0: 30-cell 2D matrix), the LLM output exceeds token budget and truncates. Six sequential HFs (OB-200 through HF-250) attempted compensating mechanisms; none advanced the product objective. This is T1-E920 (Repeated Fix Failure Is a Pattern).

Decision 158 LOCKED the architectural pivot: LLM emits compact Compositional Intent using grammar vocabulary; deterministic code constructs the PrimeNode tree from that intent.

HF-251 implements the vertical slice: intent schema → constructor library → prompt adaptation → BCL plan import → calculation verification.

**Defect class:** LLM capability ceiling on tree emission — structural, not implementable-around.

**Resolution class:** Architectural division of labor — LLM recognition + code construction.

## §3 — Phase 1: Intent Schema and Constructor Library

**Dependency rationale:** Schema and constructor are the foundation. Prompt changes (Phase 2) and integration (Phase 3) depend on these types existing.

### §3.1 — Create CompositionalIntent types

Create `web/src/lib/plan-intelligence/compositional-intent.ts`.

This file defines the CompositionalIntent interface and its constituent types. It IMPORTS PrimeNode types from `prime-grammar.ts` — does not duplicate them.

The schema follows DS-024 §3.2 exactly. Key types:

- `CompositionalIntent` — top-level per-component intent with component_id, component_name, structure, scale, output_precision, metadata
- `StructuralDescription` — discriminated union on `shape`: `banded_lookup | arithmetic | conditional | composed`
- `BandedLookupDescription` — dimensions (reference_field, reference_source, breaks[]), outputs[] flat array, optional output_derivation
- `ReferenceSource` — type discriminated: ratio | aggregate | metric | attribute | cross_data | scope_aggregate | prior_component
- `ArithmeticDescription` — operation + operands (reference | constant | structure)
- `ConditionalDescription` — condition (reference, operator, threshold) + then/else branches
- `ComposedDescription` — composition type (sum | max | min | first_match) + children

Breaks are numeric only. Labels belong in optional metadata per architect disposition.

Commit: `git add web/src/lib/plan-intelligence/compositional-intent.ts && git commit -m "HF-251 Phase 1.1: CompositionalIntent schema — Decision 158 intent format"`

### §3.2 — Create Constructor Library

Create `web/src/lib/plan-intelligence/intent-constructor.ts`.

This file exports a single public function:

```typescript
export function constructTree(intent: CompositionalIntent): PrimeNode
```

The constructor is structural recursion through the grammar. Internal helper functions, not exported:

- `constructBandedLookup(desc: BandedLookupDescription, scale: CompositionalIntent['scale']): PrimeNode` — builds nested conditionals from dimensions. For each dimension, iterates breaks in descending order, emitting `conditional(reference >= break[i], thenBranch, elseBranch)` where `thenBranch` is either a leaf (innermost dimension) or the next dimension's conditional chain. Leaf nodes are `constant(outputs[cellIndex])`. Terminal else is `constant(0)` for no-match. Half-open intervals per Decision 127: condition uses `>=` for lower bound; the nesting order with descending breaks naturally produces `[break[i], break[i+1])` semantics. Scale metadata applied per intent's `scale` field on the outermost node.

- `constructArithmetic(desc: ArithmeticDescription): PrimeNode` — emits arithmetic node, recursively constructs operands.

- `constructConditional(desc: ConditionalDescription): PrimeNode` — emits conditional node with constructed branches.

- `constructComposed(desc: ComposedDescription): PrimeNode` — emits composition wrapper with recursively constructed children.

- `constructOperand(op: OperandDescription): PrimeNode` — dispatches on type: reference → reference node, constant → constant node, structure → recursive constructTree.

The constructor validates inputs before construction:
- Banded lookup: breaks in ascending order, outputs.length === product of (breaks.length + 1) per dimension. On validation failure, throw `ConstructionError` with structured diagnostic (not silent fallback — per T1-E906 v2, failures surface as diagnostic signals).
- All shapes: no undefined or null fields in required positions.

Commit: `git add web/src/lib/plan-intelligence/intent-constructor.ts && git commit -m "HF-251 Phase 1.2: Constructor library — deterministic tree construction from intent"`

### §3.3 — Constructor Unit Tests

## §4 — Phase 2: LLM Prompt Adaptation

**Dependency rationale:** The intent schema from Phase 1 must exist before the prompt can reference it. The prompt changes cause the LLM to emit CompositionalIntent instead of PrimeNode trees.

### §4.1 — Locate the plan_interpretation prompt

```bash
grep -rn "plan_interpretation" web/src/lib/intelligence/anthropic-adapter.ts | head -20
```

Identify the system prompt and user prompt sections for the `plan_interpretation` task type. The current prompt instructs the LLM to emit complete PrimeNode DAG trees with full grammar compliance.

### §4.2 — Replace tree-emission instructions with intent-emission instructions

In the plan_interpretation system prompt, replace the PrimeNode tree emission instructions with CompositionalIntent emission instructions.

The new prompt section instructs the LLM:

- You are interpreting a plan component. Emit a CompositionalIntent JSON object describing the component's structure using the grammar's vocabulary.
- For lookup tables (1D or 2D), use shape "banded_lookup" with dimensions (reference field, how the reference is derived, break values in ascending order) and a flat output array.
- For rate-based calculations, use shape "arithmetic" with the operation and operands.
- For conditional gates, use shape "conditional" with the condition and branches.
- For composed structures (sum of sub-components), use shape "composed".
- Do NOT emit PrimeNode trees. Emit the structural description. Code will construct the tree.
- Include scale metadata: which side scales (convergence or evaluator), which field, which operation.
- Include output_precision: 0 for integer outputs, 2 for currency, per plan evidence.

Remove from the prompt:
- PrimeNode tree examples and grammar compliance instructions
- Chunking/skeleton instructions (HF-249/250 — no longer needed)
- Scale annotation placement instructions (constructor handles this)
- Half-open interval instructions (constructor handles this)
- Terminal completeness instructions (constructor handles this)

Preserve in the prompt:
- Grammar vocabulary context via `generatePromptGrammarSection()` (OB-200) — the LLM still needs to know the grammar vocabulary to describe structure accurately
- Component identification and naming
- Reference field identification (which fields the component reads from)
- Break value extraction (what thresholds define the bands)
- Output value extraction (what values the lookup produces)

The prompt SHRINKS significantly because the LLM no longer needs tree-topology instructions.

### §4.3 — Add CompositionalIntent examples to prompt

Add 2-3 concrete examples of CompositionalIntents the LLM should emit:

1. **1D tiered lookup** (like BCL C1): 4 attainment bands, ratio reference, 4 output values.
2. **2D matrix lookup** (like BCL C0): 6 attainment rows × 5 quality columns, two reference fields, 30 output values.
3. **Linear rate** (like CRP Plan 1): arithmetic multiply of revenue metric by constant rate.

Examples use the exact CompositionalIntent JSON shape from Phase 1's schema.

### §4.4 — Adapt the per-component interpretation call

In `ai-plan-interpreter.ts` (or the current per-component call site from HF-248), change the response parsing:

**Before:** Parse the LLM response as a PrimeNode tree, pass to validator, persist to rule_sets.components.

**After:**
1. Parse the LLM response as a CompositionalIntent.
2. Validate the intent structurally (breaks ascending, output count matches dimension product, reference sources well-formed). On validation failure, classify as `cognition_failure` per HF-248 error class taxonomy and retry.
3. Call `constructTree(intent)` from `intent-constructor.ts` to produce the PrimeNode tree.
4. Pass the constructed tree to the existing validator (HF-244).
5. Persist the validated tree to rule_sets.components (unchanged shape — the engine sees the same PrimeNode tree it always has).

The CompositionalIntent is also persisted alongside the tree in the component's metadata, for signal surface and debugging:
```typescript
component.metadata = {
  ...component.metadata,
  compositional_intent: intent,
  construction_method: 'intent_constructor_v1'
};
```

Commit: `git add -A && git commit -m "HF-251 Phase 2: LLM prompt adaptation — emit CompositionalIntent instead of PrimeNode tree"`

## §6A — Residuals

- **Prime count verification:** DS-024 references "nine primes" per handoff. The pipeline observatory shows 5 operational primes + 2 leaf types = 7. The constructor imports from `prime-grammar.ts` and handles whatever primes it declares. If `prime-grammar.ts` declares 9, the constructor handles 9. If it declares 7, the constructor handles 7. No hardcoded count in the constructor.

- **C2/C3 validator warnings:** Missing scale_annotation and terminal_completeness warnings observed under the prior emission pathway. Under the construction pathway, these should disappear because the constructor guarantees both properties. Verification deferred to calculation phase — if warnings persist, they indicate a constructor bug (deterministic, fixable once).

- **Mode B deprecation:** HF-250's Mode B (skeleton + chunks for large components) is no longer needed because the intent is always small enough for a single LLM call. Mode B is not removed in this HF — it is simply never triggered because no component's intent exceeds a single call's budget. Formal deprecation in HF-255.

- **Progressive performance (T1-E906 v2 read-before-derive):** Not implemented in this HF. The signal-write for successful constructions is deferred to a follow-on HF after BCL proves the construction pathway works end-to-end. The constructor works without progressive performance; progressive performance is an optimization, not a correctness requirement.
