## §1 — PROBLEM STATEMENT

The platform interprets compensation plan documents (PDF, PPTX) into executable `rule_sets.components` — the bridge between a natural-language plan and the deterministic calculation engine. HF-248 shipped a two-phase per-component orchestration that replaced the monolithic single-call path. That was a real improvement. But three gaps remain, all documented with evidence:

**Gap 1 — No self-correction.** When the LLM extracts a component incorrectly, the code accepts whatever the LLM returns and passes it downstream. Five consecutive hotfixes (HF-156 through HF-159) all failed because the prompt constrained the AI's vocabulary — the LLM picked "tiered_lookup" as the closest match for a linear formula because the prompt only offered `[matrix_lookup, tiered_lookup, percentage, flat_percentage, conditional_percentage]`. An agent can validate its extraction against the source text and re-extract on mismatch. The current path has no mechanism for this.

**Gap 2 — No cross-component reasoning during extraction.** HF-248's cross-component validation runs *after* all components are extracted. An agent extracting Component 3 can already know what it extracted for Components 1 and 2 and use that context: "Component 1 uses individual attainment as its input; Component 3 references store attainment — these are different scope levels requiring different data sources." The agent validates *during* extraction, not post-hoc.

**Gap 3 — Token-limit fragility on large plans.** The AI Architecture Briefing documents this as Backlog 7.11 and Open Item #26 (Critical): "Any plan exceeding model context limits = complete failure with silent degradation." HF-248's per-component calls mitigate this for the extraction phase, but the initial component-identification pass still sends the full document. An agent with a `read_plan_section` tool reads the document in sections, so no single LLM call needs the full text. Scales linearly with document size.

**Defect-class lineage.** The five HF-156→159 failures, the active HF-237 regression (source type enumeration at the prompt layer — pipeline observatory documents this as "registry limits what the LLM can express"), and Backlog 7.11 are all instances of the same structural class: *the plan interpretation path is code-orchestrated with no LLM-side reasoning loop.* An agent with tools, a loop, and self-validation closes the class.

**Reuse.** OB-212 shipped the agent infrastructure: `agent-runner.ts` harness, `agent_invocations` capture table, Observatory Agent Operations section, AI Substrate panel, `classification_signals` capture path, `agent_inbox` write, and the Progressive Performance `request_fingerprint` cache mechanism. This OB builds **one new agent** on that proven infrastructure — no new tables, no new Observatory sections, no harness changes.

## §2 — SUBSTRATE-BOUND DISCIPLINE APPLICATIONS

**Decision 158 (LLM Recognition + Code Construction).** The agent is a **recognizer**: it reads the plan document and emits structured per-component definitions (CompositionalIntent JSON or the current component structure — Phase 0 determines the live output contract). Deterministic code downstream (`convertComponent`, `intent-transformer.ts`) constructs the executable intent. The agent never writes to calculation tables, never constructs prime-DAG trees, never touches the engine. The boundary holds by construction: the agent's tools read plan content and return structural observations; the agent's output is a structured plan interpretation that feeds the same downstream path HF-248 feeds. If the agent's output contract matches HF-248's output contract (Phase 0 verifies), the downstream is untouched.

**Korean Test (Decision 154, LOCKED).** Every tool name is structural. Tool *descriptions* describe structural patterns ("extract a component's calculation structure from the plan document"), not domain semantics ("extract the commission formula"). The Phase 3 sweep greps for domain literals.

**Progressive Performance (constitutional).** The agent computes `request_fingerprint = hash(document_content_hash + rule_set_version)`. If the same plan document has been successfully interpreted before (same content, same rule set version), the cached interpretation is returned at $0 with no LLM call. A plan document re-uploaded unchanged gets the cached result. A plan document that has been edited gets a fresh interpretation. The EPG proves the cached second encounter empirically.

**Carry Everything, Express Contextually (T1-E902 v2).** The agent reads the full plan document (persistence-complete). Its per-component extraction is an *expression* choice (focused context for better accuracy), not a *persistence* narrowing. The full document remains available to every tool call.

**Vertical Slice.** The agent ships with its trigger (in the SCI execute path for plan-classified content units), its capture (`agent_invocations` + `classification_signals`), and its measurement (Observatory Agent Operations + AI Substrate rows go live for `plan_interpretation`). The existing reconciliation-match-rate serves as the downstream ground-truth proxy — no new ground-truth surface needed.

### §3.1 — PHASE 1: ADR + TOOLS + AGENT + ROUTE

#### §3.1.0 — Architecture Decision Record (mandatory, committed before code)

`OB-214_ADR.md` — answers two questions grounded in Phase 0:

**Q1: Replace or layer?** Does the agent *replace* HF-248's two-phase orchestration at its call site (the agent now IS the plan interpretation path), or does it *layer on top* (the agent orchestrates calls to HF-248's existing per-component extraction)? The answer depends on what Phase 0 found:
- If HF-248 is a thin orchestrator that calls the LLM per component with no validation — **replace it.** The agent subsumes its function with tools + loop + self-validation. The output contract is matched, so `convertComponent` / downstream is untouched.
- If HF-248 has substantial logic beyond LLM calls (non-trivial validation, structural transformation, variant handling) — **layer on top.** The agent uses HF-248's extraction as a tool and adds the observe-reason-act loop. Preserves the proven logic.
- If the boundary is unclear — **HALT-ADR** for architect disposition. Do not guess.

**Q2: Tool set.** Based on the document content shape (Target B) and the output contract (Target A), specify the tools with exact input/output schemas.

**Candidate tools (refined against Phase 0 findings):**
- `read_plan_section(document_ref, section_hint?)` — returns a section of the plan document text. If Target B shows the document arrives as full text, this is a substring; if it arrives as pages, this is page-ranged. The tool signature depends on Phase 0.
- `extract_component(document_ref, section_or_page_range, prior_components_summary?)` — focused single-component extraction. The `prior_components_summary` carries what previous components use (scope levels, metrics) so the LLM has cross-component context *during* extraction, not after. Output shape matches the downstream contract (Phase 0 Target A).
- `validate_extraction(component_json, source_text_excerpt)` — the self-correction tool. The LLM re-reads the relevant plan text and checks whether its extracted structure (tiers, rates, boundaries, scope) matches what the plan actually says. Returns validation result + discrepancies. If discrepancies found, the agent re-extracts.
- `validate_plan_completeness(components_json)` — after all components are extracted, checks: are component metrics distinct? Do cross-references resolve? Are there orphan references? Returns a completeness assessment.

Commit the ADR before code.

## §6 — OUT OF SCOPE

- **Prompt vocabulary constraint removal (HF-237 regression).** The agent sidesteps the constraint by reasoning over tool results rather than fitting into a response schema. A separate cleanup of the source type enumeration in the prompt is a distinct HF — the agent doesn't need it removed to work, and removing it doesn't require the agent.

The prime-DAG architecture (replacing named convenience types with composable primitives) is a separate engine evolution — the agent's output format will evolve with it but does not drive it.
- **Dispute Investigation Agent.** Deferred to its own OB (blocked on disputes table rebuild, AUD-004).
- **Multi-plan interpretation in a single agent run.** The agent interprets one plan document per invocation. Multi-plan orchestration (e.g., a workbook with two plan tabs) is a follow-on.
- **Model/provider UI switching.** The agent inherits the adapter's resolved model. The config-resolver DS (§6A forward-reference from OB-212) remains its own capability.
