## §1 — THE BINDING ENFORCEMENT

**LLM expression is not classified.** This is non-negotiable and applies to every layer of the pipeline. The LLM expresses what a column IS and what a sheet IS. That expression is stored, carried, and read by every downstream consumer. At no point does any code reduce the expression to a term from a developer-curated vocabulary, select behavior based on a classification label, or fall back to a developer-written heuristic when the expression is available.

R4 left three things alive that violate this:

**1. `data_type=target|transaction` still gates behavior.** Source_date extraction, convergence filtering, and entity-id paths read this classification to decide what to do. The classification is produced by `SCI-HC-PATTERN` from `idRepeatRatio` and `HAS/NO reference key` — deterministic heuristics that replace the LLM's expression with a developer's anticipation. The label must be removed from all behavioral gates. Downstream code reads the LLM's expression directly: "does this sheet carry temporal events?" is answered by the expression, not by `data_type=transaction`.

**2. `idRepeatRatio ≤ 1.5 → target` survives as a fallback.** R4's deviation rationale: "Meridian's `Datos_Rendimiento` (monthly actuals without a per-row event ID) would reclassify." The reclassification is a label change, not a behavioral break. Meridian's sealed figure is $556,985. If the expression-based pipeline produces $556,985 for Meridian, there is no break — regardless of what label the sheet carries. **The ratio threshold is deleted. The Meridian case is handled by the LLM expressing what the sheet IS — "records-over-time" / "monthly performance actuals by entity" — which the LLM already knows and the ratio threshold was a proxy for.**

**3. Two classifier tests assert labels.** These tests check that a specific sheet receives `data_type=target` or `data_type=transaction`. They are assertions of the classification registry. When the classification is deleted, these tests fail — and that failure is correct, not a regression. **Delete or rewrite these tests to assert behavioral outcomes** (source_dates populated, entity-id correct, calc result unchanged), not classification labels.

## §2 — CONSTRAINTS

- **Decision 158.** Recognize is not classify. The LLM recognizes what each sheet and column IS. Deterministic code verifies structural properties. No classification layer exists between recognition and verification.
- **Validation Premise Law.** `idRepeatRatio ≤ 1.5` fails the acceptance test (a developer edits the threshold). "The LLM expressed this as records-over-time and source_dates are present" passes it (structural verification of carried reality).
- **Carry Everything (T1-E902).** The LLM's expression of what the sheet IS carries through to every consumer. No consumer reads a classification label derived from a heuristic.
- **HALT-CALC is the tenant-protection gate.** Meridian $556,985. BCL $312,033. MIR Plan 2 = 210,000. These are the sealed figures. A label changing is not a break. A sealed figure moving is a break. Tests that assert labels are not protection — they are the registry encoded as test expectations.
- **C2 (fail-loud).** When the LLM's expression is absent or structurally insufficient for a downstream decision (e.g., "does this sheet carry temporal events?" and the expression doesn't say), fail loud. Never fall back to `idRepeatRatio`. Never fall back to a default classification.

## §3 — PROOF GATES

### Eradication (the enforcement)

**PG-R5-1 (No `data_type` behavioral gate).** Grep over all application code: zero code paths where `data_type === 'target'` or `data_type === 'transaction'` gates behavior (source_date extraction, entity-id selection, convergence filtering, or any other downstream decision). The field may persist as metadata/provenance — it must not be read by any `if`/`switch`/`filter` that changes what the pipeline does.

**PG-R5-2 (No `idRepeatRatio` threshold).** The `SCI-HC-PATTERN` `idRepeatRatio ≤ 1.5` heuristic is deleted. No numeric threshold on identifier cardinality determines sheet classification or any downstream behavior.

**PG-R5-3 (No label-asserting tests).** Every test that asserted a specific `data_type` classification label is either deleted or rewritten to assert a behavioral outcome (source_dates populated, entity-id correct, expression contains expected fields). Zero tests fail because a classification label changed.

**PG-R5-4 (Sheet-level expression).** The LLM expresses what each sheet IS — the expression carries enough for every downstream consumer to make its decision without a classification label. For transaction sheets: the expression conveys "rows are events/transactions with temporal ordering." For entity sheets: "rows are entity records." For monthly-actuals (the Meridian case): "rows are periodic performance records by entity over time." Downstream reads the expression. This is how the Meridian case is handled without `idRepeatRatio`.
