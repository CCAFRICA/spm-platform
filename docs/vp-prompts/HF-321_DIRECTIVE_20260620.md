## §4 — PHASE 1: SEED DATA CORRECTION

**Objective:** Regenerate the ~43,875 committed_data rows for the Sabor tenant with the complete field set that `financial-data-service.ts` expects. Preserve the existing entity hierarchy, entity_relationships, calculation_results, and tenant configuration.

### Approach: Delete-and-Reinsert

The existing committed_data rows for Sabor are deleted and replaced with new rows containing the full field schema. This is cleaner than field-by-field UPDATE because generated values must be internally consistent (food + beverage ≈ gross revenue; hourly distribution must be realistic; server attribution must reference real entity IDs).

### Generation Constraints

CC generates the seed script as a TypeScript file at `scripts/frmx-reseed-correction.ts` and executes it via `npx tsx`.

**Data realism rules:**

1. **Food/Beverage split:** `total_alimentos` + `total_bebidas` ≈ gross revenue per check (±5% for rounding). Typical restaurant ratio: 65-75% food, 25-35% beverage. Vary by brand — Cocina Dorada (full-service) higher beverage ratio than Taco Veloz (express).
2. **Time distribution:** `hora_apertura` (integer 6–23). Bimodal distribution: lunch peak 12–15, dinner peak 18–22. Express brands (Taco Veloz) heavier lunch; full-service (Cocina Dorada) heavier dinner; Mar y Brasa (seafood) heaviest dinner.
3. **Day of week:** `dia_semana` (0=Sun through 6=Sat). Weekend revenue ~30% higher than weekday. Friday/Saturday peak.
4. **Guest count:** `numero_comensales` (integer 1–8). Average ~2.5. Higher at full-service, lower at express.
5. **Server attribution:** `mesero` field referencing server name or entity external_id. Distribute checks across servers per location. Query existing `individual` entities from the entities table for server names.
6. **Discounts:** `descuentos` (numeric). ~3-5% of checks have discounts. Amount = 5-20% of check total when present.
7. **Comps/Cortesías:** `cortesias` (numeric). ~1-2% of checks have comps. Small amounts (free dessert, complimentary drink).
8. **Tax (IVA):** `iva` (numeric). Mexico IVA = 16% of pre-tax subtotal. `iva = (total_alimentos + total_bebidas) * 0.16`.
9. **Cancellations:** `cancelado` (boolean). ~0.01% of checks. Preserve approximate current count (3 cancelled).
10. **Location code:** `numero_franquicia` or equivalent location identifier matching entity external_id.

**Preserve existing aggregates:** The regenerated data should produce approximately the same totals as currently exist — MX$16,150,334.28 gross revenue, ~43,875 checks, MX$2,060,226.87 tips. Exact match not required (±2% acceptable) since this is demo data.

**Entity attribution:** Each committed_data row gets `entity_id` set to the location entity's UUID. Server attribution goes in `row_data.mesero`. CC queries the entities table for the location→server mapping via entity_relationships.

### Script Structure

CC determines the optimal script structure. The script:

1. Reads the Sabor tenant ID from `tenants` table
2. Reads all location entities (type `location`) with their UUIDs and display_names
3. Reads all individual entities (servers) and their relationship to locations
4. Deletes existing committed_data for the Sabor tenant
5. Generates ~43,875 new rows across locations with the full field schema
6. Batch-inserts using Supabase service-role client (chunk size ≤ 500 per insert)
7. Verifies row count post-insert
