## ¬ß1 ‚Äî Problem Statement

The platform does not reconcile for any tenant whose compensation metrics are *derived from generic shared data columns by filtered aggregation* ‚Äî i.e., real-world transaction data where one column (`total_amount`, `unit_price`, `quantity`) serves multiple plans and the per-plan metric is `SUM(column) WHERE category = X` or a filtered `COUNT`. This is the dominant real-world data shape. Tenants whose data columns string-match their metric names (pre-aggregated, one column per component) reconcile; tenants with shared columns do not.

This is **not a new defect**. It is the same convergence-consumption gap identified at OB-189 ("convergence doesn't produce the binding that tells the engine which column to sum and how to filter; UNIVERSAL, not tenant-specific") and named architecturally at HF-235 ("convergence must consume the plan agent's operational comprehension as authoritative derivation specifications, not as hint text for a re-derivation attempt"). Three concrete mechanisms in the current code keep the gap open. They are one defect class ‚Äî **a downstream mechanism overriding or discarding the authoritative upstream signal** ‚Äî and are treated as one HF because fixing any one in isolation leaves the calculation wrong (the redirect would still corrupt a correct mapping; a correct mapping without filters still reads the wrong rows; correct filtered bindings still don't run for a plan that skipped convergence on stale state). Dependency analysis: D1‚ÜíD2 are Complement (the redirect overwrites the very binding D2 makes filter-aware; removing one without the other does not produce a correct binding), and D3 is Precedent for D1/D2 taking effect on any plan whose `input_bindings` are already populated. No clean decomposition exists across the Complement edge.

### D1 ‚Äî The HF-263 P3.2 magnitude-proxy redirect (`convergence-service.ts`, `generateAllComponentBindings`, the `‚îÄ‚îÄ HF-263 P3.2 (HALT-B)` block)

The post-pass redirect rewrites any `cross_source_numeric` measure binding to a same-batch column selected by **magnitude proximity** (`Math.abs(log10(alt.mean) ‚àí log10(current.mean)) < 1`) where "same batch" means the batch the entity identifier was sourced from. When the entity identifier is sourced from a roster/reference batch that also carries a quota-style column, that quota column sits in the same numeric band as transaction amounts and counts, so it satisfies the proximity test for **every** revenue/count metric. The redirect therefore overwrites correct AI column mappings with the quota column. The selection criterion is *size*; the property it must preserve is *meaning*. Magnitude is not a semantic discriminator ‚Äî there is no threshold tuning that rescues this mechanism, because two columns of equal magnitude carry different meaning. The redirect is unsound by construction for the general shared-column case and is removed, not gated. (Whatever single tenant HALT-B was added to satisfy was binding the correct column only because, there, the same-batch alternative happened to be the right one; D2 makes that case resolve correctly through the validated-mapping path without a magnitude rewrite.)

### D2 ‚Äî The column-mapping AI call cannot emit filters (`convergence-service.ts`, `resolveColumnMappingsViaAI`)

HF-234 removed the categorical/filter block from this call's prompt, restricting it to flat `{metric: column}` mappings. The HF-227 binding shape (`ComponentBinding.filters`) and the engine's filter-respecting resolution already exist, and the plan-agent comprehension (`metricComprehension` signals, `signal_type='comprehension:plan_interpretation'`) is already loaded and threaded into this call ‚Äî but the prompt does not ask the AI to express, and the parser does not require, the filter that turns a bound column into the correct per-plan metric. The AI maps `consumable_revenue ‚Üí total_amount` with no filter; the engine sums *all* `total_amount` rows. The plan agent already comprehended `WHERE product_category = 'Consumables'`; convergence discards it at the mapping step. (Pass 4 / `generateAISemanticDerivations` produces filter-aware `metric_derivations`, but the engine resolves via `convergence_bindings` (Decision 111) and the bindings carry `filters: []`, so Pass-4 output is inert on the binding path ‚Äî AUD-009, confirmed by the live `0 derivations, N component bindings` footer.) The fix restores filter vocabulary to **this** call so the authoritative comprehension lands **on the binding** the engine actually reads.

### D3 ‚Äî Stale `input_bindings` skip convergence (`convergence-service.ts` consumer at the CalcAPI gate; import path)

A plan whose `rule_sets.input_bindings` is already populated takes the `HF-165: input_bindings already populated ‚Äî skipping convergence` path and binds whatever columns a prior run persisted ‚Äî including columns that no longer resolve against current data ‚Äî producing zero. New data import must invalidate stale bindings so convergence re-derives. This is the OB-195 cache-invalidation requirement; it is included because D1/D2 corrections do not take effect on any plan that skips convergence.

## ¬ß2 ‚Äî Substrate-Bound Discipline Applications

- **IGF-T1-E910 (Korean Test) / AP-25:** D1's removal eliminates a structural heuristic that, while literal-free, encodes a *semantic* assumption (magnitude ‚áí interchangeability) the substrate forbids ‚Äî structural signals may classify, never substitute meaning. D2's restored prompt and parser introduce **no** field-name or language literals: the AI receives column metadata and categorical distinct-values at runtime and returns filter values drawn from those runtime values. EPG in ¬ß3 enumerates the structural literals in the changed surface for architect inspection (DD-5).
- **IGF-T1-E952 (Adjacent-Arm Drift Discipline) / SR-34:** the defect class is "downstream override of the authoritative signal." DD-2 requires every consumer of that class be swept, not the instance of record. ¬ß3 Phase A enumerates, via directive-embedded grep, every site that rewrites or zero-fills a binding after the mapping step, so the class is closed at the layer, not at the CRP-shaped instance.
- **IGF-T1-E953 (Decision-Implementation Gap):** D2 makes the operative implementation match the long-stated decision (HF-235 / Decision 147): convergence consumes plan-agent comprehension as authoritative derivation spec. The binding the engine reads now carries that spec.
- **DD-7 (behavior preservation):** D2's prompt/parser change is **additive** ‚Äî absent filters, the binding is byte-identical to pre-HF-269 (`filters: []`, `rowMatchesFilters` returns true on empty arrays). Tenants that need no filter (identity-mapped columns) are unaffected. This is named here as the binding behavior-preservation guarantee, and ¬ß3 Phase D verifies it on an identity-mapped tenant.

### Phase B ‚Äî Restore filter vocabulary to the column-mapping call so bindings carry plan-agent intent
