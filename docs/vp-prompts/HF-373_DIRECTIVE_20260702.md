## §1 — Problem Statement

Three objectives define success, and every defect below is a live, root-caused blocker or corruptor of one of them. All evidence is from the 2026-07-01/02 CLT run on current `main` (post-HF-372).

**Objective 1 — VLTEST2 calculates correctly.** Tenant `5b078b52-55c9-4612-8f86-96038c198bfe`. The import is now correct (85 entities, 8-component plan, all rows linked and dated, one finalize claim), but the calculation produced zero for everyone:

- **D1 — Convergence produced 0 bindings; the DAG evaluated empty strings.** `[Convergence] … 0 derivations, 0 gaps, 0 component bindings` followed by hundreds of `[PrimeDAG] ordering operator 'gte' on non-numeric operands ('' vs '0.9') — returning false`. The plan's component inputs were never bound to the data columns (`Cumplimiento_Colocacion`, `Indice_Calidad_Cartera`, …), so every band comparison received `''` and every payout totaled 0. Decisive detail: convergence **loaded 7 metric-comprehension signals** — the model's recognition of what each measure column means exists — yet binding produced 0 groups. The recognition is present; the binding step is not consuming it. Historical note: the March runs on this same plan/data produced "4 component bindings, zero AI cost" — this is a restoration of a previously working surface.
- **D2 — Variant assignment excludes most of the population via name-token scavenging.** `[VARIANT-DIAG] … materializedState={"role":"Ejecutivo"}` immediately followed by `NO MATCH — excluded (tokens=[2026,bcl,mac,001,datos,5078,rosa,…])`. The matcher ignores the materialized, model-recognized `role` attribute the roster produced and instead tokenizes names/file fragments hunting for variant words. People whose role token doesn't appear are excluded from calculation entirely. Additionally both variants render the same label "(Ejecutivo Senior)" in result lines — a labeling defect in the same surface.
- **D3 — Roster entity-id field chosen by a developer heuristic over the model's recognition.** `[entity-id] HF-351 F5: 2 entity-scope candidates [ID_Gerente, ID_Empleado] → "ID_Gerente" (cold-start finest repeating identifier (distinct=13, repeat=6.5x))`. A cold-start heuristic selected the *manager* column (13 distinct values across 85 rows) as the roster's entity-id field over the actual per-row identifier. Downstream outcomes happened to survive in this run, but this is the heuristic-overrules-model class (Decision 158 violation family) sitting directly under the reconciliation, and it must be removed: the entity-id field derives from the model's recognition (`scope_role='entity'` + `nature_role='identifier'`), with the row-ordinal structural guard retained, and a loud gap when recognition is absent — never a "finest repeating" statistic.

**Objective 2 — Casa Diaz plans are precise.** Tenant `2d9979ba-5032-48a7-bccf-1928f3e6dadf`. Eight plans exist and 73 entities carry 584 assignments — structurally complete. Precision defects:

- **D4 — Indistinguishable duplicate plan names.** Two active plans both named "COMISIONES DE MAQUINARIA" (from the workbook's two machinery sheets). The HF-372 fix correctly stopped same-named plans from silently archiving each other; the residual is display precision: plans must carry a disambiguating identity drawn from their source sheet so operators can tell them apart in Plans & Canvas and everywhere else plans are listed.
- **D5 — COMISIÓN GARANTIZADA intent precision (known prior gap, in scope now).** The guaranteed-commission plan type expresses a *constant* payment intent, not a multiplicative rate intent. The interpretation contract must let the model express constant-intent components and the constructor must build them faithfully — model expression, no plan-type registry. Verify this plan's components state the guarantee correctly.

**Objective 3 — The large file processes at pace.** The 86,607×87 (~42MB) JDE extract failed at the staging step, and the failure exposed truthfulness and pace defects across the same surface:

- **D6 — Monolithic staged CSV violates the platform's own byte-budget invariant.** The truthful job record captured it exactly: `commit CSV upload failed for "Exportar Hoja de Trabajo": The object exceeded the maximum allowed size`. Staging serialized the parsed sheet as ONE CSV object; an XLSX is compressed and its CSV flattening is several times larger, so the single object exceeded a storage cap and `pulse_load_jobs` never received a single row. The architecture (parse in serverless → stage to Storage → database FDW-loads on the pg_cron worker's clock — the June 29 "Closet in the Right Warehouse" / TMB decision) is correct and stands. The defect is that staging ignores the HF-359 byte-budget invariant the loader already honors. **The fix is multi-part staging: the staged output is written as N parts, each sized within the discovered byte budget (0.8 × the effective limit), listed in the job manifest; the worker FDW-loads per part.** Parts are gzip-compressed (the S3 wrapper reads gzipped CSV natively) for pace and cost. Storage limits become architecturally irrelevant: the platform adapts to whatever cap exists. **Raising limits is not a fix and must not be the fix.** Design part loading to be parallel-ready (independent parts, per-part status in the manifest), even if the worker drains sequentially today.
- **D7 — Terminal job status lies.** The failed 86K job is recorded `status=finalized / phase=completed` with the error text attached — a job whose commit failed and produced zero rows terminating as "completed" is a truthfulness defect that defeats the entire status system. Separately, the successful workbook job is stuck at `committed / finalizing` with `completed_at=null` — the terminal transition was never written. Fix the terminal state machine: commit failure → `failed`; success → terminal status with `completed_at` stamped. The record the UI reads must be truthful at the end, not only mid-flight.
- **D8 — Summary engine times out on the large tenant.** `[SCI Finalize] summary engine failed: committed_data read: canceling statement due to statement timeout` — the post-commit summary read cannot scan a large tenant's `committed_data` within the statement timeout. Fix at the read strategy (bounded/keyed reads or the RPC/materialized path), consistent with the Materialized Serving Path principle (no raw full-table aggregate scans); a summary failure must also surface, never silently pass.
- **D9 — Multi-fire dispatch (recurring since March; the architect's standing suspicion).** One import dispatched finalize four times tonight (all coalesced by the HF-371 claim — containment works); the plan file's commit ran twice back-to-back (superseded harmlessly); the `AUDIT DIVERGENCE … fields=rows,perUnit,pulses(formula)` telemetry alarm fired on both tenants. The claim and supersession layers contain the damage; the duplicate dispatch itself is the defect. Phase 0 must enumerate EVERY dispatcher of the commit (execute-bulk) and finalize endpoints — client-side effects/handlers (including double-mount/re-render fires), server-side `waitUntil` continuations, the finalize-sweep and dispatch-jobs crons, and the pulse hand-off decision path (`shouldHandOff`) the architect specifically suspects — then make commit dispatch single-fire per import the same way finalize now is (claim/coalesce at the endpoint), and reconcile the pulse-formula bookkeeping so AUDIT DIVERGENCE stops firing on clean runs.
- **D10 — The Datos fingerprint never stores; the warm path is disabled for this sheet class.** Every Datos import logs `Skipped write (failed-outcome quality gate, HF-247): columnRoles contains 'unknown'` — some column's role resolves to `unknown`, the quality gate (correctly) refuses to memorize a flawed recognition, and every identical re-import re-pays ~50 s of full analysis. Fix at recognition completeness per Decision 158: identify which column yields `unknown` and why the model's recognition isn't carried for it (fix the carry, not the gate); the gate stays. Progressive performance must be restored: the second import of an identical Datos file uses the warm path.
- **D11 — Banded large files have no header recovery on the oversized path (known prior gap, in scope now).** The streamed/windowed parse path keys on raw row-1 headers and never applies the de-bander (`debandWorksheet` has zero call sites in `windowed-commit.ts`/`sheet-stream.ts`/`sheet-window.ts`). Fine for clean-row-1 JDE extracts; a banded human-authored large file would classify from garbage headers. Extend header recovery to the oversized path's sample window without regressing the OOM defense — closing the last known import-arc structural gap so no file class silently depends on being small.

Combined-treatment rationale: D1-D3 stand between the correct import and a correct calculation; D4-D5 are the precision gap on the delivered plans; D6-D11 are the pace, truthfulness, and determinism gaps on the same import surface. The architect's instruction is explicit: nothing on this list may be deferred, and no discovered blocker may be re-classified as out of scope.
