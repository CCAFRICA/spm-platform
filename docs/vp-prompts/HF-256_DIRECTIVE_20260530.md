## §1 — Problem Statement

### 1.1 — The capability, locked, regressed

The platform was built and proven to ingest **any file, any format, multiple files at once, through one surface**. The governing decision (Decision 82, verbatim): *"SCI upload accepts PPTX/PDF/DOCX/XLSX/CSV/TSV. Documents → Anthropic extraction → Plan Agent. Tabular → existing SCI agents. One surface, platform routes."* It was proven in production (Óptica Luminar: a PowerPoint plan, 719 people, exact-to-the-cent reconciliation — Decision 83).

The HF-239 import unification collapsed this into a **single-file, spreadsheet-shaped path**. The capability must be restored — comprehensively, in one HF. This is not a spot fix; it is the restoration of Decision 82 within the unified Storage transport.

### 1.2 — What "documents" means (the scope line that governs everything)

Decision 82 and the ingestion facility spec (DS-005: spreadsheets = *"primary data format"*; documents = *"plan documents, supporting docs"*) draw a firm line:

- **Documents (PDF/PPTX/DOCX) are PLAN sources.** Every document routes to the Plan Agent and is interpreted as a plan. The plan pipeline self-extracts a document by its format.
- **Data (entity/transaction/target/reference) arrives as spreadsheets (XLSX/CSV/TSV).** Deterministic calculation binds to structured tabular records.

Therefore **document-as-data is NOT a path and is explicitly out of scope** (§6). This HF does not add any extraction of transaction/entity data from a PDF or DOCX. A document is always a plan unit.

### 1.3 — The four regressions to restore (plain statement, then the evidence)

All against the locked design; all confirmed by the ingestion audit (AUD-0014) at HEAD `c75ad63d` — re-confirmed at current HEAD in Phase 2.

1. **Document plans crash before reaching the Plan Agent.** Execute reads every incoming file as a spreadsheet workbook unconditionally, before deciding what the file is. A PDF/PPTX/DOCX has no workbook, so it throws ("Could not find workbook") before reaching the plan pipeline that knows how to read it. *Evidence:* `execute-bulk/route.ts:160-181` (`XLSX.read`, no format guard) precedes plan dispatch at `:213`. This is the live Meridian/Óptica Luminar failure.

2. **Multiple files in one import do not work.** The client carries only the first file's storage location into execute; files two onward are uploaded but never processed. Execute downloads exactly one file per call. *Evidence:* `page.tsx:322` `const storagePath = Object.values(storagePaths)[0] || undefined;` (singular); `execute-bulk/route.ts:96,117,143-146` (one `storagePath` per call). The per-file map already exists upstream (`page.tsx:136`, passed at `:494`) and is collapsed at the execute boundary.

3. **The import's route is decided by the first file only.** Whether the import is treated as document or tabular is decided by `files[0]`; and the document analyze path handles one document per call. So if the first file is a document, files two onward are never classified; and multiple document plans collapse to the first. *Evidence:* `page.tsx:143-144` (`isDocument = files[0]...documentBase64`); `analyze-document/route.ts:85-92,:184-188` (one document → one content unit per call).

4. **Mixed-format imports produce disjoint proposals.** Document files and spreadsheet files take mutually exclusive analyze branches that each produce their own proposal and never combine; a document sent down the tabular branch arrives as an empty stub. *Evidence:* `page.tsx:219-260` (the `if (isDocument) … else …` fork, two separate `proposal` objects).

### 1.4 — What works and is the foundation (do not rebuild)

- Multi-file / multi-sheet **classification** already works at `analyze/route.ts:88` (`for (const file of files)`). The collapse is downstream (transport/execute), not in classification.
- The **format-aware plan extractor** already exists: `plan-interpretation.ts:53` dispatches by extension (pdf/pptx/docx/xlsx), downloads from a storage location (`:38`). This is the layer that reads documents; the restoration routes document plans TO it, it is not rebuilt.
- **Plan comprehension signals** (`emitPlanComprehensionSignals`, `plan-interpretation.ts:277,:558`) are transport-independent — the moat accrues for any plan that reaches the plan pipeline. Restored by routing, not by new signal code.
- **HF-255 (merged, PR #443):** documents now upload to Storage and have a storage location. This HF builds on that; it does not redo it.

- **Scale by Design.** Per-file Storage download; no row data in request bodies (AP-1). Unchanged at 10x/100x files and rows.

### §3.3 — Phase 3: Format-aware execute (restore document plans reaching the Plan Agent)

**Intent:** execute must stop assuming every file is a spreadsheet. It resolves each file's format and handles it accordingly; a document's plan unit reaches the existing plan pipeline instead of crashing at the workbook parse.

### §3.4 — Phase 4: Multi-file transport (restore multiple files in one import)

**Intent:** every file in the import reaches execute and is processed by its own format; multiple plan files each produce their own rule set; data files across the import all commit.

Files: `web/src/app/operate/import/page.tsx`, `web/src/components/sci/SCIExecution.tsx`, `web/src/app/api/import/sci/execute-bulk/route.ts`.

- **Thread the per-file storage map end-to-end.** The per-file map already exists at `page.tsx:136` and is passed at `:494`; remove the collapse to a singular location (`page.tsx:322`) as the value execute relies on. `SCIExecution` sends the per-file map (and the unified content units) to execute.
- **Execute processes every file.** For each file in the import: download it, resolve its format (Phase 3), parse-or-route accordingly. Cache the download/parse per file (do not re-download the same file per unit). Each content unit resolves to its source file's storage location and parsed content.
- **Per-file plan interpretation.** Each plan file's plan unit(s) are interpreted with that file's storage location, producing that file's rule set — the proven multi-plan shape (multiple plan files → multiple rule sets).
- **Data units** execute against their own file's parsed sheets.

The architecture is one execute call carrying the per-file map + unified content units, with an internal per-file loop. If enumeration shows this would break the proven single-file path, **HALT-2** and report rather than forcing it; the alternative (one execute call per file, orchestrated client-side) is acceptable only if it preserves the single-file XLSX behavior exactly.

Hard constraint: for an import of exactly one XLSX file, the end-to-end behavior must be byte-identical to today (EPG-4). The multi-file loop must degrade to the current single-file path for N=1 spreadsheet.

Commit: `HF-256 Phase 4: multi-file transport — per-file storage map threaded, execute processes every file by format`; push.

Hard constraint: a single-file import (one XLSX, or one document) must behave exactly as today — one file, one analyzer, one proposal (EPG-4, EPG-1).
