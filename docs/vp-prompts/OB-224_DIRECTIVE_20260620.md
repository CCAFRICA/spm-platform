## §1 Problem Statement

The platform's calculation substrate is among its richest: `calculation_results` stores per-entity payouts with component JSONB, `calculation_traces` stores per-component forensic traces (formula, inputs, output, steps), `committed_data` stores every source transaction, `entity_period_outcomes` materializes dashboard-ready breakdowns, and `disputes` supports structured dispute submission. Backend functions exist to write and read all of these (`writeCalculationTraces()`, `getCalculationTraces()`, entity period outcomes materialization).

No user can access any of it.

MC#1 (Individual Commission Statements) is P0 and NOT BUILT — the page is empty. MC#2 (Transaction-Level Detail) is P0 and NOT BUILT — the page is empty. MC#C29 confirms the commission statements BACKEND is COMPLETE but the rep-facing surface is the gap. The /stream Intelligence Stream renders total payout, component totals, entity names, and trajectory data — all as dead ends with zero click-through to supporting detail. The /operate/calculate page shows a grand total with no per-entity or per-component visibility (Platform Experience Assessment Finding #8). The reconciliation surface shows entity-level comparison with no path to the component traces that explain deltas.

Every number the platform renders is a dead end. The value proposition claims "Every calculation is auditable and explainable" and "Full transparency into every transaction." Neither is true for any user today.

**Defect class:** Expression-layer absence on a complete substrate. The fix is structural: a reusable component library implementing the five-layer drill-through chain, integrated into every surface that renders calculated data, with entity-to-user scoping for persona-appropriate access.

**Combined-treatment rationale:** MC#1, MC#2, MC#19, MC#24, MC#39, MC#43, and MC#17 are all expressions of the same underlying gap — no user-facing path from rendered number to prime data. Building them as separate HFs would produce 7 fragmented implementations reading the same tables with duplicated data-fetching code, violating AP-17 (single code path). One OB with a shared component library eliminates the duplication class entirely.

### §3.1 Database substrate queries

Write and execute a verification script at `scripts/ob-224-substrate-check.ts` using the Supabase service-role client. The script queries and reports:

**Query A — calculation_traces population:**
```sql
SELECT t.name AS tenant, COUNT(ct.*) AS trace_count,
  COUNT(DISTINCT ct.result_id) AS results_with_traces
FROM calculation_traces ct
JOIN tenants t ON ct.tenant_id = t.id
GROUP BY t.name
ORDER BY trace_count DESC;
```
If ANY tenant has trace_count > 0, the substrate is ready. If ALL tenants have zero traces, record the finding and proceed — the component will render a "No trace data available" state, and trace emission will be addressed as a §6A residual.

**Query B — entity_period_outcomes population:**
```sql
SELECT t.name AS tenant, COUNT(epo.*) AS outcome_count,
  COUNT(DISTINCT epo.entity_id) AS entities_with_outcomes
FROM entity_period_outcomes epo
JOIN tenants t ON epo.tenant_id = t.id
GROUP BY t.name;
```

**Query C — profile_scope population:**
```sql
SELECT t.name AS tenant, COUNT(ps.*) AS scope_count,
  ps.scope_type, array_length(ps.visible_entity_ids, 1) AS entity_count
FROM profile_scope ps
JOIN tenants t ON ps.tenant_id = t.id
GROUP BY t.name, ps.scope_type, array_length(ps.visible_entity_ids, 1);
```

**Query D — entity_relationships (hierarchy data):**
```sql
SELECT t.name AS tenant, er.relationship_type, COUNT(*) AS rel_count
FROM entity_relationships er
JOIN tenants t ON er.tenant_id = t.id
GROUP BY t.name, er.relationship_type;
```

**Query E — committed_data sample (data_type distribution):**
```sql
SELECT t.name AS tenant, cd.data_type, COUNT(*) AS row_count,
  COUNT(DISTINCT cd.entity_id) AS entities
FROM committed_data cd
JOIN tenants t ON cd.tenant_id = t.id
GROUP BY t.name, cd.data_type
ORDER BY row_count DESC;
```

**Query F — calculation_results component structure sample:**
```sql
SELECT t.name AS tenant, cr.entity_id,
  jsonb_array_length(cr.components) AS component_count,
  cr.total_payout
FROM calculation_results cr
JOIN tenants t ON cr.tenant_id = t.id
LIMIT 5;
```

### §3A.3 Function implementations

**`resolveEntityScope(profileId: string): Promise<EntityScope>`**
- Read `profile_scope` WHERE profile_id = profileId AND tenant_id = current tenant.
- If no row exists, return `{ visible_entity_ids: [], visible_rule_set_ids: [], visible_period_ids: [] }` — empty arrays mean "all" (admin default).
- This function is called once on page load and the result is passed to other functions.

**`getEntityResults(tenantId, scope, filters): Promise<EntityResult[]>`**
- If `entity_period_outcomes` has rows for this tenant (check Query B result): read from `entity_period_outcomes` joined with `entities` for display_name, filtered by scope.visible_entity_ids (if non-empty) and filters (period_id, batch_id).
- Fallback: if no `entity_period_outcomes` rows, read from `calculation_results` grouped by entity_id, joined with `entities`.
- Both paths produce the same `EntityResult[]` shape.
- Sort by total_payout descending by default.
- Include period label from `periods` join.
- Include lifecycle_state from `calculation_batches` join.

**`getComponentTraces(tenantId, resultId): Promise<ComponentTrace[]>`**
- Read `calculation_traces` WHERE result_id = resultId AND tenant_id = tenantId.
- If no rows, return empty array (component renders "No trace data" gracefully).
- To get the resultId for an entity+period+batch, the caller first queries `calculation_results` WHERE entity_id AND period_id AND batch_id. This lookup is part of the function: accept entity_id + period_id + batch_id as alternative parameters, resolve to result_id internally.

**`getSourceTransactions(tenantId, entityId, periodId, dataType?): Promise<SourceTransaction[]>`**
- Read `committed_data` WHERE entity_id AND period_id AND tenant_id.
- If dataType provided, filter by data_type.
- Return row_data contents as individual rows.
- Order by source_date, then created_at.

**`submitDispute(tenantId, disputeData): Promise<{ id: string }>`**
- Insert into `disputes` with: tenant_id, entity_id, period_id, batch_id, status='open', category, description, amount_disputed, filed_by (current profile_id).
- Return the created dispute id.

All functions use the Supabase browser client (user-scoped reads, RLS enforced). The `submitDispute` function uses the browser client with the user's auth context so RLS and filed_by are correct.

Commit the data layer. Verify each function compiles and returns data by writing a quick test at `scripts/ob-224-data-layer-test.ts` that calls each function with known tenant/entity ids from the substrate verification output.

### §3B.2 EntityResultsList

Props:

- `periodId?: string` (filter to specific period)
- `batchId?: string` (filter to specific batch)

Renders: A sortable, filterable table of entities with columns for display_name, total_payout (formatted per tenant currency), component_count, and lifecycle_state. ### §3B.3 ComponentCards

Props:
- `tenantId: string`
- `entityId: string`
- `periodId: string`
- `batchId?: string`
- `comparisonData?: Record<string, { expected: number; delta: number }>` (for reconciliation context)
- `onTransactionDrill?: (componentName: string, dataType: string) => void`
- `onDispute?: (componentName: string, amount: number) => void`

Renders: N cards (one per component from `calculation_traces`), each showing:
- Component name (from trace.component_name)
- Formula (from trace.formula, if present)
- Inputs (from trace.inputs — render key-value pairs, format numbers per tenant currency)
- Output (from trace.output — the component payout amount)
- Steps (from trace.steps — if present, render as an ordered list of calculation steps)
- If comparisonData is provided: show expected value and delta alongside the output
- "Show transactions" link/button on each card (triggers onTransactionDrill)
- "Dispute" action button on each card (triggers onDispute, which opens DisputeInline)

Fallback when no traces exist: render a simplified view using `calculation_results.components` JSONB directly — show component names and payout amounts without trace detail. This ensures the component works even when `calculation_traces` is empty.

Cards follow the existing Vialuce design system: use the indigo/gold palette, DM Sans typography, border-radius and spacing from the theme tokens.

### §3B.4 TransactionRows

Props:
- `tenantId: string`
- `entityId: string`
- `periodId: string`
- `dataType?: string` (filter to transactions for a specific component's metric)

Renders: A table of `committed_data` rows with columns derived dynamically from `row_data` JSONB keys. The column headers are the actual keys from the data — no hardcoding (Korean Test). Each row shows all values from row_data, plus source_date if present.

The component reads the first row's keys to determine column headers, then renders all rows with those columns. Values are formatted: numbers get currency or numeric formatting, dates get locale-appropriate formatting.

Empty state: "No source transactions found."

### §3B.5 DisputeInline

Props:
- `tenantId: string`
- `entityId: string`
- `periodId: string`
- `batchId?: string`
- `componentName: string`
- `amount: number`
- `onClose: () => void`
- `onSubmitted: (disputeId: string) => void`

Renders inline (not a modal, not a separate page — Intuitive Adjacency). A form with:
- Pre-populated and read-only: entity display_name, period label, component name, amount
- Editable: category (dropdown — read categories from the ICM domain registration's disputeCategories, but fall back to a generic list if domain registration is not available; Korean Test = no hardcoded categories), description (textarea)
- Submit button that calls `submitDispute()` from the data layer
- Success state: "Dispute submitted" with the dispute ID

### §3B.6 DrillThroughPanel (orchestrator)

This is the component that manages the progressive-disclosure state machine. It's the container that gets embedded in host pages (/stream, /calculate, etc.) and orchestrates the expand/collapse of the sub-components.

Props:
- `tenantId: string`
- `scope: EntityScope`
- `periodId?: string`
- `batchId?: string`
- `initialEntityId?: string` (for direct entity drill-in)
- `comparisonData?: ...` (for reconciliation context)
- `compact?: boolean`

State machine:
1. Initial: shows EntityResultsList
2. Entity selected: expands ComponentCards below the selected entity row
3. Component drilled: expands TransactionRows below the selected component card
4. Dispute initiated: expands DisputeInline below the selected component card

Each expansion is an accordion — selecting a new entity collapses the previous expansion. Selecting "Show transactions" on a component collapses any previous transaction expansion and opens the new one.

Back navigation: each expansion has a collapse/close control. Clicking the entity row again collapses its expansion.

**HALT-1:** If the codebase uses a data-fetching pattern that is fundamentally incompatible with the component library design (e.g., all data is fetched via server components with no client-side query capability, or the app uses a state management system that prevents component-level data fetching) — HALT and record the architectural constraint. This has never been observed but would require a design change.

**HALT-2:** If `calculation_results.components` JSONB has a structure that cannot be rendered as individual component cards (e.g., it's a flat number instead of an array of component objects) — HALT and record the actual structure. The component card design depends on per-component breakdown being present in the data.

No other halts. All other conditions (empty tables, missing hierarchy data, empty pages, dead service references) have resolution rules defined in §3.1 and §3C.7.
