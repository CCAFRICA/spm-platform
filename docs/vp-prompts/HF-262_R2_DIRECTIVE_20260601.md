## §1 — Problem Statement

Meridian's fleet component (C5, *Utilización de Flota*) pays **zero** across all entities (persisted batch `e1098ffa`, three periods). DIAG-58 (`docs/diagnostics/DIAG-58_AGGREGATE_SCOPE_CAPABILITY_OUTPUT.md`, HEAD `e85a7678`) determined **Condition A**: the fleet is the *reference-row→member projection* class, distinct from the *peer-entity aggregation* class that is complete.

**The defect is input-resolution starvation, not a missing engine primitive.** DIAG-58 §4 proved this empirically with the real-prime engine harness:
- Line `(b)` — faithful projection at live state (hub-reference entity metadata `= {}`) → **0**.
- Line `(b′)` — the *same engine* with the hub boundary present in entity metadata → **500 (matches)**.
- Line `(c2)` — the persisted fleet formula with hub metrics present → **960** (formula sound; `clamp(1200/1000 = 1.2 ≤ 1.5) × 800`).

The existing `scope` prime (`intent-executor.ts:223-238`) resolves the fleet the instant the hub-grouping boundary is present in `entities.metadata`. Live hub entities carry `metadata = {}` (AUD-005 `e85a7678` §5; `entity-resolution.ts` inserts `metadata: {}`), so `entityMetadata[boundary]` is `undefined` and never matches an employee's hub. **The fix populates the boundary; it does not add or reshape a primitive.**

**Defect-class lineage (this is a re-tread of a solved class, different boundary):** HF-190 (CRP, 2026-04-05) solved this exact starvation for the `district`/`region` boundary by spreading the structural enrichment dictionary into `entities.metadata` (`metadata: { ...meta.enrichment, ... }`) — Korean-Test compliant, zero hardcoded field names — so the `scope` prime could match. **HF-262 applies the same enrichment-spread mechanism to the hub-grouping boundary.** It does **not** introduce a hub-specific populator beside the literal store-field populator (`store-metadata-population.ts`); that literal list is itself the tenant-myopic artifact this HF must route *around*, not extend.

This is a **multi-layer defect** treated in one HF (FP-81): (D1) boundary not populated → fleet zero; (D2) the `scope`-prime `undefined === undefined` match footgun, latent today and made live the moment any boundary exists (AUD-005 §7); (D3) phantom hub payees — the assignment self-heal assigns all entities and the roster-population filter is inert because Meridian's roster sheet matches no Tier-3 literal keyword, so 12 hubs are calculated and paid.

**SR-42 (convergence-touch) does not fire.** A prior reading (the R1 ADR) contemplated re-shaping the fleet `calculationIntent` to `scope_aggregate` — a convergence/intent-emission change (SR-42 surface). DIAG-58 §3(b) shows `scope_aggregate` composes to `scope→aggregate` summing `row[field]` across **sibling entities**; the fleet projects **one** hub row onto members, which is not a peer sum — so that reshape is the **wrong shape** and is rejected (see §6). The edits in scope touch entity-metadata population, a scope-prime null-guard, and a payee filter — none touches convergence-bound calculation math or intent emission. No IRA Class A is required.

The next tenant with any member→group boundary inherits D1 for free; the guard protects every tenant; the de-literalized roster filter generalizes.
