## §1 — Problem Statement

The SCI header comprehension correctly classifies every column in every sheet. For Meridian's Datos_Rendimiento sheet:

- `No_Empleado: identifier@0.99`
- `Hub: reference_key@0.95`
- `Tipo_Coordinador: attribute@0.97`
- 12 measure columns, 2 temporal columns

The commit path ignores these classifications and independently re-derives which column is the entity key. It selects `Hub` (a reference_key) instead of `No_Empleado` (the identifier). The import log proves this:

```
commitContentUnit transaction: entity_id_field="Hub"
```

This is the same structural violation as HF-325. Comprehension recognized correctly (Decision 158: LLM recognizes). The commit code re-derived the answer independently (vestigial heuristic path). The re-derivation chose wrong.

Three consequences, all from this one root cause:

**A — Wrong entity key on transactions.** Transaction rows keyed by `Hub` instead of `No_Empleado`. The engine can't match transaction data to entities. Convergence bindings are produced correctly but per-entity resolution falls back to sheet-matching, finding no data, returning empty strings. 4 of 5 components produce $0.

**B — Empty entity metadata.** Comprehension classified `Tipo_Coordinador: attribute@0.95`. Entity resolution doesn't project comprehension-classified attributes into entity metadata. `materializedState={}` for all 67 entities. Variant discrimination produces zero tokens. All entities fall to `default_last` instead of routing to the correct variant.

**C — Convergence fallback to sheet-matching.** Because entity keys don't match, per-entity convergence resolution fails silently, falling back to sheet-matching. The engine log shows `HF-108 Resolution path: sheet-matching (fallback)` per entity despite 10 valid convergence bindings existing globally. This is a consequence of A, not a separate defect.

**The fix is subtraction.** The commit path's independent entity-key selection heuristic is the vestigial path. Remove it. The comprehension classification is the authoritative determination:

- Column classified as `identifier` → entity_id_field. One path. No heuristic.
- Columns classified as `attribute` → project into entity metadata at entity resolution time. Comprehension already determined which columns are attributes.

No new heuristic. No discriminator. No prompt change. Comprehension is authoritative.

## §2 — Constraints

**C1 — SUBTRACTION.** The change removes or bypasses the commit path's independent entity-key selection. No new selection heuristic, no new module. The comprehension classification that already exists becomes the single read path.

**C2 — COMPREHENSION IS AUTHORITATIVE.** The comprehension layer classified columns with roles (`identifier`, `reference_key`, `attribute`, `measure`, `temporal`, `name`). These classifications are the Decision 158 recognition output. The commit path must honor them, not re-derive.

**C3 — SPATIAL.** The fix applies everywhere the commit path selects an entity key or decides which columns to project into entity metadata. CC must search the SCI commit code for ALL locations where entity_id_field is assigned or where entity metadata/attributes are populated. Every location must read from comprehension classifications. No orphaned heuristic paths.

**C4 — DOMAIN-AGNOSTIC.** The fix uses structural roles (`identifier` vs `reference_key`), not field names, not language, not column positions. Korean Test (Decision 154) applies.

**C5 — NO ENGINE, CONVERGENCE, OR PLAN INTERPRETATION CHANGES.** The comprehension layer works. The convergence layer works. The engine works (proven by HF-325/BCL $312,033 EXACT). The plan interpreter works (10/10 components succeeded). Only the SCI commit/entity-resolution path changes.

**C6 — RECONCILIATION-CHANNEL SEPARATION.** CC reports computed values verbatim. No GT comparison.

**C7 — NON-REGRESSION.** BCL's import must continue to work identically. BCL's committed_data entity_id_field must remain `ID_Empleado` (its comprehension-classified identifier). Sabor's import must continue to work identically.
