## §1 — PROBLEM STATEMENT

Header Comprehension (HC) asks the LLM to characterize every column in an imported sheet. The LLM's output is compressed into a fixed enumeration of six roles: `identifier`, `name`, `temporal`, `measure`, `attribute`, `reference_key` (the `ColumnRole` type). The LLM understands that `DNI_Vendedor` is a seller's national identity document (entity identifier) and `Folio` is a transaction receipt number (transaction identifier). But the output format offers one box: `identifier`. Both go in. Downstream, `commitContentUnit` selects `entity_id_field` by confidence ranking among all `identifier`-role columns, picks `Folio` (0.99 > 0.98), and every committed_data row gets entity_id derived from a unique-per-row receipt number instead of the repeating seller DNI.

Result: 100% of MIR transaction rows (~75K) are entity-misattributed. All five plans produce wrong numbers. Clean-slate reimport reproduces the failure because the selection logic hasn't changed.

### §1.1 — Violation class

**Decision 158 (LLM recognizes; deterministic code constructs and guarantees):** The LLM's recognition was correct. The interface between recognition and construction was lossy. The fixed vocabulary is the lossy boundary.

**OB-214 / No Fixed Taxonomy:** "No fixed enumeration registry requiring developer maintenance — architecturally unacceptable." The role vocabulary is that registry.

**AUD-009 (Registry/Cherry-Pick Pattern):** The HC prompt asks the LLM to produce rich semantic understanding. The `ColumnRole` enumeration silently discards everything that doesn't fit six predefined labels.

### §1.2 — Fix class: SUBTRACTION

Remove the fixed role vocabulary. Do not widen it. Do not add `entity_identifier` / `transaction_identifier`. Every role added is a new enumeration entry that compresses the LLM's understanding.

The LLM emits its characterization as a structured-but-unconstrained object per column. The fields are semantic channels the LLM fills freely — not an enumerated type. The construction layer reads these fields directly. No pattern-matching utility between them. Same architectural move as DS-029 (No Fixed Taxonomy for plan field characterization) and the same subtraction class as HF-325 / HF-328 / HF-329.

### §2.1 — Design decision: Structured-but-unconstrained output (LOCKED for this OB)

The HC prompt instructs the LLM to emit a structured characterization object per column. The fields are semantic channels — the LLM fills each with its own assessment. No field is validated against an enumeration. Example output:

```json
{
  "characterization": "Seller's national identity document number, unique per seller, used to group all transactions by salesperson",
  "identifies": "entity",
  "data_nature": "identifier",
  "relationships": ["pairs with Nombre_Vendedor as the entity display name"]
}
```

- `characterization` — free-form description in the LLM's own words. Replaces `semanticMeaning`.
- `identifies` — the LLM's assessment of what scope this column identifies (entity, transaction, product, reference, nothing). Free-form, not enumerated — the LLM can write any value.
- `data_nature` — the LLM's assessment of the column's data nature (identifier, measure, temporal, categorical, name, computed, etc). Free-form, not enumerated.
- `relationships` — the LLM's observations about how this column relates to others in the sheet.

The construction layer reads `identifies` and `data_nature` directly. For entity_id_field selection: read `identifies` — the column where the LLM wrote an entity-scope value ("entity", "seller", "person", "employee", etc.) is the entity identifier. The construction layer verifies structurally (the selected column should repeat across rows). No intermediary matching utility. No boolean classification functions. The LLM's own words are the read surface.

This mirrors how convergence binding proposals already work: the LLM emits structured proposals with free-form field descriptions, and the construction layer reads them.

## §5 — PHASE 2: CONSUME + DIAGNOSTIC (all downstream readers)

Every function that currently `switch`es on `columnRole` or checks `=== 'identifier'` must be rewritten to read from the structured characterization. These consumers are file-disjoint, enabling ULTRACODE fan-out. CC determines the optimal parallelization from the Phase 0 inventory's file mapping.

### §5.1 — Consumer rewrite principle

Every consumer currently does: `if (role === 'identifier') { ... }`. The replacement reads: `if (char.identifies === 'entity' || /* the LLM's entity-scope assessment */) { ... }`. The construction logic is the same; the input is richer. No intermediary matching utility — each consumer reads the characterization fields directly, because each consumer knows what it's looking for (the commit path needs entity identity, the pattern classifier needs sheet-type signals, convergence needs measure/attribute distinction). The LLM's structured output is the shared surface; the consumers are domain-specific readers.

### §5.2 — Critical consumer: entity_id_field selection

This is the MIR-blocking fix. `commitContentUnit` must read `identifies` from each column's characterization. The column where the LLM assessed entity-scope identity (seller, employee, person, account) is `entity_id_field`. The column where the LLM assessed transaction-scope identity (receipt, folio, invoice) is NOT `entity_id_field`.

Structural verification after selection: the selected column should have many rows per distinct value (it repeats — 30 sellers across 4,927 transactions). If the selected column has 1:1 distinct-to-row ratio → the construction layer should log a warning but proceed (the LLM's semantic assessment takes priority; the structural check is a sanity signal, not a gate — per Decision 158).

### §5.3 — Consumer list (from inventory)

- **commitContentUnit** — entity_id_field selection (§5.2)

- **Profile enhancer / HC overrides** — temporal/name/identifier reinforcement

### §5.4 — Diagnostic log updates
