## §1 — PROBLEM STATEMENT

The platform imports data correctly (SCI proven: BCL, Meridian, CRP, MX Restaurant). After import, the user sees "Import Complete" and nothing else. `committed_data` holds 162,956+ rows. The intelligence stream on /stream reads only from `calculation_results` and `entity_period_outcomes` — tables that don't exist until a plan is created, bindings converge, and calculation runs. For any tenant without a calculation — or any tenant between imports — the platform shows nothing.

DS-029 specifies the fix: the Observation expression surface reads from the carrier. OB-205 implements Phase 1: structural carrier expression at Cold CRL tier.

**What ships:** After this OB, importing data into any tenant produces visible carrier intelligence on /stream — row counts, entity counts, classification confidence, content unit breakdown, pipeline readiness — without any plan or calculation prerequisite. The user sees what the platform understood, not just that it stored a file.

## §3 — PHASE 1: CARRIER INTELLIGENCE API

Create a server-side API route that reads carrier state for a tenant and returns a structured intelligence payload. This is the data layer — components consume it.

### 3.1 Create API route

File: `web/app/api/carrier-intelligence/route.ts`

This route accepts a GET request with the tenant_id from the authenticated session. It queries:

1. **committed_data summary**: COUNT(*) grouped by `data_type`. MIN/MAX `source_date`. COUNT(DISTINCT `entity_id`) WHERE entity_id IS NOT NULL. COUNT(DISTINCT `import_batch_id`).

2. **entities summary**: COUNT(*) grouped by `entity_type` and `status`. COUNT(*) WHERE `external_id` IS NOT NULL (bound entities).

3. **import_batches summary**: Most recent batch by `created_at`. Total batch count. Most recent `file_name`, `row_count`, `completed_at`.

4. **classification_signals summary**: AVG(`confidence`) WHERE `confidence` IS NOT NULL. COUNT(*) grouped by `classification`. COUNT(*) grouped by `decision_source` (to distinguish structural-fingerprint matches from LLM-classified).

5. **rule_sets presence**: COUNT(*) WHERE `status` != 'draft'. Presence/absence determines pipeline readiness.

6. **rule_set_assignments coverage**: COUNT(DISTINCT `entity_id`). Compared to total entity count = binding coverage percentage.

7. **periods**: Active periods by `status`. Date ranges.

8. **calculation_batches**: Most recent batch `lifecycle_state` per period. Presence/absence determines whether calculation has run.

The response shape:

```typescript
interface CarrierIntelligence {
  dataSnapshot: {
    totalRows: number;
    contentUnits: Array<{ dataType: string; rowCount: number }>;
    dateRange: { earliest: string | null; latest: string | null };
    entityBound: number;
    entityUnbound: number;
  };
  entities: {
    total: number;
    byType: Array<{ entityType: string; count: number }>;
    withExternalId: number;
  };
  imports: {
    totalBatches: number;
    latestBatch: {
      fileName: string;
      rowCount: number;
      completedAt: string | null;
      createdAt: string;
    } | null;
  };
  classification: {
    avgConfidence: number | null;
    byClassification: Array<{ classification: string; count: number }>;
    byDecisionSource: Array<{ source: string; count: number }>;
  };
  pipelineReadiness: {
    hasData: boolean;
    hasEntities: boolean;
    hasPlan: boolean;
    hasBindings: boolean;
    hasCalculation: boolean;
    latestLifecycleState: string | null;
  };
  periods: Array<{
    id: string;
    label: string;
    startDate: string;
    endDate: string;
    status: string;
  }>;
}
```

Use the existing Supabase service-role pattern from the codebase (check how `/api/calculate/route.ts` or similar routes authenticate and query). RLS applies — the route must use the tenant-scoped client, not service-role, so RLS policies on committed_data, entities, etc. gate access.

**HALT-1:** If RLS on any of these tables blocks the query for the authenticated user's tenant, HALT and report which table + which policy. Do not bypass RLS.

## §4 — PHASE 2: CARRIER INTELLIGENCE STREAM COMPONENTS

Create the React components that render carrier intelligence elements on /stream. These integrate into the existing intelligence stream layout.

### 4.1 Carrier Intelligence Hook

File: `web/lib/hooks/useCarrierIntelligence.ts`

A React hook that fetches from `/api/carrier-intelligence` and returns the `CarrierIntelligence` payload with loading/error states. Uses SWR or the existing data-fetching pattern in the codebase (check which pattern `/stream` currently uses — if it uses React Query, use React Query; if SWR, use SWR; if raw fetch + useState, match that).

### 4.2 Import Health Card

File: `web/components/stream/CarrierImportHealth.tsx`

Renders when `pipelineReadiness.hasData` is true. Shows:

- **Value:** "{totalRows} rows across {contentUnits.length} content units from {imports.totalBatches} import(s)"
- **Context:** "{entities.total} entities ({entities.withExternalId} with external ID). Classification confidence: {classification.avgConfidence}%"
- **Comparison:** Cold tier — omit comparison for now. Show "First import period — comparison available after next import." if totalBatches === 1.
- **Action:** Button "Calculate" if `pipelineReadiness.hasBindings && pipelineReadiness.hasPlan && !pipelineReadiness.hasCalculation` — navigates to `/operate/calculate`. Button "Review Data" — toggles inline expansion (Phase 4 Content Unit Browser). Button "Import More Data" — navigates to import.
- **Impact:** Derived from pipeline readiness. "Ready to calculate for {entities.total} entities." or "Missing: {what's missing} — {what to do}."

Visual: Use the existing card/element pattern from /stream. Indigo accent for admin persona. Do not invent a new card system — match the existing intelligence stream element layout.

### 4.3 Entity Landscape Card

File: `web/components/stream/CarrierEntityLandscape.tsx`

Renders when `entities.total > 0`. Shows:

- **Value:** "{entities.total} entities discovered"
- **Context:** Type breakdown: "{byType[0].count} {byType[0].entityType}, {byType[1].count} {byType[1].entityType}, ..."
- **Action:** Button "View Entities" — toggles inline expansion (Phase 4 Entity Explorer).

### 4.4 Pipeline Readiness Card

File: `web/components/stream/CarrierPipelineReadiness.tsx`

Renders always (even when no data exists — shows empty pipeline). Shows the pipeline as a horizontal stepper:

`Import → Classify → Bind → Calculate → Reconcile`

Each step shows ✓ (complete) or ○ (pending) based on `pipelineReadiness` booleans. The active step (first ○) has a primary-action button: "Import Data" / "Upload Plan" / "Configure Bindings" / "Calculate" / "Start Reconciliation" depending on which step is active. One button. One action. Thermostat, not thermometer.

### 4.5 Stream Integration

Locate the existing `/stream` page component (likely `web/app/(workspace)/stream/page.tsx` or similar). Identify where intelligence elements render. Add the three carrier intelligence cards above the existing calculation-dependent elements. Carrier cards render first because they represent the upstream pipeline — they are relevant before calculation elements become available.

**Korean Test:** No card contains hardcoded domain strings ("commission", "franchise", "ICM"). Labels derive from `data_type` values in committed_data and `entity_type` values in entities — whatever the tenant imported.

## §6 — PHASE 4: INLINE EXPANSION COMPONENTS

These components render inside the stream cards when the user clicks "Review Data" or "View Entities." They expand inline — the user does not navigate away from /stream.

### 6.1 Content Unit Browser

File: `web/components/stream/CarrierContentUnitBrowser.tsx`

Renders as an inline expansion below the Import Health Card. Shows a table/list of content units:

| Content Unit (data_type) | Rows | Date Range | Entities Bound |
|---|---|---|---|

Each row is a content unit from `dataSnapshot.contentUnits`. Row count from the carrier intelligence payload. Date range and entity binding derived from committed_data for that data_type (may require a secondary API call: `GET /api/carrier-intelligence/content-unit?dataType=X` returning row-level detail).

If secondary API needed, create: `web/app/api/carrier-intelligence/content-unit/route.ts` — accepts `dataType` query param, returns rows with date range and entity binding counts for that content unit.

Maximum 20 rows displayed per content unit expansion. Pagination if more. This is not a full data dump — it's a summary that proves the platform understood the data.

### 6.2 Entity Explorer

File: `web/components/stream/CarrierEntityExplorer.tsx`

Renders as an inline expansion below the Entity Landscape Card. Shows a list of entities:

| Entity | Type | External ID | Status | Transactions |

If entity count > 50, paginate. Show first 20 sorted by display_name.

**Korean Test:** Column header "Entity" not "Employee" or "Rep" or "Location." The display_name comes from the entity record. The entity_type comes from the entity record. No domain assumptions.

## §10A — RESIDUALS

**R1:** The carrier intelligence API may need performance optimization for tenants with 100K+ committed_data rows. COUNT queries on large tables can be slow. Materialized views or periodic count caching may be needed at scale. Not needed for current proof tenants (largest is ~163K rows).

**R2:** The Content Unit Browser's secondary API (`/api/carrier-intelligence/content-unit`) may not be needed if the primary payload includes sufficient detail. CC should assess during implementation and skip the secondary route if the main payload already carries the data.

**R3:** Entity Explorer transaction counts per entity require COUNT(*) per entity_id on committed_data. For tenants with many entities, this is N+1 queries. Consider a single grouped query or include in the main payload as `entityTransactionCounts: Record<string, number>`.
