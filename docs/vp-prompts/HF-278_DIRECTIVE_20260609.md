## §1 Problem Statement

Three hotfixes circled one structure without naming it:
- **HF-274** attached `meta.scale` to convergence-side ratio-keyed breakpoints. Correct intent (Meridian c1 floored without it) but the mechanism is wrong — see below.
- **HF-276** pre-multiplied evaluator-side ratio-keyed breakpoints by `scale.value`. Reverted (regressed BCL).
- **HF-277** omitted `meta.scale` for evaluator-side DAG-computed ratios. Correct principle, wrong scope — evaluator-only.

Production reads (architect channel, rule_sets `8afbe873` BCL and `820f46b8` Meridian) prove both c1 compare operands are identical in structure:

```
BCL c1 (Captación de Depósitos):
  {"op":"divide","prime":"arithmetic","inputs":[
     {"field":"net_new_deposits","prime":"reference"},
     {"field":"deposit_target","prime":"reference"}]}
  comp_scale = {side:"convergence", value:100}  break = 1.3

Meridian c1 (Entrega a Tiempo):
  {"op":"divide","prime":"arithmetic","inputs":[
     {"field":"on_time_deliveries","prime":"reference"},
     {"field":"total_deliveries","prime":"reference"}]}
  comp_scale = {side:"convergence", value:100}  break = 98
```

Both are `prime:arithmetic, op:divide` — a ratio computed inside the DAG from two raw `reference` operands. Both convergence-side. Both meta.scale=100.

**The principle (already aligned):** A DAG-computed ratio defines its own space. Both operands are raw data; the quotient is the value; the breakpoints are declared against that quotient. There is nothing to scale. The recognizer's `scale.value` declaration is noise — and it is noise **regardless of which side** the recognizer attached it to. Convergence-vs-evaluator is about where a scale *would* be applied; it is not a reason the ratio needs scaling. The operand structure is the whole determinant.

**The fix:** In `buildConstantWithScale`, omit `meta.scale` whenever the compare operand is a DAG divide (`prime:arithmetic, op:divide`) — on **both** sides. Scale survives only when the operand is a genuine single pre-computed `reference` column (the §6A case), where a value-space reconciliation can be meaningful.

This is one structural predicate (operand is a DAG divide) with one consequence (omit scale). It is not an enumeration of tenants, components, fields, break-spaces, or magnitudes. It is the enduring platform rule the prior three hotfixes were each approximating from one side.

## §2 Substrate-Bound Discipline

**Decision 158:** Construction consumes recognition metadata and correctly suppresses a structurally meaningless declaration rather than propagating it. The recognizer may emit any `scale.value` on any side; for a DAG divide, construction does not propagate it.

**Decision 154 (Korean Test):** The predicate is `operand.prime === 'arithmetic' && operand.op === 'divide'`. No field names, component names, tenant identifiers, magnitude constants, break-space inference, or resolved-value sniffing. Pure operand shape.

**SR-34 (class, not instance):** Any DAG-divide-keyed band, any tenant, any domain, any side. BCL c1 and Meridian c1 are the first two instances; the next divide-keyed band inherits correct behavior with zero new code.

**Not a registry (PCD):** A registry enumerates cases. This collapses them — one invariant replacing HF-274's convergence branch, HF-276's reverted evaluator branch, and HF-277's evaluator-only omission with a single side-agnostic structural rule.

**DD-7:** Single pre-computed `reference` operands (not divides) are untouched — `meta.scale` is retained for them exactly as before.

## §3B Phase 2 — Implement

In `buildConstantWithScale` (`web/src/lib/plan-intelligence/intent-constructor.ts`):

Replace the HF-277 `attach` expression with a side-agnostic rule. `meta.scale` is attached **only** when the compare operand is NOT a DAG divide:

```
const operandIsDagDivide =
  compareOperand?.prime === 'arithmetic' && compareOperand?.op === 'divide';
const attach = !operandIsDagDivide;   // scale only for single pre-computed reference operands
```

Use the same operand reference HF-277 already inspects to derive `otherSideIsRatio` — read its `prime`/`op` directly rather than introducing a new traversal. Remove `otherSideIsRatio` and the side-dependent branches entirely; they are subsumed. When `attach` is false, emit the breakpoint value with no `meta.scale`. When true (single `reference` operand), attach `ConstantScaleMeta` exactly as the pre-HF-274 path did.
