## §1 — PROBLEM STATEMENT

### 1.1 The synthesis the architect required

Across OB-205/HF-291/OB-206 the platform was patched surface-by-surface without the organizing structure that ties personas, modules, and the adaptive layer together. The architect has directed: reorganize the site around the four agents that the Capability Status Board already establishes, redesign /results per-persona-per-module (ICM and Financial), fix the routing so users reach intelligence, and make the lifecycle surface a real cockpit. These are one coherent change — a navigation spine with the persona surfaces hung on it — not four tactical fixes.

### 1.2 The four agents are the missing navigational spine

The Capability Status Board R4 organizes all 33 capabilities under four agents, each with a verb:
- **Platform Core — "Always on":** ingestion, identity, the carrier (committed_data), classification, the adaptive signal layer. This is substrate, not a destination.
- **Calculate:** the deterministic engine — import, calculate, the lifecycle.
- **Decide:** performance intelligence — results, coaching, allocation, trajectory (the persona surfaces).
- **Consolidate:** reconciliation + the Financial module views + payout/export.

The current navigation (Operate / Perform / Configure / Financial workspaces) mixes metaphors — a verb, a domain, a function — which CLT-84-F20 named as the root IA defect. The agent-verbs are one consistent organizing principle: acts the system performs. The reorganization is grouping, not new pages — every existing route folds under an agent.

### 1.3 The live failures (production, 2026-06-14)

- **F-1 Routing:** platform admin entering BCL from Observatory lands on /operate, not /stream — a middleware platform-admin branch hardcodes /operate, bypassing Decision 128 (which the roleDefaults table below it honors for every other role). OB-206 missed this branch.
- **F-2 /results is one ICM-shaped page for all personas:** the Results Proof View (total payout, empty attainment distribution, component bars, flat entity table) is an engine-verification artifact, not a persona surface. Admin needs governance + anomaly actions; Manager needs the coaching heatmap; Rep needs personal earnings + growth. The "No attainment data" panel is the F-84/attainment-not-persisted gap.
- **F-3 Financial module is absent and orphaned:** the FM 5-level hierarchy (Network→Brand→Region→Location→Check) and 6 persona roles are specified; five FM views are built but wired to the dead legacy pipeline, not committed_data. A.24 (plan-data independence) means the same transaction is revenue (Consolidate), commission (Calculate), or royalty (future) — so results cannot be one ICM shape.
- **F-4 /operate is the wrong page and was never redesigned:** Pipeline-stepper + ICM module card + "View Intelligence" button — flagged repeatedly; must become the lifecycle cockpit.
- **F-5 Dead/unintelligible action controls:** "Recognize" (Manager) leads nowhere; Acceleration Cards (Certify/Coach/Intervene) and "Investigate"/"Resolve"/"Simulate" are labels without defined behavior (CLT-167 audit lineage; F-30/F-31 dead buttons). This is the Action Proximity class — a button that does nothing is worse than no button.

### 1.4 Defect-class lineage

(a) IA-metaphor-mixing (CLT-84-F20) → the agent-nav spine resolves it. (b) Decision-Implementation Gap + incomplete trace (F-1) → the missed middleware branch. (c) "Dashboard not intelligence" / FP-67 (F-2/F-3) → persona surfaces without the Five Elements or module-awareness. (d) Action-Proximity dead controls (F-5). (e) Architect-directive deferral (F-4) → /operate redesign deferred in OB-206 R1 after the architect de-deferred it.

### 5.2 Cockpit composition (Cockpit page type: Pipeline + Action Controls + Status Panels)

/operate becomes the lifecycle cockpit, Admin operational surface:
- **THE CYCLE (pacemaker):** the period lifecycle pipeline, active stage highlighted, ONE proximate action control with an action-verb label ("Run Official," "Submit for Approval," "Review Discrepancies"). Compact multi-period timeline ("Jan ✓ PAID · Feb ✓ APPROVED · Mar ● RECONCILE · Apr ○ AWAITING") with the most-actionable period prominent.
- **THE PULSE (vital signs):** payout this period (+ delta vs prior), entities processed, data freshness ("last import: N hours ago" — the T-1 vital sign), reconciliation match %, pending actions. Reference frames + sparklines (3+ periods). Bloodwork-silent; **F-84 guard:** no synthetic budget metric.
- **THE QUEUE (action stream):** self-clearing, priority-ranked items derived from cycle state ("Review Preview results," "Import received — N employees, verify mapping," "Resolve N outliers"). Priority by cycle impact (HIGH blocks advancement / MEDIUM / LOW). Click → task surface. Empty → "All caught up — the cycle is healthy."

### 5.3 One source of truth + not the landing

Cycle/Pulse/Queue read the same lifecycle state as /stream (they never disagree on the phase). /operate is reached from sidebar/stream action — NOT the default landing (Phase 2 ensures /stream). **SR-38:** trace one computed Pulse metric in the report.

**HALT-4:** no CompensationClockService exists AND Cycle/Queue/Pulse can't wire to the lifecycle state machine without net-new infra beyond scope. Report; cockpit may render from `state-reader.ts` directly as the minimum-viable single source.
