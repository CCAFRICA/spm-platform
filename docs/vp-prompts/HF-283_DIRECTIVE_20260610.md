## §1 — PROBLEM STATEMENT

**Defect (carried verbatim-in-substance from DIAG-061; corrected premise included).** On production `6c968ad`/`db07b9cd`, tenant entry fails silently for every `role='platform'` user on every live tenant. Chain: `ObservatoryTab.handleSelectTenant (web/src/components/platform/ObservatoryTab.tsx:92)` → `setTenant (web/src/contexts/tenant-context.tsx:182)` → `loadTenant (:126)` → `loadTenantConfig (:47)` → `throw (:114)`. The tenant cookie write (`:191`) and `router.push('/operate')` (`:197`) sit AFTER `await loadTenant` (`:186`), so the throw skips both; the catches (`tenant-context.tsx:198-200`, `ObservatoryTab.tsx:102-104`) reduce to spinner-reset — silent to the user and to the console. Middleware then sees no tenant cookie and emits `redirect.tenant_select`.

**Root cause.** `loadTenantConfig` resolves a tenant via static JSON (`web/src/data/tenants/<id>/config.json` — only `frmx-demo`, `restaurantmx`, `techcorp` exist; none of the 9 live tenants maps to one) or the RLS-gated Supabase `tenants` SELECT. The governing policy `tenants_select_vl_admin` (`web/supabase/migrations/005_platform_user_nullable_tenant.sql:19-27`) requires `profiles.role='vl_admin'`; the canonical role since HF-282 is `'platform'` (`resolveRole`, `web/src/lib/auth/resolve-identity.ts`). `tenants_select_own` (`web/supabase/migrations/001_core_tables.sql:145-148`) requires a non-NULL `tenant_id`. Platform-role users therefore see **zero** tenant rows — empirically confirmed under both tester JWTs in DIAG-061 §3.2 (tdadmin 0 rows, eoadmin 0 rows, identical denial).

**Corrected premise.** There is no per-account divergence: eoadmin never entered a tenant (picker + idle); the `platform_events` 1-vs-6 asymmetry was retry count. Post-dedup (#471 applied 2026-06-10), platform@ has lost the legacy `vl_admin` row that previously satisfied the policy and is expected to exhibit the same failure.

**Defect-class lineage.** HF-282 canonicalized the role vocabulary in the **reader** (`resolveRole`: `'vl_admin'` is a legacy alias of `'platform'`) but the **RLS policy literals** (the 001/005/007/009/013 migration suite, per DIAG-061) still encode the raw `'vl_admin'` string. This is role-vocabulary drift between the app's canonical declaration and the DB policy layer — the same one-canonical-surface class HF-282 closed at the reader, now closed at its sibling surface. Closing only `tenants` would be instance closure masquerading as class closure (AP-D2); this HF re-keys the entire class.

**Combined-treatment rationale (multi-arm HF, named per DD-7).** Two arms ship together under the Vertical Slice Rule: (1) the structural fix — one canonical DB predicate replacing every raw role literal in RLS; (2) the experience arm — the silent catch is itself a defect surface (it made this class invisible and prolonged diagnosis), so a tenant-load failure becomes user-visible and a successful entry becomes observable. Arm 2 is a behavior change relative to pre-HF state and is named here explicitly as in-scope by architect decision; everything else preserves pre-HF behavior exactly.

### Phase 3 — Migration authoring (canonical path; architect-gated apply)

3.1 Author `web/supabase/migrations/20260610180000_hf283_rls_platform_predicate.sql`. **Canonical path is `web/supabase/migrations/` — the repo-root `supabase/migrations/` is DEPRECATED (HF-259 `c983db1c`); the HF-282 placement defect must not recur. Do not place the file at repo root.**

Migration content, in one transaction-safe script:

(a) **The predicate** — exactly one role-vocabulary surface DB-side:

```sql
-- HF-283: canonical platform-role predicate.
-- Mirrors resolveRole (web/src/lib/auth/resolve-identity.ts) alias map:
-- 'vl_admin' is the legacy alias of canonical 'platform'.
-- Remove the alias from BOTH surfaces together when the legacy literal is
-- purged data-side (orphan pass / Platform-Created-Users OB). Do not edit
-- this set without editing resolveRole in the same change.
CREATE OR REPLACE FUNCTION public.is_platform()
RETURNS boolean
LANGUAGE sql STABLE SECURITY DEFINER
SET search_path = ''
AS $$
  SELECT EXISTS (
    SELECT 1 FROM public.profiles
    WHERE auth_user_id = auth.uid()
      AND role IN ('platform','vl_admin')
  )
$$;
REVOKE ALL ON FUNCTION public.is_platform() FROM PUBLIC;
-- anon receives EXECUTE so any policy referencing the predicate evaluates
-- (to false: auth.uid() is NULL for anon) instead of erroring the query.
GRANT EXECUTE ON FUNCTION public.is_platform() TO anon, authenticated;
```

(b) **Re-key every policy in the Phase-1 re-key set, explicitly and by name.** Pattern, shown on the policy DIAG-061 proved against — apply the identical transformation to each remaining set member:

```sql
DROP POLICY IF EXISTS "tenants_select_vl_admin" ON public.tenants;
CREATE POLICY "tenants_select_vl_admin" ON public.tenants
  FOR SELECT USING ( public.is_platform() );
```

Rule for every re-key: the `EXISTS (... role = 'vl_admin')` clause is replaced by `public.is_platform()`; **every other clause — tenant scoping, cmd, table, additional conditions — is byte-preserved.** No policy is widened or narrowed beyond the role canonicalization (DD-7). If any policy entangles the role clause with other logic such that the byte-preserving rewrite is ambiguous → HALT-2 before authoring that re-key.

(c) **Assertion block** (transaction-bound; failure rolls back the whole migration):

```sql
DO $$
DECLARE remaining int;
BEGIN
  IF NOT EXISTS (SELECT 1 FROM pg_proc p JOIN pg_namespace n ON n.oid = p.pronamespace
                 WHERE n.nspname = 'public' AND p.proname = 'is_platform') THEN
    RAISE EXCEPTION 'HF-283 assertion: public.is_platform() missing';
  END IF;
  SELECT count(*) INTO remaining FROM pg_policies
   WHERE qual ILIKE '%vl_admin%' OR with_check ILIKE '%vl_admin%';
  IF remaining > 0 THEN
    RAISE EXCEPTION 'HF-283 assertion: % policies still reference vl_admin', remaining;
  END IF;
  -- one EXISTS assertion per expected re-keyed policy name, e.g.:
  IF NOT EXISTS (SELECT 1 FROM pg_policies WHERE tablename = 'tenants'
                 AND policyname = 'tenants_select_vl_admin') THEN
    RAISE EXCEPTION 'HF-283 assertion: tenants_select_vl_admin missing post-rekey';
  END IF;
END $$;
```

The global `pg_policies` sweep is the class-closure net: a live policy that drifted outside committed migrations aborts the apply and surfaces at HALT-5 instead of surviving silently.
